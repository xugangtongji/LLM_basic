## 零、开篇：从"单次推理"到"多步行动"

上一章的四种技巧（Zero-shot / Few-shot / CoT / Self-Consistency）都是在单次对话中引导模型推理：模型读入问题，思考，输出答案——整个过程是一个回合。

但真实世界的复杂任务往往需要**查询外部信息**（搜索最新新闻、查数据库）、**执行操作**（运行代码、发送请求）、**多步规划和执行**（先分析需求，再拆解子任务，再逐步完成）。这些超出了单次推理的范围，需要更高级的技术框架：

```
ReAct        → 推理 + 工具调用的交织循环（Agent 的基础）
ToT          → 探索多条推理路径，而非单一线性推理
Prompt 链    → 把复杂任务拆成多个小 Prompt，每步输出是下一步输入
结构化 Prompt → 工程化的 Prompt 设计，可维护、可测试、可扩展
元提示        → 让模型优化 Prompt 本身
```

## 一、ReAct：推理与行动的协同

### 1.1 初印象：用费曼视角理解

**ReAct（Reason + Act）是"边想边做，做完再想"的循环**。普通 CoT 的局限：模型只能用它"记忆中"的知识推理。如果问"今天的比特币价格是多少"，CoT 给出的推理会因知识截止日期而失效，甚至凭空捏造（幻觉）。

ReAct 的解法：让模型在推理过程中可以**调用工具**（搜索引擎、代码执行器、数据库）获取外部信息，再根据工具结果继续推理。这就是人类解决复杂问题的方式：想一想 → 查一查 → 再想一想 → 再做一步 → 直到问题解决。核心循环是 **Thought（思考）→ Action（行动）→ Observation（观察）→ 再 Thought**。

### 1.2 ReAct 的格式与完整示例

ReAct 的 prompt 包含三种特殊标记（Observation 由系统填入，不是模型生成）：

```
Thought: [模型对当前情况的推理，决定下一步做什么]
Action: [调用哪个工具，以及参数]
Observation: [工具返回的结果（由系统填入）]
... （重复 Thought/Action/Observation 循环）
Answer: [最终答案]
```

一个完整的多步示例：

```
问题：2024 年奥运会在哪里举行，那个城市的人口是多少？

Thought: 我需要查找 2024 年奥运会的举办城市。
Action: search("2024年奥运会举办城市")
Observation: 2024年夏季奥运会在法国巴黎举行，时间7月26日至8月11日。

Thought: 现在我知道是巴黎了，需要查询巴黎人口。
Action: search("巴黎人口")
Observation: 巴黎市区人口约213万，大巴黎都市区约1200万。

Thought: 我有了两个问题的答案。
Answer: 2024年奥运会在法国巴黎举行，巴黎市区人口约213万。
```

### 1.3 ReAct 的执行引擎实现

```python
import re

REACT_SYSTEM_PROMPT = """你是一个有帮助的助手，可以使用以下工具：

工具列表：
- search(query): 搜索互联网获取信息
- calculate(expression): 计算数学表达式
- get_date(): 获取今天的日期

请使用以下格式：
Thought: [分析当前情况，决定下一步]
Action: [工具名称]([参数])
Observation: [工具返回的结果]
...（可以重复多次）
Thought: [基于所有观察，得出结论]
Answer: [最终答案]
"""

class ReActAgent:
    """ReAct Agent 执行循环：每个 Action 触发一次工具调用，结果作为 Observation 填入。"""

    def __init__(self, llm_fn, tools: dict, max_steps: int = 10):
        self.llm_fn    = llm_fn
        self.tools     = tools          # {"tool_name": callable}
        self.max_steps = max_steps

    def run(self, question: str) -> str:
        history = f"问题: {question}\n"
        for _ in range(self.max_steps):
            response = self.llm_fn(REACT_SYSTEM_PROMPT + "\n\n" + history, stop=["Observation:"])

            # 已有最终答案则返回
            if "Answer:" in response:
                return response.split("Answer:")[-1].strip()

            # 解析并执行 Action
            if "Action:" in response:
                action_line = next((l for l in response.split('\n') if l.startswith('Action:')), "")
                action_str  = action_line.replace('Action:', '').strip()
                match = re.match(r'(\w+)\((.*)\)', action_str)
                if match:
                    tool_name, tool_args = match.group(1), match.group(2).strip('"\'')
                    observation = self.tools[tool_name](tool_args) \
                        if tool_name in self.tools else f"错误：工具 '{tool_name}' 不存在"
                    history += response + f"\nObservation: {observation}\n"
        return "达到最大步数，未能得到答案"


# 工具函数
def mock_search(query: str) -> str:
    db = {
        "2024年奥运会": "2024年夏季奥运会在法国巴黎举行，7月26日至8月11日",
        "巴黎人口": "巴黎市区约213万，大巴黎都市区约1200万",
    }
    for key, val in db.items():
        if key in query:
            return val
    return "未找到相关信息"

tools = {
    "search": mock_search,
    "calculate": lambda expr: str(eval(expr)),
    "get_date": lambda _: "2024年10月15日",
}
```

### 1.4 ReAct 的核心价值、优势与局限

ReAct 表现优于多个最先进的基准方法。最佳实践是将 ReAct 与 CoT 结合，允许同时使用内部知识和推理过程中获取的外部信息。

```python
react_tradeoffs = {
    "核心价值": [
        "解决幻觉：需要实时数据时通过搜索而非记忆回答",
        "可验证性：每步 Observation 可以被检查，错误容易发现",
        "可扩展性：可添加任意工具（代码执行/API/数据库）",
        "自我修正：错误可以通过工具反馈纠正",

        "复杂任务中可能陷入循环或走偏",
    ],
    "适用场景": [
        "需要实时信息的问答（新闻、价格）",
        "需要执行代码或计算的任务",
        "多步骤的信息收集和综合",
        "AI Agent 和自动化工作流",
    ],
}
```

## 二、Tree-of-Thought：探索多条推理路径

### 2.1 初印象：用费曼视角理解

**CoT 是在一条直线上思考；ToT 是同时探索多条可能的路径**。CoT 就像你只用一种方法做下去，做到一半发现走不通只能重来；ToT 就像在草稿纸上同时尝试三种解法，哪条能走通走哪条，走不通就回头换一条。ToT 可以使用广度优先（BFS）、深度优先（DFS）或 Beam 搜索等树搜索算法，并能回溯。

效果数据很有说服力：在 Game of 24（用四个数字通过加减乘除算出 24）上，标准 CoT 成功率仅 4%，而 ToT 提升到 74%；在创意写作任务中，ToT 生成文本的连贯性和创意也显著更高。

### 2.2 简化版 ToT：单 Prompt"三位专家"

完整 ToT 需要多次 LLM 调用，但有一个零成本的单 Prompt 版本——"三位专家"提示：

```python
TOT_SIMPLE_PROMPT = """想象三位不同的专家在回答这个问题。
所有专家都会写下他们思考的第一步，然后与小组分享。
然后所有专家继续下一步，以此类推。
如果任何专家意识到自己的思路有误，他们会立即退出。

问题是：{question}"""
```

### 2.3 完整 ToT：BFS + Beam Search 实现

ToT 有四个核心组件：**思考生成（Generate）** 生成多个候选下一步、**状态评估（Evaluate）** 给每个候选打分、**搜索策略（Search）** 决定探索哪些路径、**终止判断（Terminate）** 何时停止。

```python
from typing import Optional

class ThoughtNode:
    def __init__(self, content: str, parent: Optional['ThoughtNode'] = None):
        self.content  = content
        self.parent   = parent
        self.score    = 0.0
        self.depth    = 0 if parent is None else parent.depth + 1

    def get_path(self) -> list[str]:
        path, node = [], self
        while node:
            path.append(node.content)
            node = node.parent
        return list(reversed(path))


class TreeOfThought:
    """Tree-of-Thought：展开 → 评估 → Beam Search 选最优 → 继续展开"""
    def __init__(self, llm_fn, n_branches=3, max_depth=4, beam_size=2):
        self.llm_fn     = llm_fn
        self.n_branches = n_branches
        self.max_depth  = max_depth
        self.beam_size  = beam_size

    def expand(self, node: ThoughtNode, problem: str) -> list[str]:
        """生成多个下一步思路"""
        path = "\n".join(node.get_path())
        prompt = f"""问题：{problem}

已有思考：{path}

生成 {self.n_branches} 个不同的下一步方向，用"---"分隔："""
        resp = self.llm_fn(prompt)
        return [t.strip() for t in resp.split("---") if t.strip()][:self.n_branches]

    def evaluate(self, node: ThoughtNode, problem: str) -> float:
        """给当前思路打分（0-1）。评估标准：逻辑正确性、与问题相关性、是否朝正确答案前进。"""
        path = "\n".join(node.get_path())
        prompt = f"""问题：{problem}\n思路：{path}\n\n评分(0-10，仅输出数字)："""
        try:
            return float(self.llm_fn(prompt).strip().split()[0]) / 10.0
        except:
            return 0.5

    def solve(self, problem: str) -> dict:
        """BFS + Beam Search 搜索最优路径（每层保留 beam_size 个最优节点）"""
        root = ThoughtNode(f"问题：{problem}")
        beam = [root]
        for _ in range(self.max_depth):
            candidates = []
            for node in beam:
                for thought in self.expand(node, problem):
                    child = ThoughtNode(thought, parent=node)
                    child.score = self.evaluate(child, problem)
                    candidates.append(child)
            if not candidates:
                break
            candidates.sort(key=lambda n: n.score, reverse=True)
            beam = candidates[:self.beam_size]   # DFS 版本则改为递归深入单条路径

        best = beam[0] if beam else root
        best_path = best.get_path()
        final = self.llm_fn(f"问题：{problem}\n最优思路：\n{''.join(best_path)}\n\n最终答案：")
        return {"path": best_path, "answer": final, "score": best.score}
```

### 2.4 ToT 的成本与实用简化版

**成本警告**：完整 ToT 的 LLM 调用次数可能高达 `n_candidates^max_depth`——`n=3, depth=4` 最多就是 81 次调用。团队有时会在 Self-Consistency 就够用的时候错误地选择 ToT。**在采用树搜索之前，请先在实际任务上衡量准确率与成本的关系**。

| 用 Self-Consistency（更经济） | 用 ToT（更强大但更贵） |
|------|------|
| 任务有明确正确答案（数学题） | 需要探索多种解法（创意写作、策略规划） |
| 不需要回溯，线性推理可解决 | 中间步骤可能走入死胡同（需要回溯） |
| | 问题解空间很大，穷举不可行 |

实践中更常见的是"轻量级 ToT"——生成多条推理路径再综合，比完整 ToT 便宜，比 Self-Consistency 更多样化：

```python
def lightweight_tot(problem: str, llm_fn, n_paths: int = 3) -> str:
    """轻量级 ToT：用不同方法生成多条路径，再让 LLM 综合。"""
    prompt_variants = [
        f"用方法一（直接分析）解决：{problem}\n请分步推理：",
        f"用方法二（反向验证）解决：{problem}\n从结论出发验证：",
        f"用方法三（类比推理）解决：{problem}\n举一个类似问题：",
    ]
    results = [llm_fn(p) for p in prompt_variants[:n_paths]]
    synthesis_prompt = f"""问题：{problem}

以下是不同的推理路径：
路径 1：{results[0]}
路径 2：{results[1] if len(results) > 1 else '无'}
路径 3：{results[2] if len(results) > 2 else '无'}

请综合这些推理路径，给出最准确的最终答案。"""
    return llm_fn(synthesis_prompt)
```

## 三、结构化 Prompt 设计：工程化的艺术

### 3.1 初印象：用费曼视角理解

**生产级的 Prompt 不是"一段好文字"，而是一个"精心设计的软件组件"**。个人使用时 Prompt 写得随意无所谓，但当 Prompt 要服务数百万用户、输出要接入下游系统（解析 JSON、触发操作）、多人协作维护迭代、对效果做 A/B 测试时，它就需要像代码一样进行工程化设计。

### 3.2 System Prompt 的六大组成部分

| 组成部分 | 作用 |
|------|------|
| 角色定义 | 谁在说话，有什么背景和权限 |
| 核心能力/职责 | 能做什么，怎么做（可操作的清单，不要太模糊） |
| 行为约束 | 不能做什么，遇到敏感情况怎么处理 |
| 输出格式 | 回复的长度、结构、语气要求（精确的 JSON Schema 或模板） |
| 特殊情况处理 | 边缘情况和异常情况的处理规则 |
| 上下文变量 | 动态注入的信息（日期、用户信息等） |

一个生产级客服 AI 的 System Prompt 示例：

```python
customer_service_system_prompt = """
# 角色定义
你是「数码星球」电商平台的智能客服助手 Stella。
你代表公司与客户沟通，需要专业、友好、高效地解决客户问题。
今天的日期是 {current_date}。

# 核心能力
1. 查询订单状态（需要提供订单号）
2. 处理退换货申请（7 天无理由，保持原包装）
3. 解答产品使用问题
4. 解释促销活动规则
5. 转接人工客服（当问题超出你的处理范围时）

# 行为约束
- 始终用礼貌、专业的中文回复
- 不得承诺系统外的额外优惠或赔偿
- 涉及账号安全问题（密码、支付信息），必须引导用户去官方渠道，不得直接处理
- 不得讨论竞争对手的产品
- 对于投诉，先表达理解和道歉，再解决问题
- 如果问题超出权限：「您的问题需要人工专员处理，我现在为您转接，请稍候」

# 输出格式
- 每次回复控制在 200 字以内
- 需要客户提供信息时，每次最多问一个问题
- 最终解决方案以「✅」开头，转接人工以「🔄」开头

# 特殊情况处理
- 如果用户情绪激动：先用"我完全理解您的心情..."开头，再处理问题
- 如果问题涉及法律（如诈骗）：引导用户报警，提供官方联系方式
- 如果无法理解用户意思：用"我想确认一下您的问题是..."来澄清
"""
```

> 经验法则：**System Prompt 应该稳定不变，只让 User Prompt 变化**，这有助于控制输出一致性。

### 3.4 结构化输出：让 LLM 输出可解析的格式

生产系统中 LLM 输出往往要被下游程序处理，必须严格符合某种格式。有三种方法，可靠性递增：

```python
import json, re
from pydantic import BaseModel
from typing import List

# 方法一：Prompt 中直接要求 JSON（简单，但 10-20% 概率格式错误）
json_output_prompt = """分析以下产品评论，以严格的 JSON 格式输出（不要有任何额外文字）：
评论："{review}"
输出格式：
{{"sentiment": "positive|negative|neutral|mixed", "score": 1-10,
  "aspects": [{{"aspect": "...", "sentiment": "positive|negative", "quote": "..."}}],
  "summary": "一句话总结"}}"""

# 方法二：用 Pydantic 定义 Schema，注入 Prompt（更明确）
class ReviewAnalysis(BaseModel):
    sentiment: str       # positive/negative/neutral/mixed
    score: int           # 1-10
    aspects: List[dict]
    summary: str

def create_structured_prompt(schema: type) -> str:
    return f"请严格按以下 JSON Schema 输出，不要有任何额外内容：\n\n" \
           f"{json.dumps(schema.model_json_schema(), ensure_ascii=False, indent=2)}"

# 方法三：使用 API 的 tool_use 强制结构化输出（最可靠，推荐生产使用）
# client.messages.create(..., tools=[{
#     "name": "analyze_review", "description": "分析评论",
#     "input_schema": ReviewAnalysis.model_json_schema()
# }], tool_choice={"type": "tool", "name": "analyze_review"})

# 无论哪种方法，都建议加防御性解析
def parse_llm_json(response_text: str) -> dict:
    """健壮地解析 LLM 输出的 JSON（处理常见错误）"""
    text = re.sub(r'```(?:json)?\n?', '', response_text).replace('```', '').strip()
    try:
        return json.loads(text)
    except json.JSONDecodeError:
        start, end = text.find('{'), text.rfind('}') + 1   # 提取第一个 { 到最后一个 }
        if start != -1 and end > start:
            try:
                return json.loads(text[start:end])
            except json.JSONDecodeError:
                pass
    raise ValueError(f"无法解析为 JSON：{text[:100]}...")
```

LLM 输出 JSON 常见失败原因：在 JSON 前后加 Markdown 代码块（```json）；JSON 内有注释；字符串含未转义引号；输出截断导致不完整。对应解决：防御性解析 + Prompt 明确要求"不要有额外内容" + 优先用 tool_use + 失败重试。

## 四、Prompt 链（Prompt Chaining）

**把复杂任务拆成多个小 Prompt，每步输出是下一步输入。** 比起让模型在单次超长 Prompt 里同时做翻译+摘要+分类，拆分后每步更简单、更不容易出错。

```python
def document_processing_chain(english_text: str, llm_fn) -> dict:
    """多步 Prompt 链：英文文章 → 翻译 → 提取要点 → 摘要 → 分类标签。每步可单独验证。"""
    results = {}
    results["translation"] = llm_fn(
        f"将以下英文翻译成中文，保持学术用语准确：\n\n{english_text}\n\n中文翻译："
    )
    results["key_points"] = llm_fn(
        f"从以下文本提取 3-5 个核心要点，每点一句话，以'-'开头：\n\n{results['translation']}\n\n核心要点："
    )
    results["summary"] = llm_fn(
        f"根据以下要点，写 100 字以内的摘要：\n\n{results['key_points']}\n\n摘要："
    )
    results["tags"] = llm_fn(
        f"为以下摘要选择 2-3 个学科标签（机器学习/自然语言处理/计算机视觉/强化学习/其他）：\n\n"
        f"{results['summary']}\n\n标签（逗号分隔）："
    )
    return results
```

Prompt 链的优势：中间结果可以验证和修正；每步 Prompt 更简单、模型更不容易出错；便于调试（知道哪一步出问题）；独立步骤可以并行执行提速；不同步骤可以用不同模型（便宜的做简单步骤）。

## 六、Prompt 的版本管理与测试

生产级 Prompt 需要像代码一样工程化。四个支柱：**版本控制**（追踪每次修改和效果变化）、**自动化测试**（建测试集，修改后自动验证效果不下降）、**A/B 测试**（新旧并行，用真实流量比较）、**监控告警**（线上监控输出质量，异常及时告警）。

```python
# Prompt 版本管理（类似代码版本控制）
prompt_registry = {
    "customer_service_v1.0": {"created": "2024-01-01", "description": "初始版本",
                              "performance": {"accuracy": 0.82, "satisfaction": 4.1}},
    "customer_service_v1.1": {"created": "2024-02-15", "description": "增加边缘情况处理",
                              "performance": {"accuracy": 0.87, "satisfaction": 4.3}},
    "customer_service_v2.0": {"created": "2024-04-01", "description": "重构：引入 XML 结构",
                              "performance": {"accuracy": 0.91, "satisfaction": 4.5}},
}

class PromptTester:
    """自动化测试 Prompt 效果"""
    def __init__(self, prompt: str, llm_fn):
        self.prompt, self.llm, self.test_cases = prompt, llm_fn, []

    def add_test(self, input_text: str, expected: str, check_fn=None, weight: float = 1.0):
        self.test_cases.append({"input": input_text, "expected": expected,
                                "check": check_fn or self._default_check, "weight": weight})

    def _default_check(self, output: str, expected: str) -> float:
        return 1.0 if expected.lower() in output.lower() else 0.0

    def run(self) -> dict:
        total_score = total_weight = 0.0
        for case in self.test_cases:
            response = self.llm(self.prompt + "\n" + case["input"])
            total_score  += case["check"](response, case["expected"]) * case["weight"]
            total_weight += case["weight"]
        return {"overall_score": total_score / total_weight if total_weight else 0,
                "n_tests": len(self.test_cases)}
```

## 七、高级技术的选型决策

```python
def advanced_technique_selector(scenario: str) -> str:
    """根据场景推荐高级提示技术"""
    needs_tools    = any(w in scenario for w in ["搜索", "查询", "计算", "执行", "API", "实时"])
    needs_explore  = any(w in scenario for w in ["创意", "多种方案", "策略", "规划", "复杂"])
    needs_reliable = any(w in scenario for w in ["生产", "系统", "API", "接口", "格式"])
    if len(scenario) < 30:
        return "Zero-shot CoT：任务简单，基础技术够用"
    elif needs_tools:
        return "ReAct / Agent 框架：需要工具调用，用推理+行动循环"
    elif needs_explore and not needs_reliable:
        return "Tree-of-Thought（轻量版）：需要探索多条推理路径"
    elif needs_reliable:
        return "结构化 Prompt + 强制 JSON 输出：生产系统，可靠性优先"
    else:
        return "Few-shot CoT + Self-Consistency：复杂推理，追求准确率"
```

| 场景 | 推荐技术 |
|------|---------|
| 需要外部信息（实时数据/计算/数据库） | ReAct / Agent 框架 |
| 需要探索多种解决方案（创意/规划） | Tree-of-Thought |
| 任务可分解为多个独立步骤 | Prompt 链（串行或并行） |
| 需要复杂推理，CoT 效果不稳定 | Few-shot CoT + Self-Consistency |
| 输出格式要求严格（JSON/表格） | Few-shot + 格式约束 + 验证层 |
| 成本敏感，追求最低 API 调用次数 | Zero-shot 或 Few-shot |
| 准确率要求极高（医疗/法律） | Self-Consistency + 人工验证 |

## 八、面试常见问题

**Q1：ReAct 是什么？它和 CoT 有什么本质区别？**

ReAct 是 Reason + Act 的结合——模型不只是做内部推理，还能在推理过程中调用外部工具（搜索、计算器、代码执行）并观察结果，再基于观察继续推理，格式是 Thought→Action→Observation 循环。CoT 只有内部推理（Thought），没有外部行动。核心区别：CoT 是"自言自语"、依赖训练知识；ReAct 是"边想边查"、可访问实时外部信息，因此能解决幻觉。最佳实践是把 ReAct 与 CoT 结合。

**Q2：Tree-of-Thought 和 CoT、Self-Consistency 有什么区别？**

CoT 是单条线性推理链，走错路只能错到底。ToT 同时展开多条推理路径，对每条评估打分，用 BFS/DFS/Beam 搜索最优路径，遇到死胡同可以回溯。Game of 24 任务上 CoT 成功率仅 4%，ToT 达到 74%。与 Self-Consistency 的区别：Self-Consistency 对同一问题多次采样后投票，适合有明确答案、线性可解的任务，成本 n 次调用；ToT 能回溯换方向，适合需要试错的复杂任务，但成本可能高达 n^depth 次调用。Self-Consistency 够用时不要选 ToT。

**Q3：什么是 Prompt 链？什么时候比单次 Prompt 更好？**

Prompt 链把复杂任务拆成多个步骤，每步输出作为下一步输入。适用场景：任务太复杂（翻译+摘要+分类一起做容易出错）；中间结果需要验证；多个子任务相互独立可并行；需要清晰的调试路径。好处：每步 Prompt 简单准确率更高；中间结果可检查修正；独立步骤可并行加速；不同步骤可用不同模型。

**Q4：生产级 System Prompt 应该包含哪些核心部分？如何为 Claude 设计？**

六个部分：角色定义、核心能力/职责、行为约束、输出格式（精确的 JSON Schema 或模板）、特殊情况处理、上下文变量。好的 System Prompt 应稳定不变，只让 User Prompt 变化。为 Claude 设计时推荐 XML 标签结构（`<task>`、`<document>`、`<constraints>`、`<output_format>` 等）——结构清晰、便于程序化修改、Claude 训练时见过大量 XML 格式，理解效果好。

**Q5：为什么生产系统中 LLM 输出 JSON 经常失败？元提示有什么用？**

JSON 失败常因：前后加 Markdown 代码块、JSON 内有注释、字符串含未转义引号、输出截断。解决：防御性解析（先直接解析，失败则提取 {} 内容）、Prompt 明确要求"不要有额外内容"、用 API 的 tool_use 强制结构化输出（最可靠）、加重试逻辑。元提示是用 LLM 优化 Prompt 本身，用处包括：快速改进写得不好的 Prompt、生成 A/B 测试变体、对不熟悉提示工程的用户友好、结合 DSPy 实现自动优化。
