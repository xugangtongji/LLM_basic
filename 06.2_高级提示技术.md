## 零、开篇：从"单次推理"到"多步行动"

上一章的四种技巧（Zero-shot / Few-shot / CoT / Self-Consistency）都是在单次对话中引导模型推理。模型读入问题，思考，输出答案——整个过程是一个回合。

但真实世界的复杂任务往往需要：
- **查询外部信息**（搜索最新新闻、查数据库）
- **执行操作**（运行代码、发送请求）
- **多步规划和执行**（先分析需求，再拆解子任务，再逐步完成）

这些超出了单次推理的范围，需要更高级的技术框架。

本章讲三个层次的高级技术：

```
ReAct        → 推理 + 工具调用的交织循环（Agent 的基础）
ToT          → 探索多条推理路径，而非单一线性推理
结构化 Prompt → 工程化的 Prompt 设计，可维护、可测试、可扩展
```

## 一、ReAct：推理与行动的协同

### 1.1 初印象：用费曼视角理解

**ReAct（Reason + Act）是"边想边做，做完再想"的循环**。

普通 CoT 的局限：模型只能用它"记忆中"的知识推理。如果问"今天的比特币价格是多少"，CoT 给出的推理过程会因为知识截止日期而失效。

ReAct 的解法：让模型在推理过程中可以**调用工具**（搜索引擎、代码执行器、数据库）来获取外部信息，然后根据工具的结果继续推理。

这就是人类解决复杂问题的方式：想一想 → 查一查 → 再想一想 → 再做一步 → 直到问题解决。

ReAct 的核心循环：Think → 决定用哪个工具 → 观察结果 → 再 Think。

### 1.2 ReAct 的格式

ReAct 的 prompt 包含三种特殊标记：

```
Thought: [模型对当前情况的推理，决定下一步做什么]
Action: [调用哪个工具，以及参数]
Observation: [工具返回的结果（由系统填入，不是模型生成）]
... （重复 Thought/Action/Observation 循环）
Answer: [最终答案]
```

```python
# ReAct 的 Few-shot 示例 Prompt
react_prompt_template = """
你是一个有帮助的助手，可以使用以下工具：
- search(query): 搜索网络，返回相关结果摘要
- calculator(expression): 计算数学表达式，返回结果
- get_date(): 获取今天的日期

请使用以下格式：

Thought: 思考当前需要做什么
Action: 工具名(参数)
Observation: [工具返回的结果]
... （可以多次循环）
Answer: 最终答案

--- 示例 ---

问题：2024年超级碗的冠军是谁，他们赢了多少分？

Thought: 我需要查找2024年超级碗的信息，包括冠军和比分。
Action: search(2024超级碗冠军比分)
Observation: 2024年超级碗（第58届）于2024年2月11日举行，堪萨斯城酋长队以25-22击败旧金山49人队，赢得冠军。

Thought: 我找到了冠军和比分。现在可以给出完整答案。
Answer: 2024年超级碗冠军是堪萨斯城酋长队，他们以25比22击败旧金山49人队。

--- 现在回答新问题 ---

问题：{question}

Thought:"""


# ReAct 的执行引擎（简化版）
def react_agent(
    question: str,
    tools: dict,      # {"tool_name": callable}
    llm_fn,           # 调用 LLM 的函数
    max_steps: int = 8,
) -> str:
    """
    ReAct Agent 执行循环。
    在每个 Action 之后调用对应工具，把结果作为 Observation 填入。
    """
    prompt = react_prompt_template.format(question=question)
    history = prompt

    for step in range(max_steps):
        # 让 LLM 生成下一个 Thought 和 Action
        response = llm_fn(history, stop=["Observation:"])

        history += response

        # 解析 Action
        if "Action:" in response:
            action_line = [l for l in response.split('\n') if l.startswith('Action:')]
            if action_line:
                action_str = action_line[0].replace('Action:', '').strip()
                # 解析工具名和参数：tool_name(args)
                import re
                match = re.match(r'(\w+)\((.+)\)', action_str)
                if match:
                    tool_name = match.group(1)
                    tool_args = match.group(2).strip('"\'')

                    # 调用工具
                    if tool_name in tools:
                        obs = tools[tool_name](tool_args)
                    else:
                        obs = f"错误：工具 '{tool_name}' 不存在"

                    history += f"\nObservation: {obs}\n"

        # 检查是否已有最终答案
        if "Answer:" in response:
            answer_line = [l for l in response.split('\n') if l.startswith('Answer:')]
            if answer_line:
                return answer_line[0].replace('Answer:', '').strip()

    return "达到最大步数，未能得到答案"


# 模拟工具
tools = {
    "search": lambda q: f"搜索结果（模拟）：关于 '{q}' 的信息...",
    "calculator": lambda expr: str(eval(expr)),
    "get_date": lambda _: "2024年10月15日",
}

print("ReAct Agent 结构：")
print("  Thought → Action → Observation → Thought → ... → Answer")
print("  每个 Action 触发一次工具调用，Observation 是工具返回值")
print("  循环直到模型输出 Answer 或达到最大步数")
```

### 1.3 ReAct 的优势与局限

ReAct 表现优于多个最先进的基准方法。最佳方法将 ReAct 与 CoT 结合，允许同时使用内部知识和推理过程中获取的外部信息。

```python
react_tradeoffs = {
    "优势": [
        "能访问实时信息（搜索、数据库）",
        "推理过程透明可解释（每步都有 Thought）",
        "可以执行真实操作（代码、API 调用）",
        "错误可以通过工具反馈自我修正",
    ],
    "局限": [
        "需要工具基础设施（搜索 API、代码执行器等）",
        "每步都需要 LLM 调用，延迟更高",
        "Token 消耗多（每个 Observation 都进入上下文）",
        "复杂任务中可能陷入循环或走偏",
    ],
    "适用场景": [
        "需要实时信息的问答（新闻、价格）",
        "需要执行代码或计算的任务",
        "多步骤的信息收集和综合",
        "AI Agent 和自动化工作流",
    ],
}

for category, items in react_tradeoffs.items():
    print(f"\n{category}：")
    for item in items:
        print(f"  • {item}")
```

## 二、Tree-of-Thought：探索多条推理路径

### 2.1 初印象：用费曼视角理解

**CoT 是在一条直线上思考；ToT 是同时探索多条可能的路径**。

想象解一道复杂的数学竞赛题。CoT 就像你只用一种方法做下去，做到一半发现走不通，只能重来。ToT 就像在草稿纸上同时尝试三种不同的解法，哪条路能走通就走哪条，走不通就回头换一条。

Tree-of-thought 提示方式是对 CoT 的推广，通过并行生成多条推理线，能够回溯或探索其他路径。它可以使用广度优先、深度优先或 Beam 搜索等树搜索算法。

### 2.2 ToT 的四个核心组件

```python
from typing import List, Optional, Callable
import heapq

class ThoughtNode:
    """ToT 中的一个思考节点"""
    def __init__(self, thought: str, parent=None, score: float = 0.0):
        self.thought  = thought     # 这一步的思考内容
        self.parent   = parent      # 父节点
        self.score    = score       # 该节点的评分（由 LLM 评估）
        self.children = []

    def get_path(self) -> List[str]:
        """获取从根节点到当前节点的完整推理路径"""
        path = []
        node = self
        while node:
            path.append(node.thought)
            node = node.parent
        return list(reversed(path))


class TreeOfThought:
    """
    Tree-of-Thought 实现。
    四个核心组件：
    1. 思考生成（Generate）：生成多个候选下一步
    2. 状态评估（Evaluate）：给每个候选打分
    3. 搜索策略（Search）：决定探索哪些路径
    4. 终止判断（Terminate）：何时停止搜索
    """

    def __init__(
        self,
        llm_fn: Callable,           # LLM 调用函数
        n_candidates: int = 3,      # 每步生成几个候选思考
        max_depth: int = 4,         # 最大推理深度
        search_strategy: str = "bfs",  # "bfs"/"dfs"/"beam"
        beam_width: int = 2,        # beam search 的宽度
    ):
        self.llm           = llm_fn
        self.n_candidates  = n_candidates
        self.max_depth     = max_depth
        self.strategy      = search_strategy
        self.beam_width    = beam_width

    def generate_thoughts(self, problem: str, current_path: List[str]) -> List[str]:
        """生成 n 个候选的下一步思考"""
        path_str = "\n".join([f"步骤 {i+1}: {t}" for i, t in enumerate(current_path)])

        prompt = f"""问题：{problem}

已有的推理步骤：
{path_str if path_str else '（还没开始推理）'}

请生成 {self.n_candidates} 个不同的、可能的下一步推理思路。
每个思路用"---"分隔。
思路应该各不相同，覆盖不同的角度或方法。"""

        response = self.llm(prompt)
        thoughts = [t.strip() for t in response.split("---") if t.strip()]
        return thoughts[:self.n_candidates]

    def evaluate_thought(self, problem: str, path: List[str]) -> float:
        """评估当前推理路径的质量（0-10分）"""
        path_str = "\n".join(path)

        prompt = f"""问题：{problem}

推理路径：
{path_str}

请评估这条推理路径的质量，给出 0-10 分，10 分表示最优。
评估标准：逻辑正确性、与问题的相关性、是否朝着正确答案前进。
只需要返回一个数字。"""

        try:
            score_str = self.llm(prompt).strip()
            return float(score_str)
        except:
            return 5.0   # 默认中等分

    def is_final_answer(self, thought: str) -> bool:
        """判断当前思考是否已经是最终答案"""
        final_markers = ["答案是", "因此答案", "最终答案", "所以是", "the answer is"]
        return any(marker in thought.lower() for marker in final_markers)

    def solve(self, problem: str) -> dict:
        """用 ToT 解决问题，返回最优推理路径和答案"""
        root = ThoughtNode(thought="开始推理...", score=10.0)

        if self.strategy == "bfs":
            return self._bfs_solve(problem, root)
        elif self.strategy == "beam":
            return self._beam_solve(problem, root)
        else:
            return self._dfs_solve(problem, root, depth=0)

    def _beam_solve(self, problem: str, root: ThoughtNode) -> dict:
        """Beam Search：每层保留 beam_width 个最优节点"""
        current_beam = [root]
        best_final = None

        for depth in range(self.max_depth):
            candidates = []

            for node in current_beam:
                path = node.get_path()
                new_thoughts = self.generate_thoughts(problem, path[1:])  # 去掉初始节点

                for thought in new_thoughts:
                    new_path = path[1:] + [thought]
                    score = self.evaluate_thought(problem, new_path)
                    child = ThoughtNode(thought=thought, parent=node, score=score)
                    node.children.append(child)
                    candidates.append(child)

                    if self.is_final_answer(thought):
                        if best_final is None or score > best_final.score:
                            best_final = child

            # 保留 beam_width 个最高分候选
            current_beam = sorted(candidates, key=lambda x: x.score, reverse=True)[:self.beam_width]

            if best_final:
                break

        # 返回最优路径
        best_node = best_final or max(current_beam, key=lambda x: x.score)
        return {
            "path":   best_node.get_path()[1:],  # 去掉初始节点
            "score":  best_node.score,
            "answer": best_node.thought,
        }

    def _bfs_solve(self, problem, root):
        # 简化：同 _beam_solve 但不限制 beam 宽度
        return self._beam_solve(problem, root)

    def _dfs_solve(self, problem, node, depth):
        if depth >= self.max_depth:
            return {"path": node.get_path()[1:], "score": node.score, "answer": node.thought}

        path = node.get_path()
        thoughts = self.generate_thoughts(problem, path[1:])

        best_result = None
        for thought in thoughts:
            score = self.evaluate_thought(problem, path[1:] + [thought])
            child = ThoughtNode(thought=thought, parent=node, score=score)
            node.children.append(child)

            if self.is_final_answer(thought):
                return {"path": child.get_path()[1:], "score": score, "answer": thought}

            result = self._dfs_solve(problem, child, depth + 1)
            if best_result is None or result["score"] > best_result["score"]:
                best_result = result

        return best_result or {"path": [], "score": 0, "answer": "未找到答案"}


# ToT 适用场景的指导
print("ToT 使用决策指南：")
print("""
用 Self-Consistency（更经济）当：
  • 任务有明确正确答案（数学题）
  • 不需要回溯，线性推理可以解决

用 ToT（更强大但更贵）当：
  • 需要探索多种解法（创意写作、策略规划）
  • 中间步骤可能走入死胡同（需要回溯）
  • 问题解空间很大，穷举不可行

成本警告：ToT 的 LLM 调用次数 = n_candidates^max_depth
  n=3, depth=4 → 最多 81 次 LLM 调用！
""")
```

### 2.3 ToT 的实用简化版

完整的 ToT 工程成本很高。团队有时会在 Self-Consistency 就够用的时候错误地选择 ToT。在采用树搜索方法之前，请先在实际任务上衡量准确率与成本的关系。

实践中更常见的是"轻量级 ToT"：

```python
# 实用简化：Step-Back + 多路径 CoT
def lightweight_tot(problem: str, llm_fn, n_paths: int = 3) -> str:
    """
    轻量级 ToT：生成多条推理路径，选最优。
    比完整 ToT 便宜，比 Self-Consistency 更多样化。
    """
    prompt_variants = [
        f"用方法一（直接分析）解决：{problem}\n请分步推理：",
        f"用方法二（反向验证）解决：{problem}\n从结论出发验证：",
        f"用方法三（类比推理）解决：{problem}\n举一个类似问题：",
    ]

    results = []
    for prompt in prompt_variants[:n_paths]:
        response = llm_fn(prompt)
        results.append(response)

    # 让 LLM 综合多条路径给出最终答案
    synthesis_prompt = f"""问题：{problem}

以下是三条不同的推理路径：

路径 1：{results[0]}

路径 2：{results[1] if len(results) > 1 else '无'}

路径 3：{results[2] if len(results) > 2 else '无'}

请综合这些推理路径，给出最准确的最终答案。"""

    return llm_fn(synthesis_prompt)
```

## 三、结构化 Prompt 设计：工程化的艺术

### 3.1 初印象：用费曼视角理解

**生产级的 Prompt 不是"一段好文字"，而是一个"精心设计的软件组件"**。

个人使用时，Prompt 写得随意一点无所谓。但当你需要：
- Prompt 服务数百万用户（Claude.ai、ChatGPT 等）
- Prompt 的输出要接入下游系统（解析 JSON、触发操作）
- 多人协作维护和迭代 Prompt
- 对 Prompt 的效果做 A/B 测试和度量

这时候，Prompt 就需要像代码一样进行工程化设计。

### 3.2 System Prompt 的结构设计

不同模型对 Prompt 格式有不同的偏好。Claude 特别擅长理解 XML 标签结构的 Prompt。

print("Claude XML Prompt 的主要标签：")
xml_tags = {

def parse_llm_json(response_text: str) -> dict:
    """健壮地解析 LLM 输出的 JSON（处理常见错误）"""
    import re

    # 去掉 Markdown 代码块（常见问题）
    text = re.sub(r'```(?:json)?\n?', '', response_text)
    text = text.replace('```', '').strip()

    # 尝试直接解析
    try:
        return json.loads(text)
    except json.JSONDecodeError:
        # 尝试找到第一个 { 和最后一个 } 之间的内容
        start = text.find('{')
        end   = text.rfind('}') + 1
        if start != -1 and end > start:
            try:
                return json.loads(text[start:end])
            except json.JSONDecodeError:
                pass

    raise ValueError(f"无法解析为 JSON：{text[:100]}...")

# Prompt 测试套件
class PromptTester:
    """自动化测试 Prompt 效果"""

    def __init__(self, prompt: str, llm_fn):
        self.prompt = prompt
        self.llm    = llm_fn
        self.test_cases = []

    def add_test(self, input_text: str, expected_output: str,
                  check_fn=None, weight: float = 1.0):
        """添加测试用例"""
        self.test_cases.append({
            "input":    input_text,
            "expected": expected_output,
            "check":    check_fn or self._default_check,
            "weight":   weight,
        })

    def _default_check(self, output: str, expected: str) -> float:
        """默认：检查 expected 是否包含在 output 中"""
        return 1.0 if expected.lower() in output.lower() else 0.0

    def run(self) -> dict:
        """运行所有测试，返回加权得分"""
        total_score  = 0.0
        total_weight = 0.0
        results      = []

        for case in self.test_cases:
            response = self.llm(self.prompt + "\n" + case["input"])
            score    = case["check"](response, case["expected"])
            weighted = score * case["weight"]

            total_score  += weighted
            total_weight += case["weight"]
            results.append({"input": case["input"][:30], "score": score, "response": response[:50]})

        return {
            "overall_score": total_score / total_weight if total_weight > 0 else 0,
            "n_tests":       len(self.test_cases),
            "results":       results,
        }

## 六、面试常见问题

**Q1：ReAct 是什么？它和 CoT 有什么区别？**

ReAct 是 Reason + Act 的结合——模型不只是做内部推理，还能在推理过程中调用外部工具（搜索、计算器、代码执行）并观察结果，然后基于观察继续推理。CoT 只有内部推理（Thought），没有外部行动。ReAct 的格式是 Thought→Action→Observation 的循环，直到得出最终答案。最佳实践是将 ReAct 与 CoT 结合，允许同时使用内部知识和通过推理获取的外部信息。

**Q2：Tree-of-Thought 和 Self-Consistency 都能提升推理准确率，有什么区别？**

Self-Consistency：对同一问题多次采样，选最多次出现的答案（投票）。适合有明确正确答案、线性推理可以解决的任务，成本是 n 次 LLM 调用。ToT：探索多条推理路径，能回溯、换方向，类似树形搜索。适合需要探索多种解法的复杂任务，但成本可能达到 n^depth 次 LLM 调用。实践中，Self-Consistency 够用的情况下不要选 ToT。

**Q3：生产级 System Prompt 应该包含哪些核心部分？**

六个部分：角色定义（谁在说话，有什么权限背景）；核心能力（能做什么的可操作清单）；行为约束（不能做什么，遇到敏感情况如何处理）；输出格式（长度、结构、语气要求）；特殊情况处理（边缘情况和异常处理规则）；上下文变量（动态注入的信息如日期、用户信息）。

**Q4：为什么生产系统中 LLM 输出 JSON 经常失败？怎么解决？**

常见失败原因：LLM 在 JSON 前后加 Markdown 代码块（```json）；JSON 内有注释（JSON 不支持注释）；字符串中包含未转义的引号；输出截断导致 JSON 不完整。解决方案：用防御性解析（先尝试直接解析，失败则提取 {}内容）；在 Prompt 中明确说"不要有任何额外内容"；使用 API 的 tool_use 功能强制结构化输出（最可靠）；在生产系统中加重试逻辑（失败则重新请求）。

**Q5：Prompt 工程化应该如何管理？**

类比代码工程：版本控制（git 管理 Prompt 文件，追踪每次改动）；自动化测试（建立测试集，每次修改 Prompt 后自动运行测试，确保效果不下降）；A/B 测试（新旧版本并行，用真实流量比较效果）；监控（线上监控输出格式、内容质量，异常及时告警）。对于高流量的生产 Prompt，这套工程化流程与代码开发流程同样重要。
