## 零、开篇：RAG 的两块基石

上一章我们理解了 RAG 的"为什么"。这一章进入"怎么做"的第一步：搞清楚 RAG 的两块最基础的组件。

**嵌入模型（Embedding Model）**：把文字变成数字向量的机器。没有它，计算机不知道"苹果"和"Apple"有什么关系。

**向量数据库（Vector Database）**：专门存储和检索这些数字向量的数据库。普通数据库只会查"完全匹配"（找 ID=123 的那条记录），向量数据库会查"最相似"（找和这个向量最接近的那些记录）。

这两个组件是 RAG 系统的心脏——嵌入模型决定了语义理解的质量，向量数据库决定了检索的速度和规模。

## 一、向量嵌入：用数字表达语义

### 1.1 什么是嵌入

**嵌入（Embedding）是把任意内容（文字、图片、声音）转换成一组数字（向量）的过程**。

这组数字有一个关键特性：**语义相近的内容，向量在数学空间里也相近**。

想象一个三维空间（实际是几百到几千维，但三维更好理解）：
- "猫"的向量放在某个位置
- "小猫"的向量在它附近
- "汪星人"的向量稍远一点
- "汽车"的向量在完全不同的区域

这样，当你问"我家的小动物病了怎么办"，系统用同样的嵌入模型把这个问题转成向量，然后找"最近"的文档向量——内容关于猫、狗的文档会被找到，内容关于汽车的不会。

这就是语义检索的核心——不是关键词匹配，而是**语义相似度匹配**。

### 1.2 嵌入模型的发展脉络

嵌入模型不是突然出现的，它有清晰的演进线路：

**Word2Vec（2013）**：第一个大规模成功的词嵌入方法，由 Google 的 Mikolov 等人提出。把每个单词表示成向量，有了著名的发现：`king - man + woman ≈ queen`。但 Word2Vec 的问题是：同一个词只有一个固定向量，无法区分"苹果（水果）"和"苹果（手机公司）"。

**BERT 时代的嵌入（2018-2021）**：用完整的上下文来决定一个词的含义。"苹果很甜"和"苹果发布了新手机"里，"苹果"的向量会不一样。但 BERT 做嵌入的方式需要把查询和文档一起输入模型才能比较，速度太慢，不适合大规模检索。

**双编码器（Bi-encoder，2020+）**：两段文本分别单独编码，然后比较向量相似度。查询向量和文档向量可以独立计算和存储，大大加速了检索效率。这是今天主流嵌入模型的基础架构。

**OpenAI text-embedding 系列 / BGE / E5（2022+）**：更强的训练数据和方法，嵌入质量大幅提升，多语言支持越来越好。

### 1.3 主流嵌入模型选型

当前（2025 年）常见的选择分三类：

**商业 API 类**

| 模型 | 维度 | 特点 | 适合场景 |
|------|------|------|---------|
| OpenAI text-embedding-3-small | 1536 | 便宜快速，性价比高 | 快速原型、成本敏感 |
| OpenAI text-embedding-3-large | 3072 | 质量更好，价格更高 | 对质量要求高的英文场景 |
| Cohere embed-v3 | 1024 | 支持多语言，有 search/document 模式 | 多语言场景 |

**开源自托管类**

| 模型 | 维度 | 特点 | 适合场景 |
|------|------|------|---------|
| BGE-M3 | 1024 | 多语言、多功能（稠密+稀疏+多向量） | **中文场景首选**，多语言强 |
| BGE-large-zh-v1.5 | 1024 | 专门优化中文，轻量 | 中文专用，资源有限 |
| E5-large-v2 | 1024 | 需要 "query:" 和 "passage:" 前缀 | 英文检索 |
| MiniLM-L6-v2 | 384 | 极小极快，质量尚可 | 快速实验、资源极度受限 |

BGE 家族（如 bge-large-en-v1.5 和 bge-m3）由北京人工智能研究院开发，专门针对高性能检索任务。BGE 领跑多语言、长文本检索场景，其 M3 变体对于现代混合检索至关重要。

**选型建议**：

```
新项目起步，数据是中文？         → BGE-M3（开源，中文最强）
新项目起步，英文或多语言？        → text-embedding-3-small（API，简单省心）
有数据隐私要求，不能用云端 API？  → BGE 系列自托管
对延迟极度敏感，向量维度越小越好？→ MiniLM-L6-v2（384 维，极快）
已经在用 Cohere 生态？           → Cohere embed-v3
```

一个重要原则：不要一开始就过度工程化。当 Chroma 在本地运行就足够满足 MVP 需求时，不要去部署分布式 Milvus 集群。需要的时候再扩展，而不是提前扩展。同样，从 MiniLM 开始，质量不够再升级到 BGE，是合理的迭代路径。

### 1.4 一个容易忽视的细节：查询和文档的嵌入方式

很多初学者忽略了一点：**查询（用户的问题）和文档（知识库内容）的嵌入方式可能需要不同处理**。

E5 系列模型明确要求：在嵌入查询时前缀加 `"query: "`，嵌入文档时加 `"passage: "`。BGE 系列对查询建议加 `"为这个句子生成表示以用于检索相关文章："` 的前缀。不加这些前缀，检索质量会明显下降。

OpenAI 的 text-embedding-3 系列没有这个要求，使用起来更简单。

## 二、向量数据库：语义检索的基础设施

### 2.1 向量数据库做了什么

普通数据库（MySQL、PostgreSQL）的查询是精确匹配：`WHERE id = 123`，答案要么有要么没有。

向量数据库做的是**近似最近邻搜索（ANN，Approximate Nearest Neighbor）**：给定一个查询向量，在几百万、几千万个存储向量中，快速找到最相似的 K 个。

精确找到"最近的"向量在数学上需要和每个存储向量都比较一遍（O(n) 复杂度），在百万量级下太慢。ANN 算法用一些巧妙的数据结构（如 HNSW——分层可导航小世界图），以极小的精度损失换来极大的速度提升，通常能在毫秒内从千万向量中找到答案。

### 2.2 相似度度量：余弦 vs 欧氏距离

向量库提供三种常见的相似度度量：

**余弦相似度（Cosine Similarity）**：衡量两个向量方向的夹角，不关心长度。值域 -1 到 1，1 表示完全相同。大多数文本嵌入模型推荐用这个。BGE 和 OpenAI text-embedding-3 都推荐余弦相似度。

**欧氏距离（L2 Distance）**：两个向量的直线距离，值越小越相似。如果嵌入向量已经做了归一化（长度为 1），余弦相似度和欧氏距离等价。

**内积（Dot Product）**：两个向量的点积。如果向量没有归一化，点积会受长度影响，长文档可能因为向量"更长"而总是排在前面，出现偏差。

**实践原则**：用模型推荐的度量方式。嵌入模型的训练目标决定了哪种度量最合适，不要乱换。

### 2.3 主流向量数据库横向对比

| 数据库 | 类型 | 规模定位 | 核心优势 | 典型用途 |
|--------|------|---------|---------|---------|
| **Chroma** | 本地/嵌入式 | 百万以内 | 零配置，Python 原生，极速上手 | **本地实验、快速原型** |
| **pgvector** | PostgreSQL 扩展 | 千万以内 | 如果你已有 PostgreSQL，零额外基础设施 | **已有 PG 的团队，中等规模** |
| **Qdrant** | 独立服务（Rust） | 亿级 | 高性能，支持复杂过滤，云托管选项 | **生产环境推荐** |
| **Milvus** | 独立服务 | 亿级以上 | 超大规模，云原生架构，功能最全 | **超大规模生产** |
| **Pinecone** | 全托管云服务 | 任意 | 零运维，按量付费 | **不想管基础设施** |
| **Weaviate** | 独立/托管 | 亿级 | 内置模型支持，GraphQL 接口 | **需要集成多模态** |
| **FAISS** | 库（非数据库）| 任意 | Facebook 出品，纯内存，学术研究 | **研究实验** |

对于生产环境中向量数量在 1000 万以下的情况：带 HNSW 索引的 pgvector 可以很好地处理，而且少维护一个服务。对于生产环境中超过 1000 万向量：选 Qdrant 或 Pinecone，在这个规模下需要专用的向量搜索基础设施，这些都经过了实战验证。

**选型决策树**：

```
你是在做原型实验？
    是 → Chroma（零配置，装包即用）
    否 → 继续

你的团队已经在用 PostgreSQL？
    是 → pgvector（不用引入新基础设施）
    否 → 继续

你需要处理的向量数量超过 1000 万？
    是 → Qdrant（性能更强，运维合理）或 Milvus（超大规模）
    否 → Qdrant 或 pgvector 都可以

你完全不想管基础设施（付钱换省心）？
    → Pinecone 或 Qdrant Cloud 或 Zilliz Cloud
```

## 三、搭建第一个 RAG 系统

理论说完，来看最小可行 RAG 的核心代码。这里用 LlamaIndex 框架，它是目前搭建 RAG 系统最简洁的工具之一。

### 3.1 五步搭建本地 RAG

```python
# 安装依赖
# pip install llama-index llama-index-embeddings-huggingface chromadb

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.core.node_parser import SentenceSplitter

# ── 步骤一：配置嵌入模型 ──
# 用 BGE-M3（中文最强），或者换成 OpenAI API
Settings.embed_model = HuggingFaceEmbedding(
    model_name="BAAI/bge-m3",
    # 查询时加指令前缀，提升检索质量
    query_instruction="为这个句子生成表示以用于检索相关文章：",
)

# ── 步骤二：配置切块策略 ──
# chunk_size：每块多少个 token（512 是个合理起点）
# chunk_overlap：相邻块的重叠量（避免语义在边界断裂）
Settings.node_parser = SentenceSplitter(
    chunk_size=512,
    chunk_overlap=50,
)

# ── 步骤三：加载文档 ──
# SimpleDirectoryReader 支持 PDF、Word、TXT、HTML 等多种格式
documents = SimpleDirectoryReader("./my_docs").load_data()
print(f"加载了 {len(documents)} 个文档")

# ── 步骤四：构建索引（向量化并存入向量库）──
# 这一步会：把文档切块 → 每块调用嵌入模型 → 存入 ChromaDB
index = VectorStoreIndex.from_documents(
    documents,
    show_progress=True,
)

# ── 步骤五：查询 ──
query_engine = index.as_query_engine(
    similarity_top_k=5,      # 检索最相似的 5 个块
    response_mode="compact", # 压缩多个上下文，减少 Token 消耗
)

response = query_engine.query("公司的退款政策是什么？")
print(response)

# 查看来源（这是 RAG 相比纯 LLM 的核心优势）
for source in response.source_nodes:
    print(f"\n来源文档：{source.metadata.get('file_name', '未知')}")
    print(f"相关片段：{source.text[:200]}...")
    print(f"相似度分数：{source.score:.3f}")
```

### 3.2 持久化存储（向量不用每次重算）

上面的代码每次运行都会重新向量化文档，很慢也很贵。实际使用时要持久化：

```python
import chromadb
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.core import StorageContext

# 持久化到本地磁盘
chroma_client = chromadb.PersistentClient(path="./chroma_storage")
chroma_collection = chroma_client.get_or_create_collection("my_rag_docs")

vector_store = ChromaVectorStore(chroma_collection=chroma_collection)
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# 第一次运行：建立索引（慢，要调用嵌入模型）
index = VectorStoreIndex.from_documents(
    documents,
    storage_context=storage_context,
)

# 之后每次运行：直接加载（快，向量已经存好了）
index = VectorStoreIndex.from_vector_store(
    vector_store,
    storage_context=storage_context,
)
```

### 3.3 换用 OpenAI API（适合不想自托管嵌入模型的情况）

```python
import os
from llama_index.embeddings.openai import OpenAIEmbedding

# 只需要改嵌入模型配置，其余代码不变
Settings.embed_model = OpenAIEmbedding(
    model="text-embedding-3-small",
    api_key=os.environ["OPENAI_API_KEY"],
)

# 注意：需要配套使用 OpenAI 的 LLM 或者手动指定你的 LLM
from llama_index.llms.openai import OpenAI
Settings.llm = OpenAI(model="gpt-4o-mini", api_key=os.environ["OPENAI_API_KEY"])
```

## 四、新手最容易踩的坑

### 坑一：切块大小选错了

**太大的块**（1000+ token）：检索精度下降，LLM 要处理大量不相关内容，回答质量变差，而且容易触发 Token 限制。

**太小的块**（< 100 token）：每块缺少足够的上下文，导致语义不完整。想象把"这个政策适用于 2024 年 1 月 1 日之后签订的合同"切成"这个政策适用于"和"2024 年 1 月 1 日之后签订的合同"，两块单独来看都没有意义。

**经验起点**：512 token，重叠 50 token。然后根据你的文档特点调整——技术文档可以大一点（700-1000），FAQ 类可以小一点（200-300）。

### 坑二：嵌入模型和向量库的相似度度量不匹配

BGE 系列训练时用余弦相似度，你的向量数据库配置成了欧氏距离，检索结果会变差。重要：嵌入模型和相似度算法必须匹配！BGE 模型推荐余弦相似度，OpenAI text-embedding-3 系列也推荐余弦相似度。

### 坑三：把检索质量的锅扣在 LLM 头上

RAG 回答不好，通常第一反应是"换个更好的大模型"。但很多时候真正的问题是**检索到的文档和问题根本不相关**——LLM 拿着无关文档，当然给不出好答案。

应该先单独评估检索质量（检索出来的前 K 个文档，人工看一看，是否真的和问题相关），再评估生成质量。当 RAG 系统给出糟糕的答案时，团队往往会怪 LLM。但真正的问题往往出在检索步骤——正确的文本块从未被找到。应该独立评估检索和生成，建立一个有已知相关文本块的测试查询集。

### 坑四：文档质量被忽视

把扫描版 PDF 导入 RAG，OCR 识别出来的文字乱七八糟，嵌入出来的向量也是垃圾。输入数据质量决定了 RAG 质量的上限。在建立索引之前，值得花时间检查文档解析质量。

## 五、工业落地的架构演进

一个典型的 RAG 系统从 demo 到生产的演进路径：

**阶段一：Demo**（1-2 周）
- Chroma 本地向量库
- OpenAI text-embedding-3-small
- LlamaIndex 或 LangChain 框架
- 固定 512 token 切块

**阶段二：测试优化**（1-3 个月）
- 引入 RAGAS 评估框架，量化检索质量
- 尝试混合检索（向量 + BM25 关键词）
- 引入 Reranker 提升精度
- 根据评估结果调整切块策略

**阶段三：生产上线**（按需）
- 迁移到 Qdrant 或 Milvus（处理更大规模）
- 如有隐私要求，迁移到私有化部署的嵌入模型（BGE 自托管）
- 增加监控：检索耗时、召回率、用户反馈
- 定期更新索引（新文档自动入库）

## 六、面试常见问题

**Q1：什么是向量嵌入？为什么语义相近的内容向量也相近？**

嵌入是把文本转换成高维数字向量的过程。嵌入模型（通常是 Transformer 架构）在大量文本上训练，学习到了语言的语义规律：经常出现在相似上下文中的词/句子，在训练过程中会被推向相近的向量位置。这是词嵌入领域从 Word2Vec 开始就发现的分布式语义假说的体现——语义相似的内容在向量空间中自然聚集。

**Q2：如何选择嵌入模型？**

考虑四个维度：语言（中文优先 BGE 系列，英文可用 OpenAI 或 E5）；隐私（数据不能出境就必须用自托管的开源模型）；规模和成本（小规模原型用 API，大规模生产考虑自托管以节省成本）；质量要求（查 MTEB 排行榜看你的语言和任务的最新最强模型）。不要一开始就选最重的模型，从轻量模型开始，质量不够再升级。

**Q3：如何选择向量数据库？**

核心变量是规模和运维能力。实验阶段用 Chroma（零配置）；团队已有 PostgreSQL 且规模在 1000 万向量以内，用 pgvector（不增加新服务）；生产环境需要更强性能用 Qdrant；超大规模用 Milvus；完全不想运维用 Pinecone 云服务。不要用锤子找钉子：大多数企业内部 RAG 场景用 pgvector 就够了。

**Q4：切块大小如何选择？**

没有通用最优答案，取决于文档类型。经验起点是 512 token，重叠 50 token。太大会引入无关信息降低精度，太小会丢失上下文让语义不完整。建议：建立一个评估集（有问题和已知答案的测试集），用不同切块参数构建索引，对比检索召回率，选最优的。
