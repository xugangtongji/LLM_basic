## 零、开篇：数据是大模型的燃料

有一句在大模型领域广泛流传的话：**"数据的重要性不亚于模型架构"**。

一个设计精良的架构，如果用低质量数据训练，最终效果必然差强人意。反之，足够高质量的数据，即使配合相对平凡的架构，也能训练出令人惊喜的模型。FineWeb 的实验证明：在 FineWeb 数据上训练的模型，效果优于在 RefinedWeb、C4、DolmaV1.6、The Pile 和 SlimPajama 上训练的模型。

大模型预训练数据管道（Data Pipeline）是一套从"互联网原始 HTML"到"干净 token 序列"的复杂工程系统。本章把它拆解成五个环节，逐一讲清楚每步在做什么、为什么这样做、有哪些工程权衡。

## 一、数据来源：从哪里找万亿 token

### 1.1 初印象：用费曼视角理解

**预训练数据的规模令人咋舌**——LLaMA-3 训练了 15 万亿（15T）个 token，相当于大约 150 亿本书，或者整个人类已知历史中写下的所有文字的好几倍。

这些数据从哪里来？

**互联网是最大的矿山**。Common Crawl 是一个非营利组织，定期爬取整个互联网，每次爬取约 30 亿个网页，每月发布一个数据集快照。从 2008 年到现在，它积累了 100 多个 dump，总计数百 TB 的原始 HTML 数据，是大多数 LLM 预训练数据的主要来源。

## 二、第一步：原始数据清洗

### 2.1 初印象：用费曼视角理解

**Common Crawl 的原始数据就像从垃圾桶里淘宝**——里面有珍贵的书籍摘录、高质量文章、代码教程，但也充斥着：

- 导航菜单、广告文字、版权声明
- 乱码、编码错误的文字
- 数十种语言混杂
- 色情、暴力、仇恨内容
- 相同内容的无数个转载版本

清洗的目标：**保留真正有价值的文字内容，去掉所有其他噪声**。

### 2.2 基础文本清洗

```python
import re
import unicodedata

def basic_text_cleaning(text: str) -> str:
    """
    基础文本清洗：处理编码、格式、特殊字符问题。
    这是数据管道的第一道过滤。
    """
    # 1. Unicode 归一化（统一不同编码方式表示的相同字符）
    text = unicodedata.normalize("NFC", text)

    # 2. 去掉不可见字符（零宽空格、控制字符等）
    text = re.sub(r'[\x00-\x08\x0b\x0c\x0e-\x1f\x7f]', '', text)

    # 3. 统一换行符
    text = text.replace('\r\n', '\n').replace('\r', '\n')

    # 4. 合并多余空白行（3 个以上换行符合并为 2 个）
    text = re.sub(r'\n{3,}', '\n\n', text)

    # 5. 去掉行首行尾空白
    lines = [line.strip() for line in text.split('\n')]
    text = '\n'.join(lines)

    return text.strip()


def html_to_text(html: str) -> str:
    """
    从 HTML 提取纯文本（实际用 trafilatura 或 jusText 等库）。
    """
    # 实际工程中用专门的库，这里只是演示逻辑
    # pip install trafilatura
    # import trafilatura
    # return trafilatura.extract(html, include_comments=False, include_tables=False)

    # 简化示例：去掉 HTML 标签
    text = re.sub(r'<script[^>]*>.*?</script>', '', html, flags=re.DOTALL)
    text = re.sub(r'<style[^>]*>.*?</style>', '', html, flags=re.DOTALL)
    text = re.sub(r'<[^>]+>', ' ', html)
    text = re.sub(r'&nbsp;', ' ', text)
    text = re.sub(r'&amp;', '&', text)
    text = re.sub(r'&lt;', '<', text)
    text = re.sub(r'&gt;', '>', text)
    return basic_text_cleaning(text)
```

### 2.3 启发式质量过滤

清洗掉明显的低质量内容。这些规则不是百分之百准确，但简单高效：

```python
def heuristic_quality_filter(text: str) -> tuple[bool, str]:
    """
    启发式质量过滤：快速剔除明显低质量文档。
    返回 (是否保留, 拒绝原因)
    """
    # 1. 长度过滤
    words = text.split()
    if len(words) < 50:
        return False, f"太短 ({len(words)} 词)"
    if len(words) > 100000:
        return False, f"太长 ({len(words)} 词)"

    # 2. 字符比例过滤（非字母/数字/标点字符过多 → 乱码）
    alpha_ratio = sum(c.isalpha() for c in text) / max(len(text), 1)
    if alpha_ratio < 0.5:
        return False, f"字母比例过低 ({alpha_ratio:.2%})"

    # 3. 重复行过滤（大量重复行 → 模板页面、SEO 垃圾内容）
    lines = [l.strip() for l in text.split('\n') if l.strip()]
    if len(lines) > 0:
        unique_ratio = len(set(lines)) / len(lines)
        if unique_ratio < 0.7:
            return False, f"重复行过多 (唯一行比例 {unique_ratio:.2%})"

    # 4. 停用词比例（英文）：正常文本有一定比例的高频功能词
    # 比例过低 → 可能是关键词列表、代码等
    stopwords = {'the', 'a', 'an', 'is', 'are', 'was', 'were',
                 'of', 'in', 'to', 'for', 'and', 'or', 'but'}
    word_set = set(w.lower() for w in words)
    stop_overlap = len(word_set & stopwords)
    # 英文文档的停用词覆盖应 >= 3 个
    if text.isascii() and stop_overlap < 3:
        return False, f"停用词比例异常 (覆盖 {stop_overlap} 个)"

    # 5. 特殊内容标记（可扩展）
    spam_indicators = ['点击这里', 'Click here', 'Subscribe now',
                       '免费下载', '联系我们', '©']
    spam_count = sum(1 for indicator in spam_indicators if indicator in text)
    if spam_count >= 5:
        return False, f"疑似垃圾内容 ({spam_count} 个标记)"

    return True, "通过"


# 测试
samples = [
    "深度学习是机器学习的一个分支。" * 50,   # 重复内容
    "AI ML NLP CV DL GPT LLM RAG ANN CNN RNN",  # 关键词列表
    "这是一篇关于Transformer架构的详细文章，介绍了自注意力机制的工作原理..." + " 这是正常文本内容。" * 20,
]

for s in samples:
    keep, reason = heuristic_quality_filter(s)
    print(f"{'保留' if keep else '丢弃'}: {reason}  (前30字: {s[:30]}...)")
```

## 三、第二步：去重

### 3.1 初印象：用费曼视角理解

**互联网上大量内容是重复的**——同一篇新闻被几百个网站转载，同一段维基百科文字出现在无数教育网站上。

重复文本会导致过拟合，如果某内容重复出现多次，会不成比例地影响模型。比如，如果训练数据里"苹果公司是一家科技公司"重复了 10000 次，模型就会把这句话背得滚瓜烂熟，但这对理解苹果公司没有额外帮助——反而浪费了计算资源。

去重是 2024 年预训练数据处理的必要环节，常见方法包括基于哈希的精确去重，以及使用 MinHash 或 SimHash 的模糊去重，以捕捉近似重复文档。

### 3.2 精确去重：MD5/SHA 哈希

最简单的去重方式：对每篇文档计算 MD5 哈希值，相同哈希 = 完全重复，直接删除。

```python
import hashlib
from collections import defaultdict

def exact_dedup(documents: list[str]) -> list[str]:
    """
    精确去重：基于哈希，去掉完全相同的文档。
    时间复杂度 O(n)，适合大规模快速去重。
    """
    seen_hashes = set()
    unique_docs = []

    for doc in documents:
        # 对文本内容计算 MD5（归一化后，避免空白差异）
        normalized = ' '.join(doc.lower().split())
        doc_hash = hashlib.md5(normalized.encode('utf-8')).hexdigest()

        if doc_hash not in seen_hashes:
            seen_hashes.add(doc_hash)
            unique_docs.append(doc)

    removed = len(documents) - len(unique_docs)
    print(f"精确去重：{len(documents)} → {len(unique_docs)} 文档（移除 {removed} 个重复）")
    return unique_docs

# 测试
docs = [
    "深度学习是机器学习的一个分支。",
    "深度学习是机器学习的一个分支。",   # 完全重复
    "Transformer 是一种基于注意力机制的神经网络架构。",
    "深度学习是机器学习的一个分支。",   # 再次重复
    "BERT 是 Google 发布的预训练语言模型。",
]
unique = exact_dedup(docs)
```

### 3.3 模糊去重：MinHash LSH

精确去重只能处理完全相同的文档。实际上，很多"重复"是轻微修改版本——改了几个词、加了一段导航文字、调整了格式。这需要**近似去重（Near-Duplicate Detection）**。

**MinHash（最小哈希）** 是近似去重的工业标准。原理：

1. 把文档切成 n-gram（字符级或词级小片段）
2. 用多个哈希函数对 n-gram 集合计算最小哈希值 → 得到文档的"签名"（Signature）
3. 两个文档的签名相似度（Jaccard 相似度的估计）超过阈值 → 视为重复

```python
import hashlib
import random
from typing import Set

def get_ngrams(text: str, n: int = 5) -> Set[str]:
    """从文本生成字符级 n-gram 集合"""
    text = ' '.join(text.lower().split())   # 归一化
    return {text[i:i+n] for i in range(len(text) - n + 1)}

def minhash_signature(ngrams: Set[str], num_hashes: int = 128) -> list[int]:
    """
    计算 MinHash 签名。
    num_hashes 个哈希函数，每个取最小值，组成签名向量。
    """
    signature = []
    for seed in range(num_hashes):
        min_hash = float('inf')
        for ngram in ngrams:
            # 用不同 seed 的哈希函数
            h = int(hashlib.md5(f"{seed}:{ngram}".encode()).hexdigest(), 16)
            min_hash = min(min_hash, h)
        signature.append(min_hash)
    return signature

def jaccard_estimate(sig1: list[int], sig2: list[int]) -> float:
    """用 MinHash 签名估计两文档的 Jaccard 相似度"""
    matches = sum(h1 == h2 for h1, h2 in zip(sig1, sig2))
    return matches / len(sig1)

def minhash_dedup(documents: list[str],
                   similarity_threshold: float = 0.8,
                   num_hashes: int = 128) -> list[str]:
    """
    MinHash 近似去重。
    similarity_threshold：相似度超过此值视为重复（保留其中一个）。
    """
    print(f"正在计算 {len(documents)} 篇文档的 MinHash 签名...")
    signatures = []
    for doc in documents:
        ngrams = get_ngrams(doc, n=5)
        sig = minhash_signature(ngrams, num_hashes)
        signatures.append(sig)

    # 两两比较（大规模时需要 LSH 加速，这里简化为暴力比较）
    to_remove = set()
    for i in range(len(documents)):
        if i in to_remove:
            continue
        for j in range(i + 1, len(documents)):
            if j in to_remove:
                continue
            sim = jaccard_estimate(signatures[i], signatures[j])
            if sim >= similarity_threshold:
                to_remove.add(j)   # 保留 i，移除 j

    unique_docs = [doc for i, doc in enumerate(documents) if i not in to_remove]
    print(f"MinHash 去重：{len(documents)} → {len(unique_docs)} 文档")
    return unique_docs

# 测试：近似重复文档
near_dup_docs = [
    "Transformer 是一种基于注意力机制的神经网络，由 Google Brain 在 2017 年提出。",
    "Transformer 是一种基于注意力机制的神经网络架构，由 Google Brain 在 2017 年提出。",  # 只加了"架构"
    "BERT 是 Google 发布的预训练语言模型，基于 Transformer 编码器。",
    "深度学习模型需要大量数据进行训练。",
]
unique = minhash_dedup(near_dup_docs, similarity_threshold=0.7, num_hashes=64)
print("保留的文档:")
for doc in unique:
    print(f"  - {doc[:50]}...")
```

LLaMA-3 在整个数据集上使用了 MinHash 去重，同时用 CCNet 做行级去重，移除了在同一桶内重复超过六次的行。

### 3.4 去重的工程规模

真实的大规模去重不能用暴力 $O(n^2)$ 比较，需要 LSH（局部敏感哈希）加速：

```
MinHash LSH 的工作原理：
1. 把签名分成 b 个 band，每个 band 有 r 个哈希值
2. 同一 band 内签名完全相同的两个文档，放入同一个"桶"
3. 只比较同一桶内的文档对
4. 时间复杂度从 O(n²) 降到接近 O(n)

参数选择影响精度和召回的权衡：
- Falcon RefinedWeb 去重：10 亿原始页面 → 2.8TB 干净文本
  保留率约 0.28%，去掉了 99.72% 的重复和低质量内容
```

## 五、第四步：数据配比策略

### 5.1 初印象：用费曼视角理解

**数据配比是大模型训练中最"玄学"也最关键的决策之一**——不同来源的数据按什么比例混合，直接影响模型最终的能力分布。

就像培养一个学生：全学文学，数学能力弱；全学数学，语言表达差。最优比例因目标而异，没有放之四海皆准的答案。

### 5.2 配比的几个核心原则

**原则一：高质量数据要上采样**

Wikipedia、书籍、学术论文在互联网上的比例远低于普通网页，但质量更高。几乎所有大模型都会对这些来源进行上采样（多训练几遍）。

**原则二：代码数据可以大比例**

研究发现，代码训练能显著提升模型的逻辑推理和结构化思维能力，即使对非代码任务也有帮助。LLaMA-3 用了约 20% 的代码数据，远超代码在真实网络上的比例（约 1-2%）。

**原则三：语言平衡**

如果目标模型需要支持多语言，需要显式控制各语言的比例，避免英文数据完全压制其他语言（英文在 Common Crawl 里约占 50%+）。

```python
# 数据配比的工程实现：基于加权采样
import random
from dataclasses import dataclass
from typing import Generator

@dataclass
class DataSource:
    name: str
    path: str
    weight: float        # 配比权重
    epoch_multiplier: float = 1.0   # 上采样倍数（高质量数据可以 > 1）

def weighted_data_sampler(sources: list[DataSource]) -> Generator:
    """
    按权重混合多个数据源，产出训练批次。
    这是预训练数据加载的核心逻辑。
    """
    # 归一化权重
    total = sum(s.weight for s in sources)
    probs = [s.weight / total for s in sources]

    # 模拟数据流
    # 实际工程中每个 source 是一个无限迭代的数据流
    while True:
        source = random.choices(sources, weights=probs, k=1)[0]
        # yield next_batch_from(source.path)  # 从对应数据源读取
        yield f"来自 [{source.name}] 的数据批次"

# 典型配置示例（参考 LLaMA-3 / Qwen-2.5 的数据混合策略）
data_sources = [
    DataSource("通用网页", "/data/common_crawl",    weight=0.67, epoch_multiplier=1.0),
    DataSource("代码",     "/data/github",           weight=0.20, epoch_multiplier=1.0),
    DataSource("书籍",     "/data/books",            weight=0.05, epoch_multiplier=2.0),  # 上采样
    DataSource("Wikipedia", "/data/wiki",            weight=0.04, epoch_multiplier=4.0),  # 多次上采样
    DataSource("ArXiv",    "/data/arxiv",            weight=0.04, epoch_multiplier=2.0),
]

print("数据配比配置:")
for src in data_sources:
    print(f"  {src.name}: 权重={src.weight:.0%}, 上采样={src.epoch_multiplier}×")
```

## 六、第五步：最终格式化与 Token 化

### 6.1 文档格式化

清洗完成的文档，需要按模型要求格式化：添加文档分隔符、截断/填充到固定长度（或使用动态填充）。

```python
import torch

def format_and_tokenize(
    documents: list[str],
    tokenizer,
    max_length: int = 4096,
    eos_token: str = "<|endoftext|>",
) -> list[list[int]]:
    """
    把清洗后的文档 Token 化，准备送进训练。
    """
    all_token_ids = []

    for doc in documents:
        # 在文档末尾加 EOS token，作为文档边界信号
        text = doc + eos_token
        token_ids = tokenizer.encode(text)

        # 长文档切分成多个训练样本
        for i in range(0, len(token_ids), max_length):
            chunk = token_ids[i: i + max_length]
            if len(chunk) < 64:   # 太短的 chunk 丢弃
                continue
            all_token_ids.append(chunk)

    print(f"文档数: {len(documents)}, 训练样本数: {len(all_token_ids)}")
    return all_token_ids


# Pack 打包：把多个短文档拼成一个长序列（减少 padding 浪费）
def pack_documents(token_chunks: list[list[int]], seq_len: int = 4096) -> list[list[int]]:
    """
    把多个 token 序列打包成固定长度的训练样本。
    不同文档之间用 EOS token 分隔，不做额外 padding。
    这样每个训练步都能利用完整的 seq_len，没有浪费。
    """
    packed = []
    current = []

    for chunk in token_chunks:
        if len(current) + len(chunk) <= seq_len:
            current.extend(chunk)
        else:
            if current:
                # 不足 seq_len 的部分用 pad token 填充（或直接丢弃）
                packed.append(current + [0] * (seq_len - len(current)))
            current = chunk[:seq_len]

    if current:
        packed.append(current + [0] * (seq_len - len(current)))

    return packed
```

## 八、面试常见问题

**Q1：LLM 预训练数据主要来自哪里？为什么 Common Crawl 这么重要？**

LLM 预训练数据主要来自网页（60-70%）、代码（10-20%）、书籍、学术论文和百科全书。Common Crawl 是最大的数据来源，因为它从 2008 年开始持续爬取整个公开互联网，积累了超过 100 个快照，总量达数百 TB，是目前公开可用的最大文本语料库。绝大多数主流模型（LLaMA、GPT 系列、Qwen）的网页数据都来自 Common Crawl 的处理版本。

**Q2：为什么要做去重？不去重会怎样？**

重复数据有两个主要危害：浪费计算资源（重复见同一内容相当于低效学习）；导致模型"记住"重复内容而非"理解"，表现为对高频重复内容的过度自信（类似过拟合），以及基准测试污染风险（如果测试题目在训练集里大量重复出现）。去重通常能在不减少有效信息的前提下，让同等计算量训练出更好的模型。

**Q3：MinHash 是什么？为什么用它而不用精确哈希做去重？**

精确哈希（MD5/SHA）只能发现完全相同的文档，但现实中大量重复是"轻微修改版"（换了几个词、加了广告语、调整了格式）。MinHash 通过计算文档的 n-gram 集合的近似 Jaccard 相似度，能发现这类近似重复。配合 LSH（局部敏感哈希），MinHash 去重可以在 $O(n)$ 时间复杂度内处理数十亿文档规模的数据集。

**Q4：高质量数据过滤的主要方法有哪些？各有什么优缺点？**

三种主流方法：启发式规则（速度快，无需训练，但覆盖面有限，容易误判）；困惑度过滤（用小型语言模型评估文本自然度，对领域特异性文本可能误判）；分类模型过滤（质量最高，但需要人工标注数据来训练分类器）。实际大规模数据管道通常将三者结合：先用启发式快速过滤大量明显垃圾，再用分类模型精细筛选。

**Q5：为什么代码数据在 LLM 预训练中占比远超其在互联网上的实际比例？**

研究发现，代码训练能显著提升模型的逻辑推理、结构化思维和指令跟随能力，不仅在代码任务上，在数学、推理等一般任务上也有明显提升。代码的结构特性（精确的语法、明确的逻辑关系、清晰的输入输出）让模型学到了更系统的"思维方式"。因此 LLaMA-3 等模型将代码数据比例上调到 20%，远超其在互联网上约 1-2% 的真实占比。

**Q6：什么是"文档打包（Packing）"？为什么需要它？**

Packing 是把多个短文档拼接成一个固定长度的训练序列（不同文档间用 EOS 分隔），而不是为每个文档单独做 padding。原因：如果不打包，短文档会有大量 padding token（全是 0），这些位置不参与 loss 计算，浪费了算力；打包后每个训练步都能充分利用完整序列长度，训练效率显著提升。代价是需要处理跨文档的注意力边界（通常用掩码防止不同文档之间的 token 互相关注）。
