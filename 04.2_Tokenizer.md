## 零、开篇：为什么 Tokenizer 是个大问题

在大模型的完整训练流程里，Tokenizer 是最容易被忽视、却影响深远的一个环节。

**Tokenizer 的选择影响一切**：词表大小、训练速度、内存使用，以及最终的模型性能。词表大小会产生重要权衡：词表越大，序列越短，但需要更大的 Embedding 表；词表越小，序列越长，注意力计算代价越高。

举个直观的例子：

```
词表 32K（LLaMA-1）：
  "我喜欢深度学习" → ['我', '喜欢', '深度', '学习']  → 4 个 token

词表 128K（LLaMA-3）：
  "我喜欢深度学习" → ['我喜欢', '深度学习']          → 2 个 token

同样的文章，词表大的版本 token 数少一半！
同样的上下文窗口（8192 token），能装进更多内容。
```

这就是为什么 LLaMA-3 要把词表从 32K 扩大到 128K，Qwen-2.5 用了 152K 的超大词表。

## 一、为什么不直接按字或按词切分

### 1.1 初印象：用费曼视角理解

**最朴素的想法是：空格分词**——"深度 学习 很 有趣"，按空格切成四个词，每个词是一个 token。

但这有三个致命问题：

**问题一：词表会无限大**

英文单词大约有 100 万个以上，还有形态变化（play/plays/played/playing），专有名词、技术术语、网络词汇层出不穷。词表可能需要包含几百万个 token，Embedding 矩阵会大到无法接受。

**问题二：不认识新词**

训练时词表里没有的词，模型只能用 `<UNK>`（未知词）表示，完全丢失信息。

**问题三：不同语言差异巨大**

中文没有空格，不能按空格分词；德语有超长复合词（Rindfleischetikettierungsüberwachungsaufgabenübertragungsgesetz，就是一个词！）；代码有各种特殊符号。

**另一个极端：按字母（字符）切分**

每个字母是一个 token，词表极小（26 个字母 + 标点），但序列会变得超长，注意力计算代价极高——"hello world" 就是 11 个 token，一篇文章可能要几万个 token。

**最优解：子词切分（Subword Tokenization）**

把词拆成更小的有意义单元，常见词保持完整，罕见词拆成子词：
- "playing" → ["play", "##ing"]
- "unbelievable" → ["un", "##believ", "##able"]
- "我喜欢深度学习" → ["我", "喜欢", "深度", "学习"]

这样词表大小适中（3-15万），几乎不会出现 `<UNK>`，还能处理新造词。

## 二、BPE：最流行的子词切分算法

### 2.1 初印象：用费曼视角理解

**BPE（Byte Pair Encoding）原本是一种数据压缩算法，后来被用在 Tokenizer 上**。

压缩的思路：找到最频繁出现的字符对，用一个新符号代替它。重复这个过程，就能把常见的字符组合"压缩"成一个单元。

用在 Tokenizer 上：从最细的粒度（单个字符）开始，反复合并最高频的相邻单元，直到词表达到目标大小。

最终结果：高频词（"the"、"and"、"是"）保持完整，低频词被拆成多个高频子词。

### 2.2 BPE 算法：逐步手推

BPE 是 Transformer 中最流行的 Tokenizer 算法，被 Llama、Gemma、Qwen2 等模型使用。训练时从单个字符开始，迭代合并最高频相邻对。

```python
from collections import defaultdict, Counter

def train_bpe(corpus: list[str], vocab_size: int) -> tuple[dict, list]:
    """
    从零实现 BPE 训练算法。
    corpus: 训练语料（词列表）
    vocab_size: 目标词表大小
    返回: (词表, 合并规则列表)
    """
    # 第一步：统计语料中所有词的频率
    # 每个词被表示为字符序列（末尾加空格标记词边界）
    word_freqs = Counter(corpus)

    # 第二步：初始词表 = 所有字符
    vocab = set()
    splits = {}   # 每个词当前的切分状态
    for word, freq in word_freqs.items():
        chars = list(word)
        splits[word] = chars
        vocab.update(chars)

    print(f"初始词表大小: {len(vocab)} 个字符")
    print(f"初始词表: {sorted(vocab)}\n")

    merges = []   # 记录所有合并规则

    # 第三步：迭代合并最高频相邻对
    while len(vocab) < vocab_size:
        # 统计所有相邻 pair 的频率
        pair_freqs = defaultdict(int)
        for word, freq in word_freqs.items():
            current = splits[word]
            for i in range(len(current) - 1):
                pair = (current[i], current[i+1])
                pair_freqs[pair] += freq

        if not pair_freqs:
            break

        # 找最高频的 pair
        best_pair = max(pair_freqs, key=pair_freqs.get)
        best_freq = pair_freqs[best_pair]
        new_token = ''.join(best_pair)

        print(f"合并: {best_pair} → '{new_token}'  (频率: {best_freq})")

        # 在所有词中执行合并
        for word in splits:
            current = splits[word]
            new_split = []
            i = 0
            while i < len(current):
                if i < len(current)-1 and (current[i], current[i+1]) == best_pair:
                    new_split.append(new_token)
                    i += 2
                else:
                    new_split.append(current[i])
                    i += 1
            splits[word] = new_split

        vocab.add(new_token)
        merges.append(best_pair)

    return vocab, merges, splits


# 完整演示：用一个小语料训练 BPE
corpus = (
    ["low"] * 5 + ["lower"] * 2 + ["newest"] * 6 +
    ["widest"] * 3 + ["wider"] * 4
)

print("=" * 50)
print("BPE 训练过程演示")
print("=" * 50)
print(f"语料词频: {dict(Counter(corpus))}\n")

vocab, merges, final_splits = train_bpe(corpus, vocab_size=15)

print(f"\n最终词表 ({len(vocab)} 个): {sorted(vocab)}")
print(f"\n合并规则 ({len(merges)} 条):")
for i, merge in enumerate(merges, 1):
    print(f"  规则{i}: {merge[0]} + {merge[1]} → {''.join(merge)}")

print(f"\n最终切分结果:")
for word, split in final_splits.items():
    print(f"  '{word}' → {split}")
```

### 2.3 BPE 推理：如何对新文本分词

训练完成后，有了词表和合并规则。推理时：

1. 把输入文本先按字符切分
2. 按照训练时学到的合并规则，**依次**（按规则顺序）执行合并
3. 最终得到 token 序列

```python
def bpe_encode(word: str, merges: list[tuple]) -> list[str]:
    """
    用学到的合并规则对新词进行 BPE 编码。
    """
    # 初始：字符级切分
    chars = list(word)

    # 依次应用合并规则（注意：规则有顺序！先学到的规则先应用）
    for pair in merges:
        new_chars = []
        i = 0
        while i < len(chars):
            if i < len(chars)-1 and (chars[i], chars[i+1]) == pair:
                new_chars.append(''.join(pair))
                i += 2
            else:
                new_chars.append(chars[i])
                i += 1
        chars = new_chars

    return chars

# 测试
test_words = ["newest", "lower", "lowest", "newwidest"]
for word in test_words:
    encoded = bpe_encode(word, merges)
    print(f"'{word}' → {encoded}")
```

## 三、Byte-Level BPE：解决所有语言的通用方案

### 3.1 初印象：用费曼视角理解

**标准 BPE 有一个隐患：遇到词表里没有的字符怎么办？**

中文、阿拉伯文、日文、表情符号……Unicode 有超过 100 万个字符。如果初始词表只包含训练数据中出现过的字符，推理时遇到没见过的字符就会报错。

**Byte-Level BPE（字节级 BPE）** 的解法：不在字符层面操作，而在**字节**层面操作。

任何文本（不管什么语言）最终都是字节序列（UTF-8 编码后每个字符是 1-4 个字节）。字节只有 256 种（0-255），初始词表只需要 256 个，然后用 BPE 在字节序列上做合并——保证永远不会出现未知字符。

GPT-2 和 RoBERTa 的 tokenizer 不把词看作 Unicode 字符，而是字节序列。这样初始词表只有 256 个（所有可能的字节），但任何字符都能表示，永远不会出现未知 token。这个技巧叫 Byte-level BPE。

**GPT-2、GPT-3、LLaMA-1/2 全都用的是 Byte-level BPE。**

### 3.2 代码演示

```python
# Byte-level BPE 的核心思路
text = "Hello 世界 🌍"

# 先把文本编码成字节序列
byte_seq = text.encode('utf-8')
print(f"原始文本: '{text}'")
print(f"UTF-8 字节: {list(byte_seq)}")
print(f"字节长度: {len(byte_seq)}")
# "世界" 和 "🌍" 各需要多个字节

# 在 Byte-level BPE 中，初始 "字符" 就是 256 种字节值
# 然后 BPE 在字节序列上合并，最终形成对字节序列的切分
# 好处：永远不会遇到 <UNK>，任何文本都能处理
```

## 四、WordPiece 与 BERT 的 Tokenizer

### 4.1 WordPiece 和 BPE 的核心区别

WordPiece 是 Google 为 BERT 设计的 Tokenizer，和 BPE 非常相似，只有一个关键区别：

**选择合并哪对时，BPE 选频率最高的，WordPiece 选合并后"增益"最大的**：

$$\text{score}(\text{pair}) = \frac{\text{freq}(\text{pair})}{\text{freq}(A) \times \text{freq}(B)}$$

这个得分衡量的是：合并这对 token 能带来多少"惊喜"——如果 A 和 B 都很常见，但 AB 组合的频率远高于"随机概率"$\text{freq}(A) \times \text{freq}(B)$，说明这个组合有特殊含义，值得合并。

**WordPiece 的另一个特点**：子词前缀使用 `##` 标记，表示"这个片段是词的一部分而不是词的开头"：

```python
# WordPiece 分词示例（BERT 风格）
# "playing" → ["play", "##ing"]   # "##ing" 说明 ing 是词的后缀
# "unhappy" → ["un", "##happy"]

# 示意代码
def wordpiece_demo():
    examples = [
        ("playing", ["play", "##ing"]),
        ("unhappy", ["un", "##happy"]),
        ("unbelievable", ["un", "##believ", "##able"]),
        ("Transformer", ["transform", "##er"]),
    ]
    for word, tokens in examples:
        print(f"'{word}' → {tokens}")

wordpiece_demo()
```

## 五、SentencePiece：与语言无关的通用方案

### 5.1 初印象：用费曼视角理解

**BPE 和 WordPiece 都有一个隐含假设：先按空格预分词，然后在词内部做子词切分**。

但中文没有空格，阿拉伯语的词和词之间的边界也不清晰……对这些语言，"先按空格分词"的假设就失效了。

**SentencePiece 的核心理念：把整个输入字符串（包括空格）直接作为原始序列，不做任何语言特定的预处理**。

空格在 SentencePiece 里被特殊处理，用 `▁`（下划线）标记词的开头：
- "Hello World" → ["▁Hello", "▁World"]
- "我喜欢你" → ["▁我", "喜欢", "你"]（中文没有空格，▁只在开头）

SentencePiece 把输入文本作为原始字节序列处理，不依赖任何语言特定的预处理，因此能处理任意语言。

### 5.2 SentencePiece 的优势

SentencePiece 在内部可以使用 BPE 或 Unigram Language Model 作为子词切分算法：

- **SentencePiece + BPE**：T5、ALBERT、XLNet 使用
- **SentencePiece + Unigram LM**：许多多语言模型使用

```python
# 使用 sentencepiece 库（pip install sentencepiece）
```

## 六、词表大小的权衡

### 6.1 初印象：用费曼视角理解

**词表大小是 Tokenizer 设计中最重要的超参数**，它决定了一个根本性的权衡：

- **词表越大**：常见词保持完整，序列更短，注意力计算更快；但 Embedding 矩阵更大，模型参数增多
- **词表越小**：序列更长，注意力计算慢；但 Embedding 更小，更容易学好低频词

词表大小带来重要权衡：词表越大，序列越短，但需要更大的 Embedding 表；词表越小，Embedding 更小，但序列更长，注意力计算代价更高。

### 6.2 定量分析

```python
def analyze_tokenizer_tradeoff(vocab_size: int, d_model: int,
                                avg_tokens_per_char: float,
                                doc_length_chars: int = 1000):
    """
    分析不同词表大小的权衡。
    avg_tokens_per_char: 平均每个字符对应多少 token（越小越好）
    """
    # Embedding 矩阵参数量（Weight Tying 时只算一次）
    embedding_params = vocab_size * d_model

    # 文档平均 token 数
    avg_tokens = doc_length_chars * avg_tokens_per_char

    # 注意力计算量正比于 L²（序列长度的平方）
    attention_cost = avg_tokens ** 2

    print(f"词表大小: {vocab_size:,}")
    print(f"  Embedding 参数: {embedding_params/1e6:.1f}M")
    print(f"  千字符文档的平均 token 数: {avg_tokens:.0f}")
    print(f"  注意力计算代价（相对）: {attention_cost:.0f}")
    print()

# 典型配置对比
configs = [
    (32000,  4096, 0.8,  "LLaMA-1/2（32K）"),
    (50257,  768,  0.75, "GPT-2（50K）"),
    (128256, 4096, 0.5,  "LLaMA-3（128K）"),
    (152064, 3584, 0.4,  "Qwen-2.5（152K）"),
]

print("不同词表大小的权衡分析（中文文档，d_model=4096）:\n")
for vocab, dim, tpc, name in configs:
    print(f"=== {name} ===")
    analyze_tokenizer_tradeoff(vocab, 4096, tpc)
```

### 6.3 为什么 LLaMA-3 把词表从 32K 扩大到 128K

对比 LLaMA-1（32K）和 LLaMA-3（128K）处理中文的效率：

```python
# 直觉演示（实际数字因分词器实现而异）
text_zh = "今天北京的天气非常好，适合出去散步"
text_en = "The weather in Beijing today is very good, suitable for a walk"

# LLaMA-1（32K 词表）处理中文时，很多汉字拆成多个 token
# 因为中文字符在 BPE 训练时频次不够高
# "今天" 可能被切成 ["今", "天"] 或 ["今", "##天"] 等 2 个 token
# 一篇 1000 字中文文章 ≈ 1500-2000 token

# LLaMA-3（128K 词表）处理中文
# 词表有更多中文 token，常见词保持完整
# "今天" → ["今天"] 1 个 token
# 一篇 1000 字中文文章 ≈ 700-1000 token

# 实际效果：
llama1_example = ["今", "天", "北", "京", "的", "天", "气"]  # 7 tokens（示意）
llama3_example = ["今天", "北京", "的", "天气"]               # 4 tokens（示意）

print(f"LLaMA-1 风格: {llama1_example} ({len(llama1_example)} tokens)")
print(f"LLaMA-3 风格: {llama3_example} ({len(llama3_example)} tokens)")
print(f"效率提升: {len(llama1_example)/len(llama3_example):.1f}×")
```

**扩大词表的三个好处**：

1. **多语言效率提升**：中文、日文、韩文等不再需要把每个字切成多个 token
2. **更长的有效上下文**：同样 8K token 的上下文窗口，能处理更长的中文文本
3. **代码 token 增多**：编程语言的常见词组（`def `, `return `, `import ` 等）有专属 token

**代价**：Embedding 矩阵从 $32000 \times 4096 \approx 1.3$亿参数，增加到 $128256 \times 4096 \approx 5.2$亿参数。但通过 Weight Tying（Embedding 和 LM Head 共享权重），这部分参数不额外增加推理计算量。

## 七、用 HuggingFace 训练和使用 Tokenizer

### 7.1 训练自定义 Tokenizer

```python
from tokenizers import Tokenizer
from tokenizers.models import BPE
from tokenizers.trainers import BpeTrainer
from tokenizers.pre_tokenizers import Whitespace, ByteLevel
from tokenizers.normalizers import NFD, Lowercase, StripAccents, Sequence
from tokenizers.post_processors import TemplateProcessing
import json

def train_tokenizer(
    training_files: list[str],
    vocab_size: int = 32000,
    save_path: str = "my_tokenizer",
    byte_level: bool = True,
) -> Tokenizer:
    """
    使用 HuggingFace tokenizers 库训练 BPE Tokenizer。
    byte_level=True: 使用 Byte-level BPE（适合多语言）
    byte_level=False: 使用标准 BPE（适合单语言）
    """
    # 初始化 BPE 模型
    tokenizer = Tokenizer(BPE(unk_token="<|unk|>"))

    # 预分词器：Byte-level 或 空格分词
    if byte_level:
        tokenizer.pre_tokenizer = ByteLevel(add_prefix_space=False)
    else:
        tokenizer.pre_tokenizer = Whitespace()

    # 训练器配置
    special_tokens = ["<|endoftext|>", "<|unk|>", "<|pad|>", "<|bos|>", "<|eos|>"]
    trainer = BpeTrainer(
        vocab_size=vocab_size,
        special_tokens=special_tokens,
        min_frequency=2,           # 至少出现 2 次的 pair 才考虑合并
        show_progress=True,
    )

    # 从文件训练
    tokenizer.train(training_files, trainer)

    print(f"训练完成！词表大小: {tokenizer.get_vocab_size()}")

    # 保存
    tokenizer.save(f"{save_path}/tokenizer.json")
    return tokenizer
```

### 7.2 Tokenizer 的关键指标

```python
def evaluate_tokenizer(tokenizer, test_texts: list[str]) -> dict:
    """
    评估 Tokenizer 的关键指标：
    - 生育率（Fertility）：平均每个词对应多少 token（越低越好）
    - 未知率（UNK rate）：多少 token 被标记为 <UNK>（越低越好）
    - 压缩率（Compression ratio）：token 数 / 字节数（越低越好）
    """
    total_tokens = 0
    total_words = 0
    total_bytes = 0
    total_unk = 0

    for text in test_texts:
        tokens = tokenizer.tokenize(text)
        total_tokens += len(tokens)
        total_words += len(text.split())
        total_bytes += len(text.encode('utf-8'))
        total_unk += sum(1 for t in tokens if t == tokenizer.unk_token)

    return {
        "fertility": total_tokens / total_words if total_words > 0 else 0,
        "unk_rate": total_unk / total_tokens if total_tokens > 0 else 0,
        "compression": total_tokens / total_bytes if total_bytes > 0 else 0,
    }
```

## 九、面试常见问题

**Q1：BPE 算法的核心步骤是什么？**

三步：初始化（把所有词切成字符，统计字符频率，初始词表为所有字符）；迭代合并（找频率最高的相邻 pair，将其合并为新 token，加入词表；更新所有词的切分状态；重复直到词表达到目标大小）；推理（按学到的合并规则依次处理新文本）。BPE 从单个字符开始，迭代合并最高频的相邻对，直到达到目标词表大小。词表大小等于初始词表大小加上合并次数。

**Q2：WordPiece 和 BPE 有什么区别？**

两者算法框架相同，核心区别在于选择合并 pair 的标准。BPE 选频率最高的 pair；WordPiece 选"增益"最大的 pair，即合并后得分 $\text{freq}(AB) / (\text{freq}(A) \times \text{freq}(B))$ 最大的 pair。WordPiece 更倾向于合并"组合有特殊含义"的 pair，而不仅仅是频率最高的 pair。此外 WordPiece 用 `##` 标记子词，BPE 通常不做这个标记。

**Q3：为什么 LLaMA-3 要把词表从 32K 扩大到 128K？**

主要有三个原因：多语言效率——中文等非英语语言中，32K 词表的 BPE 会把很多汉字切成多个 token，128K 词表有足够多的中文 token，处理效率提升约 1.5-2 倍；有效上下文窗口——同样的 8K token 上下文，128K 词表能承载更多中文内容；代码支持——更多编程关键词有专属 token，代码理解能力更强。代价是 Embedding 矩阵从约 1.3 亿参数增加到约 5.2 亿参数。

**Q4：什么是 Byte-level BPE？解决了什么问题？**

标准 BPE 基于字符操作，如果训练语料不包含某个字符，推理时会报 `<UNK>` 错误。Byte-level BPE 把文本视为字节序列（UTF-8 编码后每个字符 1-4 字节），初始词表只有 256 个可能字节值，任何文本都能表示，永远不会出现 `<UNK>`。GPT-2、GPT-3、LLaMA 系列都使用 Byte-level BPE。

**Q5：SentencePiece 和 BPE 有什么关系？**

SentencePiece 是一个**工具库**，内部可以使用 BPE 或 Unigram Language Model 作为子词切分算法。它和标准 BPE 的主要区别在于：SentencePiece 把整个输入字符串（包括空格）作为原始序列，不做任何语言特定的预处理，用 `▁` 标记词的开始。这使它能处理任何语言（包括无空格的中文、日文等），而不依赖语言特定的分词规则。T5、ALBERT、LLaMA-1/2 都用 SentencePiece。

**Q6：词表大小对模型有哪些影响？如何选择？**

词表大小影响 Embedding 矩阵大小（$V \times d$）和序列长度（词表越大，同样文本的 token 越少）。词表越大：序列更短，注意力计算更快，但 Embedding 参数增多，低频词可能学不好；词表越小：序列更长，注意力代价高，但 Embedding 更紧凑。选择时考虑：目标语言数量（多语言需要更大词表）、上下文长度要求（越长需要越短的序列 = 越大的词表）、计算预算（Embedding 参数量的影响）。现代 LLM 通常选 50K-150K。
