## 零、开篇：提示工程是"软件工程"，不是"魔法咒语"

很多人第一次接触提示工程时，感觉它像是"和 AI 说几句特殊的话，它就变聪明了"。

这种理解是错误的。

提示工程本质上是：**在不修改模型参数的前提下，通过设计输入来引导模型的输出**。这是一门有规律可循的工程实践——和写好代码一样，好的提示需要理解底层机制、遵循设计原则、反复测试迭代。

更准确地说，提示工程不只是一种技巧，它是在指导 LLM 进行推理。对于 LLM 来说，它不能从头学习，所以我们需要通过提示来指导和控制它的行为。

本章讲四种最基础、最实用的技巧——从最简单的 Zero-shot 到需要一些设计的 CoT，再到通过多次采样提升稳定性的 Self-Consistency。

## 一、Zero-shot：直接问，不给例子

### 1.1 初印象：用费曼视角理解

**Zero-shot 就是"不给任何例子，直接问问题"**。

"zero"指的是"零个示例（examples）"。你直接把任务描述给模型，希望它凭借预训练的知识完成任务。

这是最简单的提示方式，也是很多人默认的使用方式。

### 1.2 Zero-shot 为什么有时候够用

Zero-shot 的原理：现代 LLM 在大量多样化的数据集上训练，不需要示例就能处理各种任务。这背后是预训练中的 In-Context Learning 能力——模型见过了足够多的"问题-解答"模式，能从任务描述本身推断出期望的输出格式。

```python
# Zero-shot 的典型结构

# 最基础的 Zero-shot
zero_shot_basic = """
将以下句子翻译成英文：
深度学习正在改变人工智能领域。
"""

# 更好的 Zero-shot：明确任务、角色和格式
zero_shot_better = """
你是一位专业翻译，精通中英文互译。
请将以下中文句子翻译成地道的英文：

句子：深度学习正在改变人工智能领域。

请只输出翻译结果，不需要解释。
"""

# 结构化 Zero-shot：指定输出格式
zero_shot_structured = """
分析以下产品评论的情感倾向。

评论："这款手机的拍照效果令人失望，续航也很差，但屏幕显示效果确实不错。"

请按以下格式输出：
情感：[正面/负面/混合]
主要优点：[简短描述]
主要缺点：[简短描述]
置信度：[高/中/低]
"""

print("Zero-shot 的三个层次：")
print("1. 基础：直接问")
print("2. 更好：明确角色 + 任务 + 格式要求")
print("3. 结构化：指定精确的输出格式")
```

### 1.3 Zero-shot 的局限

Zero-shot 的失效场景：
- **需要特定格式**：模型不知道你想要 JSON、Markdown 还是别的格式
- **领域特定任务**：需要特定领域的风格或惯例
- **复杂推理**：多步骤问题，模型容易跳步或出错
- **一致性要求高**：不同运行结果格式可能不一致

## 二、Few-shot：给几个例子作为示范

### 2.1 初印象：用费曼视角理解

**Few-shot 就是"给 AI 看几个例子，它就能举一反三"**。

这和人类学习很像：与其用语言解释"什么是好的客服回复"，不如直接给三个例子——AI 会从例子中归纳出模式，然后应用到新问题上。

Few-shot 的原理：LLM 从给出的示例中学习模式，并将类似的推理应用到你的查询中。可以把这看作给 AI 一个"微型训练课程"。

### 2.2 Few-shot 的格式设计

```python
# Few-shot 的标准格式：输入-输出对示例

few_shot_template = """
以下是一些情感分析的例子：

示例 1：
评论：这家餐厅的食物太棒了，服务也很热情！
情感：正面

示例 2：
评论：等了45分钟，食物还是凉的，非常失望。
情感：负面

示例 3：
评论：价格还算合理，但味道一般，不会再来了。
情感：负面

示例 4：
评论：位置方便，食物种类丰富，就是有点贵。
情感：混合

评论：咖啡味道不错，但店员态度很差，让我很不舒服。

"""

# Few-shot 的几个设计原则
few_shot_principles = {
    "示例数量": "通常 2-8 个，太少效果不稳定，太多浪费 token",
    "示例多样性": "覆盖不同情况，避免所有例子都来自同一类别",
    "示例质量": "示例要准确，错误的示例会误导模型",
    "示例顺序": "最后一个例子对模型影响最大（最近邻效应）",
    "格式一致性": "所有示例的输入输出格式必须完全一致",
}
```

## 三、Chain-of-Thought：让模型"先想后说"

### 3.1 初印象：用费曼视角理解

**CoT（思维链）就是让模型把思考过程写出来，而不是直接给答案**。

想象你在做数学题。如果直接问"32×47等于多少"，你可能需要在脑子里算好几步。但如果你把步骤写在纸上：
- 32×7 = 224
- 32×40 = 1280
- 224 + 1280 = 1504

不容易出错，也更容易检查。

CoT 给 LLM 提供了同样的"草稿纸"。

### 3.2 为什么 CoT 有效

Few-shot CoT 包含问题和详细推理链的示例，向模型展示推理步骤应该是什么样的。

从模型机制角度理解：LLM 是自回归模型，每个 token 的生成都以所有之前的 token 为条件。当模型把推理步骤写出来，后续 token 的生成就以这些推理步骤为条件——等于在推理时"在 context 里做了更多计算"。

推理步骤越明确，模型在生成最终答案时能利用的"工作空间"越大，出错率越低。

### 3.3 三种 CoT 变体

**变体一：Few-shot CoT（标准 CoT，2022）**

在示例中包含完整的推理过程：

```python
few_shot_cot = """
```

### 3.4 CoT 的效果数据

对 GPT-3 在 GSM8K 问题集上的错误分析显示：计算错误占 7%，步骤缺失错误占 12%，语义理解错误占 27%。

CoT 主要解决的是"步骤缺失"这类错误——模型直接跳到答案，中间步骤出错。对纯计算错误，CoT 帮助有限（这是 LLM 的固有弱点）。

## 四、Self-Consistency：多数投票提升可靠性

### 4.1 初印象：用费曼视角理解

**Self-Consistency 就是"问同一个问题多次，选最常见的答案"**。

一次回答可能出错，但如果问了 10 次，有 8 次都得到同样的答案，这个答案很可能是对的。

使用自一致性提示，让模型生成多个推理路径，然后选择最一致或出现最频繁的那个。

注意：Self-Consistency 通常配合 CoT 使用——每次生成时，让模型走不同的推理路径，然后选择最多路径得出的答案。

### 4.2 为什么有效

LLM 的生成有随机性（temperature > 0），同一个问题可能走不同的推理路径。有些路径会出错，但正确的答案往往在多条路径中都出现。错误往往是随机的、不一致的，正确答案是系统性的、一致的。

```python
import re
from collections import Counter
from typing import Any

def self_consistency_sample(
    prompt: str,
    model_fn,           # 调用 LLM 的函数
    n_samples: int = 10,
    temperature: float = 0.7,
    extract_answer_fn = None,
) -> dict:
    """
    Self-Consistency 采样：多次生成，取多数答案。
    
    model_fn: 接受 (prompt, temperature) 返回生成文本的函数
    extract_answer_fn: 从生成文本中提取最终答案的函数
    """
    generations = []
    answers     = []

    for i in range(n_samples):
        # 多次采样（temperature > 0 保证多样性）
        output = model_fn(prompt, temperature=temperature)
        generations.append(output)

        # 从输出中提取最终答案
        if extract_answer_fn:
            answer = extract_answer_fn(output)
        else:
            # 默认：提取最后一行作为答案
            answer = output.strip().split('\n')[-1]
        answers.append(answer)

    # 统计各答案出现次数
    answer_counts = Counter(answers)
    majority_answer, majority_count = answer_counts.most_common(1)[0]
    confidence = majority_count / n_samples

    return {
        "majority_answer": majority_answer,
        "confidence":      confidence,
        "answer_counts":   dict(answer_counts),
        "all_generations": generations,
    }


# 实际示例：数学推理 + Self-Consistency
def extract_math_answer(text: str) -> str:
    """从 CoT 推理中提取最终数字答案"""
    # 寻找 "答案是 X" 或 "= X" 这类模式
    patterns = [
        r'答案[是为：:]\s*(\d+(?:\.\d+)?)',
        r'结果[是为：:]\s*(\d+(?:\.\d+)?)',
        r'=\s*(\d+(?:\.\d+)?)\s*$',
        r'(\d+(?:\.\d+)?)\s*$',
    ]
    for pattern in patterns:
        match = re.search(pattern, text.strip())
        if match:
            return match.group(1)
    return text.strip().split()[-1]   # fallback：取最后一个词


# 模拟：同一数学题的 10 次生成结果（示意）
simulated_answers = ["504", "504", "504", "480", "504", "504", "504", "504", "480", "504"]
answer_counts = Counter(simulated_answers)
majority = answer_counts.most_common(1)[0]

print("Self-Consistency 示例（10 次采样）：")
print(f"  各答案分布: {dict(answer_counts)}")
print(f"  多数答案: {majority[0]}  （出现 {majority[1]} 次，置信度 {majority[1]/10:.0%}）")
```

## 五、四种技巧的对比与选择

### 5.1 技术全景对比

| 技巧 | Token 消耗 | 准确率提升 | 适用任务 | 何时首选 |
|------|-----------|-----------|---------|---------|
| Zero-shot | 最低 | 基准 | 简单任务 | 任务简单，格式明确 |
| Few-shot | 中等 | +10-20% | 格式敏感任务 | 需要特定风格/格式 |
| CoT | 中高 | +20-40% | 推理任务 | 多步推理，需要解释 |
| Self-Consistency | 最高（n×） | +10-20% | 精确推理 | 正确率要求高，成本可接受 |

在现代强大的 LLM 上（经过大量推理数据预训练和指令微调），Zero-shot CoT 通常能媲美甚至超过 Few-shot CoT，Few-shot 示例的主要用途变成了强制规范输出格式。这意味着对最新的大模型，直接用 "让我们一步一步思考" + 格式要求，往往就足够了。

### 5.2 决策流程

```python
def choose_prompting_strategy(task_description: str) -> str:
    """根据任务特点推荐提示策略（启发式决策）"""

    needs_reasoning = any(w in task_description for w in
                          ["计算", "推理", "分析", "比较", "为什么", "how", "why", "步骤"])
    needs_format    = any(w in task_description for w in
                          ["JSON", "表格", "格式", "结构", "列表"])
    high_accuracy   = any(w in task_description for w in
                          ["准确", "精确", "关键", "重要", "不能出错"])
    simple_task     = len(task_description) < 50 and not needs_reasoning

    if simple_task and not needs_format:
        return "Zero-shot：任务简单，直接问"
    elif needs_format and not needs_reasoning:
        return "Few-shot：给 2-4 个格式示例，确保输出格式稳定"
    elif needs_reasoning and not high_accuracy:
        return "Zero-shot CoT：加上'让我们一步一步思考'，成本低效果好"
    elif needs_reasoning and high_accuracy:
        return "CoT + Self-Consistency：多次采样投票，最高准确率"
    else:
        return "Few-shot CoT：给推理示例 + 步骤，全面覆盖"


# 测试
tasks = [
    "把这段话翻译成英文",
    "分析这道数学题：小明有3个苹果...",
    "生成一个符合 JSON 格式的用户信息",
    "证明这个逻辑推断是否正确，这个问题很关键",
]
for task in tasks:
    print(f"任务: {task[:30]}...")
    print(f"  推荐: {choose_prompting_strategy(task)}\n")
```

## 六、实战：完整的 Prompt 设计流程

```python
import anthropic

def design_robust_prompt(
    task: str,
    examples: list[dict] = None,
    use_cot: bool = True,
    output_format: str = None,
) -> str:
    """
    设计鲁棒的 Prompt（综合运用四种技巧）。
    """
    parts = []

    # 1. 角色定义（可选但有效）
    parts.append("你是一位专业的分析师，擅长逻辑推理和数据分析。")

    # 2. 任务描述（清晰明确）
    parts.append(f"\n任务：{task}")

    # 3. Few-shot 示例（如果提供）
    if examples:
        parts.append("\n以下是一些参考示例：")
        for i, ex in enumerate(examples, 1):
            parts.append(f"\n示例 {i}：")
            parts.append(f"输入：{ex['input']}")
            if use_cot and 'reasoning' in ex:
                parts.append(f"推理过程：{ex['reasoning']}")
            parts.append(f"输出：{ex['output']}")

    # 4. 输出格式要求
    if output_format:
        parts.append(f"\n请按以下格式输出：\n{output_format}")

    # 5. CoT 触发语（如果需要）
    if use_cot:
        parts.append("\n请一步一步思考，确保推理过程清晰。")

    return "\n".join(parts)
```

## 七、面试常见问题

**Q1：Zero-shot、One-shot 和 Few-shot 的区别是什么？**

区别在于 prompt 中包含的示例（examples）数量。Zero-shot 不包含任何示例，直接描述任务；One-shot 包含一个示例；Few-shot 包含多个示例（通常 2-8 个）。示例越多，模型越容易理解期望的格式和风格，但 token 消耗也越高。对强模型（如 GPT-4、Claude 3），Zero-shot 往往就够用；对弱模型或需要特定格式时，Few-shot 效果更稳定。

**Q2：Chain-of-Thought 为什么能提升推理准确率？**

CoT 通过让模型生成中间推理步骤，有效扩大了模型在生成最终答案前的"工作空间"。LLM 是自回归的，每个 token 的生成以所有前面的 token 为条件——当中间步骤被写出来，它们就成为生成答案时的有效上下文，相当于在推理时做了更多的计算。这对需要多步推理的任务（数学、逻辑推理）效果显著，主要减少的是"步骤跳跃"类错误。

**Q3：Zero-shot CoT（"让我们一步一步思考"）和 Few-shot CoT 哪个更好？**

在现代强大 LLM 上（经过大量推理数据的预训练和指令微调），Zero-shot CoT 通常能媲美甚至超过 Few-shot CoT，Few-shot 示例的主要用途已缩减为规范输出格式。实践中：对强模型（7B+，特别是指令微调版本），Zero-shot CoT 就够；对弱模型或需要特定推理风格时，Few-shot CoT 更稳定。考虑 token 成本和维护成本，先试 Zero-shot CoT，效果不够再加 Few-shot 示例。

**Q4：Self-Consistency 的原理是什么？什么时候使用？**

Self-Consistency 利用了 LLM 生成的随机性：同一问题的多次生成会走不同的推理路径，错误往往是随机的，正确答案是系统性的——通过多次采样后取多数答案（投票），可以过滤掉随机错误，提升整体准确率。适用场景：有明确正确答案的推理任务（数学、逻辑），且正确率要求高、成本预算允许多次 API 调用。不适用于开放式生成（没有"正确"答案）或成本敏感场景。

**Q5：选择 Few-shot 示例时应该注意什么？**

四个关键原则：质量第一（示例必须正确，错误示例会严重误导模型）；多样性（覆盖不同情况，不要所有示例都来自同一类别）；相关性（示例应和当前查询相似，动态检索比静态固定更好）；格式一致性（所有示例的输入输出格式必须完全统一，模型会严格模仿）。此外，最后一个示例对模型影响最大（最近邻效应），把最典型、最规范的示例放在最后。
