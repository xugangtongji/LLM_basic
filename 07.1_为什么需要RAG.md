# 7.1 为什么需要 RAG？从知识截止到实时知识

## 零、从一个真实的困境说起

2023 年，某大型银行的技术团队花了数月时间，基于 GPT-3.5 搭建了一套内部合规问答系统。系统上线后，很快遭遇了两个棘手的问题：

**问题一**：公司的合规手册每季度都会更新，但模型的训练数据早在几个月前就截止了。新的监管政策，模型根本不知道。

**问题二**：模型会自信地给出"合规建议"，但有时这些建议在公司的具体场景下根本不适用——它依据的是普遍情况，而非该银行的具体规定。

这两个问题不是因为模型"不够聪明"，而是因为模型从根本上有一个无法回避的局限：**它只知道训练时学到的知识，不了解你的最新信息、你的私有文档、以及训练截止之后发生的事情。**

RAG（Retrieval-Augmented Generation，检索增强生成）正是为了解决这个局限而生的。

---

## 一、LLM 的知识局限：三堵墙

要理解 RAG 的价值，先要清晰地认识 LLM 的三堵"知识之墙"。

### 第一堵墙：知识截止日期

每个 LLM 都有一个训练数据的截止日期。GPT-4 的训练数据截至 2023 年底，LLaMA-3 截至 2023 年底，你问它"2025 年发生了什么"，它无从得知。

这不是工程上的疏失，而是训练流程的必然：收集数据、清洗、训练、对齐，整个过程需要数月。模型上线的时候，训练数据已经是几个月前的了，部署后每过一天，这个"知识差距"就会扩大一天。

### 第二堵墙：私有知识盲区

LLM 的训练数据来自公开互联网。但世界上最有价值的知识，往往不在公开网络上：

- 公司内部的技术文档和运营手册
- 企业的历史合同和法律文件
- 医院的病历数据库
- 科研团队未发表的实验记录

这些私有知识，模型永远不会知道，除非你告诉它。

### 第三堵墙：幻觉与可信度

当 LLM 被问到它不确定的问题时，它不会说"我不知道"——它会生成一个听起来很像样的答案。这就是**幻觉（Hallucination）**。

对于一般性问题，幻觉只是麻烦。对于医疗、法律、金融等高风险场景，幻觉可能造成严重后果。更关键的是，模型生成的答案没有来源，用户无法验证。

---

## 二、RAG 的核心思路：查完再答

RAG 的解决思路非常直觉化，和人类查资料的方式几乎相同：

**在回答问题之前，先去"查一查"。**

具体流程：
1. 用户提问
2. 系统去知识库里检索最相关的文档片段
3. 把检索到的内容和用户的问题一起送给 LLM
4. LLM 基于这些"参考资料"生成回答，并注明来源

这个思路的美妙之处在于：知识库可以实时更新，模型本身不需要重新训练。更新了公司手册？往知识库里加一份。新出了一条监管政策？上传进去。LLM 下次回答时就会用到最新信息。

```
传统 LLM：
用户问题 → LLM（只有训练知识）→ 回答（可能过时/错误）

RAG：
用户问题 → 检索引擎（从知识库找相关内容）→ LLM（训练知识 + 检索到的内容）→ 有依据的回答
```

RAG 将检索机制与生成式语言模型结合，以增强输出的准确性，解决 LLM 的关键局限。

---

## 三、RAG 的演进历史

RAG 不是一个固定的技术，而是一个持续进化的方向。从 2020 年的原始论文到 2025 年的工业系统，它经历了几次重要的范式转变。

### 起源：2020 年的那篇论文

RAG 这个概念最早由 Facebook AI Research 的 Patrick Lewis 等人在 2020 年的 NeurIPS 论文中提出。他们把一个密集检索器和一个生成模型组合在一起，在知识密集型 NLP 任务（问答、事实核查）上取得了显著效果。

同年，Google 也发布了 REALM（Retrieval-Augmented Language Model Pre-training），探索在预训练阶段就引入检索的方法。

这两项工作奠定了 RAG 的理论基础：**检索和生成可以协同工作**。

### 第一代：朴素 RAG（Naive RAG，2021-2022）

朴素 RAG 代表了检索增强生成的基础实现。这些系统依赖简单的基于关键词的检索技术（如 TF-IDF 和 BM25）从静态数据集中获取文档。

朴素 RAG 的流程极其简单：
1. **索引**：把文档切成固定大小的片段，存入数据库
2. **检索**：用关键词匹配找最相关的片段
3. **生成**：把片段和问题拼在一起，送给 LLM

这个"取-读（Fetch-then-Read）"的模式，就像"Google 搜索 + ChatGPT"的简单组合——没有任何查询优化，没有复杂逻辑，只是"你问了，我找了，我答了"。

**朴素 RAG 的贡献**：提供了关键的概念验证，证明检索和生成的结合是可行的，并为后续的改进奠定了基础。

**朴素 RAG 的局限**：
- 关键词检索无法理解语义（问"苹果"可能检索到水果，而你想要的是 Apple 公司）
- 固定大小的文档切块往往把完整的语义切断
- 检索到的内容不一定和问题真正相关，LLM 会被无关信息干扰
- 朴素 RAG 面临显著挑战，包括"信息迷失在中间"现象（模型会忽略长上下文中间部分的信息）以及检索精度低的问题

### 第二代：高级 RAG（Advanced RAG，2023）

2023 年，随着 LLM 的大规模商业化，RAG 的工程问题被充分暴露，社区开始系统性地修补朴素 RAG 的缺陷。

高级 RAG 在三个环节做了改进：

**改进一：查询优化（Query Enhancement）**
- 重写用户查询（让它更适合检索）
- 查询扩展（一个问题生成多个变体，扩大检索覆盖面）
- HyDE（Hypothetical Document Embedding）：先让模型生成一个"假设的答案文档"，再用这个文档去检索，比直接用问题检索效果更好

**改进二：更好的检索**
- 从关键词检索升级到向量语义检索（理解语义而非只匹配词汇）
- 混合检索（关键词 + 向量，取长补短）
- 引入重排序模型（Reranker）：检索出候选文档后，再用专门的模型对相关性重新排序

**改进三：结果优化**
- 上下文压缩（只保留和问题最相关的部分，减少噪声）
- 引用来源（让 LLM 在回答中标注依据哪个文档）

同年，LangChain 和 LlamaIndex 两个开源框架迅速崛起，把这些改进封装成容易使用的工具，RAG 开始在工业界大规模普及。到 2023 年，RAG 范式支撑了数百篇研究论文和众多商业系统。在业界，RAG 已被迅速集成到搜索引擎、虚拟助手和企业问答应用中。

### 第三代：模块化与图 RAG（2024）

2024 年，RAG 的演进进入了一个新阶段。从朴素 RAG 到高级 RAG，再到模块化/智能体 RAG 的演进，反映了一个根本性的转变：从将检索视为简单的预处理步骤，转变为将其视为推理过程的核心智能组件。

这一年有两个里程碑事件：

**GraphRAG 开源（微软，2024 年中）**：微软将 GraphRAG 开源，这一技术把文档中的实体和关系构建成知识图谱，使得 RAG 能够处理跨文档的复杂推理问题——传统的向量检索只能找"相似文档"，而 GraphRAG 可以回答"这两个实体有什么关系"这类需要多跳推理的问题。

**Agentic RAG 兴起**：RAG 和 Agent 开始深度融合。不再是"检索一次，生成一次"的静态流程，而是 Agent 根据需要动态决定是否检索、检索什么、检索后如何继续推理。模型可以在推理过程中多次检索，自我评估检索质量，遇到不确定的地方主动补充信息。

### 现状（2025 年）

这些方法已经不再只是研究概念，而是生产级 AI 系统中的关键工具。无论你是在构建聊天机器人、虚拟助手，还是企业知识引擎，理解适合你需求的 RAG 架构现在已经是必备技能。

今天，"Long Context vs RAG"成为工程师热议的话题：随着模型上下文窗口扩展到 100K、1M token，很多场景下直接把文档塞进去是否比 RAG 更简单更好？这个争论仍在持续，没有定论，我们会在 7.4 章讨论。

## 四、RAG 在工业界的真实面貌

理解 RAG 的演进之后，有一个重要的"祛魅"认知值得建立：

**RAG 不是万能药，它解决了一些问题，同时引入了新的问题。**

**RAG 擅长解决**：
- 知识更新频繁的场景（政策、法规、产品信息）
- 需要引用可溯源信息的场景（合规、医疗）
- 私有知识库问答（企业内部文档、客服知识库）
- 降低幻觉风险（回答有参考依据）

**RAG 面临的新挑战**：
- 检索质量决定一切：检索到错误的文档，模型会生成"有理有据但完全错误"的回答
- 评估困难：RAG 系统的质量很难量化（怎么知道检索结果是否真的相关？）
- 工程复杂度高：一个生产级 RAG 系统涉及文档解析、切块策略、嵌入模型、向量数据库、检索策略、重排序、生成优化……每个环节都可能成为瓶颈
- "迷失在中间"问题：即使检索到了正确的文档，LLM 也可能忽略文档中间部分的关键信息

工业界的真实经验是：**朴素 RAG 很容易搭建，但要把它做好需要大量的工程投入。** 很多团队发现，搭建一个"能跑通"的 RAG demo 需要一天，但把它的回答质量做到生产可用，可能需要数月。

## 五、RAG 的基础架构总览

在进入后续章节的细节之前，先建立一个整体的架构印象：

```
【离线阶段：建立知识库】

原始文档（PDF/Word/网页/数据库...）
         ↓
    文档解析与清洗
         ↓
    切块（Chunking）：把文档切成合适大小的片段
         ↓
    嵌入（Embedding）：把每个片段转换成向量
         ↓
    存入向量数据库（Milvus/Qdrant/ChromaDB...）


【在线阶段：响应用户查询】

用户提问
    ↓
（可选）查询优化：重写/扩展问题
    ↓
向量检索：在数据库中找最相似的片段
    ↓
（可选）重排序：对候选片段重新评分排序
    ↓
构建上下文：把最相关的片段拼入 Prompt
    ↓
LLM 生成：基于检索内容回答问题
    ↓
返回答案（附带来源引用）
```

每个方框都有很多设计选择，每个选择都会影响最终质量。接下来的三章会依次深入：

- **7.2**：嵌入模型选什么？向量数据库怎么选？怎么快速搭起来？
- **7.3**：检索策略的细节，以及工程师在实战中踩过的坑
- **7.4**：怎么评估 RAG 的质量？Graph RAG 是什么？Long Context 会取代 RAG 吗？

## 六、面试常见问题

**Q1：什么是 RAG？解决了 LLM 的什么问题？**

RAG（检索增强生成）是在 LLM 生成回答之前，先从外部知识库中检索相关信息，将检索到的内容作为上下文一起送给模型的技术。它解决了 LLM 的三大局限：训练数据有截止日期（无法获取新信息）、不了解私有/专有知识、以及幻觉问题（没有来源依据的凭空生成）。

**Q2：RAG 经历了哪些主要的演进阶段？**

三个阶段：朴素 RAG（2020-2022，简单的关键词检索+固定切块，证明了可行性但精度低）；高级 RAG（2023，引入语义向量检索、查询优化、重排序，大幅提升检索质量）；模块化/Agentic RAG（2024+，与 Agent 深度结合，支持多步检索推理，Graph RAG 支持复杂关系推理）。

**Q3：RAG 和直接让 LLM 记忆知识（微调）相比有什么优缺点？**

RAG 的优势：知识可以实时更新（不需要重新训练）；可以处理私有数据；回答有可溯源的依据；成本低（不需要 GPU 集群微调）。RAG 的劣势：检索质量直接影响回答质量；增加了系统复杂度和延迟；检索失败时可能更糟（检索到错误信息比没有信息更危险）。微调更适合需要改变模型行为风格或让模型学习新的推理模式的场景，RAG 更适合知识更新频繁、需要引用来源的场景。

**Q4：什么是"迷失在中间"问题？**

当给 LLM 一个很长的上下文时，它对开头和结尾的信息处理得更好，但会忽略中间部分的信息。在 RAG 场景中，如果检索到的多个文档片段中，关键信息恰好在中间位置，LLM 可能忽略它。缓解方法：将最相关的内容放在上下文的开头或结尾；减少传入的文档片段数量，只保留最相关的几个。
