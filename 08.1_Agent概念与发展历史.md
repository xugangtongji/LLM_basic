## 零、开篇：从"会说话"到"会做事"

2022 年底 ChatGPT 爆红，让数亿人第一次感受到了"会说话的 AI"。它能聊天、写文章、解题——但有一个本质局限：**它只能说，不能做**。

你问它"帮我发一封邮件给张三"，它会给你写出邮件的文字，但它无法真正点击发送。你让它"查一下现在的比特币价格"，它只能告诉你训练截止日之前的数据。你让它"帮我订明天去北京的机票"，它只能给你订票的步骤，不能替你完成操作。

**Agent（智能体）** 就是为了弥合"说"与"做"之间的这道鸿沟而出现的。

一个 Agent 系统，简单说就是：让 LLM 不只生成文字，还能真正地感知环境、做出决策、执行行动，并根据结果调整下一步——形成一个真正的"计划-行动-观察-再计划"的闭环。

传统 LLM 只是在回应人类，而 Agent AI 是在替代整个工作流——它规划、行动并持续改进。

## 一、Agent 的本质：四个核心要素

理解 Agent，先理解它由什么构成。Lilian Weng（OpenAI 研究员）在她那篇被无数人引用的博客里，把 LLM Agent 系统分解为四个核心要素：

### 要素一：规划（Planning）

Agent 需要把一个复杂的目标拆解成一系列可执行的子任务。

一个普通 LLM 被问"帮我准备一份市场调研报告"，会直接开始生成报告。一个 Agent 会先规划："第一步，确定调研范围；第二步，搜索行业数据；第三步，分析竞争对手；第四步，整理成报告结构……"然后依次执行每个步骤。

这个规划能力是 Agent 能处理复杂任务的基础。

### 要素二：工具使用（Tool Use）

Agent 可以整合外部工具、API 和专业模块，执行超出其原生语言能力范围的操作。这包括代码解释器（执行代码、生成图表、完成复杂的程序性任务）、搜索引擎（实时数据检索）、数据库和向量存储（知识检索），以及协作和生产力工具（如 GitHub、Trello、Google Sheets 等）。

工具是 Agent 的"手"。没有工具，Agent 再聪明也只能纸上谈兵。

### 要素三：记忆（Memory）

单次对话的 LLM 没有记忆——每次对话都是从零开始。Agent 需要记忆来维持长任务的上下文连贯性：

- **短期记忆**：当前任务的上下文（放在 Context Window 里）
- **长期记忆**：历史对话、用户偏好、任务结果（存在外部数据库，用 RAG 检索）

### 要素四：行动（Action）

把规划变成真实的操作——调用 API、执行代码、操作文件、点击网页、向另一个 Agent 发出指令。

## 二、演进历史：从 ReAct 到 Agentic AI

### 2.1 前 LLM 时代的 Agent（1950s-2020）

Agent 的概念不是 ChatGPT 之后才有的。

**1956-1959 年**，Logic Theorist 和 General Problem Solver 是最早的"代理程序"——它们能在给定规则下推理和求解问题，但只能处理极其有限的符号世界。

**1966 年**，ELIZA 出现——世界第一个对话"机器人"，通过模式匹配和简单规则模拟心理咨询师对话。它让人第一次感受到"和机器说话"，但其实什么都不理解。

**1970-80 年代**，专家系统（如 MYCIN）兴盛——用大量的 if-then 规则编码专家知识，在医疗诊断等领域取得了实际应用。但这类 Agent 极度脆弱：规则以外的情况完全无法处理。

**2010 年代**，强化学习 Agent 崛起——AlphaGo（2016）、OpenAI Five（Dota，2018）证明了 RL Agent 能在特定游戏环境中超越人类。但这些 Agent 的"智能"被硬约束在游戏规则内，无法泛化。

这一时期所有 Agent 的共同局限：**能力被硬编码，无法理解自然语言，无法灵活应对未预见的情况**。

### 2.2 转折点：大语言模型改变了 Agent 的可能性（2020-2022）

GPT-3（2020）的出现是一个分水岭。当人们发现这个模型能遵循自然语言指令完成各种任务时，一个新的想法开始浮现：**如果让这个模型不只是生成文字，而是可以真正地与环境交互，会怎么样？**

**2022 年**，两篇论文奠定了现代 LLM Agent 的理论基础：

**ReAct**（Yao et al., Google，2022）：ReAct 通过将思维链推理与外部工具使用相结合，增强了 LLM 处理复杂任务和决策的能力。它可以被理解为一种将 LLM 的推理和行动能力整合在一起的机器学习范式。ReAct 首次证明了"让 LLM 边推理边行动"的模式是可行的，提供了 Thought-Action-Observation 的基础框架——而不是让 LLM 一次性生成所有内容。

**Toolformer**（Meta，2023 年初）：训练模型自己学习何时以及如何调用工具，把工具使用能力内化到模型权重里，而不是只靠 Prompt 引导。

### 2.3 爆炸式增长：2023 年的 Agent 热潮

2023 年，在 ChatGPT 爆红之后的几个月里，Agent 相关的实验和框架像雨后春笋一样涌现：

**AutoGPT（2023 年 3 月）**：一个 GitHub 项目，基于 GPT-4 构建了能自主分解任务、搜索网络、写代码的 Agent，在 GitHub 上在数天内获得了几万个 Star，成为技术圈的热门话题。AutoGPT 项目在 2023 年病毒式传播，它在 ReAct 思想基础上创建了一个自主的 GPT-4 Agent，能迭代地规划和执行子任务以实现给定目标。但用过它的人很快发现：大多数时候它会陷入循环、走错方向，或者花了大量 API 费用什么都没完成。

**BabyAGI（2023 年 4 月）**：一个更简洁的任务驱动型 Agent，展示了"生成任务列表 → 执行 → 评估 → 生成新任务"的循环模式。

**LangChain 和 LlamaIndex**：两个框架的崛起，把 Agent 的构建从"自己写所有代码"变成了"组合积木"。LangChain 在 2023 年成为 GitHub 上增长最快的项目之一。

**Generative Agents（2023 年）**：斯坦福和 Google 联合发布，25 个 LLM Agent 在一个模拟小镇里生活——它们有记忆、有个性、会社交，展示了 Multi-Agent 模拟人类行为的惊人潜力。

这一年的热潮也揭示了一个现实：**Agent 的概念令人兴奋，但工程落地极其困难**。早期的 GPT-4 能"勉强"规划，但它会幻觉工具调用、在多步目标上迷失方向，需要大量保护机制。

### 2.4 成熟：从研究概念到工业落地（2024-2025）

2023 年早期的 Agent 尝试很脆弱。2023 年的 GPT-4 能"大致"规划，但它会幻觉工具调用、在多步目标上迷失，需要大量保护机制。2025-2026 年具有原生思维链、改进的函数调用和更好指令跟随能力的推理模型改变了这个局面。Agent 现在能够可靠地执行 10-15 步的工作流而不会出轨。

几个标志性事件：

**OpenAI Function Calling（2023 年 6 月）**：OpenAI 发布了结构化的函数调用能力，让模型能够以标准化的 JSON 格式调用开发者定义的工具，而不是用自然语言描述行动。这是从"ReAct 风格的文字描述"到"结构化工具调用"的重要转变。

**Anthropic Claude 工具使用（2024 年）**：Claude 原生支持工具使用（Tool Use），Anthropic 同年还推出了计算机使用能力（Computer Use），让 Claude 能直接操作桌面 GUI。

**OpenAI Deep Research（2025 年）**：OpenAI 发布的深度研究功能，能在几十分钟内自主完成原本需要人类研究员几个小时的调研工作——搜索、阅读、整理、撰写报告——是 Agentic AI 工业落地的重要里程碑。

**Manus（2025 年）**：由中国团队开发的通用 Agent，在多个基准测试上取得了领先成绩，展示了 Agent 处理复杂现实任务的能力，引发广泛关注。

## 三、主流 Agent 框架对比

随着 Agent 工程化，生态系统迅速成熟。以下是目前主要框架的特点和定位：

### LangChain

**定位**：最早、最全面的 LLM 应用开发框架，功能极其丰富。

**优势**：生态最成熟，文档最多，几乎所有工具都有 LangChain 集成，学习资源丰富。

**劣势**：随着功能堆积，API 设计变得复杂，早期版本的频繁重构让很多代码很快过时，有过度工程化的倾向。

**LangGraph**（LangChain 的 Agent 扩展）：用图结构定义 Agent 的工作流，更适合构建有状态、多步骤的复杂 Agent。是目前 LangChain 体系中构建生产级 Agent 的推荐方式。

### LlamaIndex

**定位**：以 RAG 为核心，延伸到 Agent 场景。

**优势**：数据连接和检索能力最强，特别适合需要深度整合私有知识库的 Agent。

**劣势**：相比 LangChain，通用 Agent 能力稍弱，更专注于知识检索场景。

### AutoGen（微软）

**定位**：专门为 Multi-Agent 协作设计，擅长多个 Agent 互相对话和协作完成任务。

**优势**：多 Agent 场景下的最强框架，内置代码执行沙箱，特别适合数据分析和编程任务。

**劣势**：配置相对复杂，对于单 Agent 场景过于重量级。

### CrewAI

**定位**：让多个 Agent 分工协作，像一个"AI 团队"。每个 Agent 有明确的角色、目标和工具。

**优势**：抽象层次高，设计直观（你定义角色，框架处理协调），上手快。

**劣势**：灵活性相对低，复杂的自定义逻辑实现起来较麻烦。

### Dify

**定位**：面向非技术用户的可视化 Agent 构建平台，开源且支持私有化部署。

**优势**：低代码，有图形界面，不需要写大量代码就能搭建 Agent 工作流，适合企业内部快速落地。

**劣势**：灵活性低，高度定制化的场景需要绕过平台限制。

### 选型建议

```
单 Agent，需要调用各种工具和 API？
    → LangChain + LangGraph（生态最全，资料最多）

单 Agent，核心是私有知识库问答？
    → LlamaIndex（RAG 能力最强）

多 Agent 协作，需要代码执行？
    → AutoGen（多 Agent 场景最强）

多 Agent 协作，团队不太懂代码？
    → CrewAI（配置直观，上手快）

快速在企业内部部署，不想写太多代码？
    → Dify（可视化，私有化部署友好）
```

## 四、Agent 的现实局限

理解 Agent 的发展，也要清醒地认识它的局限——这是工业落地中最重要的认知。

### 局限一：可靠性问题

Agent 在简单任务上表现良好，但随着任务步骤增加，每一步都有出错的概率，错误会累积。一个需要 20 步的任务，如果每步成功率是 90%，整体成功率只有 0.9^20 ≈ 12%。

2024-2025 年的快速进步表明，Agent 正在变得更加自主、更有意识（通过记忆），以及在复杂决策上更有能力。但随之而来的是设计上的复杂性增加，以及对谨慎评估的迫切需求（确保可靠性和安全性）。

### 局限二：成本与延迟

Agent 执行一个复杂任务可能需要数十次 LLM 调用。每次调用都有费用和延迟。对于时效性要求高、成本敏感的场景，Agent 的成本可能难以接受。

### 局限三：不可预测性

Agent 的行为路径不是固定的，相同的输入可能产生不同的执行路径。这对需要审计和合规的企业场景来说是一个挑战——你不总能解释"为什么 Agent 这么做"。

### 局限四：安全边界

一个有权访问文件系统、发送邮件、执行代码的 Agent，一旦被恶意提示注入或做出错误决策，后果可能非常严重。Agent 的权限越大，安全设计的要求就越高。

## 五、面试常见问题

**Q1：什么是 LLM Agent？它和普通 LLM 有什么区别？**

普通 LLM 是一个"问答机"——给定输入，生成输出，没有状态，没有行动能力。LLM Agent 是以 LLM 为核心控制器的自主系统，具备四个要素：规划（把大目标拆成子任务）、工具使用（调用外部 API 和工具）、记忆（维护任务上下文和历史信息）、行动（真实地操作环境）。本质区别是：LLM 说，Agent 做。

**Q2：ReAct 框架是什么？它解决了什么问题？**

ReAct（Reasoning and Acting）是 2022 年 Google 提出的框架，核心思想是让 LLM 交替进行推理（Thought）和行动（Action），并观察行动结果（Observation），然后继续推理。它解决了两个问题：纯推理（CoT）没有行动能力；纯行动（直接调用工具）缺乏推理引导。ReAct 把两者结合，让 LLM 能在推理过程中随时决定"我需要查什么"，并把查询结果纳入后续推理。

**Q3：2023 年 Agent 热潮之后，为什么到 2025 年才开始真正落地？**

主要原因是基础模型能力的提升。2023 年的模型容易在工具调用上产生幻觉（调用不存在的 API、参数格式错误），在多步任务上迷失方向，可靠性不足以支撑生产环境。2025 年推理模型（o1/o3/DeepSeek-R1）和改进的 Function Calling 能力，大幅提升了 Agent 在复杂多步任务上的可靠性，才让 10-15 步的工作流在生产环境中变得可行。

**Q4：如何选择 Agent 框架？**

核心看两个维度：是否需要多 Agent 协作（是→AutoGen 或 CrewAI，否→LangChain 或 LlamaIndex）；团队的技术深度（技术团队→LangChain/LangGraph，技术较弱→CrewAI/Dify）。另外，如果任务核心是知识库检索，LlamaIndex 的 RAG 能力更强；如果需要在企业内部快速可视化配置，Dify 是好选择。
