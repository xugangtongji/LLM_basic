## 零、开篇：检索质量决定 RAG 质量的上限

有一个残酷的现实：**RAG 系统的质量上限，由检索质量而不是 LLM 质量决定**。

检索到的文档是对的，LLM 可能给出好答案；检索到的文档是错的，LLM 再厉害也无法凭空生成正确内容——它只会更自信地给出错误答案，因为它有了"来源依据"。

朴素 RAG 的失败率很高。一项分析发现，朴素 RAG 流水线在检索阶段的失败率约为 40%。这不是 LLM 的问题，而是检索问题。

本章系统介绍三种检索策略，以及工业落地中反复出现的工程陷阱。

---

## 一、三种检索策略

### 1.1 稀疏检索：BM25

**BM25（Best Matching 25）是关键词检索的工业标准**，是传统搜索引擎（Elasticsearch、Solr）的核心算法。

它的核心思想：根据词频和文档频率计算相关性分数。一个词在文档里出现越多（词频高），这个词在所有文档里出现越少（逆文档频率高），这个文档的相关性就越高。

**BM25 的优势**：
- 速度极快，毫秒级响应，无需 GPU
- 对精确词汇匹配有天然优势：型号、代码、专有名词、数字
- 完全可解释：分数是数学公式算出来的，没有黑盒

**BM25 的致命弱点**：零语义理解。

如果用户问"如何优化慢查询"，文档里写的是"数据库性能调优技术"，BM25 会认为完全不相关——因为两段话没有共同词汇。这就是所谓的**词汇鸿沟（Vocabulary Gap）**。

BM25 能在毫秒内响应数百万文档，无需 GPU。但它对语义完全无知。如果用户问"如何修复慢查询"，而文档写的是"数据库性能优化技术"，BM25 找不到任何匹配，因为两者没有重叠词汇。

### 1.2 稠密检索：向量相似度

这就是上一章介绍的向量检索：把查询和文档都转成向量，用余弦相似度找最近的邻居。

**向量检索的优势**：语义理解。"慢查询" 和 "数据库性能优化" 在向量空间里距离很近，可以正确检索。

**向量检索的致命弱点**：对精确词汇不敏感。

如果用户问 "CVE-2024-12345 漏洞的修复方案"，这是一个精确的漏洞编号，嵌入模型可能把它编码成和"安全漏洞"相近的向量，而不是和那个具体编号的文档精确匹配。

在技术查询中，BM25 能捕捉到向量搜索遗漏的精确匹配——特别是对于技术性查询。没有在检索栈中加入 BM25 的 RAG 流水线会有系统性的盲点。

### 1.3 混合检索：两者的最佳组合

**混合检索同时运行 BM25 和向量检索，然后用融合算法合并结果**。

纯向量搜索会遗漏精确关键词匹配，纯 BM25 会遗漏语义相似性。混合检索两者兼顾——这是对朴素 RAG 流水线能做的最大的单一质量提升。

融合算法最常用的是 **RRF（Reciprocal Rank Fusion，倒数排名融合）**：

```
RRF 分数 = Σ 1 / (k + rank_i)

其中 rank_i 是文档在第 i 个检索系统中的排名，k 通常取 60
```

直觉很简单：如果一个文档在 BM25 的结果里排第 1，向量检索里排第 3，它的综合得分会比只在一个系统里靠前的文档更高。以 k=60 作为 RRF 的默认参数是零配置的好起点。

**混合检索对比**：

| 维度 | BM25（稀疏）| 向量（稠密）| 混合 |
|------|-----------|-----------|------|
| 精确词汇匹配 | ✅ 强 | ❌ 弱 | ✅ |
| 语义理解 | ❌ 无 | ✅ 强 | ✅ |
| 速度 | 极快（无 GPU）| 快 | 略慢（两次检索）|
| 工程复杂度 | 低 | 低 | 中等 |
| 推荐场景 | 精确查询为主 | 语义查询为主 | **大多数生产场景** |

混合检索和重排序的两阶段流水线实验表明，Recall@5 达到 0.816，MRR@3 达到 0.605，大幅优于所有单阶段方法。

## 二、重排序（Reranker）：精度的最后一公里

### 2.1 为什么需要重排序

检索阶段（无论 BM25 还是向量）都是**双编码器（Bi-encoder）**模式：查询和文档分别独立编码，然后比较向量相似度。这很快，但不够精确——两个向量是在没有"看到对方内容"的情况下计算的。

重排序使用**交叉编码器（Cross-encoder）**：把查询和每个候选文档同时输入模型，模型在看到完整的"查询 + 文档"组合后再打分。这样的精度远高于双编码器，但速度慢很多（因为无法预先计算文档向量）。

**这就是两阶段架构的精髓**：
- 第一阶段（双编码器）：从百万文档中快速筛出 50-100 个候选（宽网撒得快）
- 第二阶段（交叉编码器 Reranker）：对这 50-100 个候选精细打分（精细筛选）

重排序是任何 RAG 系统 ROI 最高的改进。在初步检索（前 20 或前 50）之后，交叉编码器模型用完整注意力对每个文档相对于原始查询重新打分——捕捉到了嵌入相似度遗漏的相关性。

有一个关键的顺序原则：召回必须先于精度。重排序器只能对已经检索到的内容重新排序。如果稠密检索器因为缺少精确关键词而遗漏了相关文档，再多的重排序也无法找回它。

### 2.2 主流重排序模型

| 模型 | 类型 | 特点 |
|------|------|------|
| cross-encoder/ms-marco-MiniLM-L-6-v2 | 开源 | 轻量快速，英文效果好，入门首选 |
| BAAI/bge-reranker-v2-m3 | 开源 | **中文最强**，多语言支持 |
| Cohere Rerank API | 商业 API | 效果好，使用简单，按量付费 |
| ColBERT | 开源 | 延迟交互，兼顾精度和速度 |

选 Reranker 的逻辑和选嵌入模型类似：中文场景用 BGE Reranker，英文原型用 Cohere API 或 MiniLM，生产自托管考虑 BGE 或 ColBERT。

## 三、切块策略：被严重低估的关键环节

### 3.1 为什么切块策略至关重要

向量搜索质量对文档嵌入前的切割方式极其敏感。这是 RAG 流水线频繁出现无声失败的地方。

"无声失败"是最危险的：系统运行正常，没有报错，但给出的答案质量很差，而且你很难知道原因是切块出了问题。

把一个 5000 token 的文档作为一个整体向量，会把所有信息平均混合，失去检索的特异性。把文档切得太碎，每块只有 50 个 token，又会丢失语义上下文。

### 3.2 主要切块策略对比

**固定大小切块（Fixed-size Chunking）**

最简单：每 N 个 token 切一刀，相邻块有 M 个 token 的重叠。
- 优点：实现简单，适合均匀格式的文档
- 缺点：可能在句子中间、段落中间切断，破坏语义完整性

**语义切块（Semantic Chunking）**

根据语义相似度决定切割点：相邻句子语义相近就放在一块，差异变大说明话题切换了，在这里切割。
- 优点：块的语义边界更自然
- 缺点：实现更复杂，每块大小不固定，难以预测

**结构感知切块（Structure-aware Chunking）**

利用文档本身的结构（Markdown 标题、HTML 段落、PDF 章节）作为切割边界，保留标题信息。
- 适合：有清晰结构的技术文档、API 文档、政策手册
- 实践：在每个块的元数据里保存其所在的标题路径（如"产品手册 > 第三章 > 3.2 退款政策"），检索后 LLM 知道这段文字来自哪个章节

**父子文档检索（Parent-Child Chunking）**

把文档切成两个粒度：
- **小块（子）**：用于检索（512 token），精确匹配
- **大块（父）**：包含完整上下文（1500-2000 token），送给 LLM

流程：用小块检索，找到后取出其对应的大块给 LLM。既保证了检索精度，又保证了 LLM 得到足够的上下文。

**何时用哪种策略**：

```
文档类型是均匀文本（小说、报告）？      → 固定大小切块，简单起步
文档有清晰的章节/标题结构？             → 结构感知切块 + 标题元数据
上下文完整性很重要（法律/合同文档）？   → 父子文档检索
已有固定切块，回答缺少上下文？          → 升级到父子文档
```

混合检索效果最好的前提是：块是感知标题的，不能太大（会稀释 BM25 和稠密向量的信号），不能太小（会让重排序器失去上下文）。

## 四、进阶检索技术

### 4.1 HyDE：假设文档嵌入

**HyDE（Hypothetical Document Embedding）**是一个巧妙的思路：

用户的问题（"公司退款政策是什么"）和文档的表述风格（"退款申请应在购买后 30 天内提交……"）往往不匹配——一个是问句，一个是陈述句，直接用问句检索效果不好。

HyDE 的做法：先让 LLM 根据问题生成一个"假设的答案文档"，然后用这个假设文档去检索真实文档。假设文档和真实文档的风格更接近，检索效果更好。

HyDE 对一般问答任务有明显效果，但研究显示对精确数字查询帮助有限。不要把它当万能药，先测试再部署。

### 4.2 上下文检索（Contextual Retrieval）

Anthropic 在 2024 年提出的技术：在给每个块建立向量之前，先用 LLM 为每个块生成一段上下文说明（这个块是关于什么的，在文档哪个位置），把这段说明拼在块的前面再嵌入。

这样每个块的向量不只包含块本身的语义，还包含它在文档中的位置信息。研究表明，上下文检索通过文档级别的丰富化，提供了一致的性能提升。

代价：索引阶段需要调用 LLM 为每个块生成上下文（成本增加，时间增加）。

### 4.3 查询重写与多查询

**查询重写**：用 LLM 把用户的原始问题（可能模糊、口语化）重写成更适合检索的形式。

**多查询检索**：对同一个问题，自动生成 3-5 个不同表述，每种表述各自检索，合并去重。覆盖了不同的表述方式，提高召回率。

这两个技巧实现简单，效果明显，适合优先考虑。

## 五、工业落地的真实陷阱

这里汇总的是工程师们付出真实代价才换来的教训，按严重程度排序。

### 陷阱一：没有评估体系，在黑暗中优化

**症状**：调了很多参数，不知道有没有变好；或者以为优化了，实际上某些查询退步了。

**根本原因**：RAG 系统太复杂，直觉判断不可靠。一个参数改动可能让某些查询变好，另一些变差。

**解决方案**：在开始优化之前，先建立一个评估集——50 到 200 个有已知正确答案的查询，每次改动都在这个集合上评估。追踪每个阶段的指标：BM25 独立召回率 → 稠密检索独立召回率 → 混合融合后 → 加重排序后。只有能测量，才能改进。

如果不能衡量检索质量，你将不断引入回归。最基本的离线指标是 Recall@K（前 K 个结果中是否包含了答案所在的块）。从日志中抽取 50-200 个有已知参考答案的查询作为测试集，每次检索改动都在其上运行。

### 陷阱二：PDF 解析质量差，垃圾进垃圾出

**症状**：文档明明包含了答案，但 RAG 就是找不到。

**根本原因**：PDF 是一个展示格式，不是语义格式。扫描件、双栏布局、表格、图片中的文字，普通 PDF 解析器处理得很差——OCR 识别错误、段落顺序混乱、表格变成乱码。

**解决方案**：先抽样检查解析结果（打印出切块后的文本，人工看）。对于重要文档，考虑用专门的文档解析服务（如 LlamaParse、Azure Document Intelligence、Unstructured.io）。

**经验**：很多团队做 RAG 调了几个月参数，最后发现根本问题是文档解析质量太差，切块后的文本有大量乱码。

### 陷阱三：把所有文档堆在一个索引里

**症状**：用户问产品 A 的问题，检索出来的是产品 B 的文档。

**根本原因**：不同类型、不同权威级别、不同时效性的文档混在一起，检索时互相干扰。

**解决方案**：按文档类型/部门/产品线建立独立的索引，根据用户的查询意图路由到对应的索引。这需要一定的架构设计，但对准确率的提升很明显。

### 陷阱四：上下文窗口填满无关内容

**症状**：检索到了很多文档，但 LLM 的回答质量很差，有时甚至不如没有 RAG。

**根本原因**：送给 LLM 的上下文充斥了相关度低的文档，无关信息干扰了模型的判断，这比没有上下文还要糟糕。

**解决方案**：宁可少不要多。只送最相关的 3-5 个块，而不是 20 个块。加重排序，过滤掉相关度低于阈值的块。检查 Reranker 的分数分布，如果前几名分数都低，可能说明知识库里根本没有答案，这时直接回答"我不知道"比用低质量文档生成一个似是而非的答案要好得多。

### 陷阱五：索引更新策略不合理

**症状**：文档更新了，但 RAG 仍然给出旧答案。

**根本原因**：没有建立文档更新与索引更新的同步机制。文档有版本，但向量数据库里的条目没有相应删除或更新。

**解决方案**：每个块的元数据里存储文档版本、更新时间、来源 URL。文档更新时，删除旧版本的所有块（按文档 ID 过滤），重新索引新版本。建立监控，发现索引和文档来源之间的不一致。

## 六、一个完整的生产级检索流水线

把本章所有内容整合起来，一个成熟的 RAG 检索流水线应该是这样的：

```
用户输入原始查询
        ↓
【查询预处理】
  查询重写/扩展（可选，对模糊查询帮助大）
        ↓
【并行检索】（混合检索）
  BM25 稀疏检索 → Top 100 候选
  向量稠密检索 → Top 100 候选
        ↓
【融合】
  RRF 算法合并两个候选列表 → Top 50 候选
        ↓
【重排序】
  Cross-encoder Reranker 精细打分 → Top 10 精选
        ↓
【过滤】
  相关度分数低于阈值的块丢弃
  如果没有高质量结果 → 回复"我不知道"
        ↓
【上下文构建】
  把 Top 3-5 个块拼入 Prompt（含来源信息）
        ↓
LLM 生成带引用的回答
        ↓
【评估指标更新】
  记录检索分数、用户反馈，持续优化
```

这个设计有意保持"无聊"。生产环境中无聊是好事。在评估关卡后面做变更，测量尾部延迟。

## 七、面试常见问题

**Q1：BM25 和向量检索各有什么优缺点？为什么混合检索更好？**

BM25 优点：速度极快，对精确词汇（型号、代码、专有名词）匹配精准；缺点：完全无法理解语义，"慢查询"和"数据库性能优化"在 BM25 眼里没有关联。向量检索优点：语义理解强，能处理同义词和语义相关的表述；缺点：对精确关键词不敏感，可能遗漏精确匹配。混合检索用 RRF 算法融合两者结果，覆盖了两种检索模式的优势，是大多数生产场景的最佳选择。

**Q2：Reranker 的作用是什么？它为什么比直接用更高精度的嵌入模型更有效？**

Reranker 使用交叉编码器（Cross-encoder），同时看到查询和文档的完整内容再打分，注意力机制可以捕捉到细粒度的交互信息。直接用嵌入模型（双编码器）是查询和文档独立编码再比较向量，无法看到"查询遇到这个文档"的具体交互。Reranker 精度更高，但因为无法预计算，只适合对少量候选（50-100 个）精细打分，不适合从百万文档中做初步检索。两阶段是最优组合：双编码器初筛 + 交叉编码器精排。

**Q3：如何评估 RAG 系统的检索质量？**

建立一个有已知答案的测试集（50-200 个查询，每个查询有对应的正确文档块 ID）。核心指标是 Recall@K：在前 K 个检索结果中，包含正确答案的查询比例。分阶段追踪：BM25 独立召回率、向量独立召回率、混合后召回率、加 Reranker 后召回率。只有知道每个阶段的准确率，才能判断瓶颈在哪里，优化才有方向。

**Q4：父子文档检索解决了什么问题？**

标准切块有一个两难：小块检索精准但上下文不足，大块上下文丰富但检索模糊。父子文档检索用两个粒度的块解决了这个问题：用小块（200-500 token）建立向量索引，保证检索精度；检索命中后，找到小块对应的父大块（1500-2000 token），把大块送给 LLM，保证上下文完整。检索精准，生成有足够上下文，两全其美。
