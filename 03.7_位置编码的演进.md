## 零、开篇：没有位置，注意力是盲目的

3.3 章提到，Self-Attention 本身是"位置无关"的——如果把句子里所有词的顺序打乱，注意力机制得到的结果完全相同（只是对应关系变了）。

"猫吃了鱼"和"鱼吃了猫"在纯注意力机制眼里没有区别。

这显然不行。语言的顺序至关重要。

解决方案是给每个位置注入一个"坐标"——**位置编码（Positional Encoding）**。但怎么注入、注入什么，在过去几年里经历了几次重大演进：

```
第一代：正弦/余弦绝对位置编码（原始 Transformer，2017）
    ↓ 问题：处理比训练时更长的序列会"迷路"
第二代：可学习绝对位置编码（GPT-2 等，2019）
    ↓ 问题：同样无法外推，而且消耗额外参数
第三代：相对位置编码（ALiBi 等，2022）
    ↓ 问题：修改了注意力计算方式，与 KV Cache 不兼容
第四代：RoPE 旋转位置编码（RoFormer，2021；LLaMA 起全面普及）
    ✓ 当前主流，兼顾效果和工程友好性
```

## 一、第一代：正弦/余弦绝对位置编码

### 1.1 初印象：用费曼视角理解

**原始 Transformer 的位置编码思路：给每个位置分配一个独特的"指纹"**。

就像给每个座位贴一个不同颜色和形状组合的贴纸，每个座位的贴纸组合是唯一的，模型通过贴纸就能区分"这个词坐在第几号座位"。

这个"指纹"是用正弦和余弦函数生成的，不需要学习，纯数学计算。

### 1.2 数学形式

对位置 $pos$ 和维度 $i$：

$$PE(pos, 2i)   = \sin\left(\frac{pos}{10000^{2i/d}}\right)$$
$$PE(pos, 2i+1) = \cos\left(\frac{pos}{10000^{2i/d}}\right)$$

- 偶数维度用 $\sin$，奇数维度用 $\cos$
- 每个维度的频率不同：低维度变化快（高频），高维度变化慢（低频）
- 把这个编码直接加到词向量上：$\text{输入} = \text{词向量} + \text{位置编码}$

```python
import torch
import math
import matplotlib.pyplot as plt

def sinusoidal_encoding(max_len: int, d_model: int) -> torch.Tensor:
    """生成原始 Transformer 的正弦位置编码"""
    pe = torch.zeros(max_len, d_model)
    position = torch.arange(0, max_len).unsqueeze(1).float()   # (max_len, 1)

    # 每个维度对的频率（指数间隔）
    div_term = torch.exp(
        torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model)
    )   # (d_model/2,)

    pe[:, 0::2] = torch.sin(position * div_term)   # 偶数维度：sin
    pe[:, 1::2] = torch.cos(position * div_term)   # 奇数维度：cos
    return pe   # (max_len, d_model)

# 可视化：每个位置的编码"指纹"
pe = sinusoidal_encoding(max_len=50, d_model=64)

plt.figure(figsize=(12, 4))
plt.subplot(1, 2, 1)
plt.imshow(pe.numpy(), cmap="RdBu", aspect="auto")
plt.colorbar()
plt.xlabel("维度 (d_model=64)")
plt.ylabel("位置 (0-49)")
plt.title("正弦位置编码热力图")

plt.subplot(1, 2, 2)
# 展示前几个维度随位置变化的曲线
for dim in [0, 1, 4, 8, 16]:
    plt.plot(pe[:, dim].numpy(), label=f"dim={dim}")
plt.xlabel("位置"); plt.ylabel("编码值")
plt.title("不同维度的位置编码曲线")
plt.legend(); plt.grid(alpha=0.3)
plt.tight_layout()
plt.savefig("sinusoidal_pe.png", dpi=150)
```

### 1.3 正弦编码的优点和缺陷

**优点**：
- 不需要学习，无额外参数
- 位置之间的距离关系在编码里有规律地体现
- 理论上可以处理任意长度（公式可以推广到任意位置）

**缺陷（关键问题）**：虽然公式上能算出任意位置的编码，但模型**训练时从未见过超过训练长度的位置**。就像你背了 1 到 100 的乘法表，突然要算 157 × 83，公式还是那个公式，但你的"直觉"来自于 1-100 的经验，面对没见过的位置会表现糟糕。

这就是"位置外推（extrapolation）"的挑战。

## 二、第二代：可学习绝对位置编码

### 2.1 思路与实现

GPT-2、BERT 等模型放弃了固定的正弦编码，改用**直接学习每个位置的编码向量**：

```python
import torch.nn as nn

class LearnablePositionalEncoding(nn.Module):
    """可学习的绝对位置编码（GPT-2 / BERT 风格）"""
    def __init__(self, max_seq_len: int, d_model: int):
        super().__init__()
        # 每个位置一个可学习的向量
        self.pos_embed = nn.Embedding(max_seq_len, d_model)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, L, D = x.shape
        positions = torch.arange(L, device=x.device)
        return x + self.pos_embed(positions)

# GPT-2 用的就是这种，max_seq_len=1024
pos_enc = LearnablePositionalEncoding(max_seq_len=1024, d_model=768)
```

### 2.2 问题：硬性长度上限

可学习编码有一个严重问题：它只学了 `max_seq_len` 个位置的编码。如果推理时序列超过这个长度，第 1025 个位置根本没有对应的编码——模型完全不知道怎么处理。

这就是早期模型（GPT-2：1024 tokens，BERT：512 tokens）上下文长度受限的根本原因之一。

## 三、第三代：ALiBi——不编码位置，直接惩罚距离

### 3.1 初印象：用费曼视角理解

**ALiBi 的思路出人意料：干脆不给词向量加位置信息，而是直接在注意力分数上"惩罚距离"**。

逻辑是：距离近的词，注意力分数保持不变；距离远的词，直接从注意力分数里减去一个和距离成正比的惩罚值。这样模型自然地更关注近处的词，惩罚远处的词。

```
普通注意力分数：
  score(i, j) = Q_i · K_j / sqrt(d_k)

ALiBi 修改后：
  score(i, j) = Q_i · K_j / sqrt(d_k) - m × (i - j)
                                          ↑
                                    线性距离惩罚
                                    m 是每个头不同的斜率
```

其中 $i-j$ 是两个 token 的距离（$i > j$，即 $j$ 在 $i$ 的前面），$m$ 是各个注意力头预先设定好的斜率，斜率越大，对远处词的惩罚越重。

### 3.2 ALiBi 的外推优势

ALiBi 倾向于牺牲真正的长程建模能力来换取外推能力——它能处理比训练时更长的序列，因为惩罚公式不依赖于训练时见过的绝对位置，只依赖于相对距离。

```python
import torch

def alibi_bias(seq_len: int, n_heads: int) -> torch.Tensor:
    """
    生成 ALiBi 的线性距离惩罚矩阵。
    返回 (n_heads, seq_len, seq_len) 的偏置矩阵，直接加到注意力分数上。
    """
    # 每个头有不同的斜率 m（几何级数）
    slopes = torch.tensor([
        2 ** (-8 * (h + 1) / n_heads) for h in range(n_heads)
    ])   # (n_heads,)

    # 生成距离矩阵：第 i 行第 j 列 = -(i - j)（只看当前位置到之前位置的距离）
    positions = torch.arange(seq_len)
    distance = positions.unsqueeze(0) - positions.unsqueeze(1)  # (L, L)，负数=看前面
    distance = distance.clamp(max=0)   # 只保留负距离（因果方向）

    # 偏置 = 斜率 × 距离矩阵
    bias = slopes.view(n_heads, 1, 1) * distance.unsqueeze(0)
    return bias   # (n_heads, seq_len, seq_len)

# 可视化 ALiBi 偏置
bias = alibi_bias(seq_len=10, n_heads=4)
print(f"ALiBi 偏置形状: {bias.shape}")   # (4, 10, 10)
print(f"第1个头的偏置矩阵（斜率最大，对距离最敏感）:")
print(bias[0].numpy().round(2))
# 每行的惩罚值：当前位置为0，往前一步-0.5，往前两步-1.0...
```

**ALiBi 的问题**：它修改了注意力计算方式（在分数上加了偏置），这与 KV Cache 的某些高效实现不太兼容，工程复杂度略高。使用 ALiBi 的代表模型有 BLOOM、MPT 等，但后来随着 RoPE 的崛起使用率下降。

## 四、第四代：RoPE——旋转位置编码

### 4.1 初印象：用费曼视角理解

**RoPE 的核心想法是：不在输入词向量上加位置信息，而是在计算注意力分数时，把位置信息"旋转进"Q 和 K 里**。

用一个几何比喻：

想象 Q 和 K 是平面上的向量。普通注意力计算的是 Q 和 K 的内积（相似度）。RoPE 的做法是：在计算内积之前，先把每个向量**旋转一个和位置相关的角度**——位置 1 转 θ°，位置 2 转 2θ°，位置 3 转 3θ°……

旋转之后的内积有一个妙用：**两个旋转后的向量的内积，只依赖于它们的相对位置（距离），而不依赖于绝对位置**。

数学上：
$$\text{RoPE}(q, pos) \cdot \text{RoPE}(k, pos') = f(q, k, pos - pos')$$

这意味着注意力分数天然地编码了相对位置信息，而且绝对位置对模型来说是"可见的"（因为旋转角度和绝对位置相关），相对位置也是"可见的"（内积只依赖差值）。两全其美。

### 4.2 RoPE 的数学实现

RoPE 把 Q 和 K 的每两个相邻维度看作一对，对每对进行二维旋转：

$$\text{RoPE}(\mathbf{q}, m) = \begin{bmatrix} q_1 \cos m\theta_1 - q_2 \sin m\theta_1 \\ q_1 \sin m\theta_1 + q_2 \cos m\theta_1 \\ q_3 \cos m\theta_2 - q_4 \sin m\theta_2 \\ q_3 \sin m\theta_2 + q_4 \cos m\theta_2 \\ \vdots \end{bmatrix}$$

其中 $m$ 是位置索引，$\theta_i = 10000^{-2i/d}$ 是每对维度的旋转频率（和正弦编码类似，低维度频率高、高维度频率低）。

```python
import torch
import torch.nn as nn

def precompute_rope_freqs(d_head: int, max_seq_len: int, base: float = 10000.0):
    """
    预计算 RoPE 的旋转频率（cos 和 sin）。
    只需计算一次，推理时复用。
    """
    # 每对维度的旋转频率（d_head/2 个频率）
    theta = 1.0 / (base ** (torch.arange(0, d_head, 2).float() / d_head))
    # 对所有位置计算旋转角度
    positions = torch.arange(max_seq_len).float()
    freqs = torch.outer(positions, theta)   # (max_seq_len, d_head/2)
    # 返回 cos 和 sin 值
    cos_freqs = torch.cos(freqs)   # (max_seq_len, d_head/2)
    sin_freqs = torch.sin(freqs)   # (max_seq_len, d_head/2)
    return cos_freqs, sin_freqs


def apply_rope(x: torch.Tensor, cos: torch.Tensor, sin: torch.Tensor) -> torch.Tensor:
    """
    对 Q 或 K 应用 RoPE 旋转。
    x:   (B, n_heads, L, d_head)
    cos: (L, d_head/2)
    sin: (L, d_head/2)
    """
    # 把 x 的相邻维度两两配对
    x1 = x[..., 0::2]   # 偶数维度：(B, n_heads, L, d_head/2)
    x2 = x[..., 1::2]   # 奇数维度：(B, n_heads, L, d_head/2)

    # 应用旋转公式
    # [x1, x2] × rotation(θ) = [x1·cos - x2·sin, x1·sin + x2·cos]
    cos = cos.unsqueeze(0).unsqueeze(0)   # (1, 1, L, d_head/2)：广播到 batch 和头维度
    sin = sin.unsqueeze(0).unsqueeze(0)

    # 旋转后的坐标
    x_rotated_1 = x1 * cos - x2 * sin
    x_rotated_2 = x1 * sin + x2 * cos

    # 交错拼回去：(B, n_heads, L, d_head)
    x_rotated = torch.stack([x_rotated_1, x_rotated_2], dim=-1)
    return x_rotated.flatten(-2)


class RoPEAttention(nn.Module):
    """带 RoPE 的因果 Self-Attention（LLaMA 风格）"""

    def __init__(self, d_model: int, n_heads: int, max_seq_len: int = 4096, base: float = 10000.0):
        super().__init__()
        self.n_heads = n_heads
        self.d_head  = d_model // n_heads

        self.q_proj = nn.Linear(d_model, d_model, bias=False)
        self.k_proj = nn.Linear(d_model, d_model, bias=False)
        self.v_proj = nn.Linear(d_model, d_model, bias=False)
        self.o_proj = nn.Linear(d_model, d_model, bias=False)

        # 预计算 RoPE 频率，注册为 buffer（不是参数，但随模型保存）
        cos_freqs, sin_freqs = precompute_rope_freqs(self.d_head, max_seq_len, base)
        self.register_buffer("cos_freqs", cos_freqs)
        self.register_buffer("sin_freqs", sin_freqs)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, L, D = x.shape

        Q = self.q_proj(x).view(B, L, self.n_heads, self.d_head).transpose(1, 2)
        K = self.k_proj(x).view(B, L, self.n_heads, self.d_head).transpose(1, 2)
        V = self.v_proj(x).view(B, L, self.n_heads, self.d_head).transpose(1, 2)

        # 对 Q 和 K 应用 RoPE 旋转（V 不旋转）
        Q = apply_rope(Q, self.cos_freqs[:L], self.sin_freqs[:L])
        K = apply_rope(K, self.cos_freqs[:L], self.sin_freqs[:L])

        # 使用 Flash Attention（PyTorch 2.0+）
        import torch.nn.functional as F
        out = F.scaled_dot_product_attention(Q, K, V, is_causal=True)
        out = out.transpose(1, 2).contiguous().view(B, L, D)
        return self.o_proj(out)


# 验证
rope_attn = RoPEAttention(d_model=64, n_heads=4, max_seq_len=512)
x = torch.randn(2, 16, 64)
out = rope_attn(x)
print(f"RoPE Attention 输入: {x.shape}")
print(f"RoPE Attention 输出: {out.shape}")
```

### 4.3 RoPE 为什么成为标配

绝对位置编码解决了排列不变性问题，但在距离推理、长上下文稳定性和缓存效率上表现欠佳。相对位置方法（包括 ALiBi 和 RoPE）显式编码了距离关系，能更优雅地扩展序列长度，简化状态复用，这就是为什么它们取代了纯绝对位置编码，成为当今高性能 LLM 的标准。

RoPE 相比其他方案的具体优势：

| 特性 | 正弦编码 | 可学习绝对编码 | ALiBi | RoPE |
|------|---------|-------------|-------|------|
| 参数量 | 0 | max_len × d | 0 | 0 |
| 长度外推 | 差 | 不能 | 好 | 较好 |
| 相对位置感知 | 间接 | 无 | 直接 | 直接 |
| KV Cache 兼容 | 好 | 好 | 需要调整 | 好 |
| 主流使用 | 原始 Transformer | GPT-2/BERT | BLOOM/MPT | **LLaMA/Qwen/Mistral/…** |

## 五、长上下文外推：RoPE 的挑战与解决方案

### 5.1 问题：RoPE 为什么难以外推

RoPE 存在一个关键限制：每个隐藏维度对通过不同的旋转周期性来编码位置。高维度旋转慢，如果模型在训练时没有看到完整的旋转周期（比如只旋转到 90 度），就无法外推到更远的位置（超过 90 度）。

简单说：**RoPE 训练时只"见过"前 4096 个位置的旋转角度，推理时突然要处理第 8192 个位置，这个角度是"未见过"的，模型会懵**。

### 5.2 解决方案一：位置插值（Position Interpolation）

**直觉**：与其让模型处理未见过的大位置，不如把长序列的位置"压缩"到训练时见过的范围里。

如果训练长度是 4096，要处理 8192 个 token，就把所有位置除以 2：

$$pos' = pos \times \frac{\text{训练长度}}{\text{目标长度}} = pos \times \frac{4096}{8192} = pos \times 0.5$$

这样 8192 个位置被映射到 $[0, 4096]$ 的范围，全是模型见过的"熟悉区域"。

代价是精度损失（原来每一步 1 个单位，现在变成 0.5 个单位），往往需要少量微调来适应。

```python
def rope_with_position_interpolation(d_head, seq_len, train_len=4096, base=10000.0):
    """位置插值：把超长序列的位置压缩到训练范围内"""
    # 缩放因子
    scale = train_len / seq_len if seq_len > train_len else 1.0
    # 压缩后的位置索引
    positions = torch.arange(seq_len).float() * scale
    theta = 1.0 / (base ** (torch.arange(0, d_head, 2).float() / d_head))
    freqs = torch.outer(positions, theta)
    return torch.cos(freqs), torch.sin(freqs)
```

### 5.3 解决方案二：YaRN（Yet another RoPE extensioN）

YaRN 是目前效果最好的 RoPE 外推方法之一，被 LLaMA-3、Qwen-2.5 等用于扩展上下文长度。

**核心思路**：不同频率的维度用不同的处理方式：
- 高频维度（低维度）：直接用位置插值（这些维度对距离变化最敏感）
- 低频维度（高维度）：保持原始频率不变（这些维度已经能够外推）
- 中间维度：线性插值过渡

```python
def yarn_rope_freqs(d_head, seq_len, original_max_len=4096,
                    base=10000.0, alpha=1.0, beta=32.0):
    """
    YaRN RoPE 频率计算。
    alpha, beta：控制哪些维度用插值，哪些保持原始频率。
    """
    theta = 1.0 / (base ** (torch.arange(0, d_head, 2).float() / d_head))
    # 每个频率对应的波长
    wavelengths = 2 * math.pi / theta

    # 根据波长决定处理方式
    scale = seq_len / original_max_len

    # 低频（长波长）：直接外推（不插值）
    # 高频（短波长）：插值
    # 中间：线性过渡
    ratio = wavelengths / original_max_len
    factor = torch.where(
        ratio < 1 / beta,                        # 高频：插值
        torch.ones_like(ratio) * (1 / scale),
        torch.where(
            ratio > 1 / alpha,                   # 低频：保持
            torch.ones_like(ratio),
            # 中间：线性插值
            (1 - 1/scale) / (1/alpha - 1/beta) * ratio + 1/scale - (1-1/scale) * 1/beta / (1/alpha - 1/beta)
        )
    )
    adjusted_theta = theta * factor
    positions = torch.arange(seq_len).float()
    freqs = torch.outer(positions, adjusted_theta)
    return torch.cos(freqs), torch.sin(freqs)
```

## 六、NoPE：干脆不用位置编码

### 6.1 出人意料的发现

2024 年有研究发现：对于足够大的模型，**完全不使用任何位置编码（NoPE）** 也能工作，甚至在某些长上下文任务上超过 RoPE。

当 token 嵌入在初始化时彼此接近（从小方差分布中随机抽取），会导致注意力图的均匀性，而均匀混合 token 无法增加它们之间的差异，使均匀性在整个网络中持续存在。这就解释了 RoPE 和 NoPE Transformer 之间的差异。

实际上，NoPE 模型依赖 Transformer 本身（特别是因果掩码）隐式地学习位置信息。这也启示我们：位置信息未必只能通过显式编码注入，足够大的模型能从上下文中隐式感知顺序。

**NoPE 的现状**：目前还是研究阶段为主，工业界主流仍然使用 RoPE。但这个方向说明位置编码未必是 Transformer 的必要组件。

## 八、面试常见问题

**Q1：为什么 Transformer 需要位置编码？**

Self-Attention 机制本身是"排列不变"的——如果把输入 token 的顺序打乱，计算结果相同（只是对应关系变化）。但语言中词序至关重要（"猫吃鱼"和"鱼吃猫"意思完全不同），所以必须显式地给模型注入位置信息。位置编码就是这个作用——给每个位置一个独特的"坐标"，让模型能区分不同位置的 token。

**Q2：RoPE 和绝对位置编码的根本区别是什么？**

绝对位置编码把位置信息直接加到词向量上，然后一起送进注意力计算，模型间接感知位置。RoPE 不修改词向量，而是在注意力计算时把位置信息"旋转进" Q 和 K 的向量里。旋转后的内积只依赖两个 token 的相对位置（距离），而不是绝对位置，因此 RoPE 天然感知相对位置，更适合长距离建模和序列外推。

**Q3：为什么现代 LLM（LLaMA、Qwen、Mistral）几乎都用 RoPE？**

三个优势：天然感知相对位置，注意力分数直接体现两个 token 的距离关系；无额外参数，不消耗模型容量；与 KV Cache 高度兼容，推理时每次只需为新 token 计算 RoPE，不影响缓存的 K/V。相比 ALiBi 的工程复杂性和可学习编码的长度上限，RoPE 是工程友好性和效果的最佳平衡点。

**Q4：RoPE 的长度外推有什么挑战？怎么解决？**

RoPE 的主要挑战是当序列长度超过预训练长度时，某些维度会遇到训练时未见过的旋转角度，导致注意力分数失准。解决方案：位置插值（把超长序列的位置索引压缩到训练范围内）、增大 RoPE base（降低高频分量的旋转速度，使更多维度在训练时能完成完整的旋转周期）、YaRN（对不同频率维度采用不同策略）。这些方法通常需要配合少量微调以恢复完整性能。

**Q5：ALiBi 和 RoPE 各有什么优势？**

ALiBi 的优势：不修改词向量，直接在注意力分数上加线性距离惩罚，概念极简；无需额外参数；长度外推能力强，无需微调就能处理比训练时长得多的序列。RoPE 的优势：相对位置信息编码在旋转中，理论上更自然；与 KV Cache 的实现高度兼容，工程更简洁；社区支持更广泛，大量开源模型和工具都有成熟实现。目前 RoPE 使用更广泛，ALiBi 在部分强调超长外推的场景仍有应用。
