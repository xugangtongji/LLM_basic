## 零、开篇：为什么大模型工程师必须会这三件事

大模型训练几乎全部跑在 Linux GPU 服务器上。你的本地笔记本只是一个终端，真正的计算在云端或集群里。

**三件事缺一不可**：

- **Linux 命令行**：你和 GPU 服务器沟通的唯一语言。不会它就相当于不会开车却要去飙车。`nvidia-smi`、`htop`、`tmux`、`scp` 是大模型工程师每天必用的工具。
- **Git**：实验代码版本管理的基础设施。深度学习实验充满"上周那个改动跑出来的结果怎么重现"的需求，没有 Git 你永远找不回来。
- **Docker**：把环境打包进一个"集装箱"。你的代码在 Python 3.10 + PyTorch 2.3 + CUDA 12.1 上跑通了，同事的服务器是另一套版本——Docker 让你的环境变成可移植的标准品。

## 一、Linux 命令行：GPU 服务器上的生存技能

### 1.1 初印象：用费曼视角理解

**Linux 命令行就是用文字和计算机对话**。

你在 GUI 里双击文件夹，在命令行里就是输入 `ls`；你在 GUI 里拖动文件，命令行里就是 `mv`；你在任务管理器里查看进程，命令行里就是 `htop`。

区别在于：命令行可以把多个操作**串联、自动化、远程执行**，而 GUI 只能一个个点。当你的训练任务跑在一台几千公里外的 GPU 服务器上，你只能依靠命令行。

### 1.2 文件与目录操作

```bash
# 导航
pwd                        # 打印当前路径
ls -lh                     # 列出当前目录，l=详情 h=人类可读大小
ls -lh *.pt                # 列出所有 .pt checkpoint 文件
cd /data/experiments       # 切换目录
cd ~                       # 回到 home 目录
cd -                       # 回到上一个目录（非常实用）

# 文件操作
cp model.py model_backup.py    # 拷贝
cp -r results/ results_v2/     # 递归拷贝目录
mv exp_v1/ archive/            # 移动 / 重命名
rm -rf tmp/                    # 递归强制删除（小心，无法恢复！）
mkdir -p logs/2024/exp01       # 递归创建目录

# 查看文件内容
cat train.log                  # 打印全部内容
head -n 50 train.log           # 前 50 行
tail -n 100 train.log          # 后 100 行
tail -f train.log              # 实时追踪日志（训练时必用）
less train.log                 # 分页浏览（q 退出）

# 搜索
grep "val_loss" train.log      # 从日志里提取 val_loss 行
grep -r "learning_rate" .      # 在当前目录递归搜索字符串
find . -name "*.pt" -size +1G  # 找出大于 1GB 的 checkpoint
find . -name "*.py" -mtime -1  # 找出 1 天内修改的 Python 文件
```

### 1.3 GPU 监控：大模型工程师的必备命令

```bash
# 查看 GPU 状态（最常用）
nvidia-smi

# 每 0.5 秒刷新一次，持续监控（训练时开着）
watch -n 0.5 nvidia-smi

# 只显示关键信息
nvidia-smi --query-gpu=index,name,memory.used,memory.total,utilization.gpu,temperature.gpu \
           --format=csv,noheader,nounits

# 典型输出解读：
# 0, NVIDIA A100-SXM4-80GB, 78432, 81920, 98, 67
# GPU索引, 型号, 已用显存(MB), 总显存(MB), GPU利用率(%), 温度(°C)
```

**nvidia-smi 输出各字段含义**：

| 字段 | 含义 | 正常训练时 |
|------|------|-----------|
| Memory-Usage | 已用显存 / 总显存 | 应接近满载（高利用率） |
| GPU-Util | GPU 计算单元利用率 | 理想 > 90%，低说明数据加载是瓶颈 |
| Temp | GPU 温度 (°C) | A100 一般 60-85°C 正常 |
| Power | 当前功耗 / 最大功耗 | 接近 TDP 说明 GPU 在满负荷计算 |

```bash
# 查看哪些进程在用 GPU
nvidia-smi pmon -s um

# 杀掉占用 GPU 的进程（PID 来自上条命令）
kill -9 <PID>

# 系统资源监控
htop               # 交互式 CPU/内存监控（比 top 好用）
free -h            # 内存使用情况
df -h              # 磁盘使用情况（防止 /tmp 或 /data 被 checkpoint 撑满）
du -sh data/       # 查看某目录占用空间
```

### 1.4 进程管理：让训练在后台跑

```bash
# 直接跑（断开 SSH 就中断，不建议）
python train.py

# nohup：断开 SSH 后继续跑
nohup python train.py > train.log 2>&1 &
# > train.log：标准输出重定向到文件
# 2>&1：标准错误也重定向到同一文件
# &：后台运行

# 查看后台任务
jobs
ps aux | grep train.py   # 查找训练进程

# 前后台切换
Ctrl+Z      # 暂停当前进程
bg          # 把暂停的进程放到后台继续运行
fg          # 把后台进程拉回前台
```

### 1.5 tmux：长时间训练的救命工具

**tmux 是终端复用器**，它在服务器上保持一个持久会话，即使你的 SSH 断开了，里面的训练照常进行，重新 SSH 后可以恢复。

```bash
# 安装
sudo apt install tmux

# 常用操作
tmux new -s train          # 创建名为 train 的新会话
tmux ls                    # 列出所有会话
tmux attach -t train       # 恢复进入 train 会话

# 在 tmux 内的快捷键（前缀 Ctrl+b）
Ctrl+b d                   # 分离（detach）会话，SSH 断开后仍在运行
Ctrl+b c                   # 在当前 session 创建新窗口
Ctrl+b 数字                # 切换到第 N 个窗口
Ctrl+b %                   # 垂直分割窗格
Ctrl+b "                   # 水平分割窗格
Ctrl+b 方向键              # 切换窗格
```

**标准大模型训练工作流**：

```bash
# 1. SSH 连接服务器
ssh -i ~/.ssh/id_rsa user@gpu-server-ip

# 2. 创建 tmux 会话
tmux new -s train_llama

# 3. 激活环境并启动训练
conda activate llm
python train.py --config configs/llama_7b.yaml 2>&1 | tee logs/train.log

# 4. 按 Ctrl+b d 分离，关闭 SSH 本地窗口

# 5. 随时重新连接查看进度
ssh user@gpu-server-ip
tmux attach -t train_llama
```

### 1.6 文件传输

```bash
# scp：安全拷贝（本地→远程，或远程→本地）
# 上传 checkpoint 到远程
scp -r checkpoints/ user@server:/data/experiments/exp01/

# 下载评测结果到本地
scp user@server:/data/experiments/exp01/results.csv ./

# rsync：增量传输，更高效（推荐用于大文件 / 目录）
# -a 保留权限、时间戳；-v 显示进度；-z 压缩传输；-P 断点续传

```

## 二、Git：实验代码的时光机

### 2.1 初印象：用费曼视角理解

**Git 是代码的"快照记录仪"**。

每次你觉得代码到了一个稳定状态，就拍一张快照（commit）。过了两周，你不小心把代码改坏了，或者你想回去看当时的某个版本——就翻出那张快照。

更厉害的是：Git 允许你同时开多条"时间线"（分支），分别做不同的实验，最后可以合并或选择最好的那条。

**大模型工程里的常见场景**：
- "上周那个配置跑出了 72 的 MMLU，我现在找不到那个版本了" → Git tag / commit history 救你
- "我想同时试 LoRA rank=8 和 rank=16，互不干扰" → 两个分支
- "我们三个人合作开发微调代码" → PR / merge 流程

### 2.2 核心工作流

```bash
# 初始配置（只做一次）
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global core.editor "vim"      # 或 nano

# 初始化仓库
git init                        # 在当前目录新建仓库
git clone <url>                 # 克隆远程仓库
git clone <url> --depth 1       # 浅克隆（只拉最新一个 commit，省空间）

# 基本快照流程
git status                      # 查看哪些文件有变化
git add train.py configs/       # 把指定文件加入暂存区
git add .                       # 把所有变化加入暂存区
git commit -m "Add warmup scheduler and gradient clipping"  # 提交快照
git log --oneline               # 查看提交历史（简洁模式）
git log --oneline --graph       # 带分支图的历史
```

### 2.3 分支管理：实验驱动的工作流

```bash
# 分支操作
git branch                      # 查看所有本地分支
git branch lora-rank-16         # 创建新分支
git switch lora-rank-16         # 切换到该分支（Git 2.23+ 推荐用 switch）
git switch -c lora-rank-8       # 创建并切换（常用！）
git switch main                 # 切回主分支

# 大模型实验的分支命名规范建议
# feat/qlora-4bit
# exp/llama3-rope-scaling
# fix/dataloader-memory-leak
# hotfix/nan-loss-fp16

# 合并分支
git switch main
git merge lora-rank-16          # 把 lora-rank-16 的修改合并进 main

# 删除分支
git branch -d lora-rank-16      # 删除本地分支（已合并才能删）
git branch -D lora-rank-16      # 强制删除

# 查看分支差异
git diff main..lora-rank-16 -- train.py   # 对比两个分支某文件的差异
```

### 2.5 实用技巧

```bash
# 撤销操作
git restore train.py            # 丢弃工作区的修改（未 add）
git restore --staged train.py   # 取消 add（从暂存区撤回）
git revert <commit-hash>        # 撤销某次提交（生成新的 commit）
git reset --hard HEAD~1         # 彻底回滚到上一个 commit（危险！会丢失修改）

# 暂存当前工作（切分支前不想 commit）
git stash                       # 把当前修改压入栈
git stash pop                   # 恢复
git stash list                  # 查看所有 stash

# 给重要实验打标签（比 commit hash 好记）
git tag v1.0-baseline           # 打轻量标签
git tag -a v1.1-lora -m "LoRA rank=8, val_loss=0.82"  # 打注释标签
git push origin v1.1-lora       # 推送标签到远程

# 远程操作
git remote -v                   # 查看远程仓库地址
git fetch origin                # 拉取远程更新但不合并
git pull origin main            # 拉取并合并
git push origin feat/qlora      # 推送本地分支到远程

# 查找 bug 引入的 commit（二分查找）
git bisect start
git bisect bad                  # 标记当前版本有 bug
git bisect good v1.0-baseline   # 标记已知好版本
# Git 会自动切到中间 commit，你测试后告诉它好还是坏
git bisect good / git bisect bad
git bisect reset                # 完成后重置
```

## 三、Docker：可复现环境的集装箱

### 3.1 初印象：用费曼视角理解

**Docker 就是把整个"运行环境"打包进一个集装箱**。

普通情况下，代码能不能跑取决于你有没有装对版本的 Python、PyTorch、CUDA、各种依赖库……换一台机器就可能全部重来。

Docker 把这一切打包进一个"镜像"（image）。在任何装了 Docker 的机器上，用这个镜像启动一个"容器"（container），里面的环境和你打包时完全一样。**"在我这里能跑"和"在你那里也能跑"终于画上等号**。

大模型工程里 Docker 的用途：
- 统一团队开发环境
- 生产部署 LLM 推理服务
- Kubernetes 集群里的训练任务（每个任务就是一个 container）
- HuggingFace Spaces、云厂商 AI 平台的部署方式

### 3.2 核心概念

| 概念 | 类比 | 说明 |
|------|------|------|
| Image（镜像） | 程序安装包 | 只读的环境快照，可以分发 |
| Container（容器） | 运行中的程序 | 从镜像启动，可读写，独立沙箱 |
| Dockerfile | 菜谱 | 描述如何构建镜像的文本文件 |
| Registry（仓库） | App Store | 存储和分发镜像，如 Docker Hub、NVIDIA NGC |
| Volume（卷） | 外接硬盘 | 把宿主机目录挂载进容器，持久化数据 |

### 3.3 常用 Docker 命令

```bash
# 镜像操作
docker images                                  # 列出本地镜像
docker pull pytorch/pytorch:2.4.0-cuda12.1-cudnn9-runtime  # 拉取镜像
docker rmi <image_id>                          # 删除镜像

# 容器操作
docker ps                                      # 查看运行中的容器
docker ps -a                                   # 查看所有容器（含已停止）
docker stop <container_id>                     # 停止容器
docker rm <container_id>                       # 删除容器
docker logs <container_id>                     # 查看容器日志
docker logs -f <container_id>                  # 实时追踪日志

# 运行容器（最核心的命令）
docker run \
  --gpus all \                                 # 使用所有 GPU（需要 nvidia-docker）
  --rm \                                       # 运行结束后自动删除容器
  -it \                                        # 交互模式（保持 stdin 打开）
  -v /data:/workspace/data \                   # 挂载宿主机 /data 到容器 /workspace/data
  -v $(pwd):/workspace/code \                  # 挂载当前目录（代码）
  -p 8888:8888 \                               # 端口映射（Jupyter）
  --name my_train \                            # 容器命名
  pytorch/pytorch:2.4.0-cuda12.1-cudnn9-runtime \  # 镜像名
  bash                                         # 启动命令

# 进入运行中的容器
docker exec -it my_train bash
```

### 3.4 为大模型项目写 Dockerfile

```dockerfile
# Dockerfile —— LLM 微调环境
# 基础镜像：NVIDIA 提供的 PyTorch 镜像（含 CUDA）
FROM pytorch/pytorch:2.4.0-cuda12.1-cudnn9-devel

# 设置工作目录
WORKDIR /workspace

# 设置环境变量
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1 \
    PIP_NO_CACHE_DIR=1 \
    DEBIAN_FRONTEND=noninteractive

# 安装系统依赖
RUN apt-get update && apt-get install -y \
    git \
    wget \
    curl \
    vim \
    tmux \
    && rm -rf /var/lib/apt/lists/*

# 先复制依赖文件（利用 Docker 缓存层优化构建速度）
COPY requirements.txt .

# 安装 Python 依赖
RUN pip install --upgrade pip && \
    pip install -r requirements.txt

# 再复制项目代码（代码经常变，放最后避免缓存失效）
COPY . .

# 默认启动命令（可以在 docker run 时覆盖）
CMD ["python", "train.py", "--config", "configs/default.yaml"]
```

```text
# requirements.txt
torch==2.4.0
transformers==4.44.0
peft==0.12.0
datasets==2.20.0
accelerate==0.33.0
bitsandbytes==0.43.1
wandb
tqdm
```

```bash
# 构建镜像
docker build -t my-llm-trainer:v1 .

# 运行训练
docker run --gpus all --rm \
  -v /data/datasets:/workspace/data \
```

### 3.5 docker-compose：多服务编排

LLM 推理服务通常包含多个组件（模型服务 + Redis 缓存 + Nginx 代理），用 `docker-compose` 一键管理：

```yaml
# docker-compose.yml
version: "3.9"

services:
  vllm:
    image: vllm/vllm-openai:latest
    runtime: nvidia
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
    volumes:
      - /models:/models
    ports:
      - "8000:8000"
    command: >
      --model /models/llama-3-8b
      --tensor-parallel-size 2
      --max-model-len 8192
      --dtype bfloat16

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
    depends_on:
      - vllm
```

```bash
docker compose up -d           # 后台启动所有服务
docker compose logs -f vllm    # 查看 vllm 服务日志
docker compose down            # 停止并删除所有容器
```

## 四、常见陷阱

**Linux 陷阱**

1. **`rm -rf` 无法恢复**：删之前先 `ls` 确认，大文件夹可以先 `mv` 到 `/tmp` 观察几天再删。
2. **后台训练的 stdout 消失**：用 `nohup ... > train.log 2>&1 &` 把输出重定向到文件，不然训练中途什么信息都看不到。
3. **磁盘被 checkpoint 撑满**：设置 checkpoint 保存策略（只保留最近 N 个），并定期 `df -h` 和 `du -sh checkpoints/` 检查空间。
4. **CUDA OOM 后进程残留**：OOM 报错后 Python 进程有时不会自动退出，导致显存被占用。需要 `nvidia-smi pmon -s um` 找到进程 PID，再 `kill -9 <PID>`。

**Git 陷阱**

5. **把 API key 或密码 commit 进代码库**：一旦 push 到 GitHub，即使删除也会留在历史里。用 `.gitignore` 排除 `.env` 文件，或用 `git-secrets` 工具扫描。
6. **把大文件（checkpoint）commit 进 Git**：Git 不适合管理二进制大文件，会让仓库体积膨胀到几 GB，clone 变得极慢。用 `git-lfs` 或 S3/OSS 管理模型文件。
7. **`git reset --hard` 丢失工作**：这个操作会丢失未 commit 的修改，且无法通过 git 恢复。修改前先 `git stash` 备份。

**Docker 陷阱**

8. **容器内修改不持久**：容器内产生的文件（如 checkpoint）在容器删除后消失。必须用 `-v` 挂载外部卷持久化输出。
9. **镜像越来越大**：每次 `RUN apt-get install` 如果不清理，镜像会越来越臃肿。同一 RUN 命令里安装并清理：`RUN apt-get install -y xxx && rm -rf /var/lib/apt/lists/*`。
10. **GPU 访问问题**：容器里 `nvidia-smi` 报错，通常是没有安装 `nvidia-container-toolkit` 或没有 `--gpus all` 参数。

## 五、面试常见问题

**Q1：你在 GPU 服务器上怎么监控训练进度？**

用 `tmux` 开一个会话跑训练，把日志重定向到文件。开另一个窗口 `tail -f train.log` 实时查看输出，同时 `watch -n 0.5 nvidia-smi` 监控 GPU 利用率和显存占用。如果用 wandb，还能在浏览器里看实时曲线。

**Q2：训练任务跑了一半 SSH 断开怎么办？**

如果用了 `tmux`，重新 SSH 后 `tmux attach -t train` 即可恢复，任务没有中断。如果当初用的是 `nohup`，进程还在后台运行，`ps aux | grep train.py` 找到进程确认仍然存活，再 `tail -f train.log` 查看进度。

**Q3：Git 的 `git rebase` 和 `git merge` 有什么区别？**

两者都是把一个分支的修改整合进另一个分支：
- `merge` 会创建一个新的 merge commit，保留完整的分叉历史，看起来像两条线汇合。
- `rebase` 把你的 commit 逐一"搬"到目标分支的最新处，历史变成一条直线，更干净。

大模型工程里，功能分支开发完成后向 main 合并推荐用 PR + squash merge；日常拉取远端更新推荐 `git pull --rebase`。

**Q4：Docker 的镜像和容器有什么区别？**

镜像是只读的模板（像安装包），容器是从镜像启动的可运行实例（像安装完正在跑的程序）。一个镜像可以启动多个容器，容器里的修改不会影响镜像。

**Q5：为什么在 Dockerfile 里要把 `COPY requirements.txt` 和 `pip install` 分开放，之后再 `COPY .`？**

Docker 构建时按层缓存。如果 requirements.txt 没变，pip install 那层会被缓存跳过，再次构建时只重新执行代码复制层，构建速度大幅提升。如果把所有 COPY 放在一起，每次代码改动都会触发 pip install 重新跑，浪费时间。

**Q6：如何在 Docker 容器里使用 GPU？**

需要两个前提：宿主机安装了 NVIDIA 驱动和 `nvidia-container-toolkit`；`docker run` 时加 `--gpus all`（或 `--gpus '"device=0,1"'` 指定具体 GPU）。选用带 CUDA 的基础镜像（如 `pytorch/pytorch:xxx-cuda12.1`），容器内就能访问 GPU。

**Q7：`nvidia-smi` 显示 GPU 利用率很低（< 30%），但程序在正常跑，可能是什么问题？**

GPU 利用率低说明 GPU 大部分时间在"等数据"。最常见原因是数据加载（DataLoader）成为瓶颈，CPU 处理数据的速度跟不上 GPU 计算速度。解决方案：增大 `num_workers`、开启 `pin_memory=True`、把数据预处理 cache 到内存、使用更快的存储介质（NVMe SSD 比 HDD 快 10 倍）。

**Q8：你怎么管理不同项目依赖冲突的问题？**

项目级别：每个项目一个 conda 虚拟环境（`conda create -n proj_name python=3.11`），环境完全隔离，不同版本的 PyTorch / CUDA 互不影响。

跨机器复现：用 `conda env export > environment.yml` 导出环境，或者用 Docker 把整个环境打包成镜像。
