## 零、开篇：注意力的两个瓶颈

上一章我们知道，标准 Self-Attention 的复杂度是 $O(L^2)$——序列翻倍，计算量变 4 倍。这在处理长文本时是个大问题。

但其实 $O(L^2)$ 的计算量只是问题的一半。真正让 Attention 慢的，是另一个更隐蔽的原因：**内存带宽瓶颈**。

想象一条高速公路：GPU 的"计算核心"是超宽的多车道公路，但连接计算核心和"主显存（HBM）"的数据通道（内存总线）却相对狭窄。

标准注意力计算中，最大的那个中间矩阵（$L \times L$ 的注意力分数）必须先写入 HBM 再读回来，反复几次。这个"搬运数据"的过程，比实际计算本身花的时间还多。

本章讲的三大优化，分别解决了不同维度的问题：

```
Flash Attention  → 解决"内存带宽瓶颈"（让大矩阵不用反复读写 HBM）
GQA / MQA       → 解决"KV Cache 太大"（推理时共享 K/V，省显存）
稀疏注意力       → 解决"超长序列的 O(L²) 计算量"（不是所有词对都要算）
```

## 一、Flash Attention：重新设计内存访问模式

### 1.1 初印象：用费曼视角理解

**Flash Attention 的核心不是改变计算结果，而是改变计算顺序——让计算发生在更快的内存里**。

把 GPU 的内存比喻成一家工厂：

- **SRAM（片上内存）**：车间的工作台，空间小（几十 MB），但极快（TB/s 级带宽）
- **HBM（高带宽主显存）**：仓库，空间大（数十 GB），但相对慢（GB/s 级带宽）

标准注意力计算的问题：每做一步（计算 $QK^T$、Softmax、乘以 $V$），都要把结果存到仓库，下一步再从仓库取出来继续做。仓库和车间之间来回搬运的时间，远比实际计算耗时。

Flash Attention 的解法：**把整个注意力计算分成小块，让每块的数据完全在车间（SRAM）里处理完，再存结果到仓库**。不再反复进出仓库。

### 1.2 标准注意力 vs Flash Attention 的 IO 对比

**标准注意力的 HBM 读写过程**：

```
Step 1：读取 Q, K, V（各 L×d）→ 计算 S = QK^T（L×L）→ 写回 HBM
Step 2：读取 S（L×L）→ 计算 P = softmax(S)（L×L）→ 写回 HBM
Step 3：读取 P（L×L）和 V（L×d）→ 计算 O = PV（L×d）→ 写回 HBM
```

中间的 $L \times L$ 矩阵（注意力分数和权重）被完整地写入 HBM 两次、读出两次。
当 $L=4096$ 时，这个矩阵约 128 MB（BF16），反复搬运代价极高。

**Flash Attention 的做法**：

```
把 Q 分成 Br 个块，K/V 分成 Bc 个块：
  for each block of Q:
    for each block of K, V:
      在 SRAM 内完成这个小块的 QK^T、softmax、×V
      用"在线 softmax"技巧累积局部结果
  最终 O 直接写回 HBM
```

中间的 $L \times L$ 矩阵从未完整写入 HBM，HBM 读写量从 $O(L^2)$ 降到 $O(L)$。

### 1.3 在线 Softmax：分块计算的关键技巧

分块计算 Softmax 有一个数学问题：Softmax 需要看到所有分数才能归一化。分块计算时只能看到局部分数，怎么得到正确结果？

**在线 Softmax（Online Softmax）** 解决了这个问题：维护当前已处理分数的最大值和部分求和，每遇到新块时更新这两个统计量，最终能得到和标准 Softmax 完全相同的结果。

```python
import torch
import math

def online_softmax_step(prev_max, prev_sum, new_scores):
    """
    在线 Softmax 的单步更新。
    用于 Flash Attention 的分块计算：
    每处理一个 K/V 块时，更新当前的 softmax 状态。
    """
    # 新块的最大值
    new_max = torch.maximum(prev_max, new_scores.max(dim=-1, keepdim=True).values)

    # 用新的最大值更新之前的部分和（需要重新缩放）
    prev_sum_corrected = prev_sum * torch.exp(prev_max - new_max)

    # 当前块的贡献
    new_contribution = torch.exp(new_scores - new_max).sum(dim=-1, keepdim=True)

    # 合并
    total_sum = prev_sum_corrected + new_contribution
    return new_max, total_sum


def flash_attention_naive(Q, K, V, block_size=32):
    """
    Flash Attention 的简化教学实现（不含 CUDA 优化，只展示算法思路）。
    真实 Flash Attention 用 CUDA kernel 实现，速度远超此版本。
    """
    B, H, L, d = Q.shape
    scale = 1.0 / math.sqrt(d)
    O = torch.zeros_like(Q)

    for b in range(B):
        for h in range(H):
            q = Q[b, h]   # (L, d)
            k = K[b, h]   # (L, d)
            v = V[b, h]   # (L, d)

            # 初始化在线 Softmax 状态
            running_max = torch.full((L, 1), float('-inf'))
            running_sum = torch.zeros(L, 1)
            running_out = torch.zeros(L, d)

            # 分块处理 K/V
            for j in range(0, L, block_size):
                k_block = k[j:j+block_size]   # (block_size, d)
                v_block = v[j:j+block_size]   # (block_size, d)

                # 计算当前 Q 与这块 K 的注意力分数
                scores = q @ k_block.T * scale   # (L, block_size)

                # 因果掩码
                positions = torch.arange(L).unsqueeze(1)
                k_positions = torch.arange(j, j + k_block.shape[0]).unsqueeze(0)
                causal_mask = positions < k_positions
                scores = scores.masked_fill(causal_mask, float('-inf'))

                # 更新在线 Softmax 状态
                new_max = torch.maximum(running_max, scores.max(dim=-1, keepdim=True).values)
                running_out = running_out * torch.exp(running_max - new_max)
                running_sum = running_sum * torch.exp(running_max - new_max)

                exp_scores = torch.exp(scores - new_max)
                running_out = running_out + exp_scores @ v_block
                running_sum = running_sum + exp_scores.sum(dim=-1, keepdim=True)
                running_max = new_max

            # 最终归一化
            O[b, h] = running_out / running_sum

    return O


# Flash Attention 在 PyTorch 2.0+ 中已内置，自动选择最优实现
import torch.nn.functional as F

Q = torch.randn(2, 4, 64, 16)   # (B, H, L, d)
K = torch.randn(2, 4, 64, 16)
V = torch.randn(2, 4, 64, 16)

# PyTorch 的 SDPA 会自动选择 Flash Attention（如果 GPU 支持）
out_flash = F.scaled_dot_product_attention(Q, K, V, is_causal=True)
print(f"Flash Attention 输出形状: {out_flash.shape}")   # (2, 4, 64, 16)
```

## 二、MQA 与 GQA：让 KV Cache 瘦身

### 2.1 初印象：用费曼视角理解

**KV Cache 的问题**：推理时为了避免重复计算，会把每层、每个注意力头的 K 和 V 向量缓存起来。序列越长、头数越多，KV Cache 越大，占用的显存就越多。

随着模型规模增大和序列长度增加，KV Cache 消耗的内存持续增长，而 MQA 和 GQA 正是解决这个问题的两种技术。

**MHA（标准多头注意力）**：每个头有独立的 K 和 V，如果有 32 个头，就有 32 份 K 和 32 份 V。

**MQA 的想法**：只保留 1 份 K 和 1 份 V，所有 32 个查询头共享这同一份 K 和 V。

**GQA 的想法**：中间方案——把 32 个查询头分成 G 组（比如 8 组），每组共享 1 份 K 和 V，共 8 份 K 和 8 份 V。

用团队协作来比喻：
- **MHA**：32 个员工各有各的参考资料（K/V），每人查自己的
- **MQA**：32 个员工共享同一份参考资料，所有人查同一个
- **GQA**：32 个员工分成 8 个小组，每组 4 人共享一份参考资料

GQA 是 Multi-Query Attention 和 Multi-Head Attention 的插值——它实现了接近 MHA 的质量，同时保持与 MQA 相当的速度。

### 2.2 三种注意力的维度对比

设模型有 $H$ 个查询头，GQA 有 $G$ 个 KV 头（$1 \leq G \leq H$）：

| 变体 | Q 头数 | K/V 头数 | KV Cache 大小 | 模型质量 |
|------|--------|---------|-------------|---------|
| MHA | H | H | $H \times d_k \times L$ | 最好 |
| GQA | H | G（1 < G < H）| $G \times d_k \times L$ | 接近 MHA |
| MQA | H | 1 | $1 \times d_k \times L$ | 稍差 |

GQA 提供了比 MQA 略好的性能，同时仍然显著减小了 KV Cache 的大小。

### 2.3 KV Cache 节省的具体数字

以 LLaMA-3-8B 为例：$H=32$ 个查询头，GQA 用 $G=8$ 个 KV 头：

```python
def kv_cache_size(n_layers, n_kv_heads, d_head, seq_len, dtype_bytes=2):
    """计算 KV Cache 大小"""
    # K 和 V 各一份，共 2 份
    size_bytes = 2 * n_layers * n_kv_heads * d_head * seq_len * dtype_bytes
    return size_bytes

d_head = 128   # LLaMA-3-8B 的每头维度

# 比较 MHA vs GQA
print("LLaMA-3-8B，序列长度 4096，BF16:")
mha_size = kv_cache_size(32, 32, d_head, 4096)
gqa_size = kv_cache_size(32, 8,  d_head, 4096)   # LLaMA-3 用 8 个 KV 头
print(f"  MHA (32 KV头): {mha_size/1e9:.2f} GB")
print(f"  GQA (8 KV头):  {gqa_size/1e9:.2f} GB")
print(f"  节省: {(1-gqa_size/mha_size):.0%}")

print("\nLLaMA-3-8B，序列长度 128K（长文本），BF16:")
mha_128k = kv_cache_size(32, 32, d_head, 131072)
gqa_128k = kv_cache_size(32, 8,  d_head, 131072)
```

### 2.4 GQA 的代码实现

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class GroupedQueryAttention(nn.Module):
    """
    GQA（分组查询注意力）实现。
    核心技巧：K/V 头数少于 Q 头数，通过 repeat_kv 扩展 K/V 到 Q 的头数。
    """

    def __init__(self, d_model: int, n_q_heads: int, n_kv_heads: int):
        super().__init__()
        assert n_q_heads % n_kv_heads == 0, "Q 头数必须是 KV 头数的整数倍"

        self.n_q_heads  = n_q_heads
        self.n_kv_heads = n_kv_heads
        self.n_groups   = n_q_heads // n_kv_heads   # 每组的 Q 头数
        self.d_head     = d_model // n_q_heads

        # Q 投影：d_model → n_q_heads × d_head（完整）
        self.q_proj = nn.Linear(d_model, n_q_heads * self.d_head, bias=False)
        # K/V 投影：d_model → n_kv_heads × d_head（更小）
        self.k_proj = nn.Linear(d_model, n_kv_heads * self.d_head, bias=False)
        self.v_proj = nn.Linear(d_model, n_kv_heads * self.d_head, bias=False)
        self.o_proj = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, L, D = x.shape

        # Q: (B, L, n_q_heads × d_head) → (B, n_q_heads, L, d_head)
        Q = self.q_proj(x).view(B, L, self.n_q_heads, self.d_head).transpose(1, 2)

        # K/V: (B, L, n_kv_heads × d_head) → (B, n_kv_heads, L, d_head)
        K = self.k_proj(x).view(B, L, self.n_kv_heads, self.d_head).transpose(1, 2)
        V = self.v_proj(x).view(B, L, self.n_kv_heads, self.d_head).transpose(1, 2)

        # 扩展 K/V：把每个 KV 头复制 n_groups 次，凑齐 Q 的头数
        # (B, n_kv_heads, L, d_head) → (B, n_q_heads, L, d_head)
        K = K.repeat_interleave(self.n_groups, dim=1)
        V = V.repeat_interleave(self.n_groups, dim=1)

        # 使用 Flash Attention（PyTorch 2.0+）
        out = F.scaled_dot_product_attention(Q, K, V, is_causal=True)

        # 合并多头输出
        out = out.transpose(1, 2).contiguous().view(B, L, D)
        return self.o_proj(out)


# 对比参数量：GQA vs MHA
d_model = 4096
n_q_heads = 32

mha = GroupedQueryAttention(d_model, n_q_heads, n_kv_heads=32)   # 标准 MHA
gqa = GroupedQueryAttention(d_model, n_q_heads, n_kv_heads=8)    # GQA（LLaMA-3 风格）
mqa = GroupedQueryAttention(d_model, n_q_heads, n_kv_heads=1)    # MQA

def count_params(m): return sum(p.numel() for p in m.parameters())
print(f"MHA 参数量: {count_params(mha)/1e6:.1f}M")
print(f"GQA 参数量: {count_params(gqa)/1e6:.1f}M")
print(f"MQA 参数量: {count_params(mqa)/1e6:.1f}M")

# 验证
x = torch.randn(2, 16, d_model)
out = gqa(x)
print(f"\nGQA 输出形状: {out.shape}")   # (2, 16, 4096)
```

## 三、DeepSeek 的 MLA：进一步压缩 KV Cache

### 3.1 MLA（Multi-Head Latent Attention）的思路

DeepSeek-V2/V3 引入了一种更激进的 KV Cache 压缩方案：不是减少 KV 头数，而是把 K 和 V 压缩成一个低维的"潜在向量"（latent vector），推理时只需缓存这个小向量。

```
标准 GQA：每个 token 缓存 n_kv_heads × d_head × 2（K+V）个值
MLA：每个 token 只缓存 d_c 个值（d_c << n_heads × d_head）
     推理时从 d_c 维还原出完整的 K 和 V
```

在 MLA 中，单个潜在头 $c^{KV}$（维度 $d_c = 4d_h$）从 HBM 加载一次，被复用为每个查询头的 K 和 V，实现了减少数据移动和更高算术强度的目标。

MLA 的 KV Cache 大小可以压缩到 MHA 的约 5-13%，是目前最激进的 KV Cache 压缩方案，但实现复杂度也相对较高。

## 四、稀疏注意力：处理超长序列

### 4.1 初印象：用费曼视角理解

**稀疏注意力的核心思路：不是每个词都需要关注所有其他词**。

在实际语言理解中，一个词通常只需要和几类词交互：
- 附近的词（局部依赖）
- 特定重要的词（语义依赖）
- 句子开头的总结信息（全局信息）

如果只计算"真正需要的"词对的注意力，跳过大量不重要的词对，就能把复杂度从 $O(L^2)$ 降低。

### 4.2 三种主流稀疏注意力模式

```python
import torch

def visualize_attention_patterns(seq_len=16, window_size=3):
    """可视化三种注意力模式"""
    L = seq_len

    # 模式一：局部窗口注意力
    # 每个词只看附近 window_size 个词（像 CNN 的局部感受野）
    local_mask = torch.zeros(L, L, dtype=torch.bool)
    for i in range(L):
        start = max(0, i - window_size)
        end   = min(L, i + 1)   # 因果：只看左侧
        local_mask[i, start:end] = True

    # 模式二：全局 + 局部（Longformer 风格）
    # 特殊的全局 token（如 [CLS]）可以看到所有位置
    # 普通 token 只看局部窗口
    global_local_mask = local_mask.clone()
    global_local_mask[0, :] = True    # [CLS] token 可看所有
    global_local_mask[:, 0] = True    # 所有 token 都看 [CLS]

    # 模式三：滑动窗口 + Strided（BigBird 风格）
    # 局部窗口 + 随机全局连接 + 固定间隔连接
    bigbird_mask = local_mask.clone()
    stride = L // 4
    for i in range(0, L, stride):   # 固定间隔的全局连接
        bigbird_mask[:, i] = True

    return local_mask, global_local_mask, bigbird_mask


local, global_local, bigbird = visualize_attention_patterns(16, 3)

import matplotlib.pyplot as plt
fig, axes = plt.subplots(1, 3, figsize=(12, 4))
for ax, mask, title in zip(axes,
    [local, global_local, bigbird],
    ["局部窗口\n(Local Window)", "全局+局部\n(Longformer)", "稀疏混合\n(BigBird)"]):
    ax.imshow(mask.float().numpy(), cmap="Blues", vmin=0, vmax=1)
    ax.set_title(title, fontsize=11)
    ax.set_xlabel("Key 位置")
    ax.set_ylabel("Query 位置")
plt.suptitle("三种稀疏注意力模式（蓝色=参与计算，白色=跳过）")
plt.tight_layout()
plt.savefig("sparse_attention_patterns.png", dpi=150, bbox_inches="tight")
```

### 4.3 稀疏注意力 vs Flash Attention：解决不同问题

这两个技术容易混淆，但它们解决的是不同层面的问题：

| | Flash Attention | 稀疏注意力 |
|--|----------------|----------|
| 解决的问题 | 内存 IO 带宽瓶颈 | $O(L^2)$ 计算量 |
| 修改计算逻辑？ | 否（结果完全相同）| 是（跳过部分词对，结果近似）|
| 适用序列长度 | 4K-128K | 128K+（配合其他技术）|
| 工程复杂度 | 低（PyTorch 内置）| 高（需自定义 CUDA kernel）|
| 主流 LLM 使用 | 几乎所有 | 部分长上下文专用模型 |

### 4.4 现代 LLM 的做法：Flash Attention + 滑动窗口

目前处理超长序列最实用的方案是：**Flash Attention（解决 IO 效率）+ 适度的滑动窗口（限制需要参与的词对数量）**。

Mistral 采用的滑动窗口注意力（Sliding Window Attention，SWA）就是这个思路：每个词只关注最近 4096 个词，而不是所有历史词。这样 KV Cache 大小固定，计算量线性增长，但远处信息通过多层"接力"传递。

## 六、面试常见问题

**Q1：Flash Attention 为什么比标准注意力快？**

Flash Attention 快不是因为减少了浮点运算量（FLOPs 和标准注意力完全相同），而是因为它减少了 GPU 的 HBM 内存读写量。标准注意力需要把 $L \times L$ 的中间矩阵（注意力分数和权重）反复写入和读出 HBM；Flash Attention 通过分块计算和在线 Softmax，让计算全程在更快的片上 SRAM 中完成，中间结果不落 HBM，把 HBM 读写量从 $O(L^2)$ 降到 $O(L)$。

**Q2：MQA、GQA、MHA 的区别是什么？**

三者都有 H 个查询头（Q head），区别在于 KV 头数量。MHA 有 H 个 KV 头（每头独立），质量最好但 KV Cache 最大。MQA 只有 1 个 KV 头（所有查询头共享），KV Cache 最小但质量略差。GQA 是中间方案，有 G 个 KV 头（$1 < G < H$），G 个查询头共享 1 个 KV 头，在质量和 KV Cache 大小之间取得平衡，是当前主流 LLM 的选择。

**Q3：GQA 怎么节省 KV Cache？节省了多少？**

GQA 通过减少 KV 头数来节省显存。标准 MHA 的 KV Cache 大小正比于头数 $H$，GQA 的正比于 $G$（KV 头数）。LLaMA-3-8B 有 32 个 Q 头但只有 8 个 KV 头，KV Cache 节省了 75%。序列越长，节省效果越明显——处理 128K tokens 的长文本时，MHA 需要约 64 GB 的 KV Cache，GQA 只需约 16 GB。

**Q4：Flash Attention 和稀疏注意力的区别是什么？**

它们解决的是不同层面的问题。Flash Attention 改变的是计算的内存访问方式，不改变计算结果（结果和标准注意力完全一致），解决的是 IO 带宽瓶颈。稀疏注意力改变的是计算哪些词对（跳过"不重要"的词对），计算量从 $O(L^2)$ 降低（但结果是近似的），解决的是超长序列的计算量问题。实际工程中，两者经常配合使用。

**Q5：为什么推理时的 KV Cache 会随序列长度增长？**

自回归推理是逐词生成的——生成第 $t$ 个词时，需要第 $1$ 到 $t-1$ 个词的 K 和 V（用于 Attention 计算）。与其每次重新计算所有之前 token 的 K/V，不如把它们缓存起来（KV Cache），每次只计算新 token 的 K/V 并追加到缓存。所以序列越长，KV Cache 越大；层数越多、头数越多，KV Cache 也越大。GQA 通过减少 KV 头数，直接压缩每个 token 需要缓存的数据量。
