## 零、开篇：为什么工程师必须会这道"心算题"

设计大模型系统时，你会反复遇到这类问题：

- "我们有 8 张 80GB A100，能训练多大的模型？"

- "把模型从 7B 扩到 13B，显存需求怎么变？"

本章给你一套简洁的估算框架，掌握后可以在 30 秒内对任何配置的 Transformer 做出合理评估。

## 一、参数量估算

### 1.1 初印象：用费曼视角理解

**参数量就是模型里"可学习数字"的总数量**。

每一个权重矩阵的参数量 = 矩阵行数 × 矩阵列数。把模型里所有矩阵的参数量加起来，就是总参数量。

### 1.2 每层的参数构成

一个标准的 Transformer Decoder 层（不含偏置，现代 LLM 普遍无 bias）有这些组件：

**Multi-Head Attention（4 个矩阵）**：

| 矩阵 | 形状 | 参数量 |
|------|------|--------|
| $W_Q$ | $d \times d$ | $d^2$ |
| $W_K$ | $d \times d$ | $d^2$ |
| $W_V$ | $d \times d$ | $d^2$ |
| $W_O$（输出投影）| $d \times d$ | $d^2$ |
| **合计** | | $4d^2$ |

**FFN（SwiGLU，3 个矩阵）**：

| 矩阵 | 形状 | 参数量 |
|------|------|--------|
| gate_proj | $d \times d_{ff}$ | $d \cdot d_{ff}$ |
| up_proj | $d \times d_{ff}$ | $d \cdot d_{ff}$ |
| down_proj | $d_{ff} \times d$ | $d_{ff} \cdot d$ |
| **合计** | | $3d \cdot d_{ff}$ |

其中 $d_{ff} \approx \frac{8}{3}d$（SwiGLU 的标准配置），所以 FFN 参数 $\approx 8d^2$。

**RMSNorm（两个，可忽略）**：各 $d$ 个参数，合计 $2d$，相比 $d^2$ 级别的矩阵可以忽略。

**每层合计**：

$$\text{每层参数} \approx 4d^2 + 8d^2 = 12d^2$$

**全模型参数**：

$$\boxed{N \approx 12 \cdot L \cdot d^2 + V \cdot d}$$

其中 $L$ 是层数，$d$ 是隐层维度，$V$ 是词表大小（Weight Tying 时 LM Head 不额外计算）。

### 1.3 验证主流模型

```python
def estimate_params(n_layers, d_model, vocab_size,
                    ffn_ratio=8/3, weight_tying=True):
    """
    估算 Decoder-only Transformer 参数量。
    ffn_ratio: FFN 扩展比（SwiGLU 约 8/3，标准 FFN 为 4）
    """
    d_ff = int(d_model * ffn_ratio)
    d_ff = ((d_ff + 63) // 64) * 64   # 对齐 64 的倍数

    # 每层：4个 Attention 矩阵 + 3个 FFN 矩阵
    attn_per_layer = 4 * d_model * d_model
    ffn_per_layer  = 3 * d_model * d_ff
    per_layer      = attn_per_layer + ffn_per_layer

    transformer_params = per_layer * n_layers
    embedding_params   = vocab_size * d_model
    lm_head_params     = 0 if weight_tying else vocab_size * d_model

    total = transformer_params + embedding_params + lm_head_params

    print(f"L={n_layers}, d={d_model}, V={vocab_size:,}")
    print(f"  Transformer 层: {transformer_params/1e9:.2f}B  "
          f"（注意力{attn_per_layer*n_layers/transformer_params:.0%} + "
          f"FFN {ffn_per_layer*n_layers/transformer_params:.0%}）")
    print(f"  Embedding:      {embedding_params/1e9:.2f}B")
    print(f"  总参数量:        {total/1e9:.2f}B\n")
    return total

# 验证主流模型（结果应在实际值的 5% 以内）
print("=== 主流模型参数量验证 ===")
estimate_params(32, 4096, 32000)    # LLaMA-3-8B，实际约 8.0B
estimate_params(80, 8192, 32000)    # LLaMA-3-70B，实际约 70.6B
estimate_params(28, 3584, 152064)   # Qwen-2.5-7B，实际约 7.6B
estimate_params(32, 4096, 32000)    # Mistral-7B，实际约 7.2B
```

### 1.4 口算技巧

记住这个近似：

$$N \approx 12Ld^2 \quad \text{（忽略 Embedding，适用于大模型）}$$

**口算验证**：LLaMA-3-8B，$L=32$，$d=4096$：

$$N \approx 12 \times 32 \times 4096^2 = 12 \times 32 \times 16.8M \approx 6.4B$$

加上 Embedding（$32000 \times 4096 \approx 0.13B$）≈ 6.5B，实际是 8B（误差因 SwiGLU 的精确 $d_{ff}$ 取整）。数量级是对的，实际工程中这个精度完全够用。

## 二、显存估算

### 2.1 初印象：用费曼视角理解

**显存就是 GPU 的"工作台"，所有正在使用的数据都必须放在上面**。

推理和训练的显存需求差别巨大：
- 推理只需要放模型权重和少量中间计算结果
- 训练还需要放梯度、优化器状态（Adam 的 m 和 v），以及反向传播需要的中间激活值

### 2.2 数据类型与字节数

| 数据类型 | 每个参数占用字节 | 常见用途 |
|---------|----------------|---------|
| FP32 | 4 bytes | 优化器状态、精确计算 |
| BF16 | 2 bytes | 训练时的参数和梯度 |
| FP16 | 2 bytes | 推理、旧 GPU 训练 |
| INT8 | 1 byte | 量化推理 |
| INT4 | 0.5 bytes | 激进量化推理 |

### 2.3 推理显存

推理时的显存主要分两部分：

**模型权重**：

$$\text{权重显存} = N \times \text{bytes\_per\_param}$$

- 7B 模型，BF16：$7 \times 10^9 \times 2 \approx 14 \text{ GB}$
- 7B 模型，INT4 量化：$7 \times 10^9 \times 0.5 \approx 3.5 \text{ GB}$

**KV Cache**（推理时随序列增长的动态显存）：

KV Cache 存储了已处理 token 的 Key 和 Value 矩阵，避免重复计算：

$$\text{KV Cache} = 2 \times L \times d \times L_{seq} \times \text{bytes}$$

其中 2 表示 K 和 V 各一份，$L$ 是层数，$d$ 是模型维度，$L_{seq}$ 是序列长度。

对 LLaMA-3-8B（$L=32$，$d=4096$），BF16，处理 4096 个 token：

$$\text{KV Cache} = 2 \times 32 \times 4096 \times 4096 \times 2 \approx 2.1 \text{ GB}$$

```python
def estimate_inference_memory(n_layers, d_model, vocab_size,
                               seq_len=4096, dtype_bytes=2):
    """
    估算推理时的显存需求。
    dtype_bytes: BF16=2, INT8=1, INT4=0.5
    """
    # 模型权重
    n_params = 12 * n_layers * d_model**2 + vocab_size * d_model
    weight_gb = n_params * dtype_bytes / 1e9

    # KV Cache（使用 BF16 精度）
    kv_cache_gb = 2 * n_layers * d_model * seq_len * 2 / 1e9

    total_gb = weight_gb + kv_cache_gb

    print(f"模型: L={n_layers}, d={d_model}")
    print(f"  权重 ({dtype_bytes}字节/参数): {weight_gb:.1f} GB")
    print(f"  KV Cache (seq={seq_len}): {kv_cache_gb:.1f} GB")
    print(f"  推理总计: {total_gb:.1f} GB\n")

print("=== 推理显存估算 ===")
print("LLaMA-3-8B, BF16, 4096 tokens:")
estimate_inference_memory(32, 4096, 32000, seq_len=4096, dtype_bytes=2)
print("LLaMA-3-8B, INT4 量化, 4096 tokens:")
estimate_inference_memory(32, 4096, 32000, seq_len=4096, dtype_bytes=0.5)
print("LLaMA-3-70B, BF16, 4096 tokens:")
estimate_inference_memory(80, 8192, 32000, seq_len=4096, dtype_bytes=2)
```

### 2.4 训练显存

训练时的显存远大于推理，主要来自四部分：

| 部分 | 每参数字节 | 说明 |
|------|-----------|------|
| BF16 参数 | 2 | 前向/反向用 BF16 |
| BF16 梯度 | 2 | 和参数同精度 |
| FP32 Adam 一阶矩 m | 4 | **必须**用 FP32 保精度 |
| FP32 Adam 二阶矩 v | 4 | **必须**用 FP32 保精度 |
| **合计（参数相关）** | **12** | 每个参数占 12 字节 |

加上激活值（用梯度检查点可以大幅减少）：

$$\text{训练显存} \approx 12N + \text{激活值}$$

```python
def estimate_training_memory(n_layers, d_model, vocab_size,
                              seq_len=2048, batch_size=1,
                              gradient_checkpointing=False):
    """估算训练显存（AdamW，BF16 混合精度）"""
    n_params = 12 * n_layers * d_model**2 + vocab_size * d_model

    # 参数相关（12字节/参数）
    param_memory_gb = n_params * 12 / 1e9

    # 激活值（每层约 12×d×seq_len×batch × 2字节）
    activation_per_layer = 12 * d_model * seq_len * batch_size * 2
    if gradient_checkpointing:
        # 梯度检查点：只保留 sqrt(L) 层的激活，节省约 60-70%
        effective_layers = n_layers ** 0.5
    else:
        effective_layers = n_layers
    activation_gb = activation_per_layer * effective_layers / 1e9

    total_gb = param_memory_gb + activation_gb

    print(f"模型: L={n_layers}, d={d_model}, "
          f"seq={seq_len}, batch={batch_size}")
    print(f"  参数相关 (12B/param): {param_memory_gb:.1f} GB")
    print(f"  激活值 {'(梯度检查点)' if gradient_checkpointing else '(完整)'}:"
          f" {activation_gb:.1f} GB")
    print(f"  训练总计: {total_gb:.1f} GB\n")

print("=== 训练显存估算 ===")
print("LLaMA-3-8B, batch=1, seq=2048, 无梯度检查点:")
estimate_training_memory(32, 4096, 32000, seq_len=2048, batch_size=1)
print("LLaMA-3-8B, batch=1, seq=2048, 有梯度检查点:")
estimate_training_memory(32, 4096, 32000, seq_len=2048, batch_size=1,
                          gradient_checkpointing=True)
```

### 2.5 经验法则速查

| 场景 | 大约需要 |

## 三、FLOPs 估算

### 3.1 初印象：用费曼视角理解

**FLOPs（浮点运算次数）衡量模型的计算量**。知道 FLOPs 加上 GPU 的理论算力，就能估算训练需要多少时间。

最常用的经验公式来自 DeepMind 的 Chinchilla 论文：

$$\boxed{\text{训练总 FLOPs} \approx 6 \times N \times D}$$

其中 $N$ 是模型参数量（不含 Embedding），$D$ 是训练 token 数。

**这个公式的直觉**：
- 前向传播：每个参数参与约 2 次浮点运算（一次乘法 + 一次加法）→ FLOPs ≈ $2N$ per token
- 反向传播：约是前向的 2 倍 → FLOPs ≈ $4N$ per token
- 合计：约 $6N$ per token → 训练 $D$ 个 token 共 $6ND$

### 3.2 训练时间估算

```python
def estimate_training_time(
    n_params: float,       # 非 Embedding 参数量（如 7e9）
    n_tokens: float,       # 训练 token 数（如 1e12）
    n_gpus: int,
    gpu_tflops: float,     # GPU BF16 峰值算力（A100=312, H100=989）
    mfu: float = 0.4,      # 模型浮点利用率（实际/峰值，大规模约 0.35-0.45）
):
    """
    估算训练时间。
    MFU（Model FLOPs Utilization）反映真实算力 vs 峰值算力的比值，
    受通信开销、内存带宽瓶颈、pipeline bubble 等因素影响。
    """
    total_flops   = 6 * n_params * n_tokens
    actual_tflops = gpu_tflops * 1e12 * n_gpus * mfu
    seconds       = total_flops / actual_tflops
    hours         = seconds / 3600
    days          = hours / 24
    gpu_days      = days * n_gpus

    print(f"训练配置:")
    print(f"  参数量: {n_params/1e9:.1f}B，训练 {n_tokens/1e12:.1f}T tokens")
    print(f"  {n_gpus}× GPU（{gpu_tflops} TFLOPS BF16），MFU={mfu}")
    print(f"总 FLOPs: {total_flops:.2e}")
    print(f"估算时间: {hours:.0f}h = {days:.1f}天")
    print(f"GPU-天数: {gpu_days:.0f} GPU-days\n")

print("=== 训练时间估算 ===")
print("LLaMA-3-8B（1T tokens，1024×A100）:")
estimate_training_time(7.5e9, 1e12, n_gpus=1024, gpu_tflops=312, mfu=0.38)

print("LLaMA-3-8B（15T tokens，2048×A100，实际配置）:")
estimate_training_time(7.5e9, 15e12, n_gpus=2048, gpu_tflops=312, mfu=0.38)

print("假设用 256×H100 训练 7B 模型 1T tokens:")
estimate_training_time(7.5e9, 1e12, n_gpus=256, gpu_tflops=989, mfu=0.40)
```

### 3.3 Chinchilla 最优训练数据量

**一个重要的结论**：对于给定的计算预算，如何分配参数量和训练 token 数才是最优的？

DeepMind 在 Chinchilla 论文（2022）中给出了答案：

$$\text{最优 token 数} \approx 20 \times N$$

即：**模型有多少参数，就应该训练大约 20 倍数量的 token**。

```
7B 模型的最优训练数据量：7B × 20 = 140B tokens
70B 模型的最优训练数据量：70B × 20 = 1.4T tokens
```

但实际上很多模型（如 LLaMA-3-8B 训练了 15T tokens）远超 Chinchilla 建议，因为推理效率更重要——宁愿用更多数据训练一个小模型，使其在推理时更快、更便宜。

```python
def chinchilla_optimal(compute_budget_flops: float):
    """
    给定计算预算（FLOPs），计算 Chinchilla 最优的参数量和 token 数。
    结论：N ≈ sqrt(C / 12)，D ≈ 20N
    """
    import math
    # 最优参数量（近似）
    N_optimal = math.sqrt(compute_budget_flops / (6 * 20))
    D_optimal = 20 * N_optimal

    print(f"计算预算: {compute_budget_flops:.2e} FLOPs")
    print(f"Chinchilla 最优参数量: {N_optimal/1e9:.1f}B")
    print(f"Chinchilla 最优 token 数: {D_optimal/1e12:.1f}T tokens")

# 1024 张 A100，训练 30 天的计算预算
a100_flops_per_day = 312e12 * 3600 * 24 * 0.4   # 有效算力
budget = a100_flops_per_day * 1024 * 30           # 30 天的计算
chinchilla_optimal(budget)
```

## 四、综合案例：设计一个训练方案

假设你手里有 **32 张 A100-80G**，想训练一个中文大模型。该怎么决定模型大小和训练 token 数？

```python
def design_training_plan(n_gpus, gpu_memory_gb, gpu_tflops,
                          training_days, mfu=0.38):
    """
    给定计算资源，设计最优训练方案。
    """
    import math

    # 1. 估算可用计算预算
    total_flops = gpu_tflops * 1e12 * n_gpus * mfu * training_days * 86400
    print(f"{'='*50}")
    print(f"资源: {n_gpus}× A100-{gpu_memory_gb}G，{training_days}天")
    print(f"总计算量: {total_flops:.2e} FLOPs")

    # 2. 根据显存估算最大模型尺寸
    # 训练: ~16字节/参数（参数2+梯度2+m4+v4+额外2）
    max_params_from_memory = gpu_memory_gb * 1e9 * n_gpus / 16
    print(f"\n显存限制下最大参数量: {max_params_from_memory/1e9:.1f}B")

    # 3. Chinchilla 最优
    N_opt = math.sqrt(total_flops / (6 * 20))
    D_opt = 20 * N_opt
    print(f"\nChinchilla 最优参数量: {N_opt/1e9:.1f}B")
    print(f"Chinchilla 最优 token: {D_opt/1e12:.1f}T tokens")

    # 4. 实际建议（综合考虑推理效率，可能多训练数据）
    N_actual = min(N_opt, max_params_from_memory * 0.7)  # 留 30% 余量
    D_actual = total_flops / (6 * N_actual)
    print(f"\n实际建议模型大小: {N_actual/1e9:.1f}B")
    print(f"实际建议训练数据: {D_actual/1e12:.1f}T tokens")
    print(f"{'='*50}")

design_training_plan(
    n_gpus=32,
    gpu_memory_gb=80,
    gpu_tflops=312,
    training_days=30,
)
```

## 五、面试常见问题

**Q1：一个 7B 模型，推理时需要多少显存？**

BF16 精度：约 14 GB（7B × 2字节）。加上 KV Cache（4096 token，约 2 GB），共约 16 GB。一张 24GB 的 RTX 4090 可以运行。如果用 INT4 量化，权重只需约 4 GB，KV Cache 约 2 GB，共约 6 GB，中档显卡也能跑。

**Q2：训练一个 7B 模型需要多少显存？为什么比推理多那么多？**

训练约需 80-100 GB（单机需要 A100-80G）。推理只需存模型权重；训练额外需要：BF16 梯度（14 GB）+ FP32 的 Adam 一阶矩 m（28 GB）+ FP32 的 Adam 二阶矩 v（28 GB）+ 激活值（数 GB 到数十 GB）。参数相关部分就达 14+14+28+28=84 GB，是推理的 6 倍。

**Q3：训练 FLOPs 的估算公式是什么？**

$\text{FLOPs} \approx 6ND$，其中 $N$ 是非 Embedding 参数量，$D$ 是训练 token 数。前向约 $2N$ per token（每参数 1 次乘加），反向约 $4N$（梯度计算约是前向的 2 倍），合计 $6N$ per token。

**Q4：什么是 MFU？为什么不直接用 GPU 峰值算力计算训练时间？**

MFU（Model FLOPs Utilization，模型浮点利用率）是实际使用的算力 / GPU 峰值算力。大规模训练中，MFU 通常在 0.3-0.5 之间，主要损耗来自：多 GPU 通信开销（AllReduce 梯度同步）、内存带宽瓶颈（某些操作 memory-bound）、Pipeline 并行的 bubble 时间、数据加载 I/O 等。直接用峰值算力会低估训练时间 2-3 倍。

**Q5：Chinchilla 论文的核心结论是什么？**

对于给定的计算预算，最优的参数量 $N$ 和训练 token 数 $D$ 应该**等比例扩展**，最优比例约为 $D \approx 20N$。这意味着以前很多模型（如 GPT-3）训练数据远远不够，在相同计算量下用更小的模型训练更多数据效果更好。但实际工程中，推理效率的考虑使得"超训"（训练更多 token）越来越普遍。

**Q6：为什么 Adam 的优化器状态必须用 FP32，不能用 BF16？**

Adam 的一阶矩 $m$ 和二阶矩 $v$ 是梯度的累积统计量，需要非常高的数值精度来积累微小的更新。如果用 BF16（精度只有 2-3 位有效十进制数），很多微小的梯度更新会被舍入为零，导致训练不收敛或数值不稳定。FP32 提供约 7 位有效十进制数的精度，足以保证这些统计量的准确性。

**Q7：梯度检查点（Gradient Checkpointing）是什么？有什么代价？**

梯度检查点是一种以计算换显存的技术：前向传播时不保存所有中间激活值，只保存关键检查点；反向传播时遇到需要的激活值就重新计算一遍前向。这能把激活值显存从 $O(L)$（层数）降到 $O(\sqrt{L})$，显存节省约 60-70%，代价是训练速度降低约 20-30%（需要重新计算前向）。内存紧张时这是最有效的显存优化手段之一。
