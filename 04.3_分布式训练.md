## 零、开篇：一张 GPU 装不下，怎么办

训练 70B 的 LLaMA-3 需要约 1120 GB 的显存（仅模型参数加梯度加优化器状态），而当前最强的单张 GPU（H100 SXM）只有 80 GB。

这不是调调参就能解决的问题——数学上就是装不下。

分布式训练就是回答这个问题的：**把一个太大的训练任务，合理地分配到几十张、几百张、甚至几千张 GPU 上，让它们协同完成训练**。

分布式训练有四种主要并行策略，每种解决不同维度的问题：

```
数据并行（DP / DDP）：每张 GPU 存完整模型，不同 GPU 处理不同数据批次
    → 解决"批量太大、一张 GPU 跑太慢"的问题

ZeRO（零冗余优化器）：把优化器状态、梯度、参数分散存到不同 GPU
    → 解决"单张 GPU 显存装不下模型参数+优化器状态"的问题

张量并行（TP）：把单个矩阵（权重）切分到多张 GPU
    → 解决"单层就太大、连权重矩阵都切不进一张 GPU"的问题

流水线并行（PP）：不同 GPU 负责不同的 Transformer 层
    → 解决"层数极多、多层叠加后总参数超过单 GPU"的问题
```

训练大模型通常需要组合使用上述策略（称为"3D 并行"），而 ZeRO 通过在 GPU 之间分区模型状态，进一步消除冗余。

## 一、数据并行（DDP）：最简单的分布式训练

### 1.1 初印象：用费曼视角理解

**DDP（Distributed Data Parallel，分布式数据并行）是"克隆多份完整模型，每份处理不同的数据"**。

想象一个班级的作业批改：一个老师批改 200 份作业太慢。解决方法：复印 200 份"评分标准"（模型），8 个老师每人批改 25 份（数据并行）。批改完后，8 个老师碰头讨论总结（梯度同步），然后同步更新各自的评分标准。

**DDP 的工作流程**：
1. 每张 GPU 持有**完整的模型参数**副本
2. 每张 GPU 处理一个不同的 mini-batch
3. 各 GPU 独立做前向传播和反向传播，得到本地梯度
4. 用 All-Reduce 操作同步所有 GPU 的梯度（取平均）
5. 每张 GPU 用平均梯度更新自己的参数副本

结果：所有 GPU 的参数始终保持一致，每个训练步的等效批量大小 = 单 GPU 批量 × GPU 数量。

### 1.2 PyTorch DDP 代码实战

```python
import os
import torch
import torch.nn as nn
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup_distributed():
    """初始化分布式训练环境"""
    # 从环境变量获取分布式信息（torchrun 自动设置）
    local_rank  = int(os.environ.get("LOCAL_RANK",  0))
    global_rank = int(os.environ.get("RANK",        0))
    world_size  = int(os.environ.get("WORLD_SIZE",  1))

    # 初始化进程组
    dist.init_process_group(
        backend="nccl",    # GPU 训练用 nccl（最快），CPU 用 gloo
        init_method="env://",
    )

    # 设置当前进程使用的 GPU
    torch.cuda.set_device(local_rank)
    device = torch.device(f"cuda:{local_rank}")

    return local_rank, global_rank, world_size, device

def cleanup_distributed():
    dist.destroy_process_group()


class SimpleLM(nn.Module):
    """简化的语言模型（用于演示 DDP）"""
    def __init__(self, vocab_size=1000, d_model=256, n_layers=4):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)
        self.transformer = nn.TransformerDecoder(
            nn.TransformerDecoderLayer(d_model, nhead=4, batch_first=True),
            num_layers=n_layers,
        )
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)
        self.lm_head.weight = self.embedding.weight  # Weight tying

    def forward(self, x):
        emb = self.embedding(x)
        # 简化：用 emb 作为 memory（真实情况不同）
        out = self.transformer(emb, emb)
        return self.lm_head(out)


def train_with_ddp():
    """DDP 训练的完整模板"""
    local_rank, global_rank, world_size, device = setup_distributed()

    # ── 模型：先创建，再包裹 DDP ──
    model = SimpleLM().to(device)
    model = DDP(
        model,
        device_ids=[local_rank],      # 当前进程使用的 GPU
        output_device=local_rank,
        find_unused_parameters=False,  # True 时支持有条件跳过的参数（略有性能开销）
    )

    # ── 数据：用 DistributedSampler 保证不同进程处理不同数据 ──
    # dataset = YourDataset(...)
    # sampler = DistributedSampler(
    #     dataset,
    #     num_replicas=world_size,
    #     rank=global_rank,
    #     shuffle=True,
    # )
    # dataloader = DataLoader(dataset, batch_size=32, sampler=sampler)

    # ── 优化器（每个进程独立，但梯度同步保证参数一致）──
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    # ── 训练循环 ──
    model.train()
    for epoch in range(3):
        # 每个 epoch 开始时重新打乱数据（保证 epoch 间的随机性）
        # sampler.set_epoch(epoch)

        # 模拟一个 batch
        batch = torch.randint(0, 1000, (8, 64)).to(device)  # (B, L)
        labels = batch.clone()

        # 前向传播
        logits = model(batch)   # (B, L, vocab_size)

        # 计算 loss（预测下一个 token）
        loss = nn.functional.cross_entropy(
            logits[:, :-1, :].contiguous().view(-1, 1000),
            labels[:, 1:].contiguous().view(-1),
        )

        # 反向传播（DDP 自动在 backward 时 All-Reduce 梯度）
        optimizer.zero_grad()
        loss.backward()

        # 梯度裁剪
        nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        optimizer.step()

        # 只在主进程打印 log（避免重复输出）
        if global_rank == 0:
            print(f"Epoch {epoch}, Loss: {loss.item():.4f}")

    cleanup_distributed()


# 启动命令（不要直接 python 运行，用 torchrun）：
# torchrun --nproc_per_node=4 train.py
# 或者

#          --master_addr=HOST --master_port=29500 train.py
```

### 1.3 DDP 的通信：All-Reduce

DDP 的核心通信操作是 **All-Reduce**：所有进程同时参与，把每个进程的梯度相加（或求平均），并把结果同步给所有进程。

PyTorch DDP 的一个重要优化：**梯度的 All-Reduce 和反向传播重叠执行**——不是等整个反向传播完成才通信，而是某层的梯度计算完毕就立刻开始 All-Reduce，这样通信和计算并行进行，减少了等待时间。

## 二、ZeRO：显存优化的核心技术

### 2.1 初印象：用费曼视角理解

**ZeRO（Zero Redundancy Optimizer，零冗余优化器）解决的是 DDP 的显存浪费问题**。

DDP 的问题：每张 GPU 都存了完整的模型参数、梯度、优化器状态（Adam 的 m 和 v）。这对大模型来说是巨大的浪费——不同 GPU 存的是完全相同的副本。

ZeRO 的思路：**每张 GPU 只存 1/N 的优化器状态/梯度/参数，需要时从其他 GPU 获取**。

和 DDP 不同——每个 GPU 存完整模型，ZeRO 把模型状态分区到多个 GPU，每个 GPU 负责广播自己持有的那份给其他 GPU。这本质上仍然是数据并行，因为每层的张量计算不在 GPU 之间拆分。

### 2.2 ZeRO 的三个阶段

ZeRO 分三个阶段，显存越来越省，但通信量也越来越大：

| 阶段 | 分片内容 | 显存节省 | 额外通信 |
|------|---------|---------|---------|
| ZeRO-1 | 优化器状态（m, v） | 约 4× | 少量（更新后 All-Reduce 参数）|
| ZeRO-2 | 优化器状态 + 梯度 | 约 8× | 中等 |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 约 16-64× | 较多（每层前向前需 All-Gather 参数）|

```python
# ZeRO 各阶段的显存计算（以 7B 模型、AdamW BF16 为例）
def zero_memory_breakdown(n_params: float, n_gpus: int, zero_stage: int):
    """估算 ZeRO 各阶段的单 GPU 显存需求"""
    bytes_per_param = 2   # BF16 参数

    # 各组件的字节大小（全精度优化器状态）
    params_bytes  = n_params * 2     # BF16 参数
    grads_bytes   = n_params * 2     # BF16 梯度
    opt_m_bytes   = n_params * 4     # FP32 Adam m
    opt_v_bytes   = n_params * 4     # FP32 Adam v

    total_bytes   = params_bytes + grads_bytes + opt_m_bytes + opt_v_bytes

    if zero_stage == 0:  # 标准 DDP，无分片
        per_gpu = total_bytes
    elif zero_stage == 1:  # 只分片优化器状态
        per_gpu = params_bytes + grads_bytes + (opt_m_bytes + opt_v_bytes) / n_gpus
    elif zero_stage == 2:  # 分片优化器状态 + 梯度
        per_gpu = params_bytes + (grads_bytes + opt_m_bytes + opt_v_bytes) / n_gpus
    elif zero_stage == 3:  # 全部分片
        per_gpu = total_bytes / n_gpus

    print(f"ZeRO-{zero_stage}，{n_gpus} GPU：单 GPU 约需 {per_gpu/1e9:.1f} GB")
    return per_gpu

n_params = 7e9   # 7B 模型
n_gpus = 8

print(f"7B 模型，AdamW BF16 训练，{n_gpus} 张 GPU：")
for stage in [0, 1, 2, 3]:
    zero_memory_breakdown(n_params, n_gpus, stage)
```

## 三、张量并行：切分矩阵运算

### 3.1 初印象：用费曼视角理解

**张量并行（Tensor Parallelism）是"把一个大矩阵乘法切分到多张 GPU 上同时计算"**。

ZeRO-3 已经能把参数分散存储，但每次前向传播时还是需要把参数 All-Gather 到单张 GPU 上计算。当单层的权重矩阵本身就超过一张 GPU 的显存时，ZeRO 也没办法——你无法在内存不足的情况下做矩阵乘法。

张量并行直接切分矩阵乘法：

```
标准矩阵乘法：Y = XW
X: (B, L, d_model)，W: (d_model, d_ff)

列并行（把 W 按列切分到 2 张 GPU）：
  GPU 0: Y0 = X @ W[:, :d_ff//2]   → Y0: (B, L, d_ff//2)
  GPU 1: Y1 = X @ W[:, d_ff//2:]   → Y1: (B, L, d_ff//2)
  合并：Y = [Y0, Y1]                 → Y: (B, L, d_ff)

行并行（把 W 按行切分到 2 张 GPU）：
  GPU 0: Y0 = X[:, :d_model//2] @ W[:d_model//2, :]
  GPU 1: Y1 = X[:, d_model//2:] @ W[d_model//2:, :]
  合并：Y = Y0 + Y1  (All-Reduce)
```

### 3.2 Transformer 中的张量并行（Megatron-LM 风格）

Megatron-LM 是 NVIDIA 开发的参考实现，采用的方式是在 Transformer 层中对自注意力和前馈网络层进行分区，同时最小化通信开销。

```python
import torch
import torch.nn as nn
import torch.distributed as dist

class ColumnParallelLinear(nn.Module):
    """
    列并行线性层（Megatron-LM 风格）。
    把输出维度按 tensor_parallel_size 切分，每张 GPU 持有 1/tp 份输出。
    适合 FFN 第一层（扩展维度）和 MHA 的 QKV 投影。
    """
    def __init__(self, in_features: int, out_features: int, tp_size: int):
        super().__init__()
        assert out_features % tp_size == 0
        self.tp_size = tp_size
        self.tp_rank = dist.get_rank() % tp_size if dist.is_initialized() else 0
        # 每张 GPU 只存 1/tp 的权重
        self.weight = nn.Parameter(
            torch.randn(out_features // tp_size, in_features)
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # 每张 GPU 只计算自己那份输出（无需通信）
        local_out = x @ self.weight.T   # (B, L, out_features/tp)
        return local_out


class RowParallelLinear(nn.Module):
    """
    行并行线性层。
    把输入维度按 tensor_parallel_size 切分，需要 All-Reduce 汇总。
    适合 FFN 第二层（压缩维度）和 MHA 的输出投影。
    """
    def __init__(self, in_features: int, out_features: int, tp_size: int):
        super().__init__()
        assert in_features % tp_size == 0
        self.tp_size = tp_size
        # 每张 GPU 只存 1/tp 的权重
        self.weight = nn.Parameter(
            torch.randn(out_features, in_features // tp_size)
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # 每张 GPU 计算自己那份输入的贡献
        local_out = x @ self.weight.T   # (B, L, out_features)，但是部分和

        # All-Reduce：把所有 GPU 的部分和加起来得到完整输出
        if dist.is_initialized() and self.tp_size > 1:
            dist.all_reduce(local_out)   # local_out 变为完整的输出

        return local_out


# 张量并行 FFN 示意（Megatron-LM 风格）
class TensorParallelFFN(nn.Module):
    """
    FFN 的张量并行版本：
    第一个线性层用列并行（输出分到各 GPU），
    第二个线性层用行并行（输入分布在各 GPU，输出 All-Reduce 合并）。
    这样整个 FFN 只需要一次 All-Reduce 通信。
    """
    def __init__(self, d_model: int, d_ff: int, tp_size: int):
        super().__init__()
        self.fc1 = ColumnParallelLinear(d_model, d_ff, tp_size)   # 列并行
        self.act = nn.GELU()
        self.fc2 = RowParallelLinear(d_ff, d_model, tp_size)       # 行并行

    def forward(self, x):
        x = self.fc1(x)    # 各 GPU 独立计算，无通信
        x = self.act(x)
        x = self.fc2(x)    # 各 GPU 计算后 All-Reduce
        return x
```

### 3.3 张量并行的使用场景

张量并行要求 GPU 之间有**高带宽互联**（NVLink，500+ GB/s），因为每一层的前向和反向传播都需要通信。因此：

- **张量并行适合节点内（同一机器内多张 GPU）**：NVLink 带宽高，通信延迟低
- **流水线/数据并行适合跨节点**：跨机器的 InfiniBand 带宽（100-400 GB/s）低于 NVLink

为充分利用带宽，张量并行通常涉及最高的通信开销，而数据并行和流水线并行需要相对较少的通信。因此，通常的做法是在单台服务器内（有高带宽互联）应用张量并行，而在跨服务器（通信带宽受限）时使用数据并行和流水线并行。

## 四、流水线并行：层间切分

### 4.1 初印象：用费曼视角理解

**流水线并行（Pipeline Parallelism）是"不同 GPU 负责不同的层，像工厂流水线一样协作"**。

把 LLaMA-3-70B 的 80 层 Transformer 分成 4 段，每段 20 层，分配到 4 张 GPU：

```
GPU 0：第 1-20 层   GPU 1：第 21-40 层
GPU 2：第 41-60 层  GPU 3：第 61-80 层

数据流：
  Batch → GPU 0（第1-20层）→ 激活值传给 GPU 1
         GPU 1（第21-40层）→ 激活值传给 GPU 2
         GPU 2（第41-60层）→ 激活值传给 GPU 3
         GPU 3（第61-80层）→ 输出和损失
```

通信代价小：只需要在 GPU 之间传递激活值（每层的输出），数据量远小于张量并行的矩阵权重传输。

### 4.2 流水线气泡（Pipeline Bubble）问题

**直接流水线的问题**：GPU 0 做完前向传播把激活传给 GPU 1，然后 GPU 0 就空着等，等 GPU 1-3 都做完，反向传播的梯度才能传回来。这段空等时间叫 **Pipeline Bubble（流水线气泡）**，是纯粹的浪费。

**解决方案：微批次（Micro-batching）**

把一个 mini-batch 切成 M 个 micro-batch，让流水线持续保持忙碌：

```
       时间 →
GPU 0: [F1][F2][F3][F4][__][__][B4][B3][B2][B1]
GPU 1: [__][F1][F2][F3][F4][__][B4][B3][B2][__]
GPU 2: [__][__][F1][F2][F3][F4][B4][B3][__][__]
GPU 3: [__][__][__][F1][F2][F3][F4+B4][B3][__][__]

F=前向，B=反向，__=气泡时间
```

气泡比例 = $(p-1) / (M + p - 1)$，其中 $p$ 是流水线阶段数，$M$ 是 micro-batch 数量。$M$ 越大，气泡越小，但内存使用越多（需要存更多 micro-batch 的激活值）。

## 五、3D 并行：组合使用三种并行

### 5.1 三种并行的组合逻辑

模型并行在 LLM 增长过大到单个 GPU 无法容纳时变得必不可少。与数据并行（将整个模型复制到每个设备）不同，模型并行将模型本身分割到多个计算单元上。

实际训练大模型时，三种并行通常组合使用，形成 **3D 并行**：

```python
# 典型的 3D 并行配置（以 LLaMA-3-70B，256 张 H100 为例）
config_3d_parallel = {
    "tensor_parallel_size":   4,    # 节点内 4 张 GPU 做张量并行（利用 NVLink）
    "pipeline_parallel_size": 8,    # 8 个流水线阶段（跨节点）
    "data_parallel_size":     8,    # 8 路数据并行（每个 DP 组有 4×8=32 张 GPU）
    # 总 GPU 数 = 4 × 8 × 8 = 256
}

# 实际部署时的逻辑：
# - 同一机器的 8 张 GPU 分成 2 组：
#   - 每组 4 张做 TP（张量并行，NVLink 高带宽）
# - 不同机器的 8 组做 PP（流水线，InfiniBand 低延迟点对点）
# - 多个这样的 32-GPU 集合做 DP（数据并行，梯度 All-Reduce）
```

## 六、混合精度训练

### 6.1 BF16 vs FP16 vs FP32

分布式训练中，混合精度是标配：

| 精度 | 字节 | 数值范围 | 用途 |
|------|------|---------|------|
| FP32 | 4 | ~1.18×10⁻³⁸ 到 ~3.4×10³⁸ | Adam 优化器状态（精度要求高）|
| BF16 | 2 | ~1.18×10⁻³⁸ 到 ~3.4×10³⁸ | 参数、梯度（大模型标配）|
| FP16 | 2 | ~5.96×10⁻⁸ 到 65504 | 旧 GPU（A100 之前）用，容易溢出 |

**为什么大模型用 BF16 而不是 FP16？**

BF16 和 FP32 有相同的数值范围（指数位相同），只是精度更低（尾数位更少）。FP16 范围更小，大模型的某些激活值容易超出范围导致 NaN。BF16 几乎没有这个问题。

```python
import torch

# 混合精度训练（PyTorch 原生 AMP）
from torch.amp import autocast, GradScaler

model = SimpleLM().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

# BF16 不需要 GradScaler（范围足够大，不会溢出）
# FP16 需要 GradScaler 来防止梯度下溢

def train_step_amp(model, batch, labels, optimizer, use_bf16=True):
    """使用自动混合精度的训练步骤"""
    optimizer.zero_grad()

    dtype = torch.bfloat16 if use_bf16 else torch.float16

    with autocast(device_type="cuda", dtype=dtype):
        logits = model(batch)
        loss = torch.nn.functional.cross_entropy(
            logits[:, :-1].contiguous().view(-1, 1000),
            labels[:, 1:].contiguous().view(-1),
        )

    # BF16：直接 backward
    if use_bf16:
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

    # FP16：需要 GradScaler 防溢出
    else:
        scaler = GradScaler()
        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()

    return loss.item()
```

## 七、面试常见问题

**Q1：DDP 和 ZeRO 的根本区别是什么？**

DDP 每张 GPU 持有完整模型参数副本，通过 All-Reduce 同步梯度；优点是通信简单，缺点是每张 GPU 都要存完整的参数和优化器状态，显存效率低。ZeRO 把优化器状态（ZeRO-1）、梯度（ZeRO-2）、参数（ZeRO-3）分片存到不同 GPU，大幅减少每张 GPU 的显存占用；代价是需要更多的 All-Gather/Reduce-Scatter 通信。本质上 ZeRO 仍然是数据并行，只是消除了冗余存储。

**Q2：张量并行和流水线并行各自适合什么情况？**

张量并行把单个矩阵运算切分到多张 GPU，每次前向/反向传播都需要通信，要求 GPU 之间有高带宽互联（NVLink）；因此适合节点内（同一机器的多张 GPU）使用。流水线并行把不同层分给不同 GPU，通信只发生在层间激活值传递时，数据量小；适合跨节点（不同机器之间），但有流水线气泡问题，需要用 micro-batching 缓解。

**Q3：什么是流水线气泡（Pipeline Bubble）？如何减少？**

流水线气泡是流水线并行中 GPU 空闲等待的时间：GPU 0 前向传播完把激活传给 GPU 1 后，要等待反向传播的梯度传回来，这段时间 GPU 0 是空闲的。减少方法：增大 micro-batch 数量 $M$（气泡比例 $\approx (p-1)/(M+p-1)$，$M$ 越大，气泡越小）；使用更高级的调度策略（1F1B、Interleaved 等）进一步减少气泡。

**Q4：为什么大模型训练用 BF16 而不是 FP16？**

FP16 的数值范围较小（最大约 65504），大模型中某些激活值、梯度或参数容易超出范围导致 NaN（数值溢出），需要 GradScaler 动态调整缩放系数来防止，增加了工程复杂度。BF16 的指数位和 FP32 一样多，数值范围相同（只是精度/尾数位更少），几乎不会出现溢出问题，无需 GradScaler，工程更简单，A100/H100 对 BF16 有硬件加速支持。

**Q5：3D 并行是什么？各维度的通常配置是什么？**

3D 并行是同时使用张量并行（TP）、流水线并行（PP）和数据并行（DP）三种策略。典型配置：TP 在节点内使用（4-8 张 GPU，利用 NVLink 高带宽）；PP 跨节点使用（4-16 个流水线阶段，每个阶段负责几十层，通过 InfiniBand 传递激活值）；DP 在 TP×PP 的"节点组"之间使用（多个组处理不同数据）。总 GPU 数 = TP × PP × DP。LLaMA-3-70B 训练使用了类似的 3D 并行配置，部署在数千张 H100 上。

**Q6：DistributedSampler 的作用是什么？不用会怎样？**

DistributedSampler 确保在 DDP 训练时，不同进程看到的训练数据互不重叠（每个进程只处理 1/N 的数据）。如果不用 DistributedSampler，所有进程会看到相同的数据、计算相同的梯度，All-Reduce 后的结果和单进程训练完全相同——多卡训练完全没有起到加速作用，只是做了重复的工作。
