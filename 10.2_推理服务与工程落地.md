## 零、开篇：从"跑起来"到"用起来"

上一章讲的是如何让单次推理更快——量化、推测解码、KV Cache 优化。但生产环境面临的问题远不止于此：

你的服务同时面对 1000 个用户，他们的请求长度参差不齐，生成的回答长度也不可预知。你需要：高并发（同时处理大量请求）、低延迟（用户等不了太久）、高吞吐（每秒尽可能多地生成 token）、成本可控（GPU 不能空跑）。这四个目标之间存在张力，需要专门设计的推理服务框架来平衡。

## 一、连续批处理：吞吐量的核心突破

### 1.1 朴素批处理的问题

最直觉的批处理方式是"凑成一批再一起处理"：等够 10 个请求，打包成一个 batch，并行运行，处理完输出，再接受下一批。

这有一个根本缺陷：**不同请求的生成长度不同**。有人只生成 5 个 token，有人要生成 500 个。批次中所有请求必须等最慢的那个完成，生成 5 个 token 的请求要空等 500 个 token 的时间。GPU 利用率极低。

传统批处理就像一班飞机——必须等所有乘客登机才能起飞，即使大多数人已经就坐。

### 1.2 连续批处理（Continuous Batching）

连续批处理（也叫迭代级批处理，Iteration-level Batching）是 vLLM 的核心创新之一。

核心思想：**不在"请求"级别批处理，而在"每步生成"级别批处理**——每生成一个 token 之后，立即检查哪些请求已经完成，把它们从批次中移除，同时把等待队列中的新请求加入批次，让批次永远是满的。

```
传统批处理：
  Batch = [请求A(生成5个), 请求B(生成500个)]
  所有请求等B生成完才结束，A的GPU时间大量浪费

连续批处理：
  Step 1: Batch = [A, B, C, D] → 各生成1个token
  Step 5: A生成完毕，移除A，加入等待中的E
  Step 5: Batch = [B, C, D, E] → 各生成1个token
  Step 6: C生成完毕，移除C，加入F
  ...以此类推，批次永远保持最大容量
```

这让 GPU 利用率从朴素方案的 20-40% 提升到接近满负荷，吞吐量提升数倍。这是现代所有生产推理引擎的标准做法，不再是可选的优化。

## 二、主流推理框架选型

### 2.1 三强对比（2026 年现状）

2026 年的基准测试显示，SGLang 和 LMDeploy 是最快的 LLM 推理引擎，在 H100 GPU 上都能达到约 16,200 tokens/秒，vLLM 约 12,500 tokens/秒（约差 29%）。这 29% 的吞吐量差距，在每天百万次请求的规模下，大约是每月 15,000 美元的 GPU 成本差距。

**三个引擎赢在不同场景**：

**vLLM**

发布时间：2023 年，UC Berkeley

核心设计：PagedAttention、连续批处理，Python 优先加上热路径的原生 CUDA kernel。

优势：生态系统最成熟，文档最全，兼容性最好，社区最活跃，大量工具链与之集成。

适用场景：通用生产部署的首选，尤其是需要良好生态支持和多模型灵活切换的团队。

局限：在纯速度上已被 SGLang 和 LMDeploy 超越；大语言模型服务方面，v1 架构的某些设计正在被 v0.6+ 重构解决。

**SGLang**

发布时间：2024 年初，Stanford 和 UC Berkeley

核心设计：RadixAttention（基数树结构的 KV 前缀缓存）、零开销 CPU 调度器、Prefill-Decode 分离。

优势：多轮对话和 RAG 场景下性能领先（RadixAttention 让共享相同前缀的请求复用 KV Cache，大幅减少重复计算）；结构化输出（JSON/函数调用）速度最快，比 vLLM 快 3 倍以上；SGLang 在高并发下比 vLLM 快 13%。

适用场景：多轮对话、RAG（大量请求共享 System Prompt 前缀）、工具调用场景。

**LMDeploy**

发布时间：2023 年，上海 AI Lab（InternLM 团队）

核心设计：TurboMind 引擎，纯 C++ 实现，消除 Python 解释器开销。

优势：量化模型推理速度最快，INT4 推理比 FP16 快 2.4 倍；在 A100 80GB 服务 Llama 3 70B（4bit 量化，100 并发）下吞吐量最高（约 700 tokens/s）；对 InternLM 系列和国产模型支持最好。

适用场景：量化模型部署、延迟极其敏感的生产环境、国内模型服务。

### 2.2 其他值得了解的工具

**TGI（Text Generation Inference，HuggingFace）**：HuggingFace 官方推理服务器，与 HuggingFace 生态无缝集成，配置简单，适合快速实验和原型验证。对于长前缀缓存场景，TGI v3 在某些测试中处理速度是 vLLM 的 3 倍、延迟是 1/13。

**TensorRT-LLM（NVIDIA）**：NVIDIA 官方出品，针对 NVIDIA GPU 做了极致优化（算子融合、FP8 支持、CUDA Graph）。在单一模型、长期运行的生产部署上速度最快，但需要 30 分钟以上的编译时间（冷启动慢）。适合 NVIDIA-only 环境且对速度要求极端苛刻的场景。

### 2.3 选型决策树

```
你在做什么？

原型实验/快速上手？
    → vLLM（文档最多，配置最简单）
    或 TGI（HuggingFace 生态集成）

生产部署，多轮对话或 RAG 场景为主？
    → SGLang（RadixAttention 为此场景优化）

生产部署，大量量化模型服务？
    → LMDeploy（量化推理最快）

生产部署，单一模型长期运行，追求极致速度？
    → TensorRT-LLM（NVIDIA GPU，愿意接受编译等待）

需要最广泛的工具链兼容性？
    → vLLM（几乎所有 MLOps 工具都支持）
```

## 三、生产级 LLM 服务的架构设计

### 3.1 核心架构模式

一个生产级 LLM 服务通常包含以下几个层次：

```
用户请求
    ↓
API 网关（负载均衡、认证、限流、路由）
    ↓
请求队列（处理突发流量）
    ↓
推理服务集群（多个 vLLM/SGLang 实例）
  ├── GPU 实例 1：模型 A（常用小模型）
  ├── GPU 实例 2：模型 A（横向扩展）
  └── GPU 实例 3：模型 B（大模型，成本更高）
    ↓
结果缓存（对完全相同的请求复用结果）
    ↓
监控与日志
```

### 3.2 Prefill-Decode 分离：2025 年的新趋势

传统架构中，同一个 GPU 节点同时处理 Prefill（处理输入，计算密集）和 Decode（逐 token 生成，带宽密集）。这两个阶段的资源需求完全不同，放在同一个节点上会相互干扰。

**Prefill-Decode 分离**是 SGLang 和 NVIDIA 都在推进的架构创新：用专门的 GPU 节点处理 Prefill（需要高算力），用另一组节点处理 Decode（需要高带宽），两组节点专门优化各自的负载。

这在高并发场景下能显著提升整体吞吐量，但架构复杂度也随之提升。是 2025-2026 年大规模推理服务的前沿方向。

### 3.3 Prompt 缓存：减少重复计算

很多 LLM 应用都有一个固定的 System Prompt（几百到几千 token），每次请求都要重新计算它的 KV Cache。对于高并发场景，这是巨大的浪费。

**Prompt Caching（前缀缓存）**：把相同前缀的 KV Cache 存储起来，后续请求命中缓存时直接复用，跳过对这部分内容的重复计算。

Anthropic 的 Prompt Caching API 可以将输入成本降低最多 90%。SGLang 的 RadixAttention 在引擎层面自动实现这一功能，所有共享相同前缀的请求自动复用 KV Cache。

### 3.4 成本控制的实用策略

推理成本的核心公式：**成本 = token 数量 × 每 token 价格**。

输出 token 比输入 token 贵 3-10 倍（几乎所有 API 供应商都是这个比例），因为输出是串行生成的，计算密度更高。

**实用的成本控制策略**：

**模型路由（Model Routing）**：不是所有请求都需要最强的模型。简单的分类、格式化任务用小模型（便宜），复杂推理用大模型（贵）。自动路由系统根据任务复杂度决定用哪个模型，可以把平均成本降低 50-70%。

**输出长度控制**：在 Prompt 中明确要求模型给出简短回答（"请用不超过 200 字回答"），避免不必要的冗长输出。

**批量处理**：对延迟不敏感的任务（批量数据处理、离线分析），用异步批量 API，价格通常比实时 API 便宜 50%。

**KV Cache 复用**：设计 Prompt 时把固定部分（System Prompt、示例）放在开头，确保前缀缓存能命中，降低每次调用的有效计算量。

## 四、监控：你必须知道的关键指标

生产级 LLM 服务必须监控这几个指标：

**TTFT（Time to First Token，首 token 延迟）**：从用户发出请求到收到第一个 token 的时间。主要取决于 Prefill 阶段，影响用户感知的"响应速度"。

**TPOT（Time per Output Token，每 token 生成时间）**：生成每个 token 的平均时间。主要取决于 Decode 阶段效率，影响整体输出速度。

**TPS（Tokens per Second，总吞吐量）**：服务每秒能生成的总 token 数，衡量系统容量。

**GPU 利用率**：如果持续低于 70%，说明服务架构有优化空间（可能是批次太小、或者连续批处理没有正确配置）。

**请求拒绝率**：当并发请求数超过系统容量时，服务会开始拒绝请求（返回 503）。这是扩容的信号。

## 五、面试常见问题

**Q1：什么是连续批处理？为什么它对 LLM 推理这么重要？**

连续批处理（Continuous Batching）在每一步 token 生成后动态调整批次——完成的请求立即移出，等待的请求立即加入，批次永远保持最大容量。相比传统的"整批次完成再换批"方式，消除了不同长度请求相互等待的浪费，让 GPU 利用率从 20-40% 提升到接近满负荷。在高并发场景，这是吞吐量提升最大的单一优化，现在是所有生产推理框架的标准配置。

**Q2：vLLM、SGLang、LMDeploy 的核心区别是什么？如何选择？**

三者各有最优场景：vLLM 生态最成熟，文档最全，兼容性最好，是通用场景的默认选择；SGLang 的 RadixAttention 对多轮对话和 RAG 场景有突出优势（共享前缀的请求复用 KV Cache，减少重复计算），结构化输出速度也最快；LMDeploy 的纯 C++ TurboMind 引擎在量化模型推理上速度最快（INT4 比 FP16 快 2.4 倍），适合量化部署和国产模型服务。2026 年基准测试显示 SGLang 和 LMDeploy 吞吐量约 16,200 tok/s，vLLM 约 12,500 tok/s。

**Q3：Prompt 缓存是什么？什么场景最有价值？**

Prompt 缓存将共享相同前缀（通常是 System Prompt 或 Few-shot 示例）的多个请求的 KV Cache 存储复用，避免对同一前缀内容重复计算。对前缀缓存命中的 token 不收计算费或大幅打折，能降低输入成本最高 90%。最有价值的场景：System Prompt 很长且固定（RAG 的大量文档上下文、复杂的 System Prompt）、多用户共享同一 Prompt 前缀（标准化 SaaS 产品）、多轮对话（历史对话是共享前缀）。

**Q4：生产 LLM 服务应该监控哪些核心指标？**

四个必须监控的指标：TTFT（首 token 延迟，影响用户感知响应速度）；TPOT（每 token 生成时间，影响输出速度）；TPS（每秒总 token 吞吐量，衡量系统容量）；GPU 利用率（持续低于 70% 说明有优化空间，持续接近 100% 且 TTFT 上升说明需要扩容）。次要指标：请求拒绝率（超容量信号）、P99 延迟（极端情况的用户体验）。
