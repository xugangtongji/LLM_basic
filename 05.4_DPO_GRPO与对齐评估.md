## 零、开篇：更简单的对齐方法

PPO 需要四个模型、复杂的采样循环、精心调整的超参数，显存需求动辄 160 GB+。能不能有更简单的方式？

2023 年，DPO（Direct Preference Optimization）给出了一个令人惊喜的答案：**可以不用单独的奖励模型，不用强化学习，只用一个类似 SFT 的简单训练目标，就能从偏好数据中对齐 LLM**。

DPO 从 KL 约束的奖励最大化目标出发，推导出直接在离线偏好数据上的最大似然目标，绕过了显式奖励模型的训练阶段，消除了强化学习优化的需要。

截至 2026 年，DPO 已经成为开源生态中对齐微调的事实标准——不是因为它在每个任务上都最优，而是因为它足够简单、足够稳定、足够容易实现。

## 一、DPO：从 PPO 推导出来的惊人简化

### 1.1 初印象：用费曼视角理解

**DPO 的神奇之处在于：它在数学上等价于 RLHF，但不需要显式的奖励模型和强化学习**。

PPO RLHF 的流程：
```
偏好数据 → 训练奖励模型 → 奖励模型给回答打分 → PPO 更新策略
```

DPO 直接从偏好数据到策略更新：
```
偏好数据 ─────────────────────────────→ 直接更新策略
```

奖励模型"消失"了，但其实它被隐式地编码进了 LLM 本身的参数里。这是怎么做到的？

### 1.2 DPO 的推导：三步数学变换

**第一步：PPO 的最优解**

PPO 优化的目标（带 KL 约束的奖励最大化）：

$$\max_\pi \mathbb{E}_{x \sim \mathcal{D}, y \sim \pi}[r(x,y)] - \beta D_{KL}(\pi \| \pi_{ref})$$

这个优化问题有解析解（闭合形式的最优策略）：

$$\pi^*(y|x) = \frac{1}{Z(x)} \pi_{ref}(y|x) \exp\left(\frac{r(x,y)}{\beta}\right)$$

其中 $Z(x) = \sum_y \pi_{ref}(y|x) \exp(r(x,y)/\beta)$ 是归一化常数。

**第二步：从最优策略反推奖励**

对上式取对数，整理得到：

$$r(x,y) = \beta \log \frac{\pi^*(y|x)}{\pi_{ref}(y|x)} + \beta \log Z(x)$$

**关键洞见**：奖励函数可以用策略模型本身的对数概率比来表示！

**第三步：代入 Bradley-Terry 损失**

把上式代入 Bradley-Terry 偏好损失（回忆 5.2 章）：

$$\mathcal{L}_{BT} = -\mathbb{E}\left[\log \sigma(r(x, y_w) - r(x, y_l))\right]$$

$Z(x)$ 在 $r(x, y_w) - r(x, y_l)$ 中相消（两个 $\beta \log Z(x)$ 相减为零）！

最终得到 **DPO 损失**：

$$\boxed{\mathcal{L}_{DPO} = -\mathbb{E}\left[\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right)\right]}$$

**DPO 损失的直觉**：
- 第一项 $\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)}$：chosen 回答在新策略相对参考策略的"隐式奖励"
- 第二项 $\beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}$：rejected 回答的"隐式奖励"
- 最大化两者之差：让 chosen 的隐式奖励 > rejected 的隐式奖励

```python
import torch
import torch.nn.functional as F

def dpo_loss(
    policy_chosen_logprobs:   torch.Tensor,   # (B,) π_θ(y_w|x) 的对数概率之和
    policy_rejected_logprobs: torch.Tensor,   # (B,) π_θ(y_l|x) 的对数概率之和
    ref_chosen_logprobs:      torch.Tensor,   # (B,) π_ref(y_w|x) 的对数概率之和
    ref_rejected_logprobs:    torch.Tensor,   # (B,) π_ref(y_l|x) 的对数概率之和
    beta: float = 0.1,                        # KL 约束系数（通常 0.05-0.5）
) -> tuple[torch.Tensor, dict]:
    """
    DPO 损失函数。
    logprobs 是序列所有 token 对数概率的均值（或求和）。
    """
    # 计算隐式奖励（对数概率比）
    chosen_rewards   = beta * (policy_chosen_logprobs   - ref_chosen_logprobs)
    rejected_rewards = beta * (policy_rejected_logprobs - ref_rejected_logprobs)

    # DPO 损失 = -log σ(r_chosen - r_rejected)
    reward_diffs = chosen_rewards - rejected_rewards
    loss = -F.logsigmoid(reward_diffs).mean()

    # 统计指标
    accuracy = (reward_diffs > 0).float().mean()
    metrics = {
        "loss":            loss.item(),
        "accuracy":        accuracy.item(),         # chosen 隐式奖励 > rejected 的比例
        "chosen_rewards":  chosen_rewards.mean().item(),
        "rejected_rewards":rejected_rewards.mean().item(),
        "reward_margin":   reward_diffs.mean().item(),
    }
    return loss, metrics


# 测试
torch.manual_seed(42)
B = 8
policy_chosen   = torch.randn(B) * 0.5 - 2.0   # 模拟对数概率（负值）
policy_rejected = torch.randn(B) * 0.5 - 2.5
ref_chosen      = torch.randn(B) * 0.1 - 2.0
ref_rejected    = torch.randn(B) * 0.1 - 2.5

loss, metrics = dpo_loss(policy_chosen, policy_rejected, ref_chosen, ref_rejected, beta=0.1)
print("DPO 训练指标：")
for k, v in metrics.items():
    print(f"  {k}: {v:.4f}")
```

### 1.3 DPO vs PPO：核心对比

DPO 的优化目标等价于带有隐式奖励函数的 Bradley-Terry 模型，这个隐式奖励由策略模型本身参数化。与 RLHF 相比，DPO 已被证明是稳定的、高性能的。

| 维度 | PPO（RLHF）| DPO |
|------|-----------|-----|
| 模型数量 | 4 个（策略/参考/奖励/价值）| 2 个（策略/参考）|
| 奖励模型 | 显式，单独训练 | 隐式，编码在策略模型里 |
| 训练方式 | 在线（每步生成新数据）| 离线（使用固定的偏好数据）|
| 超参数 | 多（clip_eps/kl_coef/vf_coef/γ/λ）| 少（主要是 β）|
| 显存需求 | ~160 GB（7B 模型）| ~30 GB（7B 模型）|
| 探索能力 | 有（能发现 SFT 数据之外的好回答）| 无（受限于静态数据集）|
| 主要场景 | 推理、代码、需要探索的任务 | 对话对齐、风格调整 |

## 三、DPO 变体：KTO、IPO、SimPO

### 3.1 KTO：无需配对数据的偏好学习

DPO 需要 (prompt, chosen, rejected) 配对数据，但很多场景只有单条反馈：

- 用户点了"👍"或"👎"
- 人工标注"好"或"不好"，没有对比对象

**KTO（Kahneman-Tversky Optimization）** 解决了这个问题：不需要配对，只需要知道"这个回答好（desirable）"或"这个回答不好（undesirable）"。

KTO 解决了一个数据问题。DPO 需要成对偏好数据——对每个 prompt，你需要一个"chosen"和一个"rejected"回答。在实际中，大多数产品系统收集的是更简单的信号：对单个回答的点赞或点踩。

```python
def kto_loss(
    policy_logprobs:     torch.Tensor,   # (B,) 策略对每条数据的 log prob
    ref_logprobs:        torch.Tensor,   # (B,) 参考模型的 log prob
    labels:              torch.Tensor,   # (B,) 1=好（desirable），0=不好（undesirable）
    beta: float = 0.1,
    desirable_weight:   float = 1.0,
    undesirable_weight: float = 1.0,
) -> torch.Tensor:
    """
    KTO 损失：不需要配对数据，只需要单条的好/坏标签。
    """
    # 隐式奖励
    implicit_reward = policy_logprobs - ref_logprobs   # (B,)

    # KL 参考项（批内平均，近似真实的 KL）
    kl_ref = (implicit_reward).mean().detach()

    # 分别计算好/坏样本的损失
    desirable_mask   = labels.bool()
    undesirable_mask = ~labels.bool()

    loss = torch.tensor(0.0)

    if desirable_mask.any():
        # 好样本：最大化与 KL 参考的差距（这个回答比平均好多少）
        des_rewards = implicit_reward[desirable_mask] - kl_ref
        des_loss    = desirable_weight * (1 - F.sigmoid(beta * des_rewards)).mean()
        loss = loss + des_loss

    if undesirable_mask.any():
        # 坏样本：最大化 KL 参考与这个回答的差距（这个回答比平均差多少）
        und_rewards = kl_ref - implicit_reward[undesirable_mask]
        und_loss    = undesirable_weight * (1 - F.sigmoid(beta * und_rewards)).mean()
        loss = loss + und_loss

    return loss


# 测试
B = 8
policy_lp = torch.randn(B) * 0.2 - 2.0
ref_lp    = torch.randn(B) * 0.1 - 2.0
labels    = torch.randint(0, 2, (B,))  # 0 或 1

kto_l = kto_loss(policy_lp, ref_lp, labels)
print(f"KTO 损失: {kto_l.item():.4f}")
```

### 3.2 SimPO：去掉参考模型

**SimPO（Simple Preference Optimization）** 进一步简化：完全去掉参考模型，用序列长度归一化的对数概率代替对数概率比：

$$\mathcal{L}_{SimPO} = -\mathbb{E}\left[\log \sigma\left(\frac{\beta}{|y_w|}\log\pi_\theta(y_w|x) - \frac{\beta}{|y_l|}\log\pi_\theta(y_l|x) - \gamma\right)\right]$$

其中 $\gamma$ 是目标奖励差距（target reward margin）。

SimPO 的优势：不需要参考模型（省一半显存），自动解决了 DPO 的长度偏见问题（除以序列长度）。

### 3.3 DPO 家族总览

| 方法 | 需要配对数据 | 需要参考模型 | 需要奖励模型 | 主要优势 |
|------|------------|------------|------------|---------|
| PPO | 否（生成）| 是 | 是 | 在线探索，推理任务强 |
| DPO | 是 | 是 | 否 | 简单稳定，工程友好 |
| IPO | 是 | 是 | 否 | 理论更严格（防过拟合）|
| KTO | 否（单条）| 是 | 否 | 适合单标签反馈 |
| SimPO | 是 | 否 | 否 | 最简，自动处理长度偏见 |
| GRPO | 否（生成）| 是 | 规则 | 推理任务，无需 Critic |

## 四、GRPO：DeepSeek-R1 的核心算法

### 4.1 初印象：用费曼视角理解

**GRPO（Group Relative Policy Optimization）是"让同学互相比较，而不是对着标准答案"的训练方式**。

PPO 用价值模型（Critic）估计"平均水平是多少"，然后计算"这个回答比平均好/差多少"（优势函数）。

GRPO 的做法更直接：对同一个问题，让模型**生成 G 个不同的回答**，用这 G 个回答中的**相对奖励**来估计优势——不需要 Critic，只需要组内比较。

### 4.2 GRPO 算法

对每个 prompt $x$，从当前策略 $\pi_\theta$ 采样 $G$ 个回答 $\{y_1, y_2, \ldots, y_G\}$：

**第一步：计算每个回答的奖励**

$$r_i = r(x, y_i), \quad i = 1, \ldots, G$$

**第二步：组内归一化，得到优势估计**

$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_j\})}{\text{std}(\{r_j\}) + \epsilon}$$

用组内均值替代价值模型的估计——省掉了整个 Critic 网络！

**第三步：策略梯度更新（带 KL 约束）**

$$\mathcal{L}_{GRPO} = -\mathbb{E}_{i}\left[\min\left(r_t^{(i)} \hat{A}_i, \text{clip}(r_t^{(i)}, 1-\epsilon, 1+\epsilon)\hat{A}_i\right) - \beta D_{KL}(\pi_\theta \| \pi_{ref})\right]$$

```python
import torch
import torch.nn.functional as F

def grpo_loss(
    policy_logprobs_new: torch.Tensor,   # (B*G, L) 新策略的 log prob（每个回答逐 token）
    policy_logprobs_old: torch.Tensor,   # (B*G, L) 生成时策略的 log prob
    ref_logprobs:        torch.Tensor,   # (B*G, L) 参考模型的 log prob
    rewards:             torch.Tensor,   # (B*G,) 每个回答的奖励
    mask:                torch.Tensor,   # (B*G, L) 有效 token 掩码
    G: int = 8,                          # 每个 prompt 生成多少个回答
    clip_eps: float = 0.2,
    beta: float = 0.01,                  # KL 惩罚系数（GRPO 通常较小）
) -> torch.Tensor:
    """
    GRPO 损失函数。
    B*G 个样本来自 B 个 prompt，每个 prompt 有 G 个回答。
    """
    BG, L = policy_logprobs_new.shape
    B = BG // G

    # 组内归一化奖励，得到优势
    rewards_grouped = rewards.view(B, G)   # (B, G)
    mean_r = rewards_grouped.mean(dim=1, keepdim=True)   # (B, 1)
    std_r  = rewards_grouped.std(dim=1, keepdim=True)    # (B, 1)
    advantages = ((rewards_grouped - mean_r) / (std_r + 1e-8)).view(BG)   # (BG,)

    # 把 token 级优势广播（同一回答的所有 token 共享同一优势）
    advantages_per_token = advantages.unsqueeze(1).expand(-1, L)   # (BG, L)

    # PPO 裁剪更新
    log_ratio = policy_logprobs_new - policy_logprobs_old
    ratio = torch.exp(log_ratio)

    unclipped = ratio * advantages_per_token
    clipped   = torch.clamp(ratio, 1 - clip_eps, 1 + clip_eps) * advantages_per_token
    pg_loss   = -torch.min(unclipped, clipped)

    # 逐 token KL 惩罚
    kl_per_token = policy_logprobs_new - ref_logprobs

    # 总损失（只在有效 token 上）
    loss = ((pg_loss + beta * kl_per_token) * mask).sum() / mask.sum()
    return loss


# 测试
B, G, L = 4, 8, 16
BG = B * G

new_lp  = torch.randn(BG, L) * 0.1 - 2.0
old_lp  = torch.randn(BG, L) * 0.1 - 2.0
ref_lp  = torch.randn(BG, L) * 0.1 - 2.0
rewards = torch.randn(BG)
mask    = torch.ones(BG, L)

loss = grpo_loss(new_lp, old_lp, ref_lp, rewards, mask, G=G)
print(f"GRPO 损失: {loss.item():.4f}")
```

### 4.3 GRPO 的奖励函数设计（DeepSeek-R1 风格）

GRPO 的另一个关键：**对于可验证任务（数学、代码），可以用规则直接判断正确性，完全不需要奖励模型**。

基于规则的奖励信号（如正确/错误）加上大规模 GRPO，可以训练出强大的推理能力，而无需人工标注大量"思维过程"数据。

```python
def math_reward(response: str, ground_truth: str) -> float:
    """
    数学题的规则奖励（DeepSeek-R1 风格）。
    完全正确：1.0
    格式正确但答案错：0.1（鼓励正确格式）
    完全错误：0.0
    """
    # 提取最终答案（假设用 \boxed{} 包裹）
    import re
    match = re.search(r'\\boxed\{([^}]+)\}', response)
    if not match:
        return 0.0   # 没有 \boxed{} 格式

    predicted = match.group(1).strip()
    expected  = ground_truth.strip()

    if predicted == expected:
        return 1.0   # 完全正确
    else:
        return 0.1   # 格式对但答案错


def code_reward(code: str, test_cases: list[tuple]) -> float:
    """
    代码的规则奖励：能通过多少测试用例。
    """
    try:
```

**评估对齐效果是个难题**——"回答好不好"本质上是主观的，没有唯一正确答案。

## 六、第五章总结：从预训练到 Chat 模型的完整路径

```
预训练 Base Model（第四章）
    ↓ 学到了语言规律和世界知识
    
5.1 SFT 指令微调
    ↓ 学会了"按格式回答"，从原始模型变成 Instruct Model
    数据：指令-回答对（数万到数百万）
    目标：只对回答部分的 token 计算 Loss
    工具：LoRA / QLoRA（资源受限）或全量微调

5.2 奖励模型训练（可选，DPO 不需要）
    ↓ 学会了"什么是好回答"
    数据：偏好配对（chosen vs rejected）
    目标：Bradley-Terry 损失（chosen 分 > rejected 分）

5.3 RLHF / PPO（重型对齐，适合探索型任务）
    ↓ 通过在线探索，超越 SFT 数据质量上限
    需要：四个模型，复杂调参
    适合：数学推理（GRPO 版本）、代码生成、需要探索的任务

5.4 DPO / SimPO（轻量对齐，大多数场景的首选）
    ↓ 从偏好数据中直接对齐，无需 RL
    需要：两个模型（策略 + 参考），简单
    适合：对话风格调整、安全对齐、指令跟随提升

→ 最终产品：Chat Model（可以正常对话的 AI 助手）
```

## 七、面试常见问题

**Q1：DPO 的损失函数是什么？和 PPO 有什么数学关系？**

DPO 损失：$\mathcal{L}_{DPO} = -\mathbb{E}[\log\sigma(\beta\log\frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta\log\frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)})]$。它从 PPO 的 KL 约束奖励最大化目标推导而来——PPO 最优策略可以用 $\pi^*(y|x) \propto \pi_{ref}(y|x)\exp(r(x,y)/\beta)$ 表示，由此可以把奖励表达为策略和参考模型的对数概率比，代入 Bradley-Terry 损失后 $Z(x)$ 项自动消去，得到 DPO 损失。

**Q2：DPO 相比 PPO 有哪些优势？有什么局限？**

优势：只需两个模型（省 PPO 的价值和奖励模型），不需要在线采样，超参数更少，工程简单，显存需求约是 PPO 的 1/4。局限：只能用固定的离线偏好数据，无法在线探索——这限制了 DPO 在需要探索的任务（如数学推理）上的上限；DPO 的性能上限受偏好数据质量限制，不能"发现"数据集外的好答案。

**Q3：GRPO 和 PPO 的根本区别是什么？**

PPO 用价值模型（Critic）估计每个状态的预期奖励，计算优势函数（需要两个可训练模型）。GRPO 去掉了价值模型，改为对同一 prompt 生成 G 个回答，用组内奖励的均值和标准差做归一化得到优势估计，用组内相对排名代替绝对价值估计。这减少了一个需要训练的大模型，大幅降低显存和工程复杂度，DeepSeek-R1 就是用这种方法训练的。

**Q4：为什么 GRPO 适合数学和代码，而不是一般对话任务？**

GRPO 的奖励通常来自规则（数学题答案正确与否，代码能否通过测试），这类规则奖励客观、精确、不需要奖励模型。一般对话任务（"帮我写封邮件"）没有明确的对错，无法用规则奖励，需要人类偏好或奖励模型的主观判断，这时 DPO 或基于奖励模型的 PPO 更合适。

**Q5：MT-Bench 和 Chatbot Arena 有什么区别？哪个更可信？**

MT-Bench 用预设的 80 道题，用 GPT-4 打分（1-10），覆盖 8 个维度，结果稳定可复现，适合快速评估。Chatbot Arena 用真实用户的真实问题，真实人类投票（不是 LLM 判断），盲评设计，统计样本量极大（数百万对战），但需要持续时间获得显著性结果。Chatbot Arena 更可信（不受预设题库限制，没有 LLM-as-Judge 偏差），是目前公认最权威的综合排名。

**Q6：KTO 相比 DPO 的核心区别是什么？什么场景用 KTO？**

DPO 需要配对数据（每条 prompt 同时有 chosen 和 rejected 回答），KTO 只需要单条数据加好/坏标签。KTO 基于 Kahneman-Tversky 前景理论，分别优化好样本（让模型的隐式奖励高于 KL 参考）和坏样本（让模型的隐式奖励低于 KL 参考）。场景：当你只有单条反馈（用户点赞/点踩、人工标注好/坏），而没有配对对比数据时，用 KTO；有配对数据时用 DPO 或 SimPO。
