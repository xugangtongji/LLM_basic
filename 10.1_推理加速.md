## 零、开篇：训练只做一次，推理永远在跑

一个容易被忽视的事实：**超过 90% 的 LLM 运营成本来自推理，而不是训练**。训练是一次性的，完成后就结束；推理则在每一次用户请求中发生，每天数亿次，永无止境。

这意味着：推理效率提升 10 倍，等于把运营成本砍掉 90%，或者以同样的成本服务 10 倍的用户。

对于产品团队来说，推理优化不是"锦上添花"，而是商业模式是否可行的核心变量。2024 年以来，API 价格已经下降了约 80%，但总的推理支出还在上升——因为 token 用量增长的速度远快于价格下降的速度。理解推理成本结构是控制成本的第一步。

## 一、推理的真实瓶颈：内存带宽，不是算力

### 1.1 两个阶段：Prefill 和 Decode

LLM 推理分两个阶段，性质截然不同：

**Prefill（预填充）阶段**：处理用户输入的 Prompt。把所有输入 token 一次性并行处理，计算它们的 KV Cache。这个阶段是**计算密集型**（compute-bound）——GPU 的矩阵运算核满负荷运行，算力是瓶颈。

**Decode（解码）阶段**：逐 token 生成输出。每次只生成一个 token，需要读取所有之前 token 的 KV Cache 来计算注意力。这个阶段是**内存带宽密集型**（memory-bound）——GPU 要不断地从显存里读取庞大的 KV Cache 和模型权重，而实际的矩阵运算量很小，内存读写速度是瓶颈。

**这个区别非常关键**：大多数对话场景，生成的 token 远多于输入的 token，所以推理大部分时间花在 Decode 阶段，主要瓶颈是内存带宽，而不是 GPU 的浮点运算峰值。

### 1.2 为什么单 token 生成时 GPU 利用率低

想象 H100 GPU 有 3958 TFLOPS 的算力（每秒 3958 万亿次浮点运算），但内存带宽是 3.35 TB/s。

生成一个 token 时，需要把整个模型（比如 7B，BF16 约 14GB）从显存读入计算核心，然后做很少量的矩阵运算。这 14GB 在 3.35 TB/s 的带宽下需要约 4ms 来读取，而实际运算可能只需要 0.1ms。GPU 计算核心的大部分时间在等待数据，实际算力利用率可能只有 5-10%。

这就是为什么 LLM 推理"感觉"很慢——不是因为 GPU 不够强，而是因为内存带宽跟不上算力，GPU 大部分时间在等数据。

所有推理优化技术，本质上都在解决这个核心矛盾。

## 二、量化：用精度换速度和内存

### 2.1 量化的原理

量化（Quantization）是把模型权重从高精度（FP16/BF16，每个参数 2 字节）压缩到低精度（INT8/INT4，每参数 1 字节或 0.5 字节）存储，从而减少从显存读取数据的量，降低内存带宽需求。

以 7B 模型为例：
- FP16：14 GB，理论峰值速度约 4ms/token（受带宽限制）
- INT8：7 GB，速度约 2ms/token（带宽需求减半）
- INT4：3.5 GB，速度约 1ms/token，而且 70B 模型可以在单张 80G A100 上运行

量化减少了每次推理加载的字节数。INT8 能带来 1.8-2.4 倍的真实加速，但较低比特位会有精度损失——激进的联合量化（如 W4A4，权重和激活值都量化到 4bit）在多步推理任务上会造成明显性能下降。

### 2.2 权重量化 vs 激活值量化

量化有两个对象：**权重**（模型的参数）和**激活值**（推理过程中的中间计算结果）。

**权重量化（W-only Quantization）**：只量化权重，激活值保持高精度。代价是量化后的权重在计算时需要反量化为高精度，有一定开销，但精度损失相对较小。GPTQ、AWQ 是常用的权重量化方法，主要用于减小模型存储和加载带宽。

**激活值量化（W4A8/W8A8）**：权重和激活值都量化。能进一步提升速度（利用 INT8 GEMM 硬件），但对模型精度影响更大，需要更细致的校准。适合对速度要求极高的生产场景。

### 2.3 主流量化方法

**GPTQ（2022）**：用少量校准数据，通过一次前向传播优化量化误差，是最早广泛使用的 LLM 权重量化方法。支持 INT4 和 INT8，精度损失较小。

**AWQ（Activation-aware Weight Quantization，2023）**：观察到不是所有权重都同等重要——激活值较大的 channel 对应的权重更关键，量化时应该保护这些权重的精度。AWQ 在保持整体 INT4 精度的同时，通过保留关键权重的高精度来减少精度损失，在 INT4 量化模型中是当前最流行的方法之一。

**GGUF（llama.cpp 格式）**：一种将量化权重、分词器等打包在一起的文件格式，专为本地推理设计。支持 Q2、Q3、Q4、Q5、Q6、Q8 等多种量化精度，可以在 CPU 上运行，是本地运行大模型最通用的格式。

### 2.4 量化的精度损失

量化不是免费的午餐。精度损失随量化程度加深而增大：

```
BF16（无损基准）→ INT8（损失约 0.1-0.5% 精度，几乎无感）
              → INT4（损失约 1-5%，多数任务可接受）
              → INT3 以下（损失显著，不推荐用于质量敏感场景）
```

**关键原则**：量化是一次性收益，不能叠加。把 INT8 再量化到 INT4，不会得到 INT8+INT4 的叠加加速，只会进一步损失精度。选好量化精度后就固定下来，不要叠加多次量化。

## 三、推测解码：不改变输出，大幅降低延迟

### 3.1 一个反直觉的思路

推测解码（Speculative Decoding）是近年来推理加速中最有趣的方法，因为它能**在完全不改变输出质量的前提下，把延迟降低 2-3 倍**。

它的思路来自一个观察：大模型生成一个 token 和生成 10 个 token，花的时间差不多——因为主要瓶颈是内存带宽（读模型权重），而这个开销和生成多少 token 关系不大。

那能不能一次"猜"多个 token，然后让大模型一次性验证？

**推测解码的工作方式**：

1. 用一个小的"草稿模型"（Draft Model，通常 1-7B 参数）快速生成 3-8 个候选 token
2. 把原始 prompt + 这些候选 token 一起发给大的"目标模型"（Target Model）
3. 目标模型**一次并行处理**所有候选 token，验证哪些 token 是它自己也会生成的
4. 接受所有正确的 token，丢弃第一个错误的 token 及之后的所有候选
5. 循环往复

关键：目标模型的输出分布完全不变——接受的 token 必须通过严格的采样一致性检验，所以最终输出和不用推测解码完全相同。用户看到的是完全一样的答案，只是速度快了 2-3 倍。

### 3.2 实际效果数据

推测解码在 2025 年已经从实验性优化成熟为生产标准。vLLM 和 TensorRT-LLM 已内置原生支持：

- Llama 3.1-70B 配合 1B 草稿模型：**2.31 倍加速**
- Llama 3.1-405B 配合 FP8 量化：**超过 3 倍吞吐量**
- EAGLE 方法（当前最佳草稿模型训练方法）：约 80% 的 token 接受率，典型加速 **2.5-2.8 倍**

速度提升因工作负载特性而有显著差异。同步的、延迟敏感的场景（对话）能看到最大收益；高吞吐量的批量处理收益较小，因为此时算力而非顺序生成是瓶颈。

### 3.3 三种主要优化手段的对比

| 方法 | 加速倍数 | 精度影响 | 适用场景 | 是否标准配置 |
|------|---------|---------|---------|------------|
| 量化（INT8）| 1.8-2.4× | 轻微 | 大多数场景 | ✅ 是 |
| 量化（INT4）| 3-4× | 中等 | 资源受限/本地部署 | ✅ 是 |
| 推测解码 | 2-3× | **无** | 延迟敏感的对话场景 | 正在普及 |
| Flash Attention | 1.5-2× | **无** | 长上下文场景 | ✅ 是 |

KV Cache、连续批处理、张量并行现在已经是所有现代推理引擎（vLLM、SGLang、TensorRT-LLM）的标准配置。如果你的服务栈是合理现代化的，你已经拥有它们了。推测解码是不同的——它是一个真正能带来新增收益的方向。

## 四、KV Cache 管理：PagedAttention 的贡献

### 4.1 KV Cache 碎片化问题

在推理时，每个用户请求都有自己的 KV Cache，大小随生成的 token 数增长。传统实现为每个请求预分配一块连续的显存，但这带来两个问题：

**内碎片**：预分配时不知道最终会生成多少 token，往往按最大长度预分配，实际用不了那么多，浪费显存。

**外碎片**：不同大小的 KV Cache 块在显存中散布，难以合并，导致明明有足够的总显存，却无法装下新的请求。

### 4.2 PagedAttention：像操作系统管理内存一样管理 KV Cache

vLLM 的核心创新 PagedAttention 借鉴了操作系统的虚拟内存分页思想：把 KV Cache 分成固定大小的页（Page），按需分配，不连续存储，用页表记录每个请求的 KV Cache 由哪些页组成。

这个设计让显存利用率从传统方式的约 20-40% 提升到 90% 以上，极大地提高了同时处理的并发请求数，是 vLLM 能比朴素实现快几倍的核心原因。

## 五、面试常见问题

**Q1：LLM 推理的主要瓶颈是什么？为什么不是算力而是内存带宽？**

LLM 推理的 Decode 阶段（逐 token 生成）是内存带宽密集型的——每生成一个 token，需要从显存读取完整的模型权重和 KV Cache 来做注意力计算，但实际的矩阵运算量很小。以 7B 模型为例，读取 14GB 权重在 3.35 TB/s 带宽下需要约 4ms，而运算本身可能只需要 0.1ms。GPU 计算核心大部分时间在等数据，算力利用率只有 5-10%。这是所有推理优化技术的根本出发点。

**Q2：量化是如何加速推理的？INT4 和 INT8 的权衡是什么？**

量化通过降低权重精度（从 BF16 的 2 字节/参数降到 INT8 的 1 字节或 INT4 的 0.5 字节），减少从显存读取数据的量，从而降低内存带宽需求，实现加速。INT8 几乎无精度损失（约 1.8-2.4 倍加速）；INT4 精度损失中等但内存减少 75%（约 3-4 倍加速），让 70B 模型能在单张 80G GPU 上运行。关键权衡：量化程度越高，加速越大，精度损失也越大，在推理密集型任务（数学推理、代码）上损失更明显。

**Q3：推测解码为什么能在不改变输出质量的前提下加速？**

推测解码利用了 GPU 的计算特性：大模型处理 1 个 token 和 10 个 token 的时间差不多（主要瓶颈是读模型权重的带宽开销，而非运算量）。小草稿模型快速提议 3-8 个候选 token，大目标模型一次并行验证所有候选，接受其中和自己采样分布一致的 token。因为接受条件是严格的统计一致性检验，最终输出分布与不用推测解码完全相同——质量不变，速度提升 2-3 倍。

**Q4：PagedAttention 解决了什么问题？**

解决了 KV Cache 的显存碎片化问题。传统方式为每个请求预分配连续的显存块，导致内碎片（预分配超出实际用量）和外碎片（不连续空闲块无法合并），显存利用率只有 20-40%。PagedAttention 把 KV Cache 分成固定大小的页按需分配，用页表管理映射，显存利用率提升到 90% 以上，大幅增加了系统能同时处理的并发请求数。
