## 零、开篇：你怎么知道你的 RAG 系统变好了？

前几章讲了怎么搭 RAG、怎么优化检索策略。但有一个根本性的问题始终绕不开：**你怎么知道你的改动让系统变好了，而不是变差了？**

"感觉更好"不算数。手动翻 10 个问题的回答更不可靠。你需要系统性的评估框架。

RAG 系统的评估难在它有两个串联的模块——检索和生成——两个都可能出问题，而且失败方式完全不同：

- **检索失败**：找到的文档和问题无关，LLM 拿着垃圾原材料
- **生成失败**：检索到了正确的文档，但 LLM 没有忠实地利用，甚至说了文档里没有的话

所以评估也需要分别针对这两个环节，而不是只看最终答案对不对。

---

## 一、RAGAS：目前最主流的 RAG 评估框架

### 1.1 什么是 RAGAS

RAGAS（Retrieval Augmented Generation Assessment System）是一个开源 Python 框架，用于在无需大量人工标注数据的情况下评估 RAG 系统。它由研究者 Shahul Es 和 Jithin James 于 2023 年 9 月发布，并在 2024 年 EACL 会议上展示。每月处理超过 500 万次评估，被 AWS、微软、Databricks 和穆迪等公司使用。

RAGAS 的核心思路是用 LLM 来评估 LLM 的输出——它把 RAG 系统的回答和检索到的上下文一起交给一个"评判员 LLM"，让评判员打分。这样就绕开了需要大量人工标注答案的难题。

### 1.2 四个核心评估指标

RAGAS 用四个指标覆盖了 RAG 系统质量的不同维度：

**忠实度（Faithfulness）**——生成是否忠实于检索内容？

这是 RAG 最关键的指标之一。它衡量模型的回答有多少内容可以从检索到的文档中找到依据。如果模型在回答里说了文档中没有的内容，忠实度分数就会下降。

```
检索到的文档：苹果公司成立于 1976 年。
模型回答：苹果公司成立于 1976 年，目前员工超过 15 万人。

"员工超过 15 万人"在文档中没有依据 → 忠实度扣分
```

**答案相关性（Answer Relevancy）**——回答是否真正回应了用户的问题？

一个啰嗦、绕弯子、或者答非所问的回答，即使内容准确，答案相关性分数也低。

**上下文精确率（Context Precision）**——检索到的文档中，有多少比例是真正有用的？

如果检索了 5 个文档片段，但只有 1 个和问题相关，其余 4 个都是噪声，上下文精确率就很低。

**上下文召回率（Context Recall）**——正确答案所需要的信息，有没有被检索到？

如果正确答案需要用到文档 A 和文档 B 的内容，但检索只找到了文档 A，召回率就是 0.5。

### 1.3 四个指标的关系与权衡

理解四个指标的关系，才能找到系统的真正瓶颈：

```
上下文召回率低      → 检索漏掉了相关文档，改进检索策略（混合检索/扩大 Top-K）
上下文精确率低      → 检索引入了太多噪声，引入 Reranker 精排
忠实度低           → 模型没有遵循文档内容，检查 Prompt 设计或换更强的 LLM
答案相关性低        → 模型跑题了，检查 Prompt 是否明确要求"只回答问题"
```

这四个指标是 RAG 系统的"体检报告"。在做任何优化之前，先跑一遍评估，找到最薄弱的环节，优化有的放矢。

### 1.4 无需人工标注的评估

RAGAS 最大的工程价值是：**不需要大量人工标注**。

传统评估方法需要为每个测试问题准备"标准答案"，然后用 BLEU、ROUGE 等指标比较模型输出和标准答案的相似度。这有两个问题：人工标注很贵；而且对开放式问题，同样正确的答案可以有很多种说法，BLEU/ROUGE 无法识别这种多样性。

RAGAS 用 LLM 作为评判员，既不需要标准答案（对忠实度和上下文类指标），又能理解语义层面的相似性。当然，它也有局限：评判员 LLM 本身可能有偏见，对某种写作风格偏好更高的分数。

## 二、GraphRAG：当关系比内容更重要

### 2.1 传统 RAG 的根本局限

前几章讲的 RAG，无论是向量检索还是混合检索，本质上都是在做**相似性匹配**——找到和问题"最像"的文档片段。

这对"这篇文档里说了什么"这类查询很有效。但对于以下类型的问题，传统 RAG 会力不从心：

**"这个问题涉及多个实体之间的复杂关系"**

```
问题：在公司历史上，哪位高管既参与过 A 项目，
      又和 B 事件有关联，并且在 C 时期担任过职务？

这个问题需要跨越多个文档，追踪实体之间的关系。
传统 RAG 找出的是"和这个问题最像"的几个文档片段，
但这些片段很可能各自只包含部分信息，没有一个片段单独能回答这个多跳问题。
```

**"这个领域的整体全局状况是什么"**

```
问题：总结一下过去 10 年里这个行业的主要技术演进趋势。

这不是"找某个特定段落"的问题，而是需要从大量文档中综合提炼出高层规律。
传统 RAG 给的是几个具体的段落，而不是全局性的综合。
```

### 2.2 GraphRAG 的核心思路

微软的 GraphRAG 通过 LLM 生成的知识图谱和层级社区摘要，在全面性和多样性指标上实现了优于朴素 RAG 的表现，能够处理超出单个 LLM 上下文窗口的大型数据集。

GraphRAG 的基本思路是在建立索引时做更多的工作：

**索引阶段（离线，一次性）**：
1. 用 LLM 从文档中提取所有实体（人名、地名、机构、概念）和它们之间的关系
2. 把这些实体和关系构建成一张知识图谱
3. 对图谱进行社区检测，把紧密相关的实体聚成"社区"
4. 为每个社区生成摘要，从局部到全局形成层次化的知识结构

**查询阶段（在线）**：
- 局部查询（Local Search）：从图谱中找相关实体，沿着关系边追踪，汇集多跳信息
- 全局查询（Global Search）：利用社区摘要，从整体上回答宏观性问题

传统 RAG 的 LLM 必须每次从原始文本中推断结构；GraphRAG 给 LLM 提供了一张显式的结构图，让它在这张图上推理。相当于告诉模型："这是骨架，你来填充血肉。"GraphRAG 有更多的组成部分——实体提取、关系提取、图数据库操作，维护成本更高。

### 2.3 GraphRAG 什么时候值得用

GraphRAG 不是传统 RAG 的替代品，而是在特定场景下的补充：

**适合 GraphRAG 的场景**：
- 文档中有大量相互关联的实体（人物关系、组织架构、产品依赖）
- 需要多跳推理（"谁认识谁的谁"）
- 需要全局性综合分析（"这个领域的整体趋势"）
- 典型场景：企业知识图谱、医学文献分析、法律案件关系梳理、竞争情报分析

**不适合 GraphRAG 的场景**：
- 大多数"找到具体信息"的查询（传统 RAG 就够）
- 对延迟敏感的场景（GraphRAG 多个 LLM 调用，更慢更贵）
- 知识库频繁更新（图谱重建成本高）
- 团队没有图数据库运维能力

一句话总结：传统 RAG 检索的是实体、关系及其周围的内容，然后用文本支撑。当问题更像"什么和什么相连？"而不是"这在哪里提到过？"时，GraphRAG 更合适。

### 2.4 GraphRAG 的现实成本

值得注意的是，GraphRAG 的索引阶段成本相当高。一个中等规模的企业知识库（几千份文档），用 LLM 做实体和关系抽取可能需要数小时和可观的 API 费用。每次知识库更新，都要重新跑一遍这个过程。

微软在 2024 年将 GraphRAG 开源之后，社区也涌现了一些简化版本（如 nano-GraphRAG、LightRAG），降低了使用门槛，但本质的复杂性没有改变。

## 三、Long Context vs RAG：2024-2025 年最热的争论

### 3.1 争论的起源

2024 年，Gemini 1.5 Pro 把上下文窗口扩展到了 100 万 token，Claude 3 支持 20 万 token，GPT-4o 支持 12.8 万 token。

这引发了一个在工程师圈子里激烈讨论的问题：**既然模型能装这么多内容，为什么还需要 RAG？直接把所有文档塞进去不就行了？**

### 3.2 Long Context 的真实能力与限制

机械地把大量文本塞入 LLM 的上下文窗口，本质上是一种"暴力破解"策略。它不可避免地分散模型的注意力，通过"信息迷失在中间"或"信息泛滥"效应显著降低回答质量。

研究发现，LLM 处理超长上下文时，对开头和结尾的内容注意力更高，中间段落的内容往往被忽视。随着文档数量增多，这个问题会加剧。

此外，Long Context 有几个工程上的硬约束：

**成本**：100 万 token 的上下文，按主流 API 定价，每次调用费用可能高达几美元到几十美元。如果是高频查询场景，成本完全不可接受。

**延迟**：长上下文推理慢。处理 100 万 token 的上下文，即使是最快的模型也需要数十秒，很多应用场景无法接受这个延迟。

**更新**：如果知识库每天都有更新，每次查询都要重新构建超长上下文，不如用 RAG 只检索最新的相关内容。

### 3.3 两者的边界在哪里

2024 年激烈讨论的理论问题"长上下文能否取代 RAG？"在 2025 年迅速进入了实际测试阶段。一些对延迟和成本不太敏感、查询模式相对固定的场景（如某些合同审查、固定格式的报告分析），开始尝试直接使用长上下文窗口，将整批相关文档一次性输入模型。

根据 2025 年的工业实践，两者的适用边界大致如下：

```
适合 Long Context（直接塞进去）的场景：
  · 文档总量少（几十页以内），能完整放入上下文
  · 查询频率低，成本可接受
  · 需要模型理解文档间的细微关系（RAG 切块可能破坏这种关系）
  · 对延迟不敏感（比如批量处理）
  典型场景：代码库分析、合同审阅、学术论文精读

适合 RAG 的场景：
  · 知识库大（数千到数百万文档）
  · 查询频率高，成本敏感
  · 知识库频繁更新
  · 需要精确引用来源
  · 对延迟敏感（秒级响应要求）
  典型场景：企业知识库问答、客服机器人、代码搜索
```

### 3.4 未来的趋势：两者会融合

当前最前沿的方向不是"谁取代谁"，而是两者的智能融合：

- 用 RAG 先把相关文档找出来（处理规模问题）
- 再把找到的文档用长上下文传给 LLM（处理理解深度问题）
- 这样既避免了把整个知识库塞入的成本，又让 LLM 能在足够的上下文中进行深度推理

从更长远的视角看，前瞻性的观点将技术重点从"检索引擎"升级为"上下文引擎"。很多被归结为"Agent 智能"的问题，本质上仍然是在正确的时机为 LLM 找到并呈现正确的信息。如果 Agent 有一个强大的"外部大脑"帮它高效地搜索、过滤、验证和组织所有相关数据，最终的用户体验将远超单靠模型扩展来提升的效果。

## 四、第七章总结：RAG 的全局视角

读完第七章，你对 RAG 应该有了这样的立体认知：

**技术层面**：RAG 不是一个技术点，而是一套流水线，每个环节都有专门的优化空间：
- 文档解析 → 切块策略 → 嵌入模型 → 向量数据库 → 检索策略 → 重排序 → 上下文构建 → 生成

**工程层面**：从 demo 到生产是一段很长的路，大多数工程问题来自检索环节而非生成环节。评估是优化的前提，没有量化就没有方向。

**前沿层面**：RAG 在向着更智能的方向演进——从静态的"检索一次"到动态的 Agentic RAG（按需检索、多步推理）；从纯文本检索到图谱增强的知识推理。Long Context 不是 RAG 的终结者，而是它的协作者。

**认知层面**：RAG 让 LLM 从"闭卷考试"变成了"开卷考试"——它依然需要理解问题和组织答案，但可以实时查阅参考资料。这个范式转变的意义，不只是技术层面的，更是让 AI 系统在企业场景中真正可信、可验证、可维护。

## 五、面试常见问题

**Q1：RAGAS 的四个核心指标是什么？各自衡量什么？**

忠实度（Faithfulness）：回答是否有文档依据，防止 LLM 编造；答案相关性（Answer Relevancy）：回答是否真正回应了问题；上下文精确率（Context Precision）：检索结果中有用信息的比例，反映检索噪声多少；上下文召回率（Context Recall）：回答所需的关键信息是否被检索到。前两个评估生成质量，后两个评估检索质量。

**Q2：GraphRAG 和传统 RAG 的核心区别是什么？**

传统 RAG 做的是语义相似性匹配，找"和问题最像"的文档片段，适合"这篇文档说了什么"类问题。GraphRAG 在索引阶段用 LLM 提取实体关系，构建知识图谱，查询时沿关系边追踪信息，适合"这些实体之间有什么关系"和"这个领域的整体趋势"类问题。GraphRAG 更强大但也更复杂，维护成本更高，不是所有场景都值得用。

**Q3：Long Context 模型会取代 RAG 吗？**

不会完全取代，两者有不同的适用场景。Long Context 适合：文档量少（能放入上下文）、查询频率低、需要深度理解文档间关系的场景。RAG 适合：知识库大（数千到百万文档）、查询频率高（成本敏感）、需要实时更新知识、需要精确引用来源的场景。当前最佳实践是两者结合：用 RAG 先检索相关文档，再用 Long Context 深度理解检索结果。

**Q4：如何判断一个 RAG 系统"哪里出了问题"？**

用 RAGAS 分别评估检索和生成：如果上下文召回率低，说明检索漏掉了相关文档，需要优化检索策略；如果上下文精确率低，说明检索噪声太多，需要加 Reranker；如果忠实度低，说明模型在"发挥"而不是依据文档回答，需要调整 Prompt 或换更强模型；如果答案相关性低，说明模型答非所问，需要改进问题理解。
