## 零、开篇：Prompt 是与大模型沟通的语言

训练好的大模型就像一个博学多才但需要引导的顾问。它知道的东西很多，但它的回答质量很大程度上取决于你怎么问它——这就是提示工程（Prompt Engineering）的核心价值。

**一个反直觉的事实**：同样的模型，不同的 Prompt 写法，效果可以差几十个百分点。

以数学题为例：

```
写法 A：
  问：Roger 有 5 个网球。他买了 2 罐网球，每罐 3 个。现在他有几个？
  GPT-3 答：11 个。✅

写法 B（更复杂的题）：
  问：食堂原有 23 个苹果。他们做午饭用了 20 个，买了 6 个。现在有几个？
  GPT-3 答：27 个。❌（正确答案是 9 个）

写法 C（加上"让我们一步一步思考"）：
  问：食堂原有 23 个苹果。他们做午饭用了 20 个，买了 6 个。现在有几个？
  让我们一步一步思考。
  GPT-3 答：先有 23 个，用了 20 个，剩下 3 个，再买 6 个，一共 9 个。✅
```

仅仅加上一句"让我们一步一步思考"，复杂推理任务的准确率就大幅提升。

本章系统介绍四种核心技巧，它们的效果从简单到强大，工程成本也从低到高。

## 一、Zero-shot：直接问

### 1.1 初印象：用费曼视角理解

**Zero-shot（零样本）是最基础的提示方式——直接告诉模型做什么，不给任何示例**。

"Zero"指的是零个示例（example），模型完全依赖预训练时学到的知识来回答。

```
[Zero-shot 示例]

你：将以下文本的情感分类为"正面"、"负面"或"中性"：
   "这家餐厅的食物非常美味，但服务有点慢。"

模型：中性
```

### 1.2 Zero-shot 的工作原理

Zero-shot 之所以有效，是因为现代 LLM 在 SFT 指令微调阶段已经见过大量不同类型的任务描述，学会了"理解指令并执行"的能力。

Zero-Shot Chain-of-Thought 通过提示语言模型执行任务，无需特定训练或示例。给出 zero-shot 提示时，模型依赖其基于训练数据的预有知识和自然语言理解能力来理解提示并生成结果。

```python
from openai import OpenAI   # 或其他 LLM API

def zero_shot_prompt(task_description: str, input_text: str) -> str:
    """
    Zero-shot 提示的标准结构。
    """
    prompt = f"""{task_description}

输入：{input_text}

输出："""
    return prompt


# 示例：情感分析
sentiment_task = "将以下文本的情感分类为'正面'、'负面'或'中性'。只输出分类结果，不要解释。"
inputs = [
    "这款手机的电池续航非常出色，用了一天还有 40% 电量！",
    "快递三天没到，客服也不回复，太失望了。",
    "商品如描述，发货速度一般。",
]

for text in inputs:
    prompt = zero_shot_prompt(sentiment_task, text)
    print(f"输入: {text[:30]}...")
    print(f"Prompt: {prompt[:80]}...")
    print()

# Zero-shot 写作的四个要素：
elements = {
    "1. 清晰的任务描述": "告诉模型做什么，越具体越好",
    "2. 明确的输出格式": "指定输出格式（JSON、列表、单词等）",
    "3. 相关的约束条件": "字数限制、风格要求、禁止项",
    "4. 结构化分隔符":  "用 XML 标签或 ``` 区分不同部分",
}
print("Zero-shot Prompt 的四个要素：")
for k, v in elements.items():
    print(f"  {k}: {v}")
```

### 1.3 Zero-shot 的局限

Zero-shot 在以下情况效果较差：

- **领域专业任务**：医疗、法律等需要精确领域知识的任务
- **复杂推理**：多步数学、逻辑推理（需要 CoT）
- **特定格式输出**：需要精确格式（如 JSON Schema）时不稳定
- **模型不熟悉的任务模式**：训练数据中很少见的任务类型

这时需要 Few-shot 或更高级的技巧。

## 二、Few-shot：给几个例子

### 2.1 初印象：用费曼视角理解

**Few-shot（少样本）是"先给模型看几个例子，再让它做新题"**。

就像教孩子做题：与其只描述规则（Zero-shot），不如先给几道例题演示（Few-shot）。

```
[Few-shot 示例：情感分析]

文本：这款游戏太上瘾了，一玩就是好几个小时！
情感：正面

文本：等了两周，快递还没到，这什么服务。
情感：负面

文本：产品和图片差不多，价格也合理。
情感：中性

文本：{新的输入文本}
情感：
```

### 2.2 示例的选择原则

Few-shot 提示的最佳实践包括提供清晰的代表性示例，并保持格式的一致性。研究表明，即使标签被随机化，拥有示例也能显著改善性能。

这个发现很反直觉——**示例的格式和多样性往往比标签是否正确更重要**。

```python
def few_shot_prompt(
    task_description: str,
    examples: list[dict],   # [{"input": ..., "output": ...}, ...]
    new_input: str,
    separator: str = "\n\n",
) -> str:
    """
    Few-shot 提示的标准结构。
    """
    prompt = task_description + separator

    # 拼接示例
    for ex in examples:
        prompt += f"输入：{ex['input']}\n输出：{ex['output']}" + separator

    # 新任务
    prompt += f"输入：{new_input}\n输出："
    return prompt


# 示例选择的关键原则
example_selection_principles = {
    "代表性": "示例应覆盖任务的主要情况和边界条件",
    "多样性": "示例之间应有足够差异，避免模型过度关注某个特定模式",
    "数量":   "通常 3-8 个示例效果最好，更多未必更好",
    "顺序":   "最后一个示例对结果影响最大，把最相关的放最后",
    "格式":   "示例格式要和期望输出格式完全一致",
}

print("Few-shot 示例选择原则：")
for k, v in example_selection_principles.items():
    print(f"  {k}: {v}")

# 实际示例
examples = [
    {"input": "苹果的英文是什么？", "output": "Apple"},
    {"input": "香蕉的英文是什么？", "output": "Banana"},
    {"input": "西瓜的英文是什么？", "output": "Watermelon"},
]

new_question = "草莓的英文是什么？"
prompt = few_shot_prompt("将中文水果名翻译成英文：", examples, new_question)
print(f"\nFew-shot Prompt:\n{prompt}")
```

### 2.3 Few-shot 的局限

- **上下文窗口消耗**：每个示例都占用 token，示例越多越贵
- **示例选择困难**：选不好的示例可能让效果更差
- **动态任务不适用**：示例固定，无法覆盖所有情况

## 三、Chain-of-Thought（CoT）：让模型"先想后答"

### 3.1 初印象：用费曼视角理解

**Chain-of-Thought（思维链）就是让模型把推理过程写出来，而不是直接跳到结论**。

人类解决复杂问题时，通常在脑子里"演草纸"——先列出已知条件，再一步步推导。CoT 就是让模型也在"草稿纸"上推理，而不是直接说答案。

为什么有效？两个原因：

1. **序列生成的自我约束**：当模型写出"第一步...第二步..."，后续内容受到前面步骤的约束，不容易出现逻辑跳跃
2. **中间状态可见**：每个中间步骤都是模型"看到"的输入，有助于后续计算

思维链提示方法通过将复杂问题分解为一系列中间步骤，从根本上改变了我们与 LLM 的交互方式。研究发现，性能提升发生在模型非常大（约 1000 亿参数）之后，较小的模型产生的推理链看似连贯，但实际上是错误的，导致比标准提示更差的表现。

### 3.2 Zero-shot CoT：魔法咒语"一步一步来"

最简单的 CoT 不需要写示例，只需要在 Prompt 末尾加上一句话：

**"让我们一步一步思考（Let's think step by step）"**

这句话就像一个开关，激活了模型的逐步推理模式。

```python
def zero_shot_cot(question: str, cot_trigger: str = "让我们一步一步思考。") -> str:
    """
    Zero-shot CoT：只需在问题后加上思维链触发语。
    """
    return f"{question}\n\n{cot_trigger}"


# 对比：直接回答 vs CoT
math_problem = """
一个农场有鸡和兔子，共有 35 个头，94 条腿。
问：鸡和兔子各有多少只？
"""

direct_prompt = math_problem.strip()
cot_prompt = zero_shot_cot(math_problem.strip())

print("直接回答 Prompt：")
print(direct_prompt)
print("\nCoT Prompt：")
print(cot_prompt)

# CoT 触发语的变体（都有类似效果）
cot_triggers = [
    "让我们一步一步思考。",
    "Let's think step by step.",
    "请先分析问题，再给出答案。",
    "请展示你的推理过程。",
    "思考过程：",
]
print("\n常用 CoT 触发语：")
for t in cot_triggers:
    print(f"  • {t}")
```

### 3.3 Few-shot CoT：示例中包含推理过程

更强的版本：示例里不只有输入和输出，还有详细的推理过程：

```python
cot_examples = [
    {
        "question": "Roger 有 5 个网球。他买了 2 罐网球，每罐 3 个。他现在有几个网球？",
        "reasoning": "Roger 开始有 5 个网球。2 罐 × 每罐 3 个 = 6 个。5 + 6 = 11 个。",
        "answer": "11 个",
    },
    {
        "question": "食堂有 23 个苹果。做午饭用了 20 个，买了 6 个。现在有几个？",
        "reasoning": "开始有 23 个。用了 20 个，剩 23 - 20 = 3 个。又买了 6 个，3 + 6 = 9 个。",
        "answer": "9 个",
    },
]

def few_shot_cot_prompt(examples: list[dict], new_question: str) -> str:
    """构建 Few-shot CoT Prompt"""
    prompt = ""
    for ex in examples:
        prompt += f"问题：{ex['question']}\n"
        prompt += f"推理：{ex['reasoning']}\n"
        prompt += f"答案：{ex['answer']}\n\n"
    prompt += f"问题：{new_question}\n推理："
    return prompt

new_q = "一家店卖了 10 件商品，每件 5 元，又退了 2 件，每件 5 元。共收入多少？"
cot_prompt = few_shot_cot_prompt(cot_examples, new_q)
print("Few-shot CoT Prompt：")
print(cot_prompt)
```

### 3.4 CoT 的使用场景和局限

**适合 CoT 的任务**：
- 数学计算（多步运算）
- 逻辑推理（条件推断）
- 代码调试（分析错误）
- 决策分析（权衡利弊）
- 阅读理解（复杂关系提取）

**不适合 CoT 的任务**：
- 简单事实查找（"法国首都是哪里"，CoT 多余）
- 创意写作（不需要逐步推理）
- 分类任务（直接判断通常够了）

实施时选择适合你用例的 CoT 技术（如简单任务用零样本 CoT，复杂问题用少样本 CoT），并建立系统化的验证流程。

```python
# CoT 使用决策树
def should_use_cot(task_type: str, model_size: str) -> str:
    """
    判断是否应该使用 CoT。
    """
    large_model = model_size in ["7B+", "13B+", "70B+"]

    if not large_model:
        return "⚠️  小模型（< 7B）使用 CoT 可能产生看似合理但实际错误的推理链，慎用"

    complex_reasoning_tasks = ["数学", "逻辑推理", "代码调试", "多步规划"]
    if any(t in task_type for t in complex_reasoning_tasks):
        return "✅  推荐使用 CoT，尤其是 Few-shot CoT"

    simple_tasks = ["分类", "翻译", "摘要", "简单问答"]
    if any(t in task_type for t in simple_tasks):
        return "❌  不需要 CoT，直接 Zero-shot 或 Few-shot"

    return "🤔  视情况而定，先试 Zero-shot，效果差再加 CoT"

for task in ["数学推理", "情感分类", "代码调试", "文章摘要"]:
    print(f"{task}: {should_use_cot(task, '7B+')}")
```

## 四、Self-Consistency：多数投票取最优

### 4.1 初印象：用费曼视角理解

**Self-Consistency（自一致性）是"问同一个问题多次，用多数票决定最终答案"**。

一次 CoT 可能走错路。但如果你用略有不同的推理路径问同样的问题多次（通常 10-40 次），错误的推理路径往往各不相同，而正确答案会被多数路径汇聚到——这就是多数投票的力量。

```
同一道题，采样 5 次：
路径1：... 计算 ... → 答案：9   ✅
路径2：... 计算 ... → 答案：9   ✅
路径3：... 推理有误 → 答案：7   ❌
路径4：... 计算 ... → 答案：9   ✅
路径5：... 推理有误 → 答案：11  ❌

多数投票：9（3票 vs 7的1票 vs 11的1票）→ 最终答案：9 ✅
```

Self-consistency 提示生成多个输出，然后选择最一致的那个。你可以将它与 CoT 或基本上任何提示方法结合使用。

### 4.2 代码实现

```python
import random
from collections import Counter

def self_consistency(
    question: str,
    generate_fn,          # 调用 LLM 的函数
    n_samples: int = 10,  # 采样次数
    temperature: float = 0.7,   # 温度（需要 > 0 才有多样性）
    extract_answer_fn = None,   # 从回答中提取最终答案的函数
) -> dict:
    """
    Self-Consistency 实现：
    1. 用 CoT + 高温度采样多次
    2. 从每个回答中提取答案
    3. 多数投票
    """
    cot_question = f"{question}\n\n让我们一步一步思考。"
    answers = []
    full_responses = []

    for i in range(n_samples):
        # 调用 LLM（每次都有一定随机性，因为 temperature > 0）
        response = generate_fn(cot_question, temperature=temperature)
        full_responses.append(response)

        # 提取最终答案（数字、选项等）
        if extract_answer_fn:
            answer = extract_answer_fn(response)
        else:
            # 简单提取：取回答最后一行
            answer = response.strip().split('\n')[-1]
        answers.append(answer)

    # 多数投票
    vote_counts = Counter(answers)
    majority_answer, majority_votes = vote_counts.most_common(1)[0]
    confidence = majority_votes / n_samples

    return {
        "majority_answer": majority_answer,
        "confidence": confidence,
        "vote_distribution": dict(vote_counts),
        "all_answers": answers,
    }


# 模拟 LLM 调用（真实场景替换为 API 调用）
def mock_llm(prompt: str, temperature: float = 0.7) -> str:
    """模拟 LLM 输出（演示用）"""
    answers = ["9"] * 7 + ["7"] * 2 + ["11"] * 1   # 70% 正确率
    reasoning_steps = [
        "开始 23 个，用了 20 个，剩 3 个，买了 6 个，共 9 个。",
        "23 - 20 = 3，3 + 6 = 9。",
        "23 - 20 = 3，3 - 6 = -3，等等不对，应该是 3 + 6 = 9。",
    ]
    answer = random.choice(answers)
    reasoning = random.choice(reasoning_steps)
    return f"{reasoning}\n最终答案：{answer}"

import re
def extract_number(response: str) -> str:
    """从回答中提取数字答案"""
    numbers = re.findall(r'\b\d+\b', response)
    return numbers[-1] if numbers else "未知"

# 测试 Self-Consistency
random.seed(42)
question = "食堂有 23 个苹果，做午饭用了 20 个，买了 6 个。现在有几个？"
result = self_consistency(question, mock_llm, n_samples=10, extract_answer_fn=extract_number)

print(f"问题: {question}")
print(f"\nSelf-Consistency 结果 (10次采样)：")
print(f"  最终答案: {result['majority_answer']}")
print(f"  置信度: {result['confidence']:.0%}")
print(f"  投票分布: {result['vote_distribution']}")
```

### 4.3 Self-Consistency 的成本与收益

Self-Consistency 的代价是 API 调用次数增加（10次采样 = 10倍成本），但对于高准确率要求的场景，这个代价通常值得：

```python
# 成本收益分析
def sc_cost_benefit(
    base_accuracy: float,    # 单次 CoT 的准确率
    n_samples: int,           # 采样次数
    token_cost: float = 0.001 # 每 1000 token 的成本（美元）
):
    """
    Self-Consistency 的成本收益计算。
    注意：这里的准确率提升是简化估算，实际效果因任务而异。
    """
    import math
    # 多数投票准确率的粗略估算（假设每次独立）
    # P(正确) ≈ P(多数路径正确)
    # 实际提升和 n_samples、base_accuracy 的关系非线性
    majority_acc = sum(
        math.comb(n_samples, k) * (base_accuracy**k) * ((1-base_accuracy)**(n_samples-k))
        for k in range(n_samples // 2 + 1, n_samples + 1)
    )

    cost_multiplier = n_samples

    print(f"单次 CoT 准确率: {base_accuracy:.0%}")
    print(f"采样 {n_samples} 次后估算准确率: {majority_acc:.0%}")
    print(f"准确率提升: +{(majority_acc - base_accuracy):.0%}")
    print(f"成本倍数: {cost_multiplier}×")
    print(f"性价比: {'高' if (majority_acc - base_accuracy) > 0.1 else '中' if (majority_acc - base_accuracy) > 0.05 else '低'}")

# 示例场景
print("不同准确率下 Self-Consistency 的收益：")
for acc in [0.5, 0.7, 0.8, 0.9]:
    print(f"\n基准准确率 {acc:.0%}：")
    sc_cost_benefit(acc, n_samples=10)
```

## 五、四种技巧的对比与选型

### 5.1 横向对比

| 技巧 | 示例数量 | 需要推理过程 | 适合任务 | API 调用 | 成本 |
|------|---------|------------|---------|---------|------|
| Zero-shot | 0 | 否 | 简单任务 | 1次 | 低 |
| Few-shot | 3-8 | 否 | 格式要求高的任务 | 1次 | 低-中 |
| Zero-shot CoT | 0 | 是 | 中等复杂推理 | 1次 | 中 |
| Few-shot CoT | 3-8 | 是 | 复杂推理、数学 | 1次 | 中 |
| Self-Consistency | 0 | 是 | 高准确率要求 | N次 | 高 |

### 5.2 选型决策流程

```
你的任务是什么？
    ↓
简单任务（分类、翻译、简单问答）
    → Zero-shot 或 Few-shot
    → 不需要 CoT

复杂推理任务（数学、逻辑、多步规划）
    → 先试 Zero-shot CoT（"一步一步思考"）
    → 效果不好 → Few-shot CoT（加入带推理的示例）
    → 需要更高准确率 → Self-Consistency（多次采样投票）

特定格式输出（JSON、结构化数据）
    → Few-shot（示例展示期望格式）
    → 考虑 JSON mode 或结构化输出 API

不确定时
    → 先试 Zero-shot，快速迭代
    → 效果不满意再升级到 Few-shot 或 CoT
```

### 5.3 实战：Prompt 迭代示例

```python
# 一个完整的 Prompt 迭代过程示例：判断代码是否有 bug

# 版本 1：Zero-shot（可能不稳定）
v1 = """以下 Python 代码是否有 bug？如果有，是什么 bug？

代码：
def divide(a, b):
    return a / b

回答："""

# 版本 2：Zero-shot + 格式约束
v2 = """判断以下 Python 代码是否有 bug。
以 JSON 格式回答，包含 has_bug（布尔值）和 description（字符串）字段。

代码：
def divide(a, b):
    return a / b

JSON 回答："""

# 版本 3：Few-shot + 格式约束
v3 = """判断 Python 代码是否有 bug，以 JSON 格式回答。

代码：
def add(a, b):
    return a + b
JSON：{"has_bug": false, "description": "代码正确，两数相加没有问题"}

代码：
def get_first(lst):
    return lst[0]
JSON：{"has_bug": true, "description": "空列表时会抛出 IndexError，应检查 lst 是否为空"}

代码：
def divide(a, b):
    return a / b
JSON："""

print("Prompt 迭代：从简单到精确")
print(f"\n版本1（Zero-shot）：\n{v1}")
print(f"\n版本2（格式约束）：\n{v2}")
print(f"\n版本3（Few-shot+格式）：\n{v3}")

best_practices = {

        "差": "解释量子纠缠",
        "好": "你是一位善于用生活比喻解释物理的科普作家，请向一个10岁的孩子解释量子纠缠",
    },
    "输出约束": {

<text>

    },

    print(f"  ❌ 差的写法: {examples['差'][:60]}")
    print(f"  ✅ 好的写法: {examples['好'][:60]}")

## 七、面试常见问题

**Q1：什么是 Zero-shot 和 Few-shot？区别是什么？**

Zero-shot 是不提供任何示例，直接给任务描述让模型完成；Few-shot 是在任务描述前提供 3-8 个"输入-输出"示例对，让模型从示例中学习格式和规律。Few-shot 通常效果更稳定，尤其是对输出格式有严格要求时；Zero-shot 更灵活，不需要准备示例。研究发现示例的格式和多样性比示例标签是否绝对正确更重要。

**Q2：Chain-of-Thought 为什么有效？什么时候用它？**

CoT 让模型先写出中间推理步骤再给出答案，有效原因：中间步骤形成约束，减少逻辑跳跃；每个推理步骤都在上下文里，后续步骤能"看到"前面的推理。适合用于多步数学、逻辑推理、代码调试等复杂推理任务。注意：CoT 在小模型上（< 7B）效果差甚至有害，因为小模型可能生成看似合理但错误的推理链。

**Q3：Self-Consistency 的原理是什么？什么场景值得付出额外成本？**

Self-Consistency 是对同一问题用 CoT 采样多次（通常 10-40 次，temperature > 0 保证多样性），然后对所有输出的最终答案进行多数投票。原理：错误推理路径各不相同（噪声），正确推理路径收敛到相同答案（信号），多数投票过滤掉噪声。值得额外成本的场景：高准确率要求（医疗、金融、考试）；任务明确有唯一正确答案（数学、代码）；基础模型准确率在 60-80% 时提升最明显。

**Q4：Few-shot CoT 和 Zero-shot CoT 如何选择？**

Zero-shot CoT（加"一步一步思考"）是快速迭代的起点，不需要准备示例，适合快速验证 CoT 是否有帮助。如果 Zero-shot CoT 效果不够好，再升级到 Few-shot CoT——在示例里包含完整的推理步骤，展示期望的推理模式。Few-shot CoT 通常效果更好，但需要高质量的带推理过程的示例，准备成本更高。

**Q5：Prompt 工程中最重要的几个原则是什么？**

五个核心原则：清晰具体（任务描述越具体越好，避免歧义）；结构化输出（明确指定输出格式，减少解析难度）；角色设定（给模型一个适合任务的"角色"）；约束条件（字数、风格、禁止项）；分隔符（用 XML 标签或代码块区分不同部分，防止指令和内容混淆）。
