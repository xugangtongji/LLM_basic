## 零、开篇：Prompt 是与大模型沟通的语言

训练好的大模型就像一个博学多才但需要引导的顾问。它知道的东西很多，但回答质量很大程度上取决于你怎么问它——这就是提示工程（Prompt Engineering）的核心价值。

需要先破除一个误解：提示工程**不是"和 AI 说几句魔法咒语，它就变聪明了"**。它本质上是**在不修改模型参数的前提下，通过设计输入来引导模型的输出**——和写好代码一样，是一门有规律可循的工程实践，需要理解底层机制、遵循设计原则、反复测试迭代。

**一个反直觉的事实**：同样的模型，不同的 Prompt 写法，效果可以差几十个百分点。以数学题为例：

```
写法 A：
  问：Roger 有 5 个网球。他买了 2 罐网球，每罐 3 个。现在他有几个？
  GPT-3 答：11 个。✅

写法 B（更复杂的题）：
  问：食堂原有 23 个苹果。他们做午饭用了 20 个，买了 6 个。现在有几个？
  GPT-3 答：27 个。❌（正确答案是 9 个）

写法 C（加上"让我们一步一步思考"）：
  问：食堂原有 23 个苹果。他们做午饭用了 20 个，买了 6 个。现在有几个？
  让我们一步一步思考。
  GPT-3 答：先有 23 个，用了 20 个，剩下 3 个，再买 6 个，一共 9 个。✅
```

仅仅加上一句"让我们一步一步思考"，复杂推理任务的准确率就大幅提升。本章系统介绍四种核心技巧，它们的效果从简单到强大，工程成本也从低到高。

## 一、Zero-shot：直接问，不给例子

### 1.1 初印象：用费曼视角理解

**Zero-shot（零样本）是最基础的提示方式——直接告诉模型做什么，不给任何示例**。"Zero"指零个示例（example），模型完全依赖预训练时学到的知识来回答。

```
[Zero-shot 示例]
你：将以下文本的情感分类为"正面"、"负面"或"中性"：
   "这家餐厅的食物非常美味，但服务有点慢。"
模型：中性
```

### 1.2 Zero-shot 的工作原理与三个层次

Zero-shot 之所以有效，是因为现代 LLM 在 SFT 指令微调阶段已经见过大量不同类型的任务描述，学会了"理解指令并执行"的能力。这背后是预训练中的 In-Context Learning 能力——模型见过足够多的"问题-解答"模式，能从任务描述本身推断出期望的输出格式。

写好一个 Zero-shot Prompt 有三个层次，从基础到结构化：

```python
# 1. 基础：直接问
zero_shot_basic = """
将以下句子翻译成英文：
深度学习正在改变人工智能领域。
"""

# 2. 更好：明确角色 + 任务 + 格式要求
zero_shot_better = """
你是一位专业翻译，精通中英文互译。
请将以下中文句子翻译成地道的英文：

句子：深度学习正在改变人工智能领域。

请只输出翻译结果，不需要解释。
"""

# 3. 结构化：指定精确的输出格式
zero_shot_structured = """
分析以下产品评论的情感倾向。

评论："这款手机的拍照效果令人失望，续航也很差，但屏幕显示效果确实不错。"

请按以下格式输出：
情感：[正面/负面/混合]
主要优点：[简短描述]
主要缺点：[简短描述]
置信度：[高/中/低]
"""
```

一个好的 Zero-shot Prompt 通常包含四个要素：**清晰的任务描述**（越具体越好）、**明确的输出格式**（JSON、列表、单词等）、**相关的约束条件**（字数、风格、禁止项）、**结构化分隔符**（用 XML 标签或 ``` 区分不同部分）。

```python
from openai import OpenAI   # 或其他 LLM API

def zero_shot_prompt(task_description: str, input_text: str) -> str:
    """Zero-shot 提示的标准结构。"""
    prompt = f"""{task_description}

输入：{input_text}

输出："""
    return prompt

# 示例：情感分析
sentiment_task = "将以下文本的情感分类为'正面'、'负面'或'中性'。只输出分类结果，不要解释。"
for text in [
    "这款手机的电池续航非常出色，用了一天还有 40% 电量！",
    "快递三天没到，客服也不回复，太失望了。",
    "商品如描述，发货速度一般。",
]:
    print(zero_shot_prompt(sentiment_task, text))
```

### 1.3 Zero-shot 的局限

Zero-shot 在以下情况效果较差，这时需要 Few-shot 或更高级的技巧：

- **领域专业任务**：医疗、法律等需要精确领域知识或惯例的任务
- **复杂推理**：多步数学、逻辑推理（需要 CoT）
- **特定格式输出**：需要精确格式（如 JSON Schema）时不稳定
- **一致性要求高**：不同运行结果格式可能不一致

## 二、Few-shot：给几个例子作为示范

### 2.1 初印象：用费曼视角理解

**Few-shot（少样本）是"先给模型看几个例子，再让它做新题"**。就像教孩子做题：与其只描述规则（Zero-shot），不如先给几道例题演示（Few-shot）——模型会从例子中归纳出模式，然后应用到新问题上。可以把这看作给 AI 一个"微型训练课程"。

```
[Few-shot 示例：情感分析]
文本：这款游戏太上瘾了，一玩就是好几个小时！
情感：正面

文本：等了两周，快递还没到，这什么服务。
情感：负面

文本：产品和图片差不多，价格也合理。
情感：中性

文本：{新的输入文本}
情感：
```

### 2.2 示例的选择原则

Few-shot 的最佳实践包括提供清晰的代表性示例，并保持格式一致。一个反直觉的研究发现：**即使标签被随机化，拥有示例也能显著改善性能**——**示例的格式和多样性往往比标签是否正确更重要**。

```python
def few_shot_prompt(
    task_description: str,
    examples: list[dict],   # [{"input": ..., "output": ...}, ...]
    new_input: str,
    separator: str = "\n\n",
) -> str:
    """Few-shot 提示的标准结构。"""
    prompt = task_description + separator
    for ex in examples:
        prompt += f"输入：{ex['input']}\n输出：{ex['output']}" + separator
    prompt += f"输入：{new_input}\n输出："
    return prompt

# 示例选择的关键原则
example_selection_principles = {
    "代表性": "示例应覆盖任务的主要情况和边界条件",
    "多样性": "示例之间应有足够差异，避免模型过度关注某个特定模式",
    "质量":   "示例必须准确，错误的示例会严重误导模型",
    "数量":   "通常 2-8 个效果最好，更多未必更好，且浪费 token",
    "顺序":   "最后一个示例对结果影响最大（最近邻效应），把最相关的放最后",
    "格式":   "所有示例的输入输出格式必须完全一致，模型会严格模仿",
}
for k, v in example_selection_principles.items():
    print(f"  {k}: {v}")
```

### 2.3 动态选例策略

固定示例并非最优。当示例池很大（数百到数千个）时，可以针对每个新查询**动态检索最相关的示例**（如按语义相似度），比静态固定示例更有针对性。

```python
from sentence_transformers import SentenceTransformer
import numpy as np

def select_few_shot_examples(
    query: str,
    example_pool: list[dict],
    n_examples: int = 4,
    strategy: str = "similarity",   # 'similarity'（语义相似度）或 'random'
) -> list[dict]:
    """动态选择最相关的 few-shot 示例（而非静态固定示例）。"""
    if strategy == "random":
        import random
        return random.sample(example_pool, min(n_examples, len(example_pool)))
    elif strategy == "similarity":
        model = SentenceTransformer("all-MiniLM-L6-v2")
        query_emb = model.encode([query])[0]
        example_embs = model.encode([e["input"] for e in example_pool])
        similarities = np.dot(example_embs, query_emb)
        top_indices = np.argsort(similarities)[-n_examples:][::-1]
        return [example_pool[i] for i in top_indices]

# 动态 Few-shot 适用场景：示例池很大无法全放进 prompt；不同查询类型适合不同示例
```

### 2.4 Few-shot 的局限

- **上下文窗口消耗**：每个示例都占用 token，示例越多越贵
- **示例选择困难**：选不好的示例可能让效果更差
- **动态任务不适用**：固定示例无法覆盖所有情况

## 三、Chain-of-Thought（CoT）：让模型"先想后答"

### 3.1 初印象：用费曼视角理解

**Chain-of-Thought（思维链）就是让模型把推理过程写出来，而不是直接跳到结论**。人类解决复杂问题时通常在脑子里"打草稿"——比如算"32×47"：

- 32×7 = 224
- 32×40 = 1280
- 224 + 1280 = 1504

写出来不容易出错，也更容易检查。CoT 给 LLM 提供了同样的"草稿纸"。

### 3.2 为什么 CoT 有效

两个层面的原因：

1. **序列生成的自我约束**：当模型写出"第一步…第二步…"，后续内容受前面步骤约束，不容易逻辑跳跃。
2. **推理时的额外计算**：LLM 是自回归模型，每个 token 的生成都以所有之前的 token 为条件。当模型把推理步骤写出来，后续 token 就以这些步骤为条件——等于在推理时"在 context 里做了更多计算"。推理步骤越明确，模型生成最终答案时能利用的"工作空间"越大，出错率越低。

研究发现，性能提升主要发生在模型非常大（约 1000 亿参数）之后，较小的模型产生的推理链看似连贯但实际错误，反而比标准提示更差。

### 3.3 三种 CoT 变体

**变体一：Zero-shot CoT（魔法咒语"一步一步来"）。** 最简单的 CoT 不需要写示例，只需在 Prompt 末尾加一句"**让我们一步一步思考（Let's think step by step）**"，就像一个开关激活了模型的逐步推理模式。

```python
def zero_shot_cot(question: str, cot_trigger: str = "让我们一步一步思考。") -> str:
    return f"{question}\n\n{cot_trigger}"

# CoT 触发语的变体（都有类似效果）
cot_triggers = [
    "让我们一步一步思考。",
    "Let's think step by step.",
    "请先分析问题，再给出答案。",
    "请展示你的推理过程。",
]
```

**变体二：Few-shot CoT（标准 CoT，2022）。** 示例里不只有输入输出，还有详细推理过程：

```python
cot_examples = [
    {
        "question": "Roger 有 5 个网球。他买了 2 罐网球，每罐 3 个。他现在有几个网球？",
        "reasoning": "Roger 开始有 5 个网球。2 罐 × 每罐 3 个 = 6 个。5 + 6 = 11 个。",
        "answer": "11 个",
    },
    {
        "question": "食堂有 23 个苹果。做午饭用了 20 个，买了 6 个。现在有几个？",
        "reasoning": "开始有 23 个。用了 20 个，剩 23 - 20 = 3 个。又买了 6 个，3 + 6 = 9 个。",
        "answer": "9 个",
    },
]

def few_shot_cot_prompt(examples: list[dict], new_question: str) -> str:
    prompt = ""
    for ex in examples:
        prompt += f"问题：{ex['question']}\n推理：{ex['reasoning']}\n答案：{ex['answer']}\n\n"
    prompt += f"问题：{new_question}\n推理："
    return prompt
```

**变体三：Plan-and-Solve（先规划，再执行）。** 先让模型制定解题计划，再按步骤执行，适合需要全局规划的复杂任务：

```python
plan_and_solve = """
停车场有 3 排车，每排 17 辆。停车场里有多少辆车？

请先制定解题计划，然后按步骤执行：
计划：
执行：
答案：
"""
```

### 3.4 CoT 的效果数据与使用场景

对 GPT-3 在 GSM8K 问题集上的错误分析显示：计算错误占 7%，步骤缺失错误占 12%，语义理解错误占 27%。**CoT 主要解决的是"步骤缺失"这类错误**——模型直接跳到答案导致中间步骤出错。对纯计算错误，CoT 帮助有限（这是 LLM 的固有弱点）。

```python
cot_scenarios = {
    "强烈推荐 CoT": [
        "多步数学推理（GSM8K 类）", "逻辑推理（从前提推导结论）",
        "因果分析（复杂系统分析）", "代码调试（逐步分析问题）", "多步规划/决策权衡",
    ],
    "CoT 效果一般": [
        "简单事实查询（问首都、日期）", "纯计算（LLM 计算能力有限）", "创意写作（不是推理任务）",
    ],
    "CoT 可能有害": [
        "需要短答案的任务（CoT 浪费 token）", "弱模型（< 7B）：可能输出错误的推理步骤",
    ],
}

def should_use_cot(task_type: str, model_size: str) -> str:
    """判断是否应该使用 CoT。"""
    large_model = model_size in ["7B+", "13B+", "70B+"]
    if not large_model:
        return "⚠️  小模型（< 7B）使用 CoT 可能产生看似合理但实际错误的推理链，慎用"
    if any(t in task_type for t in ["数学", "逻辑推理", "代码调试", "多步规划"]):
        return "✅  推荐使用 CoT，尤其是 Few-shot CoT"
    if any(t in task_type for t in ["分类", "翻译", "摘要", "简单问答"]):
        return "❌  不需要 CoT，直接 Zero-shot 或 Few-shot"
    return "🤔  视情况而定，先试 Zero-shot，效果差再加 CoT"
```

### 3.5 高质量 CoT 推理步骤的设计要素

```python
# ❌ 糟糕的 CoT（步骤逻辑混乱，应该是 240/60，不是 60/240）
bad_cot = """
Q: 一列火车以60公里/小时的速度行驶，需要多少时间走完240公里？
A: 60除以240，等于4小时。
"""

# ✅ 好的 CoT（步骤清晰、逻辑明确、单位追踪）
good_cot = """
Q: 一列火车以60公里/小时的速度行驶，需要多少时间走完240公里？
A: 已知信息：速度 = 60 公里/小时，距离 = 240 公里
   使用公式：时间 = 距离 ÷ 速度
   时间 = 240 公里 ÷ 60 公里/小时 = 4 小时
   答案：需要 4 小时。
"""
```

好的 CoT 推理步骤的设计要素：**明确列出已知信息** → **指明使用的公式或推理规则** → **逐步展示计算过程并带单位** → **验证答案是否合理（可选）** → **用明确标记区分过程和答案（如"答案："）**。

## 四、Self-Consistency：多数投票提升可靠性

### 4.1 初印象：用费曼视角理解

**Self-Consistency（自一致性）是"问同一个问题多次，用多数票决定最终答案"**。一次 CoT 可能走错路，但如果用略有不同的推理路径问同样的问题多次（通常 10-40 次），错误的推理路径往往各不相同，而正确答案会被多数路径汇聚到——这就是多数投票的力量。注意：Self-Consistency 通常配合 CoT 使用。

```
同一道题，采样 5 次：
路径1：... 计算 ... → 答案：9   ✅
路径2：... 计算 ... → 答案：9   ✅
路径3：... 推理有误 → 答案：7   ❌
路径4：... 计算 ... → 答案：9   ✅
路径5：... 推理有误 → 答案：11  ❌

多数投票：9（3票 vs 7的1票 vs 11的1票）→ 最终答案：9 ✅
```

### 4.2 为什么有效与代码实现

LLM 的生成有随机性（temperature > 0），同一问题可能走不同的推理路径。**错误往往是随机的、不一致的（噪声），正确答案是系统性的、一致的（信号）**，多数投票过滤掉噪声、保留信号。

```python
import re
from collections import Counter

def self_consistency(
    question: str,
    generate_fn,          # 调用 LLM 的函数
    n_samples: int = 10,  # 采样次数
    temperature: float = 0.7,   # 温度（需要 > 0 才有多样性）
    extract_answer_fn = None,   # 从回答中提取最终答案的函数
) -> dict:
    """
    Self-Consistency 实现：
    1. 用 CoT + 高温度采样多次  2. 从每个回答中提取答案  3. 多数投票
    """
    cot_question = f"{question}\n\n让我们一步一步思考。"
    answers, full_responses = [], []
    for i in range(n_samples):
        response = generate_fn(cot_question, temperature=temperature)
        full_responses.append(response)
        answer = extract_answer_fn(response) if extract_answer_fn else response.strip().split('\n')[-1]
        answers.append(answer)

    vote_counts = Counter(answers)
    majority_answer, majority_votes = vote_counts.most_common(1)[0]
    return {
        "majority_answer": majority_answer,
        "confidence": majority_votes / n_samples,
        "vote_distribution": dict(vote_counts),
        "all_answers": answers,
    }

def extract_math_answer(text: str) -> str:
    """从 CoT 推理中提取最终数字答案"""
    patterns = [
        r'答案[是为：:]\s*(\d+(?:\.\d+)?)',
        r'结果[是为：:]\s*(\d+(?:\.\d+)?)',
        r'=\s*(\d+(?:\.\d+)?)\s*$',
        r'(\d+(?:\.\d+)?)\s*$',
    ]
    for pattern in patterns:
        match = re.search(pattern, text.strip())
        if match:
            return match.group(1)
    return text.strip().split()[-1]   # fallback
```

### 4.3 成本收益与使用建议

Self-Consistency 的代价是 API 调用次数增加（10 次采样 = 10 倍成本），但对高准确率场景通常值得。下面用二项分布粗略估算多数投票的准确率提升：

```python
import math

def sc_cost_benefit(base_accuracy: float, n_samples: int):
    """Self-Consistency 成本收益（假设每次独立，简化估算）"""
    majority_acc = sum(
        math.comb(n_samples, k) * (base_accuracy**k) * ((1-base_accuracy)**(n_samples-k))
        for k in range(n_samples // 2 + 1, n_samples + 1)
    )
    print(f"单次 CoT 准确率 {base_accuracy:.0%} → 采样 {n_samples} 次后约 {majority_acc:.0%}"
          f"（+{(majority_acc - base_accuracy):.0%}，成本 {n_samples}×）")

for acc in [0.5, 0.7, 0.8, 0.9]:
    sc_cost_benefit(acc, n_samples=10)
```

使用指南：

| 维度 | 建议 |
|------|------|
| 采样次数 | 经济型 3-5 次；标准型 8-12 次（收益-成本平衡点）；高精度 20-40 次 |
| 温度设置 | 0.5-0.8（太低多样性不足，太高噪声太多） |
| 适用任务 | 有明确正确答案的推理任务（数学、逻辑、代码） |
| 不适用 | 开放式创意写作（没有"正确"答案，投票无意义） |
| 替代方案 | 验证-修正（让模型自我检查答案，比 Self-Consistency 便宜） |

基础模型准确率在 60-80% 时，Self-Consistency 的提升最明显。

## 五、四种技巧的对比与选型

### 5.1 横向对比

| 技巧 | 示例数量 | Token 消耗 | 准确率提升 | 适合任务 | API 调用 |
|------|---------|-----------|-----------|---------|---------|
| Zero-shot | 0 | 最低 | 基准 | 简单任务、格式明确 | 1次 |
| Few-shot | 2-8 | 中等 | +10-20% | 格式/风格敏感任务 | 1次 |
| Zero-shot CoT | 0 | 中 | +20-40% | 中等复杂推理 | 1次 |
| Few-shot CoT | 2-8 | 中高 | +20-40% | 复杂推理、数学 | 1次 |
| Self-Consistency | 0-8 | 最高（n×） | +10-20% | 高准确率要求 | N次 |

> **重要趋势**：在现代强大的 LLM 上（经过大量推理数据预训练和指令微调），Zero-shot CoT 通常能媲美甚至超过 Few-shot CoT，Few-shot 示例的主要用途已缩减为**强制规范输出格式**。对最新大模型，直接用"让我们一步一步思考" + 格式要求往往就足够了。

### 5.2 选型决策流程

```python
def choose_prompting_strategy(task_description: str) -> str:
    """根据任务特点推荐提示策略（启发式决策）"""
    needs_reasoning = any(w in task_description for w in
                          ["计算", "推理", "分析", "比较", "为什么", "how", "why", "步骤"])
    needs_format    = any(w in task_description for w in
                          ["JSON", "表格", "格式", "结构", "列表"])
    high_accuracy   = any(w in task_description for w in
                          ["准确", "精确", "关键", "重要", "不能出错"])
    simple_task     = len(task_description) < 50 and not needs_reasoning

    if simple_task and not needs_format:
        return "Zero-shot：任务简单，直接问"
    elif needs_format and not needs_reasoning:
        return "Few-shot：给 2-4 个格式示例，确保输出格式稳定"
    elif needs_reasoning and not high_accuracy:
        return "Zero-shot CoT：加上'让我们一步一步思考'，成本低效果好"
    elif needs_reasoning and high_accuracy:
        return "CoT + Self-Consistency：多次采样投票，最高准确率"
    else:
        return "Few-shot CoT：给推理示例 + 步骤，全面覆盖"
```

### 5.4 综合实战：一个鲁棒的 Prompt 设计函数

把四种技巧综合到一个可复用的设计流程里：

```python
import anthropic

def design_robust_prompt(
    task: str,
    examples: list[dict] = None,
    use_cot: bool = True,
    output_format: str = None,
) -> str:
    """设计鲁棒的 Prompt（综合运用角色 / Few-shot / CoT / 格式约束）。"""
    parts = ["你是一位专业的分析师，擅长逻辑推理和数据分析。"]   # 1. 角色定义
    parts.append(f"\n任务：{task}")                              # 2. 任务描述
    if examples:                                                # 3. Few-shot 示例
        parts.append("\n以下是一些参考示例：")
        for i, ex in enumerate(examples, 1):
            parts.append(f"\n示例 {i}：\n输入：{ex['input']}")
            if use_cot and 'reasoning' in ex:
                parts.append(f"推理过程：{ex['reasoning']}")
            parts.append(f"输出：{ex['output']}")
    if output_format:                                           # 4. 输出格式
        parts.append(f"\n请按以下格式输出：\n{output_format}")
    if use_cot:                                                 # 5. CoT 触发语
        parts.append("\n请一步一步思考，确保推理过程清晰。")
    return "\n".join(parts)
```

## 七、面试常见问题

**Q1：Zero-shot、One-shot 和 Few-shot 的区别是什么？**

区别在于 prompt 中包含的示例数量。Zero-shot 不含任何示例，直接描述任务；One-shot 含一个示例；Few-shot 含多个（通常 2-8 个）。示例越多模型越容易理解期望格式，但 token 消耗也越高。对强模型（如 GPT-4、Claude）Zero-shot 往往就够用；对弱模型或需要特定格式时 Few-shot 更稳定。研究发现示例的格式和多样性比标签是否绝对正确更重要。

**Q2：Chain-of-Thought 为什么能提升推理准确率？什么时候用它？**

CoT 让模型生成中间推理步骤，有效扩大了生成最终答案前的"工作空间"。LLM 是自回归的，每个 token 以前面所有 token 为条件——中间步骤被写出来后就成为生成答案的有效上下文，相当于在推理时做了更多计算，同时减少逻辑跳跃。适合多步数学、逻辑推理、代码调试等任务，主要减少"步骤跳跃"类错误。注意：CoT 在小模型（< 7B）上效果差甚至有害，因为小模型可能生成看似合理但错误的推理链。

**Q3：Zero-shot CoT（"让我们一步一步思考"）和 Few-shot CoT 哪个更好？**

在现代强大 LLM 上，Zero-shot CoT 通常能媲美甚至超过 Few-shot CoT，Few-shot 示例的主要用途已缩减为规范输出格式。实践建议：对强模型（7B+，特别是指令微调版本），Zero-shot CoT 就够；对弱模型或需要特定推理风格时，Few-shot CoT 更稳定。考虑 token 成本和维护成本，先试 Zero-shot CoT，效果不够再加 Few-shot 示例。

**Q4：Self-Consistency 的原理是什么？什么场景值得付出额外成本？**

对同一问题用 CoT 采样多次（通常 10-40 次，temperature > 0 保证多样性），然后对最终答案多数投票。原理：错误推理路径各不相同（噪声），正确推理路径收敛到相同答案（信号），多数投票过滤噪声。值得额外成本的场景：高准确率要求（医疗、金融、考试）；任务有唯一正确答案（数学、代码）；基础模型准确率在 60-80% 时提升最明显。不适用于开放式生成或成本敏感场景。

**Q5：选择 Few-shot 示例时应该注意什么？**

四个关键原则：质量第一（示例必须正确，错误示例会严重误导）；多样性（覆盖不同情况，别全来自同一类别）；相关性（示例应和当前查询相似，动态检索优于静态固定）；格式一致性（所有示例输入输出格式必须统一，模型会严格模仿）。此外，最后一个示例对模型影响最大（最近邻效应），把最典型、最规范的示例放在最后。
