## 零、开篇：超越单次问答

上一章的四种技巧都在"一问一答"框架里优化。但很多现实任务不是这种模式：要查阅实时数据、调用代码执行、搜索外部知识库、或者做出需要在多个方向上深入探索的复杂决策。

这时候需要更高级的框架：让模型学会"推理 → 行动 → 观察 → 再推理"的循环（ReAct），或者同时探索多条思维路径选最优的（Tree-of-Thought）。

## 一、ReAct：推理与行动交织的 Agent 框架

### 1.1 初印象：用费曼视角理解

**ReAct 的核心思想："先想，再做，看结果，再想"的循环。**

CoT 让模型一步步推理，但推理中如果需要实时数据，只能凭空捏造（幻觉）。ReAct 在推理步骤中允许模型执行"动作"（搜索、计算、查数据库），拿到工具返回结果后继续推理。

ReAct 让 LLM 生成语言推理轨迹和任务行动，支持与外部环境（如维基百科）交互，将额外信息并入推理过程。

格式：**Thought（思考）→ Action（行动）→ Observation（观察）→ 循环**

### 1.2 ReAct 完整示例

```
问题：2024 年奥运会在哪里举行，那个城市的人口是多少？

Thought: 我需要查找 2024 年奥运会的举办城市。
Action: search("2024年奥运会举办城市")
Observation: 2024年夏季奥运会在法国巴黎举行，时间7月26日至8月11日。

Thought: 现在我知道是巴黎了，需要查询巴黎人口。
Action: search("巴黎人口")
Observation: 巴黎市区人口约213万，大巴黎都市区约1200万。

Thought: 我有了两个问题的答案。
Answer: 2024年奥运会在法国巴黎举行，巴黎市区人口约213万。
```

### 1.3 ReAct 的 Prompt 模板与实现

```python
REACT_SYSTEM_PROMPT = """你是一个有帮助的助手，可以使用以下工具：

工具列表：
- search(query: str): 搜索互联网获取信息
- calculate(expression: str): 计算数学表达式
- lookup_database(table, query): 查询数据库

请使用以下格式：
Thought: [分析当前情况，决定下一步]
Action: [工具名称]([参数])
Observation: [工具返回的结果]
...（可以重复多次）
Thought: [基于所有观察，得出结论]
Answer: [最终答案]
"""

class ReActAgent:
    """简化的 ReAct Agent"""

    def __init__(self, llm_fn, tools: dict, max_steps: int = 10):
        self.llm_fn    = llm_fn
        self.tools     = tools
        self.max_steps = max_steps

    def run(self, question: str) -> str:
        history = f"问题: {question}\n"

        for _ in range(self.max_steps):
            response = self.llm_fn(REACT_SYSTEM_PROMPT + "\n\n" + history)

            if "Answer:" in response:
                return response.split("Answer:")[-1].strip()

            if "Action:" in response:
                # 解析工具调用
                action_line = next(
                    (l for l in response.split('\n') if l.startswith('Action:')), ""
                )
                action_str = action_line.replace('Action:', '').strip()
                tool_name  = action_str.split('(')[0].strip()

                if tool_name in self.tools:
                    # 解析参数（简化：取括号内内容）
                    args = action_str[len(tool_name)+1:-1].strip('"\'')
                    observation = self.tools[tool_name](args)
                else:
                    observation = f"错误：工具 '{tool_name}' 不存在"

                history += response + f"\nObservation: {observation}\n"

        return "无法找到答案"


# 工具函数
def mock_search(query: str) -> str:
    db = {
        "2024年奥运会": "2024年夏季奥运会在法国巴黎举行，7月26日至8月11日",
        "巴黎人口": "巴黎市区约213万，大巴黎都市区约1200万",
    }
    for key, val in db.items():
        if key in query:
            return val
    return "未找到相关信息"

def mock_calculate(expr: str) -> str:
    try:
        return str(eval(expr))
    except:
        return "计算错误"

# 使用
agent = ReActAgent(
    llm_fn=lambda p: "Thought: 我知道答案了\nAnswer: 测试回答",
    tools={"search": mock_search, "calculate": mock_calculate},
)
print("ReAct Agent 工具列表:", list(agent.tools.keys()))
print("\nReAct 的核心价值：")
for k, v in {
    "解决幻觉": "需要实时数据时，通过搜索而非记忆回答",
    "可验证性": "每步 Observation 可以被检查，错误容易发现",
    "可扩展性": "可添加任意工具（代码执行/API/数据库）",
}.items():
    print(f"  {k}: {v}")
```

## 二、Tree-of-Thought：多路径探索最优解

### 2.1 初印象：用费曼视角理解

**CoT 是"一条路走到底"，ToT 是"同时开辟多条路，选最优的"。**

CoT 走错了就错到底。ToT 让模型同时展开多条推理路径，评估每条路径的质量，遇到死路可以回溯换路——就像人类解难题时会考虑多种方案一样。

在 Game of 24（用四个数字通过加减乘除算出 24）上，标准 CoT 成功率仅 4%，而 ToT 提升到了 74%。在创意写作任务中，ToT 生成的文本连贯性和创意也显著更高。

### 2.2 简化版 ToT：单 Prompt 实现

完整 ToT 需要多次 LLM 调用，但有一个简单的单 Prompt 版本——"三位专家"提示：

```python
TOT_SIMPLE_PROMPT = """想象三位不同的专家在回答这个问题。
所有专家都会写下他们思考的第一步，然后与小组分享。
然后所有专家继续下一步，以此类推。
如果任何专家意识到自己的思路有误，他们会立即退出。

问题是：{question}"""

# 使用

prompt = TOT_SIMPLE_PROMPT.format(question=q)
print("简化版 ToT Prompt（三位专家模式）：")
print(prompt)
```

### 2.3 完整 ToT：BFS 搜索实现

```python
from typing import Optional

class ThoughtNode:
    def __init__(self, content: str, parent: Optional['ThoughtNode'] = None):
        self.content  = content
        self.parent   = parent
        self.score    = 0.0
        self.depth    = 0 if parent is None else parent.depth + 1

    def get_path(self) -> list[str]:
        path, node = [], self
        while node:
            path.append(node.content)
            node = node.parent
        return list(reversed(path))


class TreeOfThought:
    """
    Tree-of-Thought：展开 → 评估 → Beam Search 选最优 → 继续展开
    """
    def __init__(self, llm_fn, n_branches=3, max_depth=4, beam_size=2):
        self.llm_fn     = llm_fn
        self.n_branches = n_branches
        self.max_depth  = max_depth
        self.beam_size  = beam_size

    def expand(self, node: ThoughtNode, problem: str) -> list[str]:
        """生成多个下一步思路"""
        path = "\n".join(node.get_path())
        prompt = f"""问题：{problem}

已有思考：{path}

生成 {self.n_branches} 个不同的下一步方向，用"---"分隔："""
        resp = self.llm_fn(prompt)
        return [t.strip() for t in resp.split("---") if t.strip()][:self.n_branches]

    def evaluate(self, node: ThoughtNode, problem: str) -> float:
        """给当前思路打分（0-1）"""
        path = "\n".join(node.get_path())
        prompt = f"""问题：{problem}\n思路：{path}\n\n评分(0-10，仅输出数字)："""
        try:
            return float(self.llm_fn(prompt).strip().split()[0]) / 10.0
        except:
            return 0.5

    def solve(self, problem: str) -> dict:
        """BFS + Beam Search 搜索最优路径"""
        root = ThoughtNode(f"问题：{problem}")
        beam = [root]

        for _ in range(self.max_depth):
            candidates = []
            for node in beam:
                for thought in self.expand(node, problem):
                    child = ThoughtNode(thought, parent=node)
                    child.score = self.evaluate(child, problem)
                    candidates.append(child)
            if not candidates:
                break
            candidates.sort(key=lambda n: n.score, reverse=True)
            beam = candidates[:self.beam_size]

        best = beam[0] if beam else root
        best_path = best.get_path()

        final = self.llm_fn(
            f"问题：{problem}\n最优思路：\n{''.join(best_path)}\n\n最终答案："
        )
        return {"path": best_path, "answer": final, "score": best.score}


# 演示（用 mock LLM）
def mock_llm(prompt: str) -> str:
    if "生成" in prompt:
        return "---\n制定学习计划\n---\n找学习伙伴\n---\n项目实践"
    if "评分" in prompt:
        return "7"
    return "综合以上思考，建议系统学习并结合项目实践..."

tot = TreeOfThought(mock_llm, n_branches=3, max_depth=2, beam_size=2)
result = tot.solve("如何在三个月内找到第一份 AI 工程师工作？")
print(f"ToT 最优路径（{len(result['path'])} 步）：")
for i, step in enumerate(result['path'], 1):
    print(f"  Step {i}: {step[:60]}")
print(f"最终答案: {result['answer'][:80]}...")
```

### 3.2 Prompt 链（Prompt Chaining）

**把复杂任务拆成多个小 Prompt，每步输出是下一步输入。**

```python
def document_processing_chain(english_text: str, llm_fn) -> dict:
    """
    多步 Prompt 链：英文文章 → 翻译 → 提取要点 → 摘要 → 分类标签
    每步都可以单独验证，出错容易定位。
    """
    results = {}

    # Step 1：翻译
    results["translation"] = llm_fn(
        f"将以下英文翻译成中文，保持学术用语准确：\n\n{english_text}\n\n中文翻译："
    )

    # Step 2：提取要点（基于翻译结果）
    results["key_points"] = llm_fn(
        f"从以下文本提取 3-5 个核心要点，每点一句话，以'-'开头：\n\n"
        f"{results['translation']}\n\n核心要点："
    )

    # Step 3：生成摘要（基于要点）
    results["summary"] = llm_fn(
        f"根据以下要点，写 100 字以内的摘要：\n\n{results['key_points']}\n\n摘要："
    )

    # Step 4：添加标签（基于摘要）
    results["tags"] = llm_fn(
        f"为以下摘要选择 2-3 个学科标签\n"
        f"（从：机器学习/自然语言处理/计算机视觉/强化学习/其他 中选）：\n\n"
        f"{results['summary']}\n\n标签（逗号分隔）："
    )

    return results


# Prompt 链 vs 单次超长 Prompt 的对比
print("Prompt 链的优势：")
for benefit in [
    "中间结果可以验证和修正",
    "每步 Prompt 更简单，模型更不容易出错",
    "便于调试（知道哪一步出了问题）",
    "独立步骤可以并行执行，提高速度",
    "不同步骤可以用不同模型（便宜的做简单步骤）",
]:
    print(f"  • {benefit}")
```

## 五、面试常见问题

**Q1：ReAct 框架是什么？和 CoT 有什么本质区别？**

CoT 让模型在单次推理中写出中间步骤，但所有信息来自模型内部知识，无法访问外部信息。ReAct 在推理过程中允许模型执行"动作"（搜索、计算、查数据库），拿到工具返回结果后继续推理，格式是 Thought → Action → Observation 循环。核心区别：CoT 是"自言自语"，ReAct 是"边想边查"；CoT 依赖训练知识，ReAct 可以访问实时外部信息。

**Q2：Tree-of-Thought 和 CoT 有什么区别？**

CoT 是单条线性推理链，一旦走错路只能错到底。ToT 同时展开多条推理路径，对每条评估分数，用 BFS/DFS 搜索最优路径，遇到死胡同可以回溯换路。Game of 24 任务上 CoT 成功率仅 4%，ToT 达到 74%，因为该任务需要试错回溯。ToT 适合有多种解法的问题，但需要更多 LLM 调用，成本更高。

**Q3：什么是 Prompt 链？什么时候比单次 Prompt 更好？**

Prompt 链把复杂任务拆成多个步骤，每步输出作为下一步输入。适用场景：任务太复杂（翻译+摘要+分类同时做容易出错）；中间结果需要验证；多个子任务相互独立可以并行；需要清晰的调试路径。好处：每步 Prompt 简单准确率更高；中间结果可检查和修正；独立步骤可并行加速。

**Q4：如何设计一个好的 System Prompt？**

六个要素：角色定义（身份和专长）；核心职责（具体任务范围）；输出格式（精确的 JSON Schema 或模板）；评判标准（如打分的具体规则）；注意事项（禁止项、边界情况）；可选示例（复杂格式可附 1-2 个完整例子）。好的 System Prompt 应该稳定不变，只让 User Prompt 变化，有助于控制输出一致性。

**Q5：元提示（Meta-Prompting）有什么用？**

元提示是用 LLM 优化 Prompt 本身，给模型"Prompt 改进专家"的角色，把模糊的原始 Prompt 改写为结构化清晰的版本。用处：快速改进写得不好的 Prompt；生成多种 Prompt 变体用于 A/B 测试；对不熟悉提示工程的用户友好（写模糊 Prompt，系统自动改进）；可以与 DSPy 等框架结合，实现 Prompt 的自动优化。
