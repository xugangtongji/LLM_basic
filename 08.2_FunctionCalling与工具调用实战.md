## 零、开篇：工具调用是 Agent 的核心原语

上一章讲了 Agent 的"大图景"——四个要素、历史演进、框架对比。这一章进入核心工程问题：**工具调用（Tool Use / Function Calling）到底是怎么工作的？**

一句话描述工具调用的本质：

**模型不执行代码，它只负责决定"调用什么工具、传什么参数"。你的应用程序接收这个决定，真正地执行，再把结果返回给模型，模型用结果继续推理。**

这个设计很关键：LLM 是"大脑"，它决策；你的代码是"手脚"，它执行。两者分工明确，模型不会直接碰你的数据库或者真的发出网络请求——这也是 Agent 可以设置安全边界的原因。

## 一、Function Calling 的工作流程

### 1.1 四步基本循环

尽管各提供商的格式不同，但模式始终相同：定义工具、把消息和工具一起发送、检测工具调用、执行函数、返回结果。

更具体地展开，是这样一个循环：

```
【第一步】你定义工具
  告诉模型：我有哪些工具，每个工具叫什么名字，
  接受什么参数，做什么事情。

【第二步】用户提问 + 工具列表 → 发给模型
  模型阅读问题和工具列表，判断：
  "这个问题我需要用工具才能回答吗？用哪个？参数是什么？"

【第三步】模型返回工具调用请求（不是最终答案）
  如果需要工具，模型不会直接生成答案，
  而是返回一个结构化的调用请求：
  {"name": "get_weather", "arguments": {"location": "北京"}}

【第四步】你的代码执行工具，把结果还给模型
  你的代码接收这个请求，真正调用天气 API，
  把结果作为新消息返回给模型。

【第五步】模型用结果生成最终回答
  模型看到天气数据，生成自然语言回答。
```

这个循环可以重复多次——模型可能先查天气，再查日历，再做计算，最后才生成最终答案。这就是"Agent 循环"（Agentic Loop）。

### 1.2 为什么这样设计

一个常见的疑问：为什么不直接让模型执行代码？

原因有三：

**安全隔离**：模型直接执行代码意味着模型有权访问你的所有系统。通过"模型决策，你执行"的分离，你可以在执行层设置权限控制——哪些工具可以调用，哪些参数值合法，是否需要人工确认。

**可靠性**：你的业务逻辑和数据验证写在你自己的代码里，不依赖模型正确地生成合法代码。

**可审计性**：每一次工具调用都经过你的代码，可以记录日志、设置超时、处理错误。

## 二、Anthropic Claude 的工具调用

### 2.1 定义工具

每个工具定义包含：名称、描述，以及 input_schema（用 JSON Schema 定义参数）。当你调用 Claude API 时包含 tools 参数，API 会从工具定义中构建一个特殊的系统提示，告诉模型如何使用指定的工具。

```python
import anthropic
import json

client = anthropic.Anthropic()

# 定义工具：告诉模型"我有什么工具"
tools = [
    {
        "name": "get_weather",
        "description": "获取指定城市的当前天气信息。当用户询问某个地方的天气时使用。",
        "input_schema": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "城市名称，例如：北京、上海、London"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "温度单位，默认使用 celsius"
                }
            },
            "required": ["location"]
        }
    },
    {
        "name": "search_web",
        "description": "搜索互联网获取最新信息。当用户询问实时信息、最新新闻或你不确定的事实时使用。",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "搜索查询词"
                }
            },
            "required": ["query"]
        }
    }
]
```

### 2.2 工具的真实实现

工具定义只是"说明书"，真正的逻辑在你的代码里：

```python
def get_weather(location: str, unit: str = "celsius") -> dict:
    """实际调用天气 API（这里用模拟数据演示）"""
    # 真实场景替换为：requests.get("https://api.weather.com/...")
    mock_data = {
        "北京": {"temperature": 22, "condition": "晴天", "humidity": "45%"},
        "上海": {"temperature": 28, "condition": "多云", "humidity": "72%"},
        "London": {"temperature": 15, "condition": "Cloudy", "humidity": "80%"},
    }
    data = mock_data.get(location, {"temperature": 20, "condition": "Unknown", "humidity": "N/A"})
    if unit == "fahrenheit":
        data["temperature"] = data["temperature"] * 9/5 + 32
    data["unit"] = unit
    data["location"] = location
    return data

def search_web(query: str) -> dict:
    """实际调用搜索 API（模拟）"""
    return {
        "results": [
            {"title": f"关于'{query}'的搜索结果1", "snippet": "这是搜索摘要..."},
            {"title": f"关于'{query}'的搜索结果2", "snippet": "另一个相关结果..."}
        ],
        "query": query
    }

# 工具注册表：工具名称 → 函数的映射
TOOL_REGISTRY = {
    "get_weather": get_weather,
    "search_web": search_web,
}

def execute_tool(tool_name: str, tool_input: dict) -> str:
    """执行工具调用，返回结果字符串"""
    if tool_name not in TOOL_REGISTRY:
        return f"错误：工具 '{tool_name}' 不存在"
    try:
        result = TOOL_REGISTRY[tool_name](**tool_input)
        return json.dumps(result, ensure_ascii=False)
    except Exception as e:
        return f"工具执行错误：{str(e)}"
```

### 2.3 完整的 Agent 循环

```python
def run_agent(user_message: str, max_turns: int = 10) -> str:
    """
    完整的 Agent 循环：
    支持多步工具调用，直到模型给出最终答案或达到最大轮数。
    """
    messages = [{"role": "user", "content": user_message}]
    
    for turn in range(max_turns):
        # 调用 Claude，带上工具列表
        response = client.messages.create(
            model="claude-opus-4-6",
            max_tokens=4096,
            tools=tools,
            messages=messages,
        )
        
        # 情况一：模型直接给出答案（不需要工具）
        if response.stop_reason == "end_turn":
            # 提取文本回答
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
            return "（无文本回答）"
        
        # 情况二：模型需要调用工具
        if response.stop_reason == "tool_use":
            # 把模型的回应加入对话历史
            messages.append({"role": "assistant", "content": response.content})
            
            # 执行所有工具调用（可能同时调用多个工具）
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"[Agent] 调用工具: {block.name}，参数: {block.input}")
                    result = execute_tool(block.name, block.input)
                    print(f"[Agent] 工具结果: {result[:100]}...")
                    
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            
            # 把工具结果作为新消息加入对话历史
            messages.append({"role": "user", "content": tool_results})
            # 继续下一轮，让模型基于工具结果继续推理
            continue
        
        # 其他情况（异常）
        break
    
    return "Agent 达到最大轮数限制，未能完成任务"


# 测试
if __name__ == "__main__":
    questions = [
        "北京今天天气怎么样？适合出去运动吗？",
        "帮我比较一下北京和上海的天气，然后推荐去哪个城市旅游",
    ]
    
    for q in questions:
        print(f"\n{'='*50}")
        print(f"用户：{q}")
        print(f"{'='*50}")
        answer = run_agent(q)
        print(f"Agent：{answer}")
```

## 三、工具定义的工程艺术

### 3.1 工具描述比实现更重要

Claude Sonnet 3.5 在 SWE-bench Verified 评估上实现了最先进的性能，是在我们对工具描述进行精确细化之后，大幅降低了错误率并提高了任务完成率。

工具能不能被模型正确使用，80% 取决于描述写得好不好。JSON Schema 定义了结构上合法的内容，但无法表达使用模式：何时应该包含可选参数，哪些组合有意义，或者你的 API 期望什么约定。

**好的工具描述应该回答三个问题**：

1. **什么时候用它**："当用户询问...时使用"
2. **不该什么时候用它**（如果容易混淆）："注意：只用于查询实时数据，不用于历史数据"
3. **参数的具体含义**：每个参数的格式、范围、示例

**差的工具描述（模糊）**：
```json
{
    "name": "query_db",
    "description": "查询数据库"
}
```

**好的工具描述（清晰）**：
```json
{
    "name": "query_customer_orders",
    "description": "查询指定客户的订单历史。当用户询问订单状态、购买记录或退款情况时使用。只查询过去 12 个月的数据。",
    "input_schema": {
        "type": "object",
        "properties": {
            "customer_id": {
                "type": "string",
                "description": "客户 ID，格式为 'CUST-' 加 6 位数字，例如 'CUST-123456'"
            },
            "status_filter": {
                "type": "string",
                "enum": ["all", "pending", "completed", "cancelled"],
                "description": "订单状态过滤器，不指定时返回所有状态"
            }
        },
        "required": ["customer_id"]
    }
}
```

### 3.2 工具响应的设计原则

设计工具响应时应只返回高信号信息。返回语义上稳定的标识符（如 slug 或 UUID）而非不透明的内部引用，且只包含 Claude 推理下一步所需的字段。臃肿的响应会浪费上下文并使 Claude 更难提取关键信息。

工具返回的数据会进入 LLM 的上下文。如果返回 1 万行 JSON，大量 token 被浪费，而且模型可能从海量信息中找不到关键点。

**原则**：工具只返回模型下一步推理所需的最少信息。

```python
# 差的工具响应（信息过多）
def get_user_info_bad(user_id: str) -> dict:
    user = database.get_user(user_id)
    return user  # 返回用户的所有字段，包括内部 ID、哈希值、日志...

# 好的工具响应（只返回关键信息）
def get_user_info_good(user_id: str) -> dict:
    user = database.get_user(user_id)
    return {
        "name": user.name,
        "email": user.email,
        "account_status": user.status,
        "last_order_date": user.last_order_date,
        # 省略内部 ID、密码哈希、操作日志等无关字段
    }
```

### 3.3 错误处理：给模型有意义的错误信息

当工具遇到错误时，在 tool_result 块的内容中向 Claude 发送清晰且具有信息量的错误消息。不要只返回通用的"失败"，而应该返回类似"错误：在天气数据库中找不到位置 'Atlantis'"这样的信息，这有助于 Claude 理解工具为何失败，并可能再次尝试或正确告知用户。

```python
def get_weather_with_error_handling(location: str) -> str:
    try:
        result = call_weather_api(location)
        return json.dumps(result)
    except LocationNotFoundError:
        # 给模型有意义的错误，让它能自我纠正
        return f"错误：找不到位置 '{location}'。请检查城市名称是否正确，或尝试使用英文名称。"
    except APIRateLimitError:
        return "错误：天气 API 请求频率超限，请稍后重试。"
    except Exception as e:
        return f"错误：获取天气信息失败 - {str(e)}"
```

## 四、MCP：工具调用的标准化协议

### 4.1 为什么需要 MCP

在 MCP（Model Context Protocol）出现之前，每个工具都要为每个模型提供商单独适配：同样的数据库查询工具，要为 OpenAI 写一份、为 Claude 写一份、为 Gemini 写一份。这是大量重复工作。

更深层的问题是：OpenAI 的函数调用将每次函数调用视为孤立的交互，需要外部编排来维护状态或管理多步骤工作流。 对于复杂的多步任务，你需要自己维护状态，自己管理工具调用的上下文连续性。

### 4.2 MCP 做了什么

模型上下文协议（MCP）是函数调用领域自 OpenAI 于 2023 年引入该功能以来最重要的发展。MCP 定义了将 AI 模型连接到外部工具和数据源的标准协议。一次构建集成，即可在 Claude、GPT、Gemini 和任何兼容 MCP 的客户端上运行，无需针对每个提供商编写适配代码。

简单说，MCP 做了三件事：

**标准化工具格式**：工具的定义、调用、响应有统一的协议，不再需要为不同模型提供商单独适配。

**服务器化工具**：工具作为独立的 MCP 服务器运行，可以被任何兼容的客户端调用。已经有大量开源的 MCP 服务器（GitHub、Slack、数据库、浏览器控制等），可以直接接入。

**解决状态管理**：MCP 在模型的上下文窗口中嵌入工具使用，支持更丰富和连续的交互模式，维护跨多次调用的持久状态，使变量、中间结果和工具状态在任务过程中能够一致使用。

### 4.3 MCP 的现实意义

OpenAI 和 Google 都已采用 MCP，OpenAI 已弃用其 Assistants API，支持 MCP，并计划在 2026 年中期关闭前者。现在基于 MCP 构建意味着随着生态系统趋同，需要的迁移工作将更少。

对于开发者来说，MCP 正在成为工具集成的事实标准。今天花时间学 MCP，是对未来的投资。

**使用 MCP 工具（Claude SDK 示例）**：

```python
import anthropic

client = anthropic.Anthropic()

# 使用 Anthropic 提供的服务端工具（无需自己处理执行）
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    # 服务端工具：Anthropic 的基础设施直接执行，不需要你的代码处理
    tools=[
        {"type": "web_search_20260209", "name": "web_search"},  # 网页搜索
    ],
    messages=[{
        "role": "user",
        "content": "2025 年诺贝尔物理学奖是谁获得的？"
    }]
)

# 服务端工具的结果直接在 response 里，无需 Agent 循环
print(response.content)
```

## 五、工程实践：从单工具到生产级 Agent

### 5.1 工具数量的权衡

工具越多，Agent 越强大，但也越复杂：

- 工具太少：Agent 能做的事情有限
- 工具太多：模型选择工具的准确性下降（"选择困难"），推理成本增加

上下文中工具太多会导致"上下文污染"——当 Claude 分析一个 10MB 的日志文件寻找错误模式时，整个文件都进入了上下文窗口，即使 Claude 只需要错误频率的摘要。

**实践原则**：从最少的工具开始，只有当 Agent 明显需要某个能力时才添加。一个功能聚焦的 Agent（只做一件事，工具精简）通常比一个功能全面但混乱的 Agent 更可靠。

### 5.2 需要人工确认的操作

不是所有工具调用都应该自动执行。删除数据、发送邮件、进行支付这类"不可逆"操作，应该在执行前请求人工确认：

```python
HIGH_RISK_TOOLS = {"delete_file", "send_email", "make_payment", "drop_table"}

def execute_tool_with_confirmation(tool_name: str, tool_input: dict) -> str:
    """高风险工具在执行前请求人工确认"""
    if tool_name in HIGH_RISK_TOOLS:
        print(f"\n⚠️  高风险操作请求确认")
        print(f"工具：{tool_name}")
        print(f"参数：{json.dumps(tool_input, ensure_ascii=False, indent=2)}")
        confirm = input("是否执行？(y/n): ")
        if confirm.lower() != 'y':
            return "操作已被用户取消"
    
    return execute_tool(tool_name, tool_input)
```

### 5.3 常见的工程问题和解决方案

| 问题 | 根因 | 解决方案 |
|------|------|---------|
| 模型调用了不存在的工具 | 工具描述模糊 | 完善工具描述，加示例 |
| 模型传了错误格式的参数 | Schema 没有明确约束 | 加 enum 和 pattern 约束，加参数说明 |
| Agent 进入无限循环 | 工具调用错误没有终止条件 | 设置 max_turns 上限，检测重复调用 |
| 工具返回太多 token | 响应没有过滤 | 只返回必要字段，加分页或截断 |
| Agent 忽略工具错误继续推进 | 错误信息不够明确 | 给模型结构化的错误和重试建议 |

## 六、面试常见问题

**Q1：Function Calling 的工作原理是什么？模型真的"执行"了函数吗？**

不，模型不执行函数。模型的工作是决策：分析用户问题和工具列表，判断是否需要工具、调用哪个、传什么参数，然后以结构化 JSON 格式返回这个决策。真正的执行在你的应用代码里。你收到模型的决策，调用真实的函数或 API，把结果以新消息的形式返回给模型，模型再基于结果继续推理或生成最终答案。这个"模型决策-代码执行-结果反馈"的循环就是 Agent Loop。

**Q2：工具定义里最重要的部分是什么？**

描述（description）比参数 Schema 更重要。Schema 告诉模型"参数格式是什么"，但描述告诉模型"什么时候应该用这个工具"。如果描述写得不清楚，模型要么该用的时候不用，要么不该用的时候乱用。好的描述应该明确说明适用场景、参数的具体含义和格式要求，对于容易混淆的工具还要说明不适用的情况。

**Q3：Agent 循环中应该如何处理工具执行错误？**

不要返回通用的"失败"，而要返回具体的、可操作的错误信息，让模型能理解发生了什么，并可能自我纠正。例如"错误：找不到位置 'Atlantis'，请检查城市名称"比"天气查询失败"有用得多。在代码层面，用 try-catch 包装工具执行，捕获不同类型的异常，分别返回对应的有意义错误信息。

**Q4：什么是 MCP？它解决了什么问题？**

MCP（Model Context Protocol）是 Anthropic 于 2024 年 11 月发布的工具调用标准协议，现已被 OpenAI 和 Google 采用。它解决了两个主要问题：标准化（工具一次定义，在所有兼容模型上运行，不需要为每个提供商单独适配）；状态管理（MCP 维护跨多次调用的持久状态，使复杂多步任务的上下文管理更简单）。OpenAI 已宣布弃用 Assistants API 转向 MCP，MCP 正在成为工具集成的事实标准。
