## 一、GPU vs CPU：为什么 GPU 能做深度学习

### 1.1 初印象：用费曼视角理解

**CPU 是一个"全能专家"，GPU 是一个"人海战术工厂"**。

CPU 通常有 8 到 128 个核心，每个核心很强，能处理复杂的逻辑分支、乱序执行、大缓存——它适合做各种复杂的事情，但同时只能并行做很少的任务。

GPU 有成千上万个小核心（NVIDIA A100 有 6912 个 CUDA Core），每个核心很简单，只会做基本运算，但它们**同时**一起干活。矩阵乘法这种"对很多数做同样的事"的任务，简直是为 GPU 量身定做的。

深度学习的本质就是反复做矩阵乘法，所以 GPU 是天作之合。

### 1.2 GPU 架构的三层结构

理解 GPU 性能，需要记住三层结构：

```
GPU
├── SM（Streaming Multiprocessor，流多处理器）×N
│   ├── CUDA Core（FP32 计算单元）×128
│   ├── Tensor Core（矩阵专用计算单元）×4
│   ├── 寄存器（Register File）      <- 最快，但极少
│   └── 共享内存（Shared Memory）    <- 快，可编程
└── 全局显存（HBM）                  <- 慢，但大
```

**主流 GPU 参数对比**：

| GPU | SM 数量 | CUDA Cores | Tensor Cores | 显存 | 显存带宽 | FP16 峰值算力 |
|-----|---------|-----------|-------------|------|---------|-------------|
| V100 | 80 | 5120 | 640 | 32/80 GB HBM2 | 900 GB/s | 125 TFLOPS |
| A100 | 108 | 6912 | 432 | 40/80 GB HBM2e | 2000 GB/s | 312 TFLOPS |
| H100 SXM | 132 | 16896 | 528 | 80 GB HBM3 | 3350 GB/s | 989 TFLOPS |
| RTX 4090 | 128 | 16384 | 512 | 24 GB GDDR6X | 1008 GB/s | 330 TFLOPS |

**关键比率**：显存带宽 / 算力 = "算术强度门槛"。操作的算术强度（FLOP/Byte）高于此值时，瓶颈在算力（compute-bound）；低于此值时，瓶颈在显存带宽（memory-bound）。

### 1.3 两类运算瓶颈

**算力瓶颈（Compute-bound）**：
- 典型操作：大矩阵乘法（`nn.Linear` 的前向、Attention 的 $QK^T$、$PV$）
- 特征：GPU 利用率接近 100%，带宽利用率低
- 优化方向：Tensor Core、量化（更高吞吐的数据类型）、算子融合

**带宽瓶颈（Memory-bound）**：
- 典型操作：Softmax、LayerNorm、逐元素操作（激活函数、残差加法）
- 特征：GPU 大量时间在读写显存，计算单元闲置
- 优化方向：算子融合（减少读写次数）、Flash Attention（在 SRAM 里完成计算）

```python
import torch
import time

device = "cuda"
N = 4096

# 算力瓶颈：大矩阵乘法（compute-bound）
A = torch.randn(N, N, device=device, dtype=torch.float16)
B = torch.randn(N, N, device=device, dtype=torch.float16)

torch.cuda.synchronize()
t0 = time.perf_counter()
for _ in range(100):
    C = A @ B
torch.cuda.synchronize()
matmul_time = (time.perf_counter() - t0) / 100

flops = 2 * N ** 3
throughput = flops / matmul_time / 1e12
print(f"矩阵乘法：{matmul_time*1000:.2f}ms  算力利用：{throughput:.1f} TFLOPS")

# 带宽瓶颈：逐元素操作（memory-bound）
x = torch.randn(N, N, device=device, dtype=torch.float16)

torch.cuda.synchronize()
t0 = time.perf_counter()
for _ in range(100):
    y = torch.exp(x)
torch.cuda.synchronize()
exp_time = (time.perf_counter() - t0) / 100

bytes_moved = x.numel() * 2 * 2  # 读 + 写，fp16 = 2 bytes
bandwidth = bytes_moved / exp_time / 1e9
print(f"逐元素 exp：{exp_time*1000:.2f}ms  带宽利用：{bandwidth:.1f} GB/s")
```

## 二、显存模型：OOM 的根本原因

### 2.1 初印象：用费曼视角理解

**显存就是 GPU 的工作台**，你要处理的材料（参数、激活值、梯度）都得摆在工作台上才能加工。工作台大小固定，装不下就 OOM（Out Of Memory，显存溢出）。

大模型训练时工作台上放了什么？比你想象的多得多。

### 2.2 训练时显存的四大构成

以一个 7B 参数模型为例，用 BF16 训练：

**1. 模型参数（Parameters）**

$$\text{显存} = N_{\text{params}} \times \text{bytes per param}$$

- FP32：7B × 4 bytes = **28 GB**
- BF16/FP16：7B × 2 bytes = **14 GB**

**2. 梯度（Gradients）**

每个参数对应一个同形状的梯度张量：

- FP32：7B × 4 bytes = 28 GB
- BF16：7B × 2 bytes = 14 GB（大模型训练常用 BF16 梯度）

**3. 优化器状态（Optimizer States）**

Adam/AdamW 要为每个参数保存一阶矩 $m$ 和二阶矩 $v$，**必须是 FP32** 以保证数值精度：

$$\text{AdamW 状态} = N_{\text{params}} \times 2 \times 4 \text{ bytes} = 7\text{B} \times 8 \text{ bytes} = 56 \text{ GB}$$

**4. 激活值（Activations）**

前向传播的中间结果，用于反向传播计算梯度。这部分随 batch size 和序列长度线性增长：

$$\text{粗估} \approx 12 \times \text{seq\_len} \times \text{hidden\_size} \times \text{n\_layers} \times \text{batch\_size} \times 2 \text{ bytes}$$

LLaMA-7B，seq=2048，batch=1，BF16：约 12 GB

**汇总表（7B 模型，BF16 训练，AdamW）**：

| 组成 | 大小 |
|------|------|
| 参数 (BF16) | 14 GB |
| 梯度 (BF16) | 14 GB |
| 优化器状态 (FP32) | 56 GB |
| 激活值 (batch=1, seq=2048) | ~12 GB |
| **合计** | **~96 GB** |

这就是为什么微调一个 7B 模型**至少需要一张 80GB A100**，而且还很紧张。

## 三、Tensor Core：深度学习性能的秘密武器

### 3.1 初印象：用费曼视角理解

**CUDA Core 是通用计算员，Tensor Core 是专职矩阵乘法机器**。

普通 CUDA Core 一个时钟周期做一次乘加运算（FMA：`a = b * c + d`）。Tensor Core 一个时钟周期做一个 4×4 矩阵乘法——相当于 64 次乘法加上 64 次加法，**效率高出几十倍**。

代价是：Tensor Core 只会做 GEMM（矩阵乘加），而且**对矩阵维度有特殊要求**（必须是 8 的倍数，最好是 64 的倍数）。

### 3.2 Tensor Core 的工作原理

Tensor Core 执行的核心操作是：

$$D = A \times B + C$$

其中 $A, B$ 是低精度输入（BF16/FP16/INT8/FP8），$C, D$ 是高精度累加（FP32）。

这就是混合精度的物理基础：**输入矩阵用 BF16 存储和传输（省带宽），内部累加用 FP32（保精度），输出再转回 BF16**。

**各代 GPU 的 Tensor Core 规格**：

| 架构 | GPU | BF16 峰值 | FP8 峰值 | 特点 |
|------|-----|----------|---------|------|
| Volta (2017) | V100 | 不支持 | 不支持 | 首代 Tensor Core，仅 FP16 |
| Ampere (2020) | A100 | 312 TFLOPS | 不支持 | BF16 登场，TF32 模式 |
| Hopper (2022) | H100 | 989 TFLOPS | 1979 TFLOPS | FP8 训练、Transformer Engine |
| Blackwell (2024) | B200 | ~2x H100 | ~4x H100 | FP4 支持 |

**使矩阵维度对齐以激活 Tensor Core**（实践规则）：

```python
# Tensor Core 对维度的要求：必须是 8 的倍数，推荐 64 的倍数
# hidden_size = 4096  √（64 的倍数）
# hidden_size = 4100  ✗（不是 8 的倍数，退回到 CUDA Core）

# 检查你的模型配置是否满足对齐
def check_tensor_core_friendly(config):
    checks = {
        "hidden_size": config.hidden_size % 64 == 0,
        "num_heads":   config.num_attention_heads % 8 == 0,
        "head_dim":    (config.hidden_size // config.num_attention_heads) % 8 == 0,
        "ffn_size":    config.intermediate_size % 64 == 0,
        "vocab_size":  config.vocab_size % 64 == 0,
    }
    for name, ok in checks.items():
        status = "OK" if ok else "NOT aligned - Tensor Core will not activate!"
        print(f"  {name}: {status}")

# 矩阵维度对齐的好处：实测 2-4 倍速度提升
```

### 3.3 TF32：A100 的隐藏加速

A100 引入了 TF32（TensorFloat-32）模式：对 FP32 数据做矩阵乘法时，**内部截断成 10 bit 尾数（类似 FP16 精度）进行运算，结果按 FP32 返回**。

```python
# TF32 默认只对卷积启用，对矩阵乘法需要手动开
torch.backends.cuda.matmul.allow_tf32 = True   # 矩阵乘法用 TF32
torch.backends.cudnn.allow_tf32 = True          # 卷积用 TF32

# TF32 不需要改 dtype，代码无感知，自动加速 FP32 矩阵乘法
# A100 上：FP32 CUDA Core ~19 TFLOPS → TF32 Tensor Core ~156 TFLOPS
```

## 四、混合精度训练：BF16 vs FP16 深度解析

### 4.1 初印象：用费曼视角理解

**混合精度训练就是"在需要精度的地方用高精度，在能省的地方用低精度"**。

不能全用 FP32：太慢、太费显存。
不能全用 FP16/BF16：某些操作数值不稳定（如 softmax、loss 计算、累加）。
解决方案：**参数存 FP32（Master Weight），前向/反向用 BF16，优化器在 FP32 权重上更新**。

### 4.2 浮点数格式的本质

三种格式的位分配对比：

```
FP32  (32 bit): [1 符号][8 指数][23 尾数]  范围: ±3.4×10^38  精度: ~7位十进制
BF16  (16 bit): [1 符号][8 指数][7 尾数]   范围: ±3.4×10^38  精度: ~2位十进制
FP16  (16 bit): [1 符号][5 指数][10 尾数]  范围: ±6.5×10^4   精度: ~3位十进制
```

**关键区别**：

- **BF16 的指数位和 FP32 相同（8 bit）**，所以数值范围和 FP32 完全一致，不会溢出。代价是尾数少（精度粗）。
- **FP16 的指数只有 5 bit**，范围小得多（最大约 65504）。大梯度、大激活值容易溢出，训练时需要 **Loss Scaling**（把 loss 先乘大一个因子，再反向传播，让小梯度不被舍入到 0）。

```python
import torch

# BF16 和 FP16 的实际数值范围差异
print("BF16 最大值:", torch.finfo(torch.bfloat16).max)  # 3.39e+38
print("FP16 最大值:", torch.finfo(torch.float16).max)    # 6.55e+04

# 模拟 FP16 溢出
x = torch.tensor(100000.0, dtype=torch.float16)
print("FP16 下 100000:", x)  # inf（溢出！）

x = torch.tensor(100000.0, dtype=torch.bfloat16)
print("BF16 下 100000:", x)  # 100000.（正常）
```

### 4.3 完整的混合精度训练流程

```
前向传播：
  参数(FP32 Master) → 转换 → 参数(BF16)
  BF16 参数 → Tensor Core 计算 → BF16 激活值

反向传播：
  BF16 激活值 → 反向计算 → BF16 梯度

参数更新：
  BF16 梯度 → 转换 → FP32 梯度
  FP32 Master Weight + FP32 梯度 → AdamW → 更新后的 FP32 Master Weight
  FP32 Master Weight → 转换 → BF16（下一次前向用）
```

PyTorch 的 `autocast` 把前向和反向的转换自动完成：

```python
from torch.amp import autocast, GradScaler

# BF16 方案（A100/H100 推荐，不需要 GradScaler）
scaler = GradScaler(enabled=False)   # BF16 不需要 loss scaling

for batch in train_loader:
    optimizer.zero_grad()

    with autocast("cuda", dtype=torch.bfloat16):
        loss = model(**batch).loss

    loss.backward()
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()

# FP16 方案（旧 GPU，需要 GradScaler）
scaler = GradScaler()

for batch in train_loader:
    optimizer.zero_grad()

    with autocast("cuda", dtype=torch.float16):
        loss = model(**batch).loss

    scaler.scale(loss).backward()  # 自动 loss scaling
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()
```

**autocast 的选择原则**：

| 条件 | 推荐方案 |
|------|---------|
| A100 / H100 / RTX 40 系 | BF16，无需 GradScaler |
| V100 / RTX 30 系 | FP16 + GradScaler |
| 复杂模型（Transformer 为主） | 优先 BF16 |
| 有 Loss NaN 问题 | 检查 FP16 溢出，换 BF16 或调整 Loss Scaling |

## 五、CUDA 计算流与核函数：工程师够用的底层知识

### 5.1 初印象：用费曼视角理解

**CUDA 是 NVIDIA 的 GPU 编程语言和平台，它定义了 CPU（host）和 GPU（device）如何分工**。

CPU 负责"发号施令"：准备数据、调用函数。GPU 负责"批量执行"：用几千个线程同时跑同一段代码（kernel）。

就像一个项目经理（CPU）把任务分配给一个几千人的施工队（GPU），施工队每个人做相同的操作但处理不同的数据——这叫 SIMT（Single Instruction Multiple Threads）。

### 5.2 CUDA 的三层并行层次

```
Grid（网格，一次 kernel 调用）
└── Block（线程块）×N
    └── Thread（线程）×M
```

- **Thread**：最小的执行单位，对应一个 CUDA Core
- **Block**：线程的分组，同一 Block 内的线程共享 Shared Memory，可以同步
- **Warp**：32 个线程的物理执行单位，同一 Warp 内的线程步调一致

```python
# PyTorch 的每个操作都在调用 CUDA kernel
# 但你通常不需要直接写 CUDA，只需理解其后果：

# 例：为什么算子融合（kernel fusion）能加速？
# 未融合：3 次 kernel 调用，每次都要读写显存
y = model.norm(x)   # 1 次读 x、写 y
y = model.attn(y)   # 1 次读 y、写 y
y = model.drop(y)   # 1 次读 y、写 y
# 共 6 次显存 I/O

# 融合后：1 次 kernel 调用，中间结果在 SRAM 里流转
y = fused_norm_attn_drop(x)  # 只读 x 和写最终 y
# 共 2 次显存 I/O  → 带宽省 3 倍

# 这正是 Flash Attention 快的根本原因：
# 把 Attention 的多个步骤（Q@K, softmax, P@V）融合成 1 个 kernel，
# 全程在 Shared Memory 里计算，不频繁读写 HBM
```

### 5.3 CUDA Stream：计算与传输重叠

```python
import torch

# 默认情况下所有操作串行
# GPU 完成上一个操作才开始下一个

# 用 Stream 让计算和数据传输并行
stream1 = torch.cuda.Stream()
stream2 = torch.cuda.Stream()

# 实际场景：训练时，当前 batch 在 GPU 上计算
# 同时把下一个 batch 从 CPU 传到 GPU
with torch.cuda.stream(stream1):
    # 用 non_blocking=True 异步传输（不阻塞 CPU）
    batch1_gpu = batch1_cpu.to("cuda", non_blocking=True)

with torch.cuda.stream(stream2):
    loss = model(current_batch)
    loss.backward()

# 同步等待两个 stream 都完成
torch.cuda.synchronize()
```

### 5.4 Flash Attention：带宽优化的典范

Flash Attention 是理解"把理论变成工程加速"的最佳案例：

**普通 Attention 的显存瓶颈**（序列长度 $L$，头维度 $d$）：

```
Q, K, V:  各 (L, d)，占 3Ld 显存
S = Q@K:  (L, L)，占 L² 显存  ← 瓶颈所在
P = softmax(S): (L, L)，再占 L² 显存
O = P@V:  (L, d)，占 Ld 显存
```

$L=8192$ 时，$S$ 矩阵占 8192² × 2 bytes = **128 MB**（每个注意力头！），16 头 × 32 层 = 64 GB，根本存不下。

**Flash Attention 的思路**：不把完整的 $(L, L)$ 注意力矩阵写入 HBM，改为分块（tiling），每次只在 SRAM（共享内存，比 HBM 快 10 倍）里处理一小块，处理完立刻扔掉：

```
for each block of Q:
    for each block of K, V:
        在 SRAM 里计算这个小块的 attention
        用 online softmax 技巧累积全局结果
        不写回 HBM 中间结果
输出 O
```

结果：$O(L^2)$ 的 HBM 读写降到 $O(L)$，对序列长度的显存依赖从二次降到线性，速度提升 2-4 倍。

```python
# PyTorch 2.0+ 自动使用 Flash Attention（SDPA）
# 无需手动调用，框架会自动选择
from torch.nn.functional import scaled_dot_product_attention

# 自动选择最优 backend（Flash Attention / Memory Efficient / Math）
out = scaled_dot_product_attention(q, k, v, attn_mask=None, dropout_p=0.0, is_causal=True)

# 也可以显式控制
with torch.backends.cuda.sdp_kernel(enable_flash=True, enable_math=False):
    out = scaled_dot_product_attention(q, k, v, is_causal=True)
```

## 六、常见陷阱

1. **维度不对齐导致 Tensor Core 失效**：模型 hidden_size 不是 64 的倍数时，PyTorch 退化为普通 CUDA Core 计算，速度可能慢 3-4 倍。Profile 时用 `torch.profiler` 确认是否触发 `ampere_h16gemm_bf16` 之类的 Tensor Core kernel。

2. **OOM 后显存没有被释放**：Python 的垃圾回收不及时，`del tensor` 后调 `torch.cuda.empty_cache()` 可以释放缓存。但这不增加实际可用显存，只是把 PyTorch 预留的碎片还给 CUDA。

3. **忘记 `torch.cuda.synchronize()` 导致计时错误**：GPU 操作是异步的，Python 的 `time.perf_counter()` 不会等 GPU 完成，必须先 `synchronize` 再停表。

4. **BF16 精度不足的场景**：某些操作对精度极敏感（如 log_softmax、大累加求和），在 BF16 下可能产生 NaN。Transformers 库通常已经处理，自己写 kernel 时要注意。

5. **mixed precision 和 LayerNorm**：LayerNorm 在低精度下不稳定，autocast 内部会自动把 LayerNorm 保持 FP32，不需要手动处理，但自己实现的 norm 要注意。

6. **多卡时忘记 NCCL 通信开销**：DDP 的 AllReduce 在每次 backward 后同步梯度，通信量和参数量成正比。如果 GPU 利用率高但整体吞吐低，通信可能是瓶颈。

## 七、面试常见问题

**Q1：GPU 比 CPU 快在哪里？有什么限制？**

GPU 的优势是极高的并行度（几千个核心同时干活），特别适合矩阵乘法这种"对大量数据做同样运算"的任务。限制是：GPU 不擅长复杂控制流（if-else 分支会导致"线程分歧"）、内核启动有固定开销（小矩阵乘反而比 CPU 慢）、数据从 CPU 传输到 GPU 的 PCIe 带宽有限。

**Q2：训练时显存由哪几部分组成？哪部分最大？**

四部分：参数、梯度、优化器状态、激活值。最重的通常是优化器状态——AdamW 需要保存 FP32 的一阶和二阶矩，约等于参数量的两倍（按 FP32 算）。7B 模型的 AdamW 状态就达到 56 GB，远超参数本身的 14 GB（BF16）。

**Q3：Tensor Core 是什么？怎么让它工作？**

Tensor Core 是 GPU 里专门做矩阵乘加（GEMM）的硬件单元，比普通 CUDA Core 做矩阵乘法快 10 倍以上。让它工作的条件：使用 BF16/FP16 等低精度数据类型，且矩阵维度是 8 的倍数（最好 64 的倍数）。在 PyTorch 里用 `autocast` 开混合精度后，所有线性层的矩阵乘会自动走 Tensor Core。

**Q4：BF16 和 FP16 怎么选？**

A100/H100 及以上用 BF16，它的数值范围和 FP32 一样宽（指数 8 bit），不会溢出，不需要 Loss Scaling。V100 等较老 GPU 用 FP16，需要配合 GradScaler 做 Loss Scaling 防止梯度下溢。出现 Loss NaN 时，首先怀疑 FP16 溢出，换 BF16 或调大 GradScaler 的初始 scale。

**Q5：Flash Attention 为什么快？它解决了什么问题？**

普通 Attention 要把 $L \times L$ 的注意力矩阵写入显存（HBM），这是带宽瓶颈，而且显存占用随序列长度二次增长。Flash Attention 通过分块计算（tiling）和 online softmax，让中间注意力矩阵始终留在 SRAM（片上高速缓存）里，不写回 HBM，把 HBM 读写从 $O(L^2)$ 降到 $O(L)$，速度快 2-4 倍，还解锁了超长序列（长文本）的训练。

**Q6：`compute-bound` 和 `memory-bound` 是什么意思？**

算力瓶颈（compute-bound）：GPU 的计算单元满负荷，数据加载跟得上。典型操作：大矩阵乘法。这时优化要提高算力（量化、Tensor Core 对齐）。带宽瓶颈（memory-bound）：GPU 大部分时间在等显存数据，计算单元闲置。典型操作：softmax、LayerNorm、激活函数。这时优化要减少显存读写（算子融合、Flash Attention）。`nvidia-smi` 里 GPU-Util 低但带宽接近上限，就是 memory-bound 的特征。

**Q7：什么是 Loss Scaling，为什么 BF16 不需要？**

FP16 的最小正数约 $6 \times 10^{-8}$，很多小梯度会被舍入为 0（梯度消失）。Loss Scaling 是把 loss 先乘以一个大因子（如 1024），反向得到放大后的梯度，在更新参数前再除回去，让小梯度能被表示。BF16 的指数位数和 FP32 相同，最小正数约 $10^{-38}$，和 FP32 一样低，不会有梯度下溢，所以不需要 Loss Scaling。

**Q8：怎么排查训练速度慢的问题？**

步骤：先看 `nvidia-smi` 的 GPU-Util。如果低于 80%，多半是数据加载瓶颈（加 `num_workers`、`pin_memory`）；如果接近 100%，用 `torch.profiler` 看耗时最长的 kernel 是矩阵乘（compute-bound）还是内存操作（memory-bound）；确认是否开了混合精度和 Tensor Core 对齐；检查是否有无意义的 CPU-GPU 同步点（如频繁 `.item()` 调用）。
