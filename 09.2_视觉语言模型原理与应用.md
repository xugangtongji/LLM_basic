## 零、开篇：LLM 看不懂图片，但可以学会看

LLM 的本质是处理 token 序列——输入是 token，输出也是 token。图片不是 token，所以 LLM 本身看不懂图片。

**VLM 解决的核心问题是：如何把图像信息转化成 LLM 能理解的 token 表示？**

这个问题的答案决定了 VLM 的架构设计。过去几年里，研究者探索了多种方案，从复杂的交叉注意力到极简的线性投影，最终形成了几个主流的设计范式。

---

## 一、VLM 的三层架构

几乎所有 VLM 都由三个部分组成，只是每部分的具体选择不同：

### 第一层：视觉编码器

负责把原始图像转换成视觉特征向量序列。

**ViT（Vision Transformer）** 是当前最主流的选择。它把图像切割成固定大小的小方块（patch，通常 14×14 或 16×16 像素），把每个 patch 展平成一个向量，然后用 Transformer 处理这个 patch 序列，就像处理文本 token 序列一样。

一张 224×224 的图像，如果用 14×14 的 patch，就被切成 (224/14)² = 256 个视觉 token。ViT 的输出是这 256 个 patch 经过 Transformer 处理后的特征向量序列。

几乎所有现代 VLM 都用 CLIP 预训练的 ViT 作为视觉编码器。这是因为 CLIP 的 ViT 已经学会了图文对齐的表示，它对图像的"理解"和文本的语义空间比较兼容，接入 LLM 时更容易对齐。

### 第二层：连接器（Connector / Projector）

这是 VLM 架构中最关键的设计选择：**如何把视觉 token 转化成 LLM 能处理的语言 token 空间表示？**

视觉编码器输出的特征向量维度（比如 1024）和 LLM 的词嵌入维度（比如 4096）不一样，而且语义空间也不完全对齐。连接器负责弥合这个 gap。

历史上出现过三种主要设计：

**方案一：线性投影层（LLaVA 的选择）**

最简单的方案——一个线性层把视觉特征向量投影到 LLM 的 token 维度。LLaVA 1.0 用的就是这个，简单到令人吃惊，但效果出奇地好。后来 LLaVA 1.5 换成了两层 MLP，效果有进一步提升。

**方案二：Q-Former（BLIP-2 的选择）**

BLIP-2 引入了 Q-Former（Querying Transformer）——一组可学习的查询向量，通过交叉注意力机制从视觉特征中"查询"最相关的信息。Q-Former 的好处是可以把 256 个视觉 token 压缩成 32 个查询 token，大幅减少进入 LLM 的视觉 token 数量，节省计算。缺点是可能损失一些细粒度的视觉信息。

**方案三：将视觉 token 直接拼入 LLM 输入序列（当前主流）**

把视觉编码器输出的 token 经过投影后，直接和文本 token 拼接，作为 LLM 的输入序列。图像部分对应的 position 里放视觉 token，文字部分放文字 token，LLM 用统一的注意力机制处理整个序列。

这是目前 LLaVA 系列、InternVL、Qwen-VL 等主流开源 VLM 的通用做法。简洁、有效，也便于扩展到多图或高分辨率场景。

### 第三层：大语言模型（LLM 主干）

接收拼接了视觉 token 的序列，按照语言模型的方式生成回答。LLM 主干通常是 LLaMA、Qwen 或 Mistral 等已经经过指令微调的语言模型。

## 二、高分辨率和动态分辨率：2024 年的关键改进

早期 VLM 把所有图像统一缩放到 224×224 分辨率，这对需要看清细节的任务（OCR、表格识别、小字体）是致命的——缩放后文字模糊不清。

2024 年，几乎所有主流 VLM 都引入了高分辨率处理方案：

**分块处理（Tiling）**：把高分辨率图像分割成多个 336×336 或 448×448 的子图，每个子图分别用视觉编码器处理，再把所有子图的 token 拼接。一张 1920×1080 的截图可能被切成十几块，每块单独编码，最终 LLM 接收到的是几千个视觉 token。

**动态分辨率**：根据图像内容自适应地决定切分方式——简单图像用低分辨率（省 token），文档、表格等细节密集的图像用高分辨率（切更多块）。InternVL 2 和 Qwen-VL 2 都实现了这种动态分辨率处理。

这个改进让 VLM 在文档理解、OCR、图表分析等任务上的表现大幅提升，是从"能看图"到"看得清、看得准"的关键跨越。

## 三、主流 VLM 横向对比

### 闭源商业模型

| 模型 | 机构 | 亮点 |
|------|------|------|
| GPT-4o | OpenAI | 综合能力最强，原生多模态，支持语音；价格较高 |
| Gemini 1.5 Pro | Google | 超长上下文（100万 token），视频理解好；需要 Google API |
| Claude 3.5 Sonnet | Anthropic | 文档理解和代码视觉分析出色；隐私保护好 |

### 开源模型

| 模型 | 机构 | 亮点 |
|------|------|------|
| InternVL 2（8B/26B/76B）| 上海 AI Lab | 开源 VLM 的新标杆，多个基准接近 GPT-4V |
| Qwen-VL 2 | 阿里 | 中文多模态最强，文档/OCR/表格处理优秀 |
| LLaVA-Next / LLaVA-OneVision | 哈工大+微软 | 开源生态最活跃，架构简洁，资料最多 |
| MiniCPM-V | 面壁智能 | 端侧轻量级 VLM，3B 参数能在手机上运行 |

**选型原则**：
- 需要最强综合能力，不介意成本 → GPT-4o
- 中文文档、OCR、表格分析场景 → Qwen-VL 2
- 需要开源自托管，追求质量 → InternVL 2
- 端侧/轻量级部署 → MiniCPM-V

## 四、工业落地场景

### 4.1 文档理解：VLM 最成熟的应用

传统的文档处理是"OCR 提取文字 + NLP 理解文字"的两步流程，在表格、图表、混排复杂文档上容易出错。

VLM 直接接受文档图像，不需要 OCR 这一步，模型在看到图像的同时就能理解布局、识别表格结构、读取图表数据。

**典型场景**：
- 发票/合同自动审核：识别金额、日期、签名位置
- 财务报告分析：从年报的图表中自动提取数据
- 医疗影像报告：识别 CT/MRI 报告中的文字和图像标注

**落地工具**：Azure Document Intelligence、AWS Textract 都已集成了基于 VLM 的文档理解能力；国内的阿里云、腾讯云也有对应产品。

### 4.2 图表分析与数据可视化理解

给定一个折线图、柱状图、饼图，提问"2022 年的增长率是多少"或"哪个类别占比最大"——这类任务对纯文字模型完全不可能，但对 VLM 来说是基础能力。

**应用场景**：
- 商业智能：自动分析 Dashboard 截图，生成文字报告
- 学术研究：自动解读论文中的实验图表
- 企业数据分析：非技术人员用自然语言查询可视化报告

### 4.3 医学影像辅助诊断

这是 VLM 最具价值、也最需要谨慎对待的应用领域。

**可以做的**：
- 放射科报告草稿生成（医生审核后使用）
- 病变区域的自动标注辅助
- 医学图像分类（有监督微调后）

**必须注意的**：医疗场景的 VLM 必须经过严格的临床验证，不能直接用通用 VLM 输出作为诊断依据。目前的 VLM 在医学影像上的表现参差不齐，幻觉（生成看起来合理但错误的描述）的风险在高风险医疗场景中是不可接受的。AI 辅助诊断的定位应该是"提高医生效率"而非"替代医生判断"。

### 4.4 自动驾驶的感知模块

理解路面情况、识别交通标志、判断行人意图——这些是自动驾驶感知层的核心任务。

传统自动驾驶用专门训练的目标检测模型（YOLO、Faster R-CNN 等）。基于 VLM 的新方法可以在更开放的方式下理解场景："前方有一辆自行车正在减速，骑车人似乎要向右转"——这类开放性的场景描述是传统检测模型做不到的。

Tesla 的 FSD（Full Self-Driving）和国内的多家智能驾驶公司都在探索把 VLM 融入感知-决策链路。但目前 VLM 在实时性（延迟）和鲁棒性（边界情况）上还有差距，更多是辅助工具而非主控模块。

## 五、VLM 的现实局限

### 空间关系理解的困难

"图中红色方块在蓝色圆圈的左边还是右边？"——这类精确的空间关系问题，当前 VLM 经常出错。视觉编码器学到的是全局语义特征，对精确位置关系的表示并不强。

### 细粒度计数和精确识别

"图中有多少只鸟？"——数量超过 5-6 时，VLM 的准确率显著下降。精确的字符识别（手写潦草文字、特殊符号）也是弱项。

### 对抗性视觉攻击

给图像加入人眼不可见的细微扰动，可以让 VLM 把一辆汽车识别成一只猫，或者在图片中嵌入不可见的文字指令来劫持模型行为（视觉版提示注入）。这在安全敏感场景是一个需要认真对待的风险。

## 六、面试常见问题

**Q1：VLM 的基本架构是什么？三层分别做什么？**

三层架构：视觉编码器（通常是 CLIP 的 ViT，把图像切成 patch，处理成视觉特征向量序列）；连接器（把视觉特征投影到 LLM 的 token 空间，可以是线性层、MLP 或 Q-Former）；LLM 主干（接收拼接了视觉 token 和文字 token 的序列，生成文字回答）。核心设计问题是连接器的选择，它决定了视觉信息如何被 LLM 消化。

**Q2：为什么 CLIP 的 ViT 成为几乎所有 VLM 的视觉编码器标配？**

因为 CLIP 是用图文对比学习训练的，ViT 输出的视觉特征已经和文本语义空间对齐。用它初始化视觉编码器，接入 LLM 时两种模态的"语言"更接近，训练更容易收敛。从零训练一个视觉编码器再接 LLM，对齐难度更大，也需要更多数据和计算。

**Q3：高分辨率处理（分块 Tiling）解决了什么问题？**

早期 VLM 把图像统一缩放到 224×224，细节（小字、密集表格、OCR 内容）在缩放后严重失真，模型看不清楚。分块处理把高分辨率图像切成多个子图，每块单独编码，保留细节信息，最终拼接所有块的 token 给 LLM。代价是 token 数量大幅增加（更贵、更慢），但对文档理解、OCR 等细节密集任务来说是必要的。

**Q4：VLM 在医疗场景的应用有什么需要特别注意的地方？**

核心是：AI 辅助诊断的定位必须是"辅助"而非"替代"。通用 VLM 在医学影像上容易产生听起来合理但临床上错误的幻觉描述，这在高风险医疗场景不可接受。应用于医疗场景的 VLM 必须在对应领域的数据上专门微调，经过严格临床验证，并且始终有医生作为最终决策者。
